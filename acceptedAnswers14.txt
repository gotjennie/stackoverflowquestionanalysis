V<p>Correctly detecting the encoding all times is <strong>impossible</strong>.</p>\u000a\u000a<p>(From chardet FAQ:)</p>\u000a\u000a<blockquote>\u000a  <p>However, some encodings are optimized\u000a  for specific languages, and languages\u000a  are not random. Some character\u000a  sequences pop up all the time, while\u000a  other sequences make no sense. A\u000a  person fluent in English who opens a\u000a  newspaper and finds \u201ctxzqJv 2!dasd0a\u000a  QqdKjvz\u201d will instantly recognize that\u000a  that isn't English (even though it is\u000a  composed entirely of English letters).\u000a  By studying lots of \u201ctypical\u201d text, a\u000a  computer algorithm can simulate this\u000a  kind of fluency and make an educated\u000a  guess about a text's language.</p>\u000a</blockquote>\u000a\u000a<p>There is the <a href="http://pypi.python.org/pypi/chardet" rel="nofollow noreferrer">chardet</a> library that uses that study to try to detect encoding. chardet is a port of the auto-detection code in Mozilla. </p>\u000a\u000a<p>You can also use <a href="https://www.crummy.com/software/BeautifulSoup/bs4/doc/#unicode-dammit" rel="nofollow noreferrer">UnicodeDammit</a>. It will try the following methods:</p>\u000a\u000a<ul>\u000a<li>An encoding discovered in the document itself: for instance, in an XML declaration or (for HTML documents) an http-equiv META tag. If Beautiful Soup finds this kind of encoding within the document, it parses the document again from the beginning and gives the new encoding a try. The only exception is if you explicitly specified an encoding, and that encoding actually worked: then it will ignore any encoding it finds in the document.</li>\u000a<li>An encoding sniffed by looking at the first few bytes of the file. If an encoding is detected at this stage, it will be one of the UTF-* encodings, EBCDIC, or ASCII.</li>\u000a<li>An encoding sniffed by the <a href="http://pypi.python.org/pypi/chardet" rel="nofollow noreferrer">chardet</a> library, if you have it installed.</li>\u000a<li>UTF-8</li>\u000a<li>Windows-1252 </li>\u000a</ul>\u000a

V<p>Since asking this question I have settled on a solution, which I use when developing small application with a small team of people.</p>\u000a\u000a<p><a href="https://github.com/AGWA/git-crypt" rel="nofollow noreferrer">git-crypt</a></p>\u000a\u000a<p>git-crypt uses GPG to transparently encrypt files when their names match certain patterns. For intance, if you add to your <code>.gitattributes</code> file...</p>\u000a\u000a<pre><code>*.secret.* filter=git-crypt diff=git-crypt\u000a</code></pre>\u000a\u000a<p>...then a file like <code>config.secret.json</code> will always be pushed to remote repos with encryption, but remain unencrypted on your local file system.</p>\u000a\u000a<p>If I want to add a new GPG key (a person) to your repo which can decrypt the protected files then run <code>git-crypt add-gpg-user &lt;gpg_user_key&gt;</code>. This creates a new commit. The new user will be able to decrypt subsequent commits.</p>\u000a

V<p>I've found one way to achieve what I want, by using proxy models to get around the fact that each model may be registered only once.</p>\u000a\u000a<pre><code>class PostAdmin(admin.ModelAdmin):\u000a    list_display = ('title', 'pubdate','user')\u000a\u000aclass MyPosts(Post):\u000a    class Meta:\u000a        proxy = True\u000a\u000aclass MyPostAdmin(PostAdmin):\u000a    def get_queryset(self, request):\u000a        return self.model.objects.filter(user = request.user)\u000a\u000a\u000aadmin.site.register(Post, PostAdmin)\u000aadmin.site.register(MyPost, MyPostAdmin)\u000a</code></pre>\u000a\u000a<p>Then the default PostAdmin would be accessible at /admin/myapp/post and the list of posts owned by the user would be at /admin/myapp/myposts.</p>\u000a\u000a<p>After looking at <a href="http://code.djangoproject.com/wiki/DynamicModels" rel="noreferrer">http://code.djangoproject.com/wiki/DynamicModels</a>, I've come up with the following function utility function to do the same thing:</p>\u000a\u000a<pre><code>def create_modeladmin(modeladmin, model, name = None):\u000a    class  Meta:\u000a        proxy = True\u000a        app_label = model._meta.app_label\u000a\u000a    attrs = {'__module__': '', 'Meta': Meta}\u000a\u000a    newmodel = type(name, (model,), attrs)\u000a\u000a    admin.site.register(newmodel, modeladmin)\u000a    return modeladmin\u000a</code></pre>\u000a\u000a<p>This can be used as follows:</p>\u000a\u000a<pre><code>class MyPostAdmin(PostAdmin):\u000a    def get_queryset(self, request):\u000a        return self.model.objects.filter(user = request.user)\u000a\u000acreate_modeladmin(MyPostAdmin, name='my-posts', model=Post)\u000a</code></pre>\u000a

V<p>An example usage:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; s = 'Hello world'\u000a&gt;&gt;&gt; t = buffer(s, 6, 5)\u000a&gt;&gt;&gt; t\u000a&lt;read-only buffer for 0x10064a4b0, size 5, offset 6 at 0x100634ab0&gt;\u000a&gt;&gt;&gt; print t\u000aworld\u000a</code></pre>\u000a\u000a<p>The buffer in this case is a sub-string, starting at position 6 with length 5, and it doesn't take extra storage space - it references a slice of the string.</p>\u000a\u000a<p>This isn't very useful for short strings like this, but it can be necessary when using large amounts of data. V<p>Here the <a href="http://en.wikipedia.org/wiki/Token_bucket" rel="noreferrer">simplest algorithm</a>, if you want just to drop messages when they arrive too quickly (instead of queuing them, which makes sense because the queue might get arbitrarily large):</p>\u000a\u000a<pre><code>rate = 5.0; // unit: messages\u000aper  = 8.0; // unit: seconds\u000aallowance = rate; // unit: messages\u000alast_check = now(); // floating-point, e.g. usec accuracy. Unit: seconds\u000a\u000awhen (message_received):\u000a  current = now();\u000a  time_passed = current - last_check;\u000a  last_check = current;\u000a  allowance += time_passed * (rate / per);\u000a  if (allowance &gt; rate):\u000a    allowance = rate; // throttle\u000a  if (allowance &lt; 1.0):\u000a    discard_message();\u000a  else:\u000a    forward_message();\u000a    allowance -= 1.0;\u000a</code></pre>\u000a\u000a<p>There are no datastructures, timers etc. in this solution and it works cleanly :) To see this, 'allowance' grows at speed 5/8 units per seconds at most, i.e. at most five units per eight seconds. Every message that is forwarded deducts one unit, so you can't send more than five messages per every eight seconds.</p>\u000a\u000a<p>Note that <code>rate</code> should be an integer, i.e. without non-zero decimal part, or the algorithm won't work correctly (actual rate will not be <code>rate/per</code>). E.g. <code>rate=0.5; per=1.0;</code> does not work because <code>allowance</code> will never grow to 1.0. But <code>rate=1.0; per=2.0;</code> works fine.</p>\u000a

V<p>Logits simply means that the function operates on the unscaled output of earlier layers and that the relative scale to understand the units is linear.  It means, in particular, the sum of the inputs may not equal 1, that the values are <em>not</em> probabilities (you might have an input of 5).</p>\u000a\u000a<p><code>tf.nn.softmax</code> produces just the result of applying the <a href="https://en.wikipedia.org/wiki/Softmax_function" rel="noreferrer">softmax function</a> to an input tensor.  The softmax "squishes" the inputs so that sum(input) = 1;  it's a way of normalizing.  The shape of output of a softmax is the same as the input - it just normalizes the values.  The outputs of softmax <em>can</em> be interpreted as probabilities.</p>\u000a\u000a<pre><code>a = tf.constant(np.array([[.1, .3, .5, .9]]))\u000aprint s.run(tf.nn.softmax(a))\u000a[[ 0.16838508  0.205666    0.25120102  0.37474789]]\u000a</code></pre>\u000a\u000a<p>In contrast, <code>tf.nn.softmax_cross_entropy_with_logits</code> computes the cross entropy of the result after applying the softmax function (but it does it all together in a more mathematically careful way).  It's similar to the result of:</p>\u000a\u000a<pre><code>sm = tf.nn.softmax(x)\u000ace = cross_entropy(sm)\u000a</code></pre>\u000a\u000a<p>The cross entropy is a summary metric - it sums across the elements.  The output of <code>tf.nn.softmax_cross_entropy_with_logits</code> on a shape <code>[2,5]</code> tensor is of shape <code>[2,1]</code> (the first dimension is treated as the batch).</p>\u000a\u000a<p>If you want to do optimization to minimize the cross entropy, AND you're softmaxing after your last layer, you should use <code>tf.nn.softmax_cross_entropy_with_logits</code> instead of doing it yourself, because it covers numerically unstable corner cases in the mathematically right way.  Otherwise, you'll end up hacking it by adding little epsilons here and there.</p>\u000a\u000a<p>(Edited 2016-02-07: If you have single-class labels, where an object can only belong to one class, you might now  consider using <code>tf.nn.sparse_softmax_cross_entropy_with_logits</code> so that you don't have to convert your labels to a dense one-hot array.  This function was added after release 0.6.0.)</p>\u000a

V<p><strong>TL;DR - <a href="http://bugs.python.org/issue21118" rel="noreferrer">ISSUE 21118</a></strong></p>\u000a\u000a<hr>\u000a\u000a<p><strong>The long Story</strong></p>\u000a\u000a<p>Josh Rosenberg found out that the <code>str.translate()</code> function is very slow compared to the <code>bytes.translate</code>, he raised an <a href="http://bugs.python.org/issue21118" rel="noreferrer">issue</a>, stating that:</p>\u000a\u000a<blockquote>\u000a  <p>In Python 3, <code>str.translate()</code> is usually a performance pessimization, not optimization.</p>\u000a</blockquote>\u000a\u000a<h3>Why was <code>str.translate()</code> slow?</h3>\u000a\u000a<p>The main reason for <code>str.translate()</code> to be very slow was that the lookup used to be in a Python dictionary.</p>\u000a\u000a<p>The usage of <code>maketrans</code> made this problem worse. The similar approach using <a href="https://docs.python.org/3/library/functions.html#bytes" rel="noreferrer"><code>bytes</code></a> builds a C array of 256 items to fast table lookup. Hence the usage of higher level Python <code>dict</code> makes the <code>str.translate()</code> in Python 3.4 very slow.</p>\u000a\u000a<h3>What happened now?</h3>\u000a\u000a<p>The first approach was to add a small patch, <a href="http://bugs.python.org/file34691/translate_writer.patch" rel="noreferrer">translate_writer</a>, However the speed increase was not that pleasing. However from 3.5  ASCII case is much faster. </p>\u000a\u000a<p>Earlier ASCII vs non-ascii used to be almost same, however now we can see a great change in the performance. </p>\u000a\u000a<p>It can be an improvement from 71.6\u03bcs to 2.33\u03bcs as seen in this <a href="http://stackoverflow.com/questions/29998052/deleting-consonants-from-a-string-in-python/29998062#29998062">answer</a>.</p>\u000a\u000a<p>The following code demonstrates this </p>\u000a\u000a<pre><code>python3.5 -m timeit -s "text = 'mJssissippi'*100; d=dict(J='i')" "text.translate(d)"\u000a100000 loops, best of 3: 2.3 usec per loop\u000apython3.5 -m timeit -s "text = 'm\u005cU0001F602ssissippi'*100; d={'\u005cU0001F602': 'i'}" "text.translate(d)"\u000a10000 loops, best of 3: 117 usec per loop\u000a\u000apython3 -m timeit -s "text = 'm\u005cU0001F602ssissippi'*100; d={'\u005cU0001F602': 'i'}" "text.translate(d)"\u000a10000 loops, best of 3: 91.2 usec per loop\u000apython3 -m timeit -s "text = 'mJssissippi'*100; d=dict(J='i')" "text.translate(d)"\u000a10000 loops, best of 3: 101 usec per loop\u000a</code></pre>\u000a\u000a<p>Tabulation of the results:</p>\u000a\u000a<pre><code>         Python 3.4    Python 3.5  \u000aAscii     91.2          2.3 \u000aUnicode   101           117\u000a</code></pre>\u000a

V<h2>General idea</h2>\u000a\u000a<p>Option 1: Load both images as arrays (<code>scipy.misc.imread</code>) and calculate an element-wise (pixel-by-pixel) difference. Calculate the norm of the difference.</p>\u000a\u000a<p>Option 2: Load both images. Calculate some feature vector for each of them (like a histogram). Calculate distance between feature vectors rather than images.</p>\u000a\u000a<p>However, there are some decisions to make first.</p>\u000a\u000a<h2>Questions</h2>\u000a\u000a<p>You should answer these questions first:</p>\u000a\u000a<ul>\u000a<li><p>Are images of the same shape and dimension?</p>\u000a\u000a<p>If not, you may need to resize or crop them. PIL library will help to do it in Python.</p>\u000a\u000a<p>If they are taken with the same settings and the same device, they are probably the same.</p></li>\u000a<li><p>Are images well-aligned?</p>\u000a\u000a<p>If not, you may want to run cross-correlation first, to find the best alignment first. SciPy has functions to do it.</p>\u000a\u000a<p>If the camera and the scene are still, the images are likely to be well-aligned.</p></li>\u000a<li><p>Is exposure of the images always the same? (Is lightness/contrast the same?)</p>\u000a\u000a<p>If not, you may want <a href="http://en.wikipedia.org/wiki/Normalization_(image_processing)" rel="noreferrer">to normalize</a> images.</p>\u000a\u000a<p>But be careful, in some situations this may do more wrong than good. For example, a single bright pixel on a dark background will make the normalized image very different.</p></li>\u000a<li><p>Is color information important?</p>\u000a\u000a<p>If you want to notice color changes, you will have a vector of color values per point, rather than a scalar value as in gray-scale image. You need more attention when writing such code.</p></li>\u000a<li><p>Are there distinct edges in the image? Are they likely to move?</p>\u000a\u000a<p>If yes, you can apply edge detection algorithm first (e.g. calculate gradient with Sobel or Prewitt transform, apply some threshold), then compare edges on the first image to edges on the second.</p></li>\u000a<li><p>Is there noise in the image?</p>\u000a\u000a<p>All sensors pollute the image with some amount of noise. Low-cost sensors have more noise. You may wish to apply some noise reduction before you compare images. Blur is the most simple (but not the best) approach here.</p></li>\u000a<li><p>What kind of changes do you want to notice?</p>\u000a\u000a<p>This may affect the choice of norm to use for the difference between images.</p>\u000a\u000a<p>Consider using Manhattan norm (the sum of the absolute values) or zero norm (the number of elements not equal to zero) to measure how much the image has changed. The former will tell you how much the image is off, the latter will tell only how many pixels differ.</p></li>\u000a</ul>\u000a\u000a<h2>Example</h2>\u000a\u000a<p>I assume your images are well-aligned, the same size and shape, possibly with different exposure. For simplicity, I convert them to grayscale even if they are color (RGB) images.</p>\u000a\u000a<p>You will need these imports:</p>\u000a\u000a<pre><code>import sys\u000a\u000afrom scipy.misc import imread\u000afrom scipy.linalg import norm\u000afrom scipy import sum, average\u000a</code></pre>\u000a\u000a<p>Main function, read two images, convert to grayscale, compare and print results:</p>\u000a\u000a<pre><code>def main():\u000a    file1, file2 = sys.argv[1:1+2]\u000a    # read images as 2D arrays (convert to grayscale for simplicity)\u000a    img1 = to_grayscale(imread(file1).astype(float))\u000a    img2 = to_grayscale(imread(file2).astype(float))\u000a    # compare\u000a    n_m, n_0 = compare_images(img1, img2)\u000a    print "Manhattan norm:", n_m, "/ per pixel:", n_m/img1.size\u000a    print "Zero norm:", n_0, "/ per pixel:", n_0*1.0/img1.size\u000a</code></pre>\u000a\u000a<p>How to compare. <code>img1</code> and <code>img2</code> are 2D SciPy arrays here:</p>\u000a\u000a<pre><code>def compare_images(img1, img2):\u000a    # normalize to compensate for exposure difference, this may be unnecessary\u000a    # consider disabling it\u000a    img1 = normalize(img1)\u000a    img2 = normalize(img2)\u000a    # calculate the difference and its norms\u000a    diff = img1 - img2  # elementwise for scipy arrays\u000a    m_norm = sum(abs(diff))  # Manhattan norm\u000a    z_norm = norm(diff.ravel(), 0)  # Zero norm\u000a    return (m_norm, z_norm)\u000a</code></pre>\u000a\u000a<p>If the file is a color image, <code>imread</code> returns a 3D array, average RGB channels (the last array axis) to obtain intensity. No need to do it for grayscale images (e.g. <code>.pgm</code>):</p>\u000a\u000a<pre><code>def to_grayscale(arr):\u000a    "If arr is a color image (3D array), convert it to grayscale (2D array)."\u000a    if len(arr.shape) == 3:\u000a        return average(arr, -1)  # average over the last axis (color channels)\u000a    else:\u000a        return arr\u000a</code></pre>\u000a\u000a<p>Normalization is trivial, you may choose to normalize to [0,1] instead of [0,255]. <code>arr</code> is a SciPy array here, so all operations are element-wise:</p>\u000a\u000a<pre><code>def normalize(arr):\u000a    rng = arr.max()-arr.min()\u000a    amin = arr.min()\u000a    return (arr-amin)*255/rng\u000a</code></pre>\u000a\u000a<p>Run the <code>main</code> function:</p>\u000a\u000a<pre><code>if __name__ == "__main__":\u000a    main()\u000a</code></pre>\u000a\u000a<p>Now you can put this all in a script and run against two images. If we compare image to itself, there is no difference:</p>\u000a\u000a<pre><code>$ python compare.py one.jpg one.jpg\u000aManhattan norm: 0.0 / per pixel: 0.0\u000aZero norm: 0 / per pixel: 0.0\u000a</code></pre>\u000a\u000a<p>If we blur the image and compare to the original, there is some difference:</p>\u000a\u000a<pre><code>$ python compare.py one.jpg one-blurred.jpg \u000aManhattan norm: 92605183.67 / per pixel: 13.4210411116\u000aZero norm: 6900000 / per pixel: 1.0\u000a</code></pre>\u000a\u000a<p>P.S. Entire <a href="http://gist.github.com/626356" rel="noreferrer">compare.py</a> script.</p>\u000a\u000a<h2>Update: relevant techniques</h2>\u000a\u000a<p>As the question is about a video sequence, where frames are likely to be almost the same, and you look for something unusual, I'd like to mention some alternative approaches which may be relevant:</p>\u000a\u000a<ul>\u000a<li>background subtraction and segmentation (to detect foreground objects)</li>\u000a<li>sparse optical flow (to detect motion)</li>\u000a<li>comparing histograms or some other statistics instead of images</li>\u000a</ul>\u000a\u000a<p>I strongly recommend taking a look at  \u201cLearning OpenCV\u201d book, Chapters 9 (Image parts and segmentation) and 10 (Tracking and motion). The former teaches to use Background subtraction method, the latter gives some info on optical flow methods. All methods are implemented in OpenCV library. If you use Python, I suggest to use OpenCV \u2265 2.3, and its <code>cv2</code> Python module.</p>\u000a\u000a<p>The most simple version of the background subtraction:</p>\u000a\u000a<ul>\u000a<li>learn the average value \u03bc and standard deviation \u03c3 for every pixel of the background</li>\u000a<li>compare current pixel values to the range of (\u03bc-2\u03c3,\u03bc+2\u03c3) or (\u03bc-\u03c3,\u03bc+\u03c3)</li>\u000a</ul>\u000a\u000a<p>More advanced versions make take into account time series for every pixel and handle non-static scenes (like moving trees or grass).</p>\u000a\u000a<p>The idea of optical flow is to take two or more frames, and assign velocity vector to every pixel (dense optical flow) or to some of them (sparse optical flow). To estimate sparse optical flow, you may use <a href="http://en.wikipedia.org/wiki/Lucas%E2%80%93Kanade_method" rel="noreferrer">Lucas-Kanade method</a> (it is also implemented in OpenCV). Obviously, if there is a lot of flow (high average over max values of the velocity field), then something is moving in the frame, and subsequent images are more different.</p>\u000a\u000a<p>Comparing histograms may help to detect sudden changes between consecutive frames. This approach was used in <a href="http://www.sciencedirect.com/science/article/pii/S0967066110000808" rel="noreferrer">Courbon et al, 2010</a>:</p>\u000a\u000a<blockquote>\u000a  <p><em>Similarity of consecutive frames.</em> The distance between two consecutive frames is measured. If it is too high, it means that the second frame is corrupted and thus the image is eliminated. The <a href="https://en.wikipedia.org/wiki/Kullback%E2%80%93Leibler_divergence" rel="noreferrer">Kullback\u2013Leibler distance</a>, or mutual entropy, on the histograms of the two frames:</p>\u000a  \u000a  <p><img src="https://i.imgur.com/hdeh8ni.gif" alt="$$ d(p,q) = \u005csum_i p(i) \u005clog (p(i)/q(i)) $$"></p>\u000a  \u000a  <p>where <em>p</em> and <em>q</em> are the histograms of the frames is used. The threshold is fixed on 0.2.</p>\u000a</blockquote>\u000a

V<p><strong>Python 2.x</strong></p>\u000a\u000a<p>There were two functions to get user input, called <a href="https://docs.python.org/2/library/functions.html#input"><code>input</code></a> and <a href="https://docs.python.org/2/library/functions.html#raw_input"><code>raw_input</code></a>. The difference between them is, <code>raw_input</code> doesn't evaluate the data and returns as it is, in string form. But, <code>input</code> will evaluate whatever you entered and the result of evaluation will be returned. For example,</p>\u000a\u000a<pre><code>&gt;&gt;&gt; import sys\u000a&gt;&gt;&gt; sys.version\u000a'2.7.6 (default, Mar 22 2014, 22:59:56) \u005cn[GCC 4.8.2]'\u000a&gt;&gt;&gt; data = input("Enter a number: ")\u000aEnter a number: 5 + 17\u000a&gt;&gt;&gt; data, type(data)\u000a(22, &lt;type 'int'&gt;)\u000a</code></pre>\u000a\u000a<p>The data <code>5 + 17</code> is evaluated and the result is <code>22</code>. When it evaluates the expression <code>5 + 17</code>, it detects that you are adding two numbers and so the result will also be of the same <code>int</code> type. So, the type conversion is done for free and <code>22</code> is returned as the result of <code>input</code> and stored in <code>data</code> variable. You can think of <code>input</code> as the <code>raw_input</code> composed with an <a href="https://docs.python.org/2/library/functions.html#eval"><code>eval</code></a> call.</p>\u000a\u000a<pre><code>&gt;&gt;&gt; data = eval(raw_input("Enter a number: "))\u000aEnter a number: 5 + 17\u000a&gt;&gt;&gt; data, type(data)\u000a(22, &lt;type 'int'&gt;)\u000a</code></pre>\u000a\u000a<p><strong>Note:</strong> you should be careful when you are using <code>input</code> in Python 2.x. I explained why one should be careful when using it, in <a href="http://stackoverflow.com/a/21122817/1903116">this answer</a>.</p>\u000a\u000a<p>But, <code>raw_input</code> doesn't evaluate the input and returns as it is, as a string.</p>\u000a\u000a<pre><code>&gt;&gt;&gt; import sys\u000a&gt;&gt;&gt; sys.version\u000a'2.7.6 (default, Mar 22 2014, 22:59:56) \u005cn[GCC 4.8.2]'\u000a&gt;&gt;&gt; data = raw_input("Enter a number: ")\u000aEnter a number: 5 + 17\u000a&gt;&gt;&gt; data, type(data)\u000a('5 + 17', &lt;type 'str'&gt;)\u000a</code></pre>\u000a\u000a<p><strong>Python 3.x</strong></p>\u000a\u000a<p>Python 3.x's <a href="https://docs.python.org/3/library/functions.html#input"><code>input</code></a> and Python 2.x's <code>raw_input</code> are similar and <code>raw_input</code> is not available in Python 3.x. </p>\u000a\u000a<pre><code>&gt;&gt;&gt; data = int(input("Enter a number: "), 8)\u000aEnter a number: 777\u000a&gt;&gt;&gt; data\u000a511\u000a&gt;&gt;&gt; data = int(input("Enter a number: "), 16)\u000aEnter a number: FFFF\u000a&gt;&gt;&gt; data\u000a65535\u000a&gt;&gt;&gt; data = int(input("Enter a number: "), 2)\u000aEnter a number: 10101010101\u000a&gt;&gt;&gt; data\u000a1365\u000a</code></pre>\u000a\u000a<p>The second parameter tells what is the base of the numbers entered and then internally it understands and converts it. If the entered data is wrong it will throw a <code>ValueError</code>.</p>\u000a\u000a<pre><code>&gt;&gt;&gt; data = int(input("Enter a number: "), 2)\u000aEnter a number: 1234\u000aTraceback (most recent call last):\u000a  File "&lt;input&gt;", line 1, in &lt;module&gt;\u000aValueError: invalid literal for int() with base 2: '1234'\u000a</code></pre>\u000a\u000a<hr>\u000a\u000a<p>Apart from that, your program can be changed a little bit, like this</p>\u000a\u000a<pre><code>while True:\u000a    ...\u000a    ...\u000a    if input("Play again? ") == "no":\u000a        break\u000a</code></pre>\u000a\u000a<p>You can get rid of the <code>play</code> variable by using <code>break</code> and <code>while True</code>. </p>\u000a\u000a<p><strong>PS</strong>: Python doesn't expect <code>;</code> at the end of the line :)</p>\u000a

V<p>Dictionary views are essentially what their name says: <strong>views are simply like a window</strong> on the keys and values (or items) of a dictionary.  Here is an excerpt from the <a href="http://docs.python.org/py3k/library/stdtypes.html#dictionary-view-objects">official documentation</a> for Python3:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; dishes = {'eggs': 2, 'sausage': 1, 'bacon': 1, 'spam': 500}\u000a&gt;&gt;&gt; keys = dishes.keys()\u000a&gt;&gt;&gt; values = dishes.values()\u000a\u000a&gt;&gt;&gt; # view objects are dynamic and reflect dict changes\u000a&gt;&gt;&gt; del dishes['eggs']\u000a&gt;&gt;&gt; keys  # No eggs anymore!\u000adict_keys(['sausage', 'bacon', 'spam'])\u000a\u000a&gt;&gt;&gt; values  # No eggs value (2) anymore!\u000adict_values([1, 1, 500])\u000a</code></pre>\u000a\u000a<p>(The Python2 equivalent uses <code>dishes.viewkeys()</code> and <code>dishes.viewvalues()</code>.)</p>\u000a\u000a<p>This example shows the <strong>dynamic character of views</strong>: the keys view is <em>not</em> a copy of the keys at a given point in time, but rather a simple window that shows you the keys; if they are changed, then what you see through the window does change as well.  This feature can be useful in some circumstances (for instance, one can work with a view on the keys in multiple parts of a program instead of recalculating the current list of keys each time they are needed).</p>\u000a\u000a<p>One advantage is that <em>looking</em> at, say, the keys uses only <strong>a small and fixed amount of memory</strong> and requires <strong>a small and fixed amount of processor time</strong>, as there is no creation of a list of keys (Python2, on the other hand, often unnecessarily creates a new list, as quoted by Rajendran T, which takes memory and time in an amount proportional to the length of the list).  To continue the window analogy, if you want to see a landscape behind a wall, you simply make an opening in it (you build a window); copying the keys into a list would correspond to instead painting a copy of the landscape on your wall\u2014the copy takes time, space, and does not update itself.</p>\u000a\u000a<p>To summarize, views are simply\u2026 views (windows) on your dictionary, which show the contents of the dictionary even after it changes.  They offer features that differ from those of lists: a list of keys contain a <em>copy</em> of the dictionary keys at a given point in time, while a view is dynamic and is much faster to obtain, as it does not have to copy any data (keys or values) in order to be created.</p>\u000a

V<p>Python's GIL is intended to serialize access to interpreter internals from different threads. On multi-core systems, it means that multiple threads can't effectively make use of multiple cores. (If the GIL didn't lead to this problem, most people wouldn't care about the GIL - it's only being raised as an issue because of the increasing prevalence of multi-core systems.) If you want to understand it in detail, you can view <a href="https://www.youtube.com/watch?v=ph374fJqFPE" rel="noreferrer">this video</a> or look at <a href="http://www.dabeaz.com/python/GIL.pdf" rel="noreferrer">this set of slides</a>. It might be too much information, but then you did ask for details :-)</p>\u000a\u000a<p>Note that Python's GIL is only really an issue for CPython, the reference implementation. Jython and IronPython don't have a GIL. As a Python developer, you don't generally come across the GIL unless you're writing a C extension. C extension writers need to release the GIL when their extensions do blocking I/O, so that other threads in the Python process get a chance to run.</p>\u000a\u000a<p><strong>Update:</strong> Updated link to video to point to Youtube, as the earlier <code>blip.tv</code> link had rotted.</p>\u000a

V<p>I suppose I should expand this answer, now that I'm older and wiser and know what's going on.  Better late than never.</p>\u000a\u000a<p>You <em>can</em> add a property to a class dynamically.  But that's the catch: you have to add it to the <em>class</em>.</p>\u000a\u000a<pre><code>&gt;&gt;&gt; class Foo(object):\u000a...     pass\u000a... \u000a&gt;&gt;&gt; foo = Foo()\u000a&gt;&gt;&gt; foo.a = 3\u000a&gt;&gt;&gt; Foo.b = property(lambda self: self.a + 1)\u000a&gt;&gt;&gt; foo.b\u000a4\u000a</code></pre>\u000a\u000a<p>A <code>property</code> is actually a simple implementation of a thing called a <a href="http://docs.python.org/2/reference/datamodel.html#implementing-descriptors" rel="noreferrer">descriptor</a>.  It's an object that provides custom handling for a given attribute, <em>on a given class</em>.  Kinda like a way to factor a huge <code>if</code> tree out of <code>__getattribute__</code>.</p>\u000a\u000a<p>When I ask for <code>foo.b</code> in the example above, Python sees that the <code>b</code> defined on the class implements the <em>descriptor protocol</em>\u2014which just means it's an object with a <code>__get__</code>, <code>__set__</code>, or <code>__delete__</code> method.  The descriptor claims responsibility for handling that attribute, so Python calls <code>Foo.b.__get__(foo, Foo)</code>, and the return value is passed back to you as the value of the attribute.  In the case of <code>property</code>, each of these methods just calls the <code>fget</code>, <code>fset</code>, or <code>fdel</code> you passed to the <code>property</code> constructor.</p>\u000a\u000a<p>Descriptors are really Python's way of exposing the plumbing of its entire OO implementation.  In fact, there's another type of descriptor even more common than <code>property</code>.</p>\u000a\u000a<pre><code>&gt;&gt;&gt; class Foo(object):\u000a...     def bar(self):\u000a...         pass\u000a... \u000a&gt;&gt;&gt; Foo().bar\u000a&lt;bound method Foo.bar of &lt;__main__.Foo object at 0x7f2a439d5dd0&gt;&gt;\u000a&gt;&gt;&gt; Foo().bar.__get__\u000a&lt;method-wrapper '__get__' of instancemethod object at 0x7f2a43a8a5a0&gt;\u000a</code></pre>\u000a\u000a<p>The humble method is just another kind of descriptor.  Its <code>__get__</code> tacks on the calling instance as the first argument; in effect, it does this:</p>\u000a\u000a<pre><code>def __get__(self, instance, owner):\u000a    return functools.partial(self.function, instance)\u000a</code></pre>\u000a\u000a<p>Anyway, I suspect this is why descriptors only work on classes: they're a formalization of the stuff that powers classes in the first place.  They're even the exception to the rule: you can obviously assign descriptors to a class, and classes are themselves instances of <code>type</code>!  In fact, trying to read <code>Foo.b</code> still calls <code>property.__get__</code>; it's just idiomatic for descriptors to return themselves when accessed as class attributes.</p>\u000a\u000a<p>I think it's pretty cool that virtually all of Python's OO system can be expressed in Python.  :)</p>\u000a\u000a<p>Oh, and I wrote a <a href="http://me.veekun.com/blog/2012/05/23/python-faq-descriptors/" rel="noreferrer">wordy blog post about descriptors</a> a while back if you're interested.</p>\u000a

See <a href="http://stackoverflow.com/questions/3217673/why-use-argparse-rather-than-optparse">this answer</a> for more information.</strong></p>\u000a\u000a<p>As other people pointed out, you are better off going with optparse over getopt.  getopt is pretty much a one-to-one mapping of the standard getopt(3) C library functions, and not very easy to use.</p>\u000a\u000a<p>optparse, while being a bit more verbose, is much better structured and simpler to extend later on.</p>\u000a\u000a<p>Here's a typical line to add an option to your parser:</p>\u000a\u000a<pre><code>parser.add_option('-q', '--query',\u000a            action="store", dest="query",\u000a            help="query string", default="spam")\u000a</code></pre>\u000a\u000a<p>It pretty much speaks for itself; at processing time, it will accept -q or --query as options, store the argument in an attribute called query and has a default value if you don't specify it.  It is also self-documenting in that you declare the help argument (which will be used when run with -h/--help) right there with the option.</p>\u000a\u000a<p>Usually you parse your arguments with:</p>\u000a\u000a<pre><code>options, args = parser.parse_args()\u000a</code></pre>\u000a\u000a<p>This will, by default, parse the standard arguments passed to the script (sys.argv[1:])</p>\u000a\u000a<p>options.query will then be set to the value you passed to the script.</p>\u000a\u000a<p>You create a parser simply by doing</p>\u000a\u000a<pre><code>parser = optparse.OptionParser()\u000a</code></pre>\u000a\u000a<p>These are all the basics you need.  Here's a complete Python script that shows this:</p>\u000a\u000a<pre><code>import optparse\u000a\u000aparser = optparse.OptionParser()\u000a\u000aparser.add_option('-q', '--query',\u000a    action="store", dest="query",\u000a    help="query string", default="spam")\u000a\u000aoptions, args = parser.parse_args()\u000a\u000aprint 'Query string:', options.query\u000a</code></pre>\u000a\u000a<p>5 lines of python that show you the basics.</p>\u000a\u000a<p>Save it in sample.py, and run it once with</p>\u000a\u000a<pre><code>python sample.py\u000a</code></pre>\u000a\u000a<p>and once with</p>\u000a\u000a<pre><code>python sample.py --query myquery\u000a</code></pre>\u000a\u000a<p>Beyond that, you will find that optparse is very easy to extend.\u000aIn one of my projects, I created a Command class which allows you to nest subcommands in a command tree easily.  It uses optparse heavily to chain commands together.  It's not something I can easily explain in a few lines, but feel free to <a href="https://thomas.apestaart.org/moap/trac/browser/trunk/moap/extern/command/command.py" rel="noreferrer">browse around in my repository</a> for the main class, as well as <a href="https://thomas.apestaart.org/moap/trac/browser/trunk/moap/command/doap.py" rel="noreferrer">a class that uses it and the option parser</a></p>\u000a

V<p>In Python versions prior to 3.0 there are two kinds of strings "plain strings" and "unicode strings". Plain strings (<code>str</code>) cannot represent characters outside of the Latin alphabet (ignoring details of code pages for simplicity). Unicode strings (<code>unicode</code>) can represent characters from any alphabet including some fictional ones like Klingon.</p>\u000a\u000a<p>So why have two kinds of strings, would it not be better to just have Unicode since that would cover all the cases? Well it is better to have only Unicode but Python was created before Unicode was the preferred method for representing strings. It takes time to transition the string type in a language with many users, in Python 3.0 it is finally the case that all strings are Unicode.</p>\u000a\u000a<p>The inheritance hierarchy of Python strings pre-3.0 is:</p>\u000a\u000a<pre><code>          object\u000a             |\u000a             |\u000a         basestring\u000a            / \u005c\u000a           /   \u005c\u000a         str  unicode\u000a</code></pre>\u000a\u000a<p>'basestring' introduced in Python 2.3 can be thought of as a step in the direction of string unification as it can be used to check whether an object is an instance of <code>str</code> or <code>unicode</code> </p>\u000a\u000a<pre><code>&gt;&gt;&gt; string1 = "I am a plain string"\u000a&gt;&gt;&gt; string2 = u"I am a unicode string"\u000a&gt;&gt;&gt; isinstance(string1, str)\u000aTrue\u000a&gt;&gt;&gt; isinstance(string2, str)\u000aFalse\u000a&gt;&gt;&gt; isinstance(string1, unicode)\u000aFalse\u000a&gt;&gt;&gt; isinstance(string2, unicode)\u000aTrue\u000a&gt;&gt;&gt; isinstance(string1, basestring)\u000aTrue\u000a&gt;&gt;&gt; isinstance(string2, basestring)\u000aTrue\u000a</code></pre>\u000a

V<p>Alright! I've finally managed to get something working consistently! This problem pulled me in for several days... Fun stuff! Sorry for the length of this answer, but I need to elaborate a bit on some things... (Though I may set a record for the longest non-spam stackoverflow answer ever!)</p>\u000a\u000a<p>As a side note, I'm using the full dataset that Ivo <a href="http://cid-0e685da213fd5935.skydrive.live.com/redir.aspx?page=browse&amp;resid=E685DA213FD5935!148&amp;type=6&amp;authkey=WOUgehAzQ!o%24&amp;Bsrc=EMSHGM&amp;Bpub=SN.Notifications" rel="noreferrer">provided a link to</a> in his <a href="http://stackoverflow.com/questions/3684484/peak-detection-in-a-2d-array">original question</a>.  It's a series of rar files (one-per-dog) each containing several different experiment runs stored as ascii arrays. Rather than try to copy-paste stand-alone code examples into this question, here's a <a href="https://bitbucket.org/joferkington/paw-analysis/overview" rel="noreferrer">bitbucket mercurial repository</a> with full, stand-alone code. You can clone it with </p>\u000a\u000a<p><code>hg clone https://joferkington@bitbucket.org/joferkington/paw-analysis</code></p>\u000a\u000a<hr>\u000a\u000a<p><strong>Overview</strong></p>\u000a\u000a<p>There are essentially two ways to approach the problem, as you noted in your question. I'm actually going to use both in different ways.</p>\u000a\u000a<ol>\u000a<li>Use the (temporal and spatial) order of the paw impacts to determine which paw is which.</li>\u000a<li>Try to identify the "pawprint" based purely on its shape.</li>\u000a</ol>\u000a\u000a<p>Basically, the first method works with the dog's paws follow the trapezoidal-like pattern shown in Ivo's question above, but fails whenever the paws don't follow that pattern. It's fairly easy to programatically detect when it doesn't work.  </p>\u000a\u000a<p>Therefore, we can use the measurements where it did work to build up a training dataset (of ~2000 paw impacts from ~30 different dogs) to recognize which paw is which, and the problem reduces to a supervised classification (With some additional wrinkles... Image recognition is a bit harder than a "normal" supervised classification problem).</p>\u000a\u000a<hr>\u000a\u000a<p><strong>Pattern Analysis</strong></p>\u000a\u000a<p>To elaborate on the first method, when a dog is walking (not running!) normally (which some of these dogs may not be), we expect paws to impact in the order of: Front Left, Hind Right, Front Right, Hind Left, Front Left, etc. The pattern may start with either the front left or front right paw.</p>\u000a\u000a<p>If this were always the case, we could simply sort the impacts by initial contact time and use a modulo 4 to group them by paw.  </p>\u000a\u000a<p><img src="https://i.stack.imgur.com/gLhbt.png" alt="Normal Impact Sequence"></p>\u000a\u000a<p>However, even when everything is "normal", this doesn't work.  This is due to the trapezoid-like shape of the pattern. A hind paw spatially falls behind the previous front paw.</p>\u000a\u000a<p>Therefore, the hind paw impact after the initial front paw impact often falls off the sensor plate, and isn't recorded.  Similarly, the last paw impact is often not the next paw in the sequence, as the paw impact before it occured off the sensor plate and wasn't recorded.</p>\u000a\u000a<p><img src="https://i.stack.imgur.com/th87Q.png" alt="Missed Hind Paw"></p>\u000a\u000a<p>Nonetheless, we can use the shape of the paw impact pattern to determine when this has happened, and whether we've started with a left or right front paw.  (I'm actually ignoring problems with the last impact here. It's not too hard to add it, though.)</p>\u000a\u000a<pre><code>def group_paws(data_slices, time):   \u000a    # Sort slices by initial contact time\u000a    data_slices.sort(key=lambda s: s[-1].start)\u000a\u000a    # Get the centroid for each paw impact...\u000a    paw_coords = []\u000a    for x,y,z in data_slices:\u000a        paw_coords.append([(item.stop + item.start) / 2.0 for item in (x,y)])\u000a    paw_coords = np.array(paw_coords)\u000a\u000a    # Make a vector between each sucessive impact...\u000a    dx, dy = np.diff(paw_coords, axis=0).T\u000a\u000a    #-- Group paws -------------------------------------------\u000a    paw_code = {0:'LF', 1:'RH', 2:'RF', 3:'LH'}\u000a    paw_number = np.arange(len(paw_coords))\u000a\u000a    # Did we miss the hind paw impact after the first \u000a    # front paw impact? If so, first dx will be positive...\u000a    if dx[0] &gt; 0: \u000a        paw_number[1:] += 1\u000a\u000a    # Are we starting with the left or right front paw...\u000a    # We assume we're starting with the left, and check dy[0].\u000a    # If dy[0] &gt; 0 (i.e. the next paw impacts to the left), then\u000a    # it's actually the right front paw, instead of the left.\u000a    if dy[0] &gt; 0: # Right front paw impact...\u000a        paw_number += 2\u000a\u000a    # Now we can determine the paw with a simple modulo 4..\u000a    paw_codes = paw_number % 4\u000a    paw_labels = [paw_code[code] for code in paw_codes]\u000a\u000a    return paw_labels\u000a</code></pre>\u000a\u000a<p>In spite of all of this, it frequently doesn't work correctly. Many of the dogs in the full dataset appear to be running, and the paw impacts don't follow the same temporal order as when the dog is walking. (Or perhaps the dog just has severe hip problems...)</p>\u000a\u000a<p><img src="https://i.stack.imgur.com/DwXZ5.png" alt="Abnormal Impact Sequence"></p>\u000a\u000a<p>Fortunately, we can still programatically detect whether or not the paw impacts follow our expected spatial pattern:</p>\u000a\u000a<pre><code>def paw_pattern_problems(paw_labels, dx, dy):\u000a    """Check whether or not the label sequence "paw_labels" conforms to our\u000a    expected spatial pattern of paw impacts. "paw_labels" should be a sequence\u000a    of the strings: "LH", "RH", "LF", "RF" corresponding to the different paws"""\u000a    # Check for problems... (This could be written a _lot_ more cleanly...)\u000a    problems = False\u000a    last = paw_labels[0]\u000a    for paw, dy, dx in zip(paw_labels[1:], dy, dx):\u000a        # Going from a left paw to a right, dy should be negative\u000a        if last.startswith('L') and paw.startswith('R') and (dy &gt; 0):\u000a            problems = True\u000a            break\u000a        # Going from a right paw to a left, dy should be positive\u000a        if last.startswith('R') and paw.startswith('L') and (dy &lt; 0):\u000a            problems = True\u000a            break\u000a        # Going from a front paw to a hind paw, dx should be negative\u000a        if last.endswith('F') and paw.endswith('H') and (dx &gt; 0):\u000a            problems = True\u000a            break\u000a        # Going from a hind paw to a front paw, dx should be positive\u000a        if last.endswith('H') and paw.endswith('F') and (dx &lt; 0):\u000a            problems = True\u000a            break\u000a        last = paw\u000a    return problems\u000a</code></pre>\u000a\u000a<p>Therefore, even though the simple spatial classification doesn't work all of the time, we can determine when it does work with reasonable confidence.  </p>\u000a\u000a<p><strong>Training Dataset</strong></p>\u000a\u000a<p>From the pattern-based classifications where it worked correctly, we can build up a very large training dataset of correctly classified paws (~2400 paw impacts from 32 different dogs!).  </p>\u000a\u000a<p>We can now start to look at what an "average" front left, etc, paw looks like.</p>\u000a\u000a<p>To do this, we need some sort of "paw metric" that is the same dimensionality for any dog. (In the full dataset, there are both very large and very small dogs!)  A paw print from an Irish elkhound will be both much wider and much "heavier" than a paw print from a toy poodle.  We need to rescale each paw print so that a) they have the same number of pixels, and b) the pressure values are standardized.  To do this, I resampled each paw print onto a 20x20 grid and rescaled the pressure values based on the maximum, mininum, and mean pressure value for the paw impact. </p>\u000a\u000a<pre><code>def paw_image(paw):\u000a    from scipy.ndimage import map_coordinates\u000a    ny, nx = paw.shape\u000a\u000a    # Trim off any "blank" edges around the paw...\u000a    mask = paw &gt; 0.01 * paw.max()\u000a    y, x = np.mgrid[:ny, :nx]\u000a    ymin, ymax = y[mask].min(), y[mask].max()\u000a    xmin, xmax = x[mask].min(), x[mask].max()\u000a\u000a    # Make a 20x20 grid to resample the paw pressure values onto\u000a    numx, numy = 20, 20\u000a    xi = np.linspace(xmin, xmax, numx)\u000a    yi = np.linspace(ymin, ymax, numy)\u000a    xi, yi = np.meshgrid(xi, yi)  \u000a\u000a    # Resample the values onto the 20x20 grid\u000a    coords = np.vstack([yi.flatten(), xi.flatten()])\u000a    zi = map_coordinates(paw, coords)\u000a    zi = zi.reshape((numy, numx))\u000a\u000a    # Rescale the pressure values\u000a    zi -= zi.min()\u000a    zi /= zi.max()\u000a    zi -= zi.mean() #&lt;- Helps distinguish front from hind paws...\u000a    return zi\u000a</code></pre>\u000a\u000a<p>After all of this, we can finally take a look at what an average left front, hind right, etc paw looks like.  Note that this is averaged across >30 dogs of greatly different sizes, and we seem to be getting consistent results!</p>\u000a\u000a<p><img src="https://i.stack.imgur.com/1rcSk.png" alt="Average Paws"></p>\u000a\u000a<p>However, before we do any analysis on these, we need to subtract the mean (the average paw for all legs of all dogs).</p>\u000a\u000a<p><img src="https://i.stack.imgur.com/y6sbC.png" alt="Mean Paw"></p>\u000a\u000a<p>Now we can analyize the differences from the mean, which are a bit easier to recognize:</p>\u000a\u000a<p><img src="https://i.stack.imgur.com/JkNR7.png" alt="Differential Paws"></p>\u000a\u000a<p><strong>Image-based Paw Recognition</strong></p>\u000a\u000a<p>Ok... We finally have a set of patterns that we can begin to try to match the paws against.  Each paw can be treated as a 400-dimensional vector (returned by the <code>paw_image</code> function) that can be compared to these four 400-dimensional vectors.  </p>\u000a\u000a<p>Unfortunately, if we just use a "normal" supervised classification algorithm (i.e. find which of the 4 patterns is closest to a particular paw print using a simple distance), it doesn't work consistently. In fact, it doesn't do much better than random chance on the training dataset.</p>\u000a\u000a<p>This is a common problem in image recognition. Due to the high dimensionality of the input data, and the somewhat "fuzzy" nature of images (i.e. adjacent pixels have a high covariance), simply looking at the difference of an image from a template image does not give a very good measure of the similarity of their shapes.  </p>\u000a\u000a<p><strong>Eigenpaws</strong></p>\u000a\u000a<p>To get around this we need to build a set of "eigenpaws" (just like "eigenfaces" in facial recognition), and describe each paw print as a combination of these eigenpaws.  This is identical to principal components analysis, and basically provides a way to reduce the dimensionality of our data, so that distance is a good measure of shape.</p>\u000a\u000a<p>Because we have more training images than dimensions (2400 vs 400), there's no need to do "fancy" linear algebra for speed. We can work directly with the covariance matrix of the training data set:</p>\u000a\u000a<pre><code>def make_eigenpaws(paw_data):\u000a    """Creates a set of eigenpaws based on paw_data.\u000a    paw_data is a numdata by numdimensions matrix of all of the observations."""\u000a    average_paw = paw_data.mean(axis=0)\u000a    paw_data -= average_paw\u000a\u000a    # Determine the eigenvectors of the covariance matrix of the data\u000a    cov = np.cov(paw_data.T)\u000a    eigvals, eigvecs = np.linalg.eig(cov)\u000a\u000a    # Sort the eigenvectors by ascending eigenvalue (largest is last)\u000a    eig_idx = np.argsort(eigvals)\u000a    sorted_eigvecs = eigvecs[:,eig_idx]\u000a    sorted_eigvals = eigvals[:,eig_idx]\u000a\u000a    # Now choose a cutoff number of eigenvectors to use \u000a    # (50 seems to work well, but it's arbirtrary...\u000a    num_basis_vecs = 50\u000a    basis_vecs = sorted_eigvecs[:,-num_basis_vecs:]\u000a\u000a    return basis_vecs\u000a</code></pre>\u000a\u000a<p>These <code>basis_vecs</code> are the "eigenpaws".</p>\u000a\u000a<p><img src="https://i.stack.imgur.com/NamaF.png" alt="Eigenpaws"></p>\u000a\u000a<p>To use these, we simply dot (i.e. matrix multiplication) each paw image (as a 400-dimensional vector, rather than a 20x20 image) with the basis vectors. This gives us a 50-dimensional vector (one element per basis vector) that we can use to classify the image. Instead of comparing a 20x20 image to the 20x20 image of each "template" paw, we compare the 50-dimensional, transformed image to each 50-dimensional transformed template paw.  This is much less sensitive to small variations in exactly how each toe is positioned, etc, and basically reduces the dimensionality of the problem to just the relevant dimensions.</p>\u000a\u000a<p><strong>Eigenpaw-based Paw Classification</strong></p>\u000a\u000a<p>Now we can simply use the distance between the 50-dimensional vectors and the "template" vectors for each leg to classify which paw is which:</p>\u000a\u000a<pre><code>codebook = np.load('codebook.npy') # Template vectors for each paw\u000aaverage_paw = np.load('average_paw.npy')\u000abasis_stds = np.load('basis_stds.npy') # Needed to "whiten" the dataset...\u000abasis_vecs = np.load('basis_vecs.npy')\u000apaw_code = {0:'LF', 1:'RH', 2:'RF', 3:'LH'}\u000adef classify(paw):\u000a    paw = paw.flatten()\u000a    paw -= average_paw\u000a    scores = paw.dot(basis_vecs) / basis_stds\u000a    diff = codebook - scores\u000a    diff *= diff\u000a    diff = np.sqrt(diff.sum(axis=1))\u000a    return paw_code[diff.argmin()]\u000a</code></pre>\u000a\u000a<p>Here are some of the results:\u000a<img src="https://i.stack.imgur.com/FS0vD.png" alt="alt text">\u000a<img src="https://i.stack.imgur.com/AtzDu.png" alt="alt text">\u000a<img src="https://i.stack.imgur.com/0Ref6.png" alt="alt text"></p>\u000a\u000a<p><strong>Remaining Problems</strong></p>\u000a\u000a<p>There are still some problems, particularly with dogs too small to make a clear pawprint... (It works best with large dogs, as the toes are more clearly seperated at the sensor's resolution.) Also, partial pawprints aren't recognized with this system, while they can be with the trapezoidal-pattern-based system. </p>\u000a\u000a<p>However, because the eigenpaw analysis inherently uses a distance metric, we can classify the paws both ways, and fall back to the trapezoidal-pattern-based system when the eigenpaw analysis's smallest distance from the "codebook" is over some threshold. I haven't implemented this yet, though.</p>\u000a\u000a<p>Phew... That was long! My hat is off to Ivo for having such a fun question!</p>\u000a

V<p>There are a number of ways of achieving your result. This is only going to work if each of your <code>print</code> statements prints the same string. Note that it works for multiplication of any length string (e.g. <code>'foo' * 20</code> works).</p>\u000a\u000a<pre><code>&gt;&gt;&gt; print 'a' * 20\u000aaaaaaaaaaaaaaaaaaaaa\u000a</code></pre>\u000a\u000a<p>If you want to do this in general, build up a string and then print it once. This will consume a bit of memory for the string, but only make a single call to <code>print</code>. Note that string concatenation using <code>+=</code> is now linear in the size of the string you're concatenating so this will be fast.</p>\u000a\u000a<pre><code>&gt;&gt;&gt; for i in xrange(20):\u000a...     s += 'a'\u000a... \u000a&gt;&gt;&gt; print s\u000aaaaaaaaaaaaaaaaaaaaa\u000a</code></pre>\u000a\u000a<p>Or you can do it more directly using <a href="http://docs.python.org/library/sys.html#sys.stdout" rel="noreferrer">sys.stdout</a>.<a href="http://docs.python.org/library/stdtypes.html#file.write" rel="noreferrer">write()</a>, which <code>print</code> is a wrapper around. This will write only the raw string you give it, without any formatting. Note that no newline is printed even at the end of the 20 <code>a</code>s.</p>\u000a\u000a<pre><code>&gt;&gt;&gt; for i in xrange(20):\u000a...     sys.stdout.write('a')\u000a... \u000aaaaaaaaaaaaaaaaaaaaa&gt;&gt;&gt; \u000a</code></pre>\u000a\u000a<p>Python 3 changes the <code>print</code> statement into a <a href="http://docs.python.org/library/functions.html#print" rel="noreferrer">print() function</a>, which allows you to set an <code>end</code> parameter. You can use it in >=2.6 by importing from <code>__future__</code>. I'd avoid this in any serious 2.x code though, as it will be a little confusing for those who have never used 3.x. However, it should give you a taste of some of the goodness 3.x brings.</p>\u000a\u000a<pre><code>&gt;&gt;&gt; from __future__ import print_function\u000a&gt;&gt;&gt; for i in xrange(20):\u000a...     print('a', end='')\u000a... \u000aaaaaaaaaaaaaaaaaaaaa&gt;&gt;&gt; \u000a</code></pre>\u000a

V<p>The most popular way to manage python packages (if you're not using your system package manager) is to use setuptools and easy_install. It is probably already installed on your system. Use it like this:</p>\u000a\u000a<pre><code>easy_install django\u000a</code></pre>\u000a\u000a<p>easy_install uses the <a href="http://pypi.python.org" rel="noreferrer">Python Package Index</a>  which is an amazing resource for python developers. Have a look around to see what packages are available.</p>\u000a\u000a<p>A better option is <a href="http://pypi.python.org/pypi/pip" rel="noreferrer">pip</a>, which is gaining traction, as it attempts to <a href="http://www.b-list.org/weblog/2008/dec/14/packaging/" rel="noreferrer">fix a lot of the problems</a> associated with easy_install. Pip uses the same package repository as easy_install, it just works better. Really the only time use need to use easy_install is for this command:</p>\u000a\u000a<pre><code>easy_install pip\u000a</code></pre>\u000a\u000a<p>After that, use:</p>\u000a\u000a<pre><code>pip install django\u000a</code></pre>\u000a\u000a<p>At some point you will probably want to learn a bit about <a href="http://pypi.python.org/pypi/virtualenv" rel="noreferrer">virtualenv</a>. If you do a lot of python development on projects with conflicting package requirements, virtualenv is a godsend. It will allow you to have completely different versions of various packages, and switch between them easily depending your needs.</p>\u000a\u000a<p>Regarding which python to use, sticking with Apple's python will give you the least headaches, but If you need a newer version (Leopard is 2.5.1 I believe), I would go with the <a href="http://trac.macports.org/browser/trunk/dports/lang/python26/Portfile" rel="noreferrer">macports</a> python 2.6.</p>\u000a
