V<p>You cannot return two values, but you can return a <code>tuple</code> or a <code>list</code> and unpack it after the call:</p>\u000a\u000a<pre><code>def select_choice():\u000a    ...\u000a    return i, card  # or [i, card]\u000a\u000amy_i, my_card = select_choice()\u000a</code></pre>\u000a\u000a<p>In line <code>return i, card</code> <code>i, card</code> means creating a tuple. You can also use parenthesis like <code>return (i, card)</code>, but tuples are created by comma, so parens are not mandatory. But you can use parens to make your code more readable or to split the tuple over multiple lines. The same applies to line <code>my_i, my_card = select_choice()</code>.</p>\u000a\u000a<p>If you want to return more than two values, consider using a <a href="https://docs.python.org/3.6/library/collections.html#collections.namedtuple" rel="noreferrer">named tuple</a>. It will allow the caller of the function to access fields of the returned value by name, which is more readable. You can still access items of the tuple by index. For example in <code>Schema.loads</code> method Marshmallow framework returns a <a href="http://marshmallow.readthedocs.io/en/latest/api_reference.html#marshmallow.UnmarshalResult" rel="noreferrer"><code>UnmarshalResult</code></a> which is a <a href="https://github.com/marshmallow-code/marshmallow/blob/fe5469998a6f1f297f8c85acfba8c48ff7bc3d7d/marshmallow/schema.py#L29" rel="noreferrer"><code>namedtuple</code></a>. So you can do:</p>\u000a\u000a<pre><code>data, errors = MySchema.loads(request.json())\u000aif errors:\u000a    ...\u000a</code></pre>\u000a\u000a<p>or</p>\u000a\u000a<pre><code>result = MySchema.loads(request.json())\u000aif result.errors:\u000a    ...\u000aelse:\u000a    # use `result.data`\u000a</code></pre>\u000a\u000a<p>In other cases you may return a <code>dict</code> from your function:</p>\u000a\u000a<pre><code>def select_choice():\u000a    ...\u000a    return {'i': i, 'card': card, 'other_field': other_field, ...}\u000a</code></pre>\u000a\u000a<p>But you might want consider to return an instance of a utility class, which wraps your data:</p>\u000a\u000a<pre><code>class ChoiceData():\u000a    def __init__(self, i, card, other_field, ...):\u000a        # you can put here some validation logic\u000a        self.i = i\u000a        self.card = card\u000a        self.other_field = other_field\u000a        ...\u000a\u000adef select_choice():\u000a    ...\u000a    return ChoiceData(i, card, other_field, ...)\u000a\u000achoice_data = select_choice()\u000aprint(choice_data.i, choice_data.card)\u000a</code></pre>\u000a

V<h3>Short version</h3>\u000a\u000a<p>ABCs offer a higher level of semantic contract between clients and the implemented classes.</p>\u000a\u000a<h3>Long version</h3>\u000a\u000a<p>There is a contract between a class and its callers. The class promises to do certain things and have certain properties. </p>\u000a\u000a<p>There are different levels to the contract. </p>\u000a\u000a<p>At a very low level, the contract might include the name of a method or its number of parameters. </p>\u000a\u000a<p>In a staticly-typed language, that contract would actually be enforced by the compiler. In Python, you can use EAFP or introspection to confirm that the unknown object meets this expected contract.</p>\u000a\u000a<p>But there are also higher-level, semantic promises in the contract. </p>\u000a\u000a<p>For example, if there is a <code>__str__()</code> method, it is expected to return a string representation of the object. It <em>could</em> delete all contents of the object, commit the transaction and spit a blank page out of the printer... but there is a common understanding of what it should do, described in the Python manual.</p>\u000a\u000a<p>That's a special case, where the semantic contract is described in the manual.  What should the <code>print()</code> method do? Should it write the object to a printer or a line to the screen, or something else? It depends - you need to read the comments to understand the full contract here. A piece of client code that simply checks that the <code>print()</code> method exists has confirmed part of the contract - that a method call can be made, but not that there is agreement on the higher level semantics of the call.</p>\u000a\u000a<p>Defining an Abstract Base Class (ABC) is a way of producing a contract between the class implementers and the callers. It isn't just a list of method names, but a shared understanding of what those methods should do. If you inherit from this ABC, you are promising to follow all the rules described in the comments, including the semantics of the <code>print()</code> method.</p>\u000a\u000a<p>Python's duck-typing has many advantages in flexibility over static-typing, but it doesn't solve all the problems. ABCs offer an intermediate solution between the free-form of Python and the bondage-and-discipline of a staticly-typed language. </p>\u000a

This has been, by far, the easiest way I've found for doing what you ask.</p>\u000a\u000a<pre><code>import ctypes\u000a\u000a# Load DLL into memory.\u000a\u000ahllDll = ctypes.WinDLL ("c:\u005c\u005cPComm\u005c\u005cehlapi32.dll")\u000a\u000a# Set up prototype and parameters for the desired function call.\u000a# HLLAPI\u000a\u000ahllApiProto = ctypes.WINFUNCTYPE (\u000a    ctypes.c_int,      # Return type.\u000a    ctypes.c_void_p,   # Parameters 1 ...\u000a    ctypes.c_void_p,\u000a    ctypes.c_void_p,\u000a    ctypes.c_void_p)   # ... thru 4.\u000ahllApiParams = (1, "p1", 0), (1, "p2", 0), (1, "p3",0), (1, "p4",0),\u000a\u000a# Actually map the call ("HLLAPI(...)") to a Python name.\u000a\u000ahllApi = hllApiProto (("HLLAPI", hllDll), hllApiParams)\u000a\u000a# This is how you can actually call the DLL function.\u000a# Set up the variables and call the Python name with them.\u000a\u000ap1 = ctypes.c_int (1)\u000ap2 = ctypes.c_char_p (sessionVar)\u000ap3 = ctypes.c_int (1)\u000ap4 = ctypes.c_int (0)\u000ahllApi (ctypes.byref (p1), p2, ctypes.byref (p3), ctypes.byref (p4))\u000a</code></pre>\u000a\u000a<p>The <code>ctypes</code> stuff has all the C-type data types (<code>int</code>, <code>char</code>, <code>short</code>, <code>void*</code>, and so on) and can pass by value or reference. V<p><strong><em>As of today, there are four available approaches, two of them requiring a certain storage backend:</em></strong></p>\u000a\u000a<ol>\u000a<li><p><strong><a href="https://github.com/mvpdev/django-eav">Django-eav</a></strong> (the original package is no longer mantained but has some <strong><a href="https://github.com/mvpdev/django-eav/network">thriving forks</a></strong>) </p>\u000a\u000a<p>This solution is based on <a href="https://en.wikipedia.org/wiki/Entity-attribute-value_model">Entity Attribute Value</a> data model, essentially, it uses several tables to store dynamic attributes of objects. Great parts about this solution is that it:</p>\u000a\u000a<ul>\u000a<li>uses several pure and simple Django models to represent dynamic fields, which makes it simple to understand and database-agnostic; </li>\u000a<li><p>allows you to effectively attach/detach dynamic attribute storage to Django model with simple commands like:</p>\u000a\u000a<pre><code>eav.unregister(Encounter)\u000aeav.register(Patient)\u000a</code></pre></li>\u000a<li><p><strong><a href="https://github.com/mvpdev/django-eav/blob/master/eav/admin.py">Nicely integrates with Django admin</a></strong>;</p></li>\u000a<li><p>At the same time being really powerful.</p></li>\u000a</ul>\u000a\u000a<p>Downsides:</p>\u000a\u000a<ul>\u000a<li>Not very efficient.  This is more of a criticism of the EAV pattern itself, which requires manually merging the data from a column format to a set of key-value pairs in the model.</li>\u000a<li>Harder to maintain.  Maintaining data integrity requires a multi-column unique key constraint, which may be inefficient on some databases.</li>\u000a<li>You will need to select <a href="https://github.com/mvpdev/django-eav/network">one of the forks</a>, since the official package is no longer maintained and there is no clear leader.</li>\u000a</ul>\u000a\u000a<p>The usage is datatype=Attribute.TYPE_TEXT)\u000aAttribute.objects.create(name='country', datatype=Attribute.TYPE_TEXT)\u000a\u000aself.yes = EnumValue.objects.create(value='yes')\u000aself.no = EnumValue.objects.create(value='no')\u000aself.unkown = EnumValue.objects.create(value='unkown')\u000aynu = EnumGroup.objects.create(name='Yes / No / Unknown')\u000aynu.enums.add(self.yes)\u000aynu.enums.add(self.no)\u000aynu.enums.add(self.unkown)\u000a\u000aAttribute.objects.create(name='fever', datatype=Attribute.TYPE_ENUM,\u005c\u000a                                       enum_group=ynu)\u000a\u000a# When you register a model within EAV,\u000a# you can access all of EAV attributes:\u000a\u000aPatient.objects.create(name='Bob', eav__age=12,\u000a                           eav__fever=no, eav__city='New York',\u000a                           eav__country='USA')\u000a# You can filter queries based on their EAV JSONB fields in PostgreSQL</strong></p>\u000a\u000a<p>PostgreSQL supports several more complex data types.  Most are supported via third-party packages, but in recent years Django has adopted them into django.contrib.postgres.fields.</p>\u000a\u000a<p><strong>HStoreField</strong>:</p>\u000a\u000a<p><a href="https://github.com/jordanm/django-hstore">Django-hstore</a> was originally a third-party package, but Django 1.8 added <strong><a href="https://docs.djangoproject.com/en/1.8/ref/contrib/postgres/fields/#hstorefield">HStoreField</a></strong> as a built-in, along with several other PostgreSQL-supported field types.</p>\u000a\u000a<p>This approach is good in a sense that it lets you have the best of both worlds: dynamic fields and relational database. And is inspired by incredible but somewhat hackish solutions by <a href="http://dynamic-models.readthedocs.org/en/latest/index.html">Will Hardy</a> and Michael Hall.</p>\u000a\u000a<p>All of these are based on Django South hooks, which, according to <a href="http://blip.tv/djangocon-europe-2011/wednesday-1415-will-hardy-5311186">Will Hardy's talk at DjangoCon 2011</a> <strong>(watch it!)</strong> are nevertheless robust and tested in production (<a href="http://dynamic-models.readthedocs.org/en/latest/">relevant source code</a>).</p>\u000a\u000a<p>First to <a href="https://bitbucket.org/mhall119/dynamo/overview">implement this</a> was <a href="http://mhall119.com/2011/02/fun-with-django-meta-classes-and-dynamic-models/">Michael Hall</a>.</p>\u000a\u000a<p>Yes, this is magic, with these approaches you can achieve <strong>fully dynamic Django apps, models and fields</strong> with any relational database backend. But at what cost? Will stability of application suffer upon heavy use? These are the questions to be considered. You need to be sure to maintain a proper <a href="http://stackoverflow.com/questions/1123200/how-to-lock-a-critical-section-in-django">lock</a> in order to allow simultaneous database altering requests.</p>\u000a\u000a<p>If you are using Michael Halls lib, your code will look like this:</p>\u000a\u000a<pre><code>from dynamo import models\u000a\u000atest_app, created = models.DynamicApp.objects.get_or_create(\u000a                      name='dynamo'\u000a                    )\u000atest, created = models.DynamicModel.objects.get_or_create(\u000a                  name='Test',\u000a                  verbose_name='Test Model',\u000a                  app=test_app\u000a               )\u000afoo, created = models.DynamicModelField.objects.get_or_create(\u000a                  name = 'foo',\u000a                  verbose_name = 'Foo Field',\u000a                  model = test,\u000a                  field_type = 'dynamiccharfield',\u000a                  null = True,\u000a                  blank = True,\u000a                  unique = False,\u000a                  help_text = 'Test field for Foo',\u000a               )\u000abar, created = models.DynamicModelField.objects.get_or_create(\u000a                  name = 'bar',\u000a                  verbose_name = 'Bar Field',\u000a                  model = test,\u000a                  field_type = 'dynamicintegerfield',\u000a                  null = True,\u000a                  blank = True,\u000a                  unique = False,\u000a                  help_text = 'Test field for Bar',\u000a               )\u000a</code></pre></li>\u000a</ol>\u000a

V<p>Here are some key differences to me:</p>\u000a\u000a<ol>\u000a<li><p>Ruby has blocks; Python does not.</p></li>\u000a<li><p>Python has functions; Ruby does not.  In Python, you can take any function or method and pass it to another function.  In Ruby, everything is a method, and methods can't be directly passed.  Instead, you have to wrap them in Proc's to pass them.</p></li>\u000a<li><p>Ruby and Python both support closures, but in different ways.  In Python, you can define a function inside another function.  The inner function has read access to variables from the outer function, but not write access.  In Ruby, you define closures using blocks.  The closures have full read and write access to variables from the outer scope.</p></li>\u000a<li><p>Python has list comprehensions, which are pretty expressive.  For example, if you have a list of numbers, you can write</p>\u000a\u000a<pre><code>[x*x for x in values if x &gt; 15]\u000a</code></pre>\u000a\u000a<p>to get a new list of the squares of all values greater than 15.  In Ruby, you'd have to write the following:</p>\u000a\u000a<pre><code>values.select {|v| v &gt; 15}.map {|v| v * v}\u000a</code></pre>\u000a\u000a<p>The Ruby code doesn't feel as compact.  It's also not as efficient since it first converts the values array into a shorter intermediate array containing the values greater than 15.  Then, it takes the intermediate array and generates a final array containing the squares of the intermediates.  The intermediate array is then thrown out.  So, Ruby ends up with 3 arrays in memory during the computation; Python only needs the input list and the resulting list.</p>\u000a\u000a<p>Python also supplies similar map comprehensions.</p></li>\u000a<li><p>Python supports tuples; Ruby doesn't.  In Ruby, you have to use arrays to simulate tuples.</p></li>\u000a<li><p>Ruby supports switch/case statements; Python does not.</p></li>\u000a<li><p><del>Ruby supports the standard <code>expr ? val1 : val2</code> ternary operator; Python does not.</del></p></li>\u000a<li><p>Ruby supports only single inheritance.  If you need to mimic multiple inheritance, you can define modules and use mix-ins to pull the module methods into classes.  Python supports multiple inheritance rather than module mix-ins.</p></li>\u000a<li><p>Python supports only single-line lambda functions.  Ruby blocks, which are kind of/sort of lambda functions, can be arbitrarily big.  Because of this, Ruby code is typically written in a more functional style than Python code.  For example, to loop over a list in Ruby, you typically do</p>\u000a\u000a<pre><code>collection.each do |value|\u000a  ...\u000aend\u000a</code></pre>\u000a\u000a<p>The block works very much like a function being passed to <code>collection.each</code>.  If you were to do the same thing in Python, you'd have to define a named inner function and then pass that to the collection each method (if list supported this method):</p>\u000a\u000a<pre><code>def some_operation(value):\u000a  ...\u000a\u000acollection.each(some_operation)\u000a</code></pre>\u000a\u000a<p>That doesn't flow very nicely.  So, typically the following non-functional approach would be used in Python:</p>\u000a\u000a<pre><code>for value in collection:\u000a  ...\u000a</code></pre></li>\u000a<li><p>Using resources in a safe way is quite different between the two languages.  Here, the problem is that you want to allocate some resource (open a file, obtain a database cursor, etc), perform some arbitrary operation on it, and then close it in a safe manner even if an exception occurs.</p>\u000a\u000a<p>In Ruby, because blocks are so easy to use (see #9), you would typically code this pattern as a method that takes a block for the arbitrary operation to perform on the resource.</p>\u000a\u000a<p>In Python, passing in a function for the arbitrary action is a little clunkier since you have to write a named, inner function (see #9).  Instead, Python uses a <code>with</code> statement for safe resource handling.  See <a href="http://stackoverflow.com/questions/865115/how-do-i-correctly-clean-up-a-python-object/865272#865272">How do I correctly clean up a Python object?</a> for more details.</p></li>\u000a</ol>\u000a

V<p>I don't know where that "Stackless is 10% faster" on the Wiki came from, but then again I've never tried to measure those performance numbers. I can't think of what Stackless does to make a difference that big.</p>\u000a\u000a<p>Stackless is an amazing tool with several organizational/political problems.</p>\u000a\u000a<p>The first comes from history. Christian Tismer started talking about what eventually became Stackless about 10 years ago. He had an idea of what he wanted, but had a hard time explaining what he was doing and why people should use it. This is partially because his background didn't have the CS training regarding ideas like coroutines and because his presentations and discussion are very implementation oriented, which is hard for anyone not already hip-deep in continuations to understand how to use it as a solution to their problems.</p>\u000a\u000a<p>For that reason, the initial documentation was poor. There were some descriptions of how to use it, with the best from third-party contributors. At PyCon 2007 I gave a talk on "<a href="http://dalkescientific.com/StacklessPyCon2007-Dalke.pdf" rel="noreferrer">Using Stackless</a>" which went over quite well, according to the PyCon survey numbers. Richard Tew has done a great job collecting these, updating <a href="http://stackless.com" rel="noreferrer">stackless.com</a>, and maintaining the distribution when new Python releases comes up. He's an employee of <a href="http://ccpgames.com/" rel="noreferrer">CCP Games</a>, developers of EVE Online, which uses Stackless as an essential part of their gaming system.</p>\u000a\u000a<p>CCP games is also the biggest real-world example people use when they talk about Stackless. The main tutorial for Stackless is Grant Olson's "<a href="http://www.grant-olson.net/projects/1970/01/02/introduction-to-stackless-python.html" rel="noreferrer">Introduction to Concurrent Programming with Stackless Python</a>", which is also game oriented. I think this gives people a skewed idea that Stackless is games-oriented, when it's more that games are more easily continuation oriented.</p>\u000a\u000a<p>Another difficulty has been the source code. In its original form it required changes to many parts of Python, which made Guido van Rossum, the Python lead, wary. Part of the reason, I think, was support for call/cc that was later removed as being "too much like supporting a goto when there are better higher-level forms." I'm not certain about this history, so just read this paragraph as "Stackless used to require too many changes."</p>\u000a\u000a<p>Later releases didn't require the changes, and Tismer continued to push for its inclusion in Python. While there was some consideration, the official stance (as far as I know) is that CPython is not only a Python implementation but it's meant as a reference implementation, and it won't include Stackless functionality because it can't be implemented by Jython or Iron Python.</p>\u000a\u000a<p>There are absolutely no plans for "<em>significant changes to the code base</em>". That quote and  reference hyperlink from Arafangion's (see the comment) are from roughly 2000/2001. The structural changes have long been done, and it's what I mentioned above. Stackless as it is now is stable and mature, with only minor tweaks to the code base over the last several years. </p>\u000a\u000a<p>One final limitation with Stackless - there is no strong advocate for Stackless. Tismer is now deeply involved with <a href="http://pypy.org/" rel="noreferrer">PyPy</a>, which is an implementation of Python for Python. He has implemented the Stackless functionality in PyPy and considers it much superior to Stackless itself, and feels that PyPy is the way of the future. Tew maintains Stackless but he isn't interested in advocacy. I considered being in that role, but couldn't see how I could make an income from it.</p>\u000a\u000a<p>Though if you want training in Stackless, feel free to <a href="http://mailto:dalke@dalkescientific.com" rel="noreferrer">contact me</a>! :)</p>\u000a

V<p>Thanks to bits and pieces from various replies, I think we can stitch up an explanation. </p>\u000a\u000a<p>By trying to print an unicode string, u'\u005cxe9', Python implicitly try to encode that string using the encoding scheme currently stored in sys.stdout.encoding. Python actually picks up this setting from the environment it's been initiated from. If it can't find a proper encoding from the environment, only then does it revert to its <em>default</em>, ASCII.</p>\u000a\u000a<p>For example, I use a bash shell which encoding defaults to UTF-8. If I start Python from it, it picks up and use that setting:</p>\u000a\u000a<pre><code>$ python\u000a\u000a&gt;&gt;&gt; import sys\u000a&gt;&gt;&gt; print sys.stdout.encoding\u000aUTF-8\u000a</code></pre>\u000a\u000a<p>Let's for a moment exit the Python shell and set bash's environment with some bogus encoding:</p>\u000a\u000a<pre><code>$ export LC_CTYPE=klingon\u000a# we should get some error message here, just ignore it.\u000a</code></pre>\u000a\u000a<p>Then start the python shell again and verify that it does indeed revert to its default ascii encoding.</p>\u000a\u000a<pre><code>$ python\u000a\u000a&gt;&gt;&gt; import sys\u000a&gt;&gt;&gt; print sys.stdout.encoding\u000aANSI_X3.4-1968\u000a</code></pre>\u000a\u000a<p>Bingo! </p>\u000a\u000a<p>If you now try to output some unicode character outside of ascii you should get a nice error message</p>\u000a\u000a<pre><code>&gt;&gt;&gt; print u'\u005cxe9'\u000aUnicodeEncodeError: 'ascii' codec can't encode character u'\u005cxe9' \u000ain position 0: ordinal not in range(128)\u000a</code></pre>\u000a\u000a<hr>\u000a\u000a<p>Lets exit Python and discard the bash shell. </p>\u000a\u000a<p>We'll now observe what happens after Python outputs strings. For this we'll first start a bash shell within a graphic terminal (I use Gnome Terminal) and we'll set the terminal to decode output with ISO-8859-1 aka latin-1 (graphic terminals usually have an option to <em>Set Character Encoding</em> in one of their dropdown menus). Note that this doesn't change the actual <em>shell environment's</em> encoding, it only changes the way the <em>terminal</em> itself will decode output it's given, a bit like a web browser does. You can therefore change the terminal's encoding, independantly from the shell's environment. Let's then start Python from the shell and verify that sys.stdout.encoding is set to the shell environment's encoding (UTF-8 for me):</p>\u000a\u000a<pre><code>$ python\u000a\u000a&gt;&gt;&gt; import sys\u000a\u000a&gt;&gt;&gt; print sys.stdout.encoding\u000aUTF-8\u000a\u000a&gt;&gt;&gt; print '\u005cxe9' # (1)\u000a\u000a&gt;&gt;&gt; print u'\u005cxe9' # (2)\u000a챕\u000a&gt;&gt;&gt; print u'\u005cxe9'.encode('latin-1') # (3)\u000a\u000a&gt;&gt;&gt;\u000a</code></pre>\u000a\u000a<p>(1) python outputs binary string as is, terminal receives it and tries to match its value with latin-1 character map. In latin-1, 0xe9 or 233 yields the character "" and so that's what the terminal displays.</p>\u000a\u000a<p>(2) python attempts to <em>implicitly</em> encode the Unicode string with whatever scheme is currently set in sys.stdout.encoding, in this instance it's "UTF-8". After UTF-8 encoding, the resulting binary string is '\u005cxc3\u005cxa9' (see later explanation). Terminal receives the stream as such and tries to decode 0xc3a9 using latin-1, but latin-1 goes from 0 to 255 and so, only decodes streams 1 byte at a time. 0xc3a9 is 2 bytes long, latin-1 decoder therefore interprets it as 0xc3 (195) and 0xa9 (169) and that yields 2 characters:  and .</p>\u000a\u000a<p>(3) python encodes unicode code point u'\u005cxe9' (233) with the latin-1 scheme. Turns out latin-1 code points range is 0-255 and points to the exact same character as Unicode within that range. Therefore, Unicode code points in that range will yield the same value when encoded in latin-1. So u'\u005cxe9' (233) encoded in latin-1 will also yields the binary string '\u005cxe9'. Terminal receives that value and tries to match it on the latin-1 character map. Just like case (1), it yields "" and that's what's displayed. </p>\u000a\u000a<p>Let's now change the terminal's encoding settings to UTF-8 from the dropdown menu (like you would change your web browser's encoding settings). No need to stop Python or restart the shell. The terminal's encoding now matches Python's. Let's try printing again:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; print '\u005cxe9' # (4)\u000a\u000a&gt;&gt;&gt; print u'\u005cxe9' # (5)\u000a\u000a&gt;&gt;&gt; print u'\u005cxe9'.encode('latin-1') # (6)\u000a\u000a&gt;&gt;&gt;\u000a</code></pre>\u000a\u000a<p>(4) python outputs a <em>binary</em> string as is. Terminal attempts to decode that stream with UTF-8. But UTF-8 doesn't understand the value 0xe9 (see later explanation) and is therefore unable to convert it to a unicode code point. No code point found, no character printed. </p>\u000a\u000a<p>(5) python attempts to <em>implicitly</em> encode the Unicode string with whatever's in sys.stdout.encoding. Still "UTF-8". The resulting binary string is '\u005cxc3\u005cxa9'. Terminal receives the stream and attempts to decode 0xc3a9 also using UTF-8. It yields back code value 0xe9 (233), which on the Unicode character map points to the symbol "". Terminal displays "".</p>\u000a\u000a<p>(6) python encodes unicode string with latin-1, it yields a binary string with the same value '\u005cxe9'. Again, for the terminal this is pretty much the same as case (4).</p>\u000a\u000a<p>Conclusions:\u000a- Python outputs non-unicode strings as raw data, without considering its default encoding. The terminal just happens to display them if its current encoding matches the data. \u000a- Python outputs Unicode strings after encoding them using the scheme specified in sys.stdout.encoding. \u000a- Python gets that setting from the shell's environment.\u000a- the terminal displays output according to its own encoding settings.\u000a- the terminal's encoding is independant from the shell's.</p>\u000a\u000a<hr>\u000a\u000a<p><strong>More details on unicode, UTF-8 and latin-1:</strong></p>\u000a\u000a<p>Unicode is basically a table of characters where some keys (code points) have been conventionally assigned to point to some symbols. e.g. by convention it's been decided that key 0xe9 (233) is the value pointing to the symbol ''. ASCII and Unicode use the same code points from 0 to 127, as do latin-1 and Unicode from 0 to 255. That is, 0x41 points to 'A' in ASCII, latin-1 and Unicode, 0xc8 points to '' in latin-1 and Unicode, 0xe9 points to '' in latin-1 and Unicode. </p>\u000a\u000a<p>When working with electronic devices, Unicode code points need an efficient way to be represented electronically. That's what encoding schemes are about. Various Unicode encoding schemes exist (utf7, UTF-8, UTF-16, UTF-32). The most intuitive and straight forward encoding approach would be to simply use a code point's value in the Unicode map as its value for its electronic form, but Unicode currently has over a million code points, which means that some of them require 3 bytes to be expressed. To work efficiently with text, a 1 to 1 mapping would be rather impractical, since it would require that all code points be stored in exactly the same amount of space, with a minimum of 3 bytes per character, regardless of their actual need.</p>\u000a\u000a<p>Most encoding schemes have shortcomings regarding space requirement, the most economic ones don't cover all unicode code points, for example ascii only covers the first 128, while latin-1 covers the first 256. Others that try to be more comprehensive end up also being wasteful, since they require more bytes than necessary, even for common "cheap" characters. UTF-16 for instance, uses a minimum of 2 bytes per character, including those in the ascii range ('B' which is 65, still requires 2 bytes of storage in UTF-16). UTF-32 is even more wasteful as it stores all characters in 4 bytes. </p>\u000a\u000a<p>UTF-8 happens to have cleverly resolved the dilemma, with a scheme able to store code points with a variable amount of byte spaces. As part of its encoding strategy, UTF-8 laces code points with flag bits that indicate (presumably to decoders) their space requirements and their boundaries.</p>\u000a\u000a<p><strong>UTF-8 encoding of unicode code points in the ascii range (0-127):</strong></p>\u000a\u000a<pre><code>0xxx xxxx  (in binary)\u000a</code></pre>\u000a\u000a<ul>\u000a<li>the x's show the actual space reserved to "store" the code point during encoding </li>\u000a<li>The leading 0 is a flag that indicates to the UTF-8 decoder that this code point will only require 1 byte. </li>\u000a<li>upon encoding, UTF-8 doesn't change the value of code points in that specific range (i.e. 65 encoded in UTF-8 is also 65). Considering that Unicode and ASCII are also compatible in the same range, it incidentally makes UTF-8 and ASCII also compatible in that range.</li>\u000a</ul>\u000a\u000a<p>e.g. Unicode code point for 'B' is '0x42' or 0100 0010 in binary (as we said, it's the same in ASCII). After encoding in UTF-8 it becomes:</p>\u000a\u000a<pre><code>0xxx xxxx  &lt;-- UTF-8 encoding for Unicode code points 0 to 127\u000a*100 0010  &lt;-- Unicode code point 0x42\u000a0100 0010  &lt;-- UTF-8 encoded (exactly the same)\u000a</code></pre>\u000a\u000a<p><strong>UTF-8 encoding of Unicode code points above 127 (non-ascii):</strong></p>\u000a\u000a<pre><code>110x xxxx 10xx xxxx            &lt;-- (from 128 to 2047)\u000a1110 xxxx 10xx xxxx 10xx xxxx  &lt;-- (from 2048 to 65535)\u000a</code></pre>\u000a\u000a<ul>\u000a<li>the leading bits '110' indicate to the UTF-8 decoder the beginning of a code point encoded in 2 bytes, whereas '1110' indicates 3 bytes, 11110 would indicate 4 bytes and so forth.</li>\u000a<li>the inner '10' flag bits are used to signal the beginning of an inner byte.</li>\u000a<li>again, the x's mark the space where the Unicode code point value is stored after encoding.</li>\u000a</ul>\u000a\u000a<p>e.g. '' Unicode code point is 0xe9 (233).</p>\u000a\u000a<pre><code>1110 1001    &lt;-- 0xe9\u000a</code></pre>\u000a\u000a<p>When UTF-8 encodes this value, it determines that the value is larger than 127 and less than 2048, therefore should be encoded in 2 bytes:</p>\u000a\u000a<pre><code>110x xxxx 10xx xxxx   &lt;-- UTF-8 encoding for Unicode 128-2047\u000a***0 0011 **10 1001   &lt;-- 0xe9\u000a1100 0011 1010 1001   &lt;-- '' after UTF-8 encoding\u000aC    3    A    9\u000a</code></pre>\u000a\u000a<p>The 0xe9 Unicode code points after UTF-8 encoding becomes 0xc3a9. Which is exactly how the terminal receives it. If your terminal is set to decode strings using latin-1 (one of the non-unicode legacy encodings), you'll see 챕, because it just so happens that 0xc3 in latin-1 points to  and 0xa9 to .</p>\u000a

And, since the first child is\u000a# a session leader without a controlling terminal, it's possible for\u000a# it to acquire one by opening a terminal in the future (System V-\u000a# based systems). V<p>Here is what is going on.</p>\u000a\u000a<p>First, the only global variables Python really has are module-scoped variables.  You cannot make a variable that is truly global; all you can do is make a variable in a particular scope.  (If you make a variable inside the Python interpreter, and then import other modules, your variable is in the outermost scope and thus global within your Python session.)</p>\u000a\u000a<p>All you have to do to make a module-global variable is just assign to a name.</p>\u000a\u000a<p>Imagine a file called foo.py, containing this single line:</p>\u000a\u000a<pre><code>X = 1\u000a</code></pre>\u000a\u000a<p>Now imagine you import it.</p>\u000a\u000a<pre><code>import foo\u000aprint(foo.X)  # prints 1\u000a</code></pre>\u000a\u000a<p>However, let's suppose you want to use one of your module-scope variables as a global inside a function, as in your example.  Python's default is to assume that function variables are local.  You simply add a <code>global</code> declaration in your function, before you try to use the global.</p>\u000a\u000a<pre><code>def initDB(name):\u000a    global __DBNAME__  # add this line!\u000a    if __DBNAME__ is None: # see notes below; explicit test for None\u000a        __DBNAME__ = name\u000a    else:\u000a        raise RuntimeError("Database name has already been set.")\u000a</code></pre>\u000a\u000a<p>By the way, for this example, the simple <code>if not __DBNAME__</code> test is adequate, because any string value other than an empty string will evaluate true, so any actual database name will evaluate true.  But for variables that might contain a number value that might be 0, you can't just say <code>if not variablename</code>; in that case, you should explicitly test for <code>None</code> using the <code>is</code> operator.  I modified the example to add an explicit <code>None</code> test.  The explicit test for <code>None</code> is never wrong, so I default to using it.</p>\u000a\u000a<p>Finally, as others have noted on this page, two leading underscores signals to Python that you want the variable to be "private" to the module.  If you ever do an <code>import * from mymodule</code>, Python will not import names with two leading underscores into your name space.  But if you just do a simple <code>import mymodule</code> and then say <code>dir(mymodule)</code> you will see the "private" variables in the list, and if you explicitly refer to <code>mymodule.__DBNAME__</code> Python won't care, it will just let you refer to it.  The double leading underscores are a major clue to users of your module that you don't want them rebinding that name to some value of their own.</p>\u000a\u000a<p>It is considered best practice in Python not to do <code>import *</code>, but to minimize the coupling and maximize explicitness by either using <code>mymodule.something</code> or by explicitly doing an import like <code>from mymodule import something</code>.</p>\u000a\u000a<p>EDIT: If, for some reason, you need to do something like this in a very old version of Python that doesn't have the <code>global</code> keyword, there is an easy workaround.  Instead of setting a module global variable directly, use a mutable type at the module global level, and store your values inside it.</p>\u000a\u000a<p>In your functions, the global variable name will be read-only; you won't be able to rebind the actual global variable name.  (If you assign to that variable name inside your function it will only affect the local variable name inside the function.)  But you can use that local variable name to access the actual global object, and store data inside it.</p>\u000a\u000a<p>You can use a <code>list</code> but your code will be ugly:</p>\u000a\u000a<pre><code>__DBNAME__ = [None] # use length-1 list as a mutable\u000a\u000a# later, in code:  \u000aif __DBNAME__[0] is None:\u000a    __DBNAME__[0] = name\u000a</code></pre>\u000a\u000a<p>A <code>dict</code> is better.  But the most convenient is a class instance, and you can just use a trivial class:</p>\u000a\u000a<pre><code>class Box:\u000a    pass\u000a\u000a__m = Box()  # m will contain all module-level values\u000a__m.dbname = None  # database name global in module\u000a\u000a# later, in code:\u000aif __m.dbname is None:\u000a    __m.dbname = name\u000a</code></pre>\u000a\u000a<p>(You don't really need to capitalize the database name variable.)</p>\u000a\u000a<p>I like the syntactic sugar of just using <code>__m.dbname</code> rather than <code>__m["DBNAME"]</code>; it seems the most convenient solution in my opinion.  But the <code>dict</code> solution works fine also.</p>\u000a\u000a<p>With a <code>dict</code> you can use any hashable value as a key, but when you are happy with names that are valid identifiers, you can use a trivial class like <code>Box</code> in the above.</p>\u000a

V<p>Both ways aren't correct or incorrect, they are just two different kind of class elements:</p>\u000a\u000a<ul>\u000a<li>Elements outside the <code>__init__</code> method are static elements, it means, they belong to the class.</li>\u000a<li>Elements inside the <code>__init__</code> method are elements of the object (<code>self</code>), they don't belong to the class.</li>\u000a</ul>\u000a\u000a<p>You'll see it more clearly with some code:</p>\u000a\u000a<pre><code>class MyClass:\u000a    static_elem = 123\u000a\u000a    def __init__(self):\u000a        self.object_elem = 456\u000a\u000ac1 = MyClass()\u000ac2 = MyClass()\u000a\u000a# Initial values of both elements\u000a&gt;&gt;&gt; print c1.static_elem, c1.object_elem \u000a123 456\u000a&gt;&gt;&gt; print c2.static_elem, c2.object_elem\u000a123 456\u000a\u000a# Nothing new so far ...\u000a\u000a# Let's try changing the static element\u000aMyClass.static_elem = 999\u000a\u000a&gt;&gt;&gt; print c1.static_elem, c1.object_elem\u000a999 456\u000a&gt;&gt;&gt; print c2.static_elem, c2.object_elem\u000a999 456\u000a\u000a# Now, let's try changing the object element\u000ac1.object_elem = 888\u000a\u000a&gt;&gt;&gt; print c1.static_elem, c1.object_elem\u000a999 888\u000a&gt;&gt;&gt; print c2.static_elem, c2.object_elem\u000a999 456\u000a</code></pre>\u000a\u000a<p>As you can see, when we changed the class element, it changed for both objects. But, when we changed the object element, the other object remained unchanged.</p>\u000a

The idea is this: as soon as we find two elements that are not in the right order, we set <code>sorted</code> back to <code>False</code>. <code>sorted</code> will remain <code>True</code> <em>only if there were no elements in the wrong order</em>.</p>\u000a\u000a<pre><code>sorted = False  # We haven't started sorting yet\u000a\u000awhile not sorted:\u000a    sorted = True  # Assume the list is now sorted\u000a    for element in range(0, length):\u000a        if badList[element] &gt; badList[element + 1]:\u000a            sorted = False  # We found two elements in the wrong order\u000a            hold = badList[element + 1]\u000a            badList[element + 1] = badList[element]\u000a            badList[element] = hold\u000a    # We went through the whole list. At this point, if there were no elements\u000a    # in the wrong order, sorted is still True. Otherwise, it's false, and the\u000a    # while loop executes again.\u000a</code></pre>\u000a\u000a<p>There are also minor little issues that would help the code be more efficient or readable.</p>\u000a\u000a<ul>\u000a<li><p>In the <code>for</code> loop, you use the variable <code>element</code>. Technically, <code>element</code> is not an element; it's a number representing a list index. Also, it's quite long.
