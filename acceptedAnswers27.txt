V<p>Backticks are a deprecated alias for <code>repr()</code>. Don't use them any more, the syntax was removed in Python 3.0.</p>\u000a\u000a<p>Using backticks seems to be faster than using <code>repr(num)</code> or <code>num.__repr__()</code> in version 2.x. I guess it's because additional dictionary lookup is required in the global namespace (for <code>repr</code>), or in the object's namespace (for <code>__repr__</code>), respectively.</p>\u000a\u000a<hr>\u000a\u000a<p>Using the <code>dis</code> module proves my assumption:</p>\u000a\u000a<pre><code>def f1(a):\u000a    return repr(a)\u000a\u000adef f2(a):\u000a    return a.__repr__()\u000a\u000adef f3(a):\u000a    return `a`\u000a</code></pre>\u000a\u000a<p>Disassembling shows:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; import dis\u000a&gt;&gt;&gt; dis.dis(f1)\u000a  3           0 LOAD_GLOBAL              0 (repr)\u000a              3 LOAD_FAST                0 (a)\u000a              6 CALL_FUNCTION            1\u000a              9 RETURN_VALUE\u000a&gt;&gt;&gt; dis.dis(f2)\u000a  6           0 LOAD_FAST                0 (a)\u000a              3 LOAD_ATTR                0 (__repr__)\u000a              6 CALL_FUNCTION            0\u000a              9 RETURN_VALUE        \u000a&gt;&gt;&gt; dis.dis(f3)\u000a  9           0 LOAD_FAST                0 (a)\u000a              3 UNARY_CONVERT       \u000a              4 RETURN_VALUE   \u000a</code></pre>\u000a\u000a<p><code>f1</code> involves a global lookup for <code>repr</code>, <code>f2</code> an attribute lookup for <code>__repr__</code>, whereas the backtick operator is implemented in a separate opcode. Since there is no overhead for dictionary lookup (<code>LOAD_GLOBAL</code>/<code>LOAD_ATTR</code>) nor for function calls (<code>CALL_FUNCTION</code>), backticks are faster.</p>\u000a\u000a<p>I guess that the Python folks decided that having a separate low-level operation for <code>repr()</code> is not worth it, and having both <code>repr()</code> and backticks violates the principle</p>\u000a\u000a<blockquote>\u000a  <p>"There should be one-- and preferably only one --obvious way to do it"</p>\u000a</blockquote>\u000a\u000a<p>so the feature was removed in Python 3.0.</p>\u000a

V<p>I don't know whether this is <em>function</em> chaining as much as it's <em>callable</em> chaining, but, since functions <em>are</em> callables I guess there's no harm done. Either way, there's two ways I can think of doing this:</p>\u000a\u000a<h3>Sub-classing <code>int</code> and defining <code>__call__</code>:</h3>\u000a\u000a<p>The first way would be with a custom <code>int</code> subclass that defines <a href="https://docs.python.org/3/reference/datamodel.html#object.__call__" rel="nofollow noreferrer"><code>__call__</code></a> which returns a new instance of itself with the updated value:</p>\u000a\u000a<pre><code>class CustomInt(int):\u000a    def __call__(self, v):\u000a        return CustomInt(self + v)\u000a</code></pre>\u000a\u000a<p>Function <code>add</code> can now be defined to return a <code>CustomInt</code> instance, which, as a callable that returns an updated value of itself, can be called in succession:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; def add(v):\u000a...    return CustomInt(v)\u000a&gt;&gt;&gt; add(1)\u000a1\u000a&gt;&gt;&gt; add(1)(2)\u000a3\u000a&gt;&gt;&gt; add(1)(2)(3)(44)  # and so on..\u000a50\u000a</code></pre>\u000a\u000a<p>In addition, as an <code>int</code> subclass, the returned value retains the <code>__repr__</code> and <code>__str__</code> behavior of <code>int</code>s. <em>For more complex operations though, you should define other dunders appropriately</em>. </p>\u000a\u000a<p>As @Caridorc noted in a comment, <code>add</code> could also be simply written as:</p>\u000a\u000a<pre><code>add = CustomInt \u000a</code></pre>\u000a\u000a<p>Renaming the class to <code>add</code> instead of <code>CustomInt</code> also works similarly.</p>\u000a\u000a<hr>\u000a\u000a<h3>Define a closure, requires extra call to yield value:</h3>\u000a\u000a<p>The only other way I can think of involves a nested function that requires an extra empty argument call in order to return the result. I'm <strong>not</strong> using <code>nonlocal</code> and opt for attaching attributes to the function objects to make it portable between Pythons:</p>\u000a\u000a<pre><code>def add(v):\u000a    def _inner_adder(val=None):  \u000a        """ \u000a        if val is None we return _inner_adder.v \u000a        else we increment and return ourselves\u000a        """\u000a        if val is None:    \u000a            return _inner_adder.v\u000a        _inner_adder.v += val\u000a        return _inner_adder\u000a    _inner_adder.v = v  # save value\u000a    return _inner_adder \u000a</code></pre>\u000a\u000a<p>This continuously returns itself (<code>_inner_adder</code>) which, if a <code>val</code> is supplied, increments it (<code>_inner_adder += val</code>) and if not, returns the value as it is. Like I mentioned, it requires an extra <code>()</code> call in order to return the incremented value:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; add(1)(2)()\u000a3\u000a&gt;&gt;&gt; add(1)(2)(3)()  # and so on..\u000a6\u000a</code></pre>\u000a

V<p>My results were similar to yours: the code using variables was pretty consistently 10-20 % faster. However when I used IPython on the very same Python 3.4, I got these results:</p>\u000a\u000a<pre><code>In [1]: %timeit -n10000 -r20 tuple(range(2000)) == tuple(range(2000))\u000a10000 loops, best of 20: 74.2 µs per loop\u000a\u000aIn [2]: %timeit -n10000 -r20 a = tuple(range(2000));  b = tuple(range(2000)); a==b\u000a10000 loops, best of 20: 75.7 µs per loop\u000a</code></pre>\u000a\u000a<p>Notably, I never managed to get even close to the 74.2 µs for the former when I used <code>-mtimeit</code> from the command line.</p>\u000a\u000a<hr>\u000a\u000a<p>So I decided to run the command with <code>strace</code> and indeed there is something fishy going on:</p>\u000a\u000a<pre><code>% strace -o withoutvars python3 -m timeit "tuple(range(2000)) == tuple(range(2000))"\u000a10000 loops, best of 3: 134 usec per loop\u000a% strace -o withvars python3 -mtimeit "a = tuple(range(2000));  b = tuple(range(2000)); a==b"\u000a10000 loops, best of 3: 75.8 usec per loop\u000a% grep mmap withvars|wc -l\u000a46\u000a% grep mmap withoutvars|wc -l\u000a41149\u000a</code></pre>\u000a\u000a<p>Now that is a good reason for the difference. The code that does not use variables causes the <code>mmap</code> system call be called almost 1000x more than the one that uses intermediate variables.</p>\u000a\u000a<p>The <code>withoutvars</code> is full of <code>mmap</code>/<code>munmap</code> for a 256k region; these same lines are repeated over and over again:</p>\u000a\u000a<pre><code>mmap(NULL, 262144, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANONYMOUS, -1, 0) = 0x7f32e56de000\u000amunmap(0x7f32e56de000, 262144)          = 0\u000ammap(NULL, 262144, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANONYMOUS, -1, 0) = 0x7f32e56de000\u000amunmap(0x7f32e56de000, 262144)          = 0\u000ammap(NULL, 262144, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANONYMOUS, -1, 0) = 0x7f32e56de000\u000amunmap(0x7f32e56de000, 262144)          = 0\u000a</code></pre>\u000a\u000a<hr>\u000a\u000a<p>The <code>mmap</code> call seems to be coming from the function <code>_PyObject_ArenaMmap</code> from <code>Objects/obmalloc.c</code>; the <code>obmalloc.c</code> also contains the macro <code>ARENA_SIZE</code>, which is <code>#define</code>d to be <code>(256 &lt;&lt; 10)</code> (that is <code>262144</code>); similarly the <code>munmap</code> matches the <code>_PyObject_ArenaMunmap</code> from <code>obmalloc.c</code>.</p>\u000a\u000a<p><code>obmalloc.c</code> says that</p>\u000a\u000a<blockquote>\u000a  <p>Prior to Python 2.5, arenas were never <code>free()</code>'ed. V<p>This is one of those subtle differences among various mapping collections that can bite you.</p>\u000a\u000a<p>In Python (and apparently in Lua) the keys to a mapping (dictionary or table, respectively) are object references.  In Python they must be immutable types, or they must be objects which implement a <code>__hash__</code> method.  (The Lua docs suggest that it automatically uses the object's  ID as a hash/key even for mutable objects and relies on string interning to ensure that equivalent strings map to the same objects).</p>\u000a\u000a<p>In Perl, Javascript, awk and many other languages the keys for hashes, associative arrays or whatever they're called for the given language, are strings (or "scalars" in Perl).  In perl <code>$foo{1}, $foo{1.0}, and $foo{"1"}</code> are all references to the same mapping in <code>%foo</code> --- the key is <em>evaluated</em> as a scalar!</p>\u000a\u000a<p>JSON started as a Javascript serialization technology. (JSON stands for [J]ava[S]cript [o]bject [n]otation.)  Naturally it implements semantics for its mapping notation which are consistent with its mapping semantics.</p>\u000a\u000a<p>If both ends of your serialization are going to be Python then you'd be better off using pickles.  If you really need to convert these back from JSON into native Python objects I guess you have a couple of choices.  First you could try (<code>try: ... except: ...</code>) to convert any key to a number in the event of a dictionary look-up failure.  Alternatively, if you add  code to the other end (the serializer or generator of this JSON data) then you could have it  perform a JSON serialization on each of the key values --- providing those as a list of keys.  (Then your Python code would first iterate over the list of keys, instantiating/deserializing them into native Python objects ... and then use those for access the values out of the mapping).</p>\u000a

V<p>"Best practices" are very context-dependent, so I won't claim my practices are best, just that they work for me. I work on mostly small sites, so no multiple-server deployments, CDNs etc. I do need to support Webfaction shared hosting deployment, as some clients need the cheapest hosting they can find. I do often have to deploy sites multiple times in different environments, so repeatable scripted deploys are critical.</p>\u000a\u000a<ul>\u000a<li>I don't use pip bundles, I install from a requirements.txt. I do run my own <a href="http://github.com/ask/chishop" rel="nofollow noreferrer">chishop</a> server with sdists of everything I need, so there aren't multiple single points of failure in the build process. I also use PIP_DOWNLOAD_CACHE on my development machines to speed up bootstrapping project environments, since most of my projects' requirements overlap quite a bit.</li>\u000a<li>I have <a href="http://fabfile.org" rel="nofollow noreferrer">Fabric</a> scripts that can automatically set up and configure nginx + Apache/mod_wsgi on an Ubuntu VPS, or configure the equivalent on <a href="http://webfaction.com" rel="nofollow noreferrer">Webfaction</a> shared hosting, and then deploy the project.</li>\u000a<li>I do not use --no-site-packages with virtualenv, because I prefer having slow-moving compiled packages (Python Imaging Library, psycopg2) installed at the system level; too slow and troublesome to do inside every virtualenv. I have not had trouble with polluted system site-packages, because I generally don't pollute it. And in any case, you can install a different version of something in the virtualenv and it will take precedence.</li>\u000a<li>Each project has its own virtualenv. I have some bash scripts (not <a href="http://www.doughellmann.com/projects/virtualenvwrapper/" rel="nofollow noreferrer">virtualenvwrapper</a>, though a lot of people use that and love it) that automate deploying the virtualenv for a given project to a known location and installing that project's requirements into it.</li>\u000a<li>The entire deployment process, from a bare Ubuntu server VPS or Webfaction shared hosting account to a running website, is scripted using Fabric.</li>\u000a<li>Fabric scripts are part of the project source tree, and I run them from a local development checkout.</li>\u000a<li>I have no need for SCons (that I am aware of).</li>\u000a</ul>\u000a\u000a<h2>Deployment</h2>\u000a\u000a<p>At the moment a fresh deployment is split into these steps:</p>\u000a\u000a<ul>\u000a<li><code>fab staging bootstrap</code> (server setup and initial code deploy)</li>\u000a<li><code>fab staging enable</code> (enable the Apache/nginx config for this site)</li>\u000a<li><code>fab staging reload_server</code> (reload Apache/nginx config).</li>\u000a</ul>\u000a\u000a<p>Those can of course be combined into a single command line <code>fab staging bootstrap enable reload_server</code>.</p>\u000a\u000a<p>Once these steps are done, updating the deployment with new code is just <code>fab staging deploy</code>.</p>\u000a\u000a<p>If I need to roll back an update, <code>fab staging rollback</code>. Nothing particularly magical in the rollback; it just rolls back the code to the last-deployed version and migrates the database to the previous state (this does require recording some metadata about the migration state of the DB post-deploy, I just do that in a text file).</p>\u000a\u000a<h2>Examples</h2>\u000a\u000a<p>I haven't used the Fabric scripts described in this answer for a few years, so they aren't maintained at all and I disclaim responsibility for their quality :-) But you can see them at <a href="https://bitbucket.org/carljm/django-project-template" rel="nofollow noreferrer">https://bitbucket.org/carljm/django-project-template</a> - in <code>fabfile.py</code> in the repo root, and in the <code>deploy/</code> subdirectory.</p>\u000a

V<p>The Python interpreter assigns the last expression value to <code>_</code>.</p>\u000a\u000a<p>This behaviour is limited to the <a href="http://en.wikipedia.org/wiki/Read%E2%80%93eval%E2%80%93print_loop">REPL interpreter</a> only, and is intended to assist in interactive coding sessions:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; import math\u000a&gt;&gt;&gt; math.pow(3.0, 5)\u000a243.0\u000a&gt;&gt;&gt; result = _\u000a&gt;&gt;&gt; result\u000a243.0\u000a</code></pre>\u000a\u000a<p>The <em>standard</em> Python interpreter goes to some length to not trample on user-defined values though; if you yourself assign something <em>else</em> to <code>_</code> then the interpreter will not overwrite that (technically speaking, the <a href="http://stackoverflow.com/questions/1538832/is-this-single-underscore-a-built-in-variable-in-python"><code>_</code> variable is a <code>__builtin__</code> attribute</a>, your own assignments are 'regular' globals). You are not using the standard Python interpreter though; you are using IPython, and that interpreter is not that careful.</p>\u000a\u000a<p>IPython <a href="http://ipython.org/ipython-doc/dev/interactive/reference.html#output-caching-system">documents this behaviour explicitly</a>:</p>\u000a\u000a<blockquote>\u000a  <p>The following GLOBAL variables always exist (so don\u2019t overwrite them!):</p>\u000a  \u000a  <ul>\u000a  <li><code>[_]</code> (a single underscore) : stores previous output, like Python\u2019s default interpreter.  </li>\u000a  </ul>\u000a  \u000a  <p><em>[...]</em></p>\u000a</blockquote>\u000a\u000a<p>Outside of the Python interpreter, <code>_</code> is by convention used as the name of the translatable text function (see the <a href="http://docs.python.org/2/library/gettext.html#internationalizing-your-programs-and-modules"><code>gettext</code> module</a>; external tools look for that function to extract translatable strings).</p>\u000a\u000a<p>In loops, using <code>_</code> as an assignment target tells readers of your code that you are going to ignore that value; e.g. <code>[random.random() for _ in range(5)]</code> to generate a list of 5 random float values.</p>\u000a

V<p>Let's look at the <a href="http://docs.python.org/2.7/reference/expressions.html#calls">language specification</a>:</p>\u000a\u000a<pre><code>call                 ::=  primary "(" [argument_list [","]\u000a                          | expression genexpr_for] ")"\u000aargument_list        ::=  positional_arguments ["," keyword_arguments]\u000a                            ["," "*" expression] ["," keyword_arguments]\u000a                            ["," "**" expression]\u000a                          | keyword_arguments ["," "*" expression]\u000a                            ["," "**" expression]\u000a                          | "*" expression ["," "*" expression] ["," "**" expression]\u000a                          | "**" expression\u000apositional_arguments ::=  expression ("," expression)*\u000akeyword_arguments    ::=  keyword_item ("," keyword_item)*\u000akeyword_item         ::=  identifier "=" expression\u000a</code></pre>\u000a\u000a<p>Let's sift down to the parts we care about:</p>\u000a\u000a<pre><code>call                 ::=  primary "(" [argument_list [","]] ")"\u000aargument_list        ::=  positional_arguments ["," keyword_arguments]\u000a                            ["," "*" expression] ["," keyword_arguments]\u000a                            ["," "**" expression]\u000apositional_arguments ::=  expression ("," expression)*\u000akeyword_arguments    ::=  keyword_item ("," keyword_item)*\u000akeyword_item         ::=  identifier "=" expression\u000a</code></pre>\u000a\u000a<p>So, it looks like after any arguments to a function call, we're allowed an extra <code>,</code>. So this looks like a bug in the cpython implementation.</p>\u000a\u000a<p>Something like: <code>f(1, *(2,3,4), )</code> should work according to this grammar, but doesn't in CPython.</p>\u000a\u000a<hr>\u000a\u000a<p>In an earlier answer, <a href="http://stackoverflow.com/users/102441/eric">Eric</a> linked to the <a href="http://docs.python.org/2/reference/grammar.html">CPython grammar specification</a>, which includes the CPython implementation of the above grammar. Here it is below:</p>\u000a\u000a<pre><code>arglist: (argument ',')* ( argument [',']\u000a                         | '*' test (',' argument)* [',' '**' test] \u000a                         | '**' test\u000a                         )\u000a</code></pre>\u000a\u000a<p>Note, that this grammar is <strong>not the same</strong> as the one proposed by the language specification. I'd consider this an implementation bug.</p>\u000a\u000a<hr>\u000a\u000a<p>Note that there are additional issues with the CPython implementation. This should also be supported: <code>f(*(1,2,3), *(4,5,6))</code></p>\u000a\u000a<p>Oddly though, the specification does not allow <code>f(*(1,2,3), *(4,5,6), *(7,8,9))</code></p>\u000a\u000a<p><strong>As I look at this more,</strong> I think this part of the specification needs some fixing. This is allowed: <code>f(x=1, *(2,3))</code>, but this isn't: <code>f(x=1, 2, 3)</code>.</p>\u000a\u000a<hr>\u000a\u000a<p>And to perhaps be helpful to the original question, in CPython, you can have a trailing comma if you don't use the <code>*args</code> or the <code>**kwargs</code> feature. I agree that this is lame.</p>\u000a

V<p>It depends what you want your tasks to be doing, if you need to distribute them, and how you want to manage them.</p>\u000a\u000a<p>A crontab is capable of executing a script every N intervals. It runs, and then returns. Essentially you get a single execution each interval. You could just direct a crontab to execute a django management command and get access to the entire django environment, so celery doesn't really help you there.</p>\u000a\u000a<p>What celery brings to the table, with the help of a message queue, is distributed tasks. Many servers can join the pool of workers and each receive a work item without fear of double handling. It's also possible to execute a task as soon as it is ready. With cron, you're limited to a minimum of one minute.</p>\u000a\u000a<p>As an example, imagine you've just launched a new web application and you're receiving hundreds of sign ups that require an email to be sent to each user. Sending an email may take a long time (comparatively) so you decide that you'll handle activation emails via tasks.</p>\u000a\u000a<p>If you were using cron, you'd need to ensure that every minute cron is able to process all of the emails that need to be sent. If you have several servers you now need to make sure that you aren't sending multiple activation emails to the same user - you need some kind of synchronization.</p>\u000a\u000a<p>With celery, you add a task to the queue. You may have several workers per server so you've already scaled ahead of a cronjob. You may also have several servers allowing you to scale even more. Synchronization is handled as part of the 'queue'.</p>\u000a\u000a<p>You <em>can</em> use celery as a cron replacement but that's not really its primary use. It is used for farming out asynchronous tasks across a distributed cluster.</p>\u000a\u000a<p>And of course, celery has a <a href="http://docs.celeryproject.org/en/master/getting-started/introduction.html#features" rel="noreferrer">big list of features</a> that cron does not.</p>\u000a

V<p>Each of these return a dictionary:</p>\u000a\u000a<ul>\u000a<li><code>globals()</code> <em>always</em> returns the dictionary of the <em>module</em> namespace</li>\u000a<li><code>locals()</code> <em>always</em> returns <em>a</em> dictionary of the <em>current</em> namespace</li>\u000a<li><code>vars()</code> returns <em>either a</em> dictionary of the current namespace (if called with no argument) or <em>the</em> dictionary of the argument.</li>\u000a</ul>\u000a\u000a<p><code>locals</code> and <code>vars</code> could use some more explanation.  if <code>locals()</code> is called inside a function it constructs a dictionary of the function namespace as of that moment and returns it -- any further name assignments are <em>not</em> reflected in the returned dictionary, and any assignments <em>to</em> the dictionary are <em>not</em> reflected in the actual local namespace:</p>\u000a\u000a<pre><code>def test():\u000a    a = 1\u000a    b = 2\u000a    huh = locals()\u000a    c = 3\u000a    print(huh)\u000a    huh['d'] = 4\u000a    print(d)\u000a</code></pre>\u000a\u000a<p>gives us:</p>\u000a\u000a<pre><code>{'a': 1, 'b': 2}\u000aTraceback (most recent call last):\u000a  File "test.py", line 30, in &lt;module&gt;\u000a    test()\u000a  File "test.py", line 26, in test\u000a    print(d)\u000aNameError: global name 'd' is not defined\u000a</code></pre>\u000a\u000a<p>Two notes:</p>\u000a\u000a<ol>\u000a<li>This behavior is CPython specific -- other Pythons may allow the updates to make it back to the local namespace</li>\u000a<li>In CPython 2.x it is possible to make this work by putting an <code>exec "pass"</code> line in the function.</li>\u000a</ol>\u000a\u000a<p>If <code>locals()</code> is called <em>outside</em> a function it returns the actual dictionary that is the current namespace.  Further changes to the namespace <em>are</em> reflected in the dictionary, and changes to the dictionary <em>are</em> reflected in the namespace:</p>\u000a\u000a<pre><code>class Test(object):\u000a    a = 'one'\u000a    b = 'two'\u000a    huh = locals()\u000a    c = 'three'\u000a    huh['d'] = 'four'\u000a    print huh\u000a</code></pre>\u000a\u000a<p>gives us:</p>\u000a\u000a<pre><code>{\u000a  'a': 'one',\u000a  'b': 'two',\u000a  'c': 'three',\u000a  'd': 'four',\u000a  'huh': {...},\u000a  '__module__': '__main__',\u000a}\u000a</code></pre>\u000a\u000a<p>So far, everything I've said about <code>locals()</code> is also true for <code>vars()</code>... here's the difference:  <code>vars()</code> accepts a single object as its argument, and if you give it an object it returns the <code>__dict__</code> of that object.  If that object was <em>not</em> a function the <code>__dict__</code> returned is that object's namespace:</p>\u000a\u000a<pre><code>class Test(object):\u000a    a = 'one'\u000a    b = 'two'\u000a    def frobber(self):\u000a        print self.c\u000at = Test()\u000ahuh = vars(t)\u000ahuh['c'] = 'three'\u000at.frobber()\u000a</code></pre>\u000a\u000a<p>which gives us:</p>\u000a\u000a<pre><code>three\u000a</code></pre>\u000a\u000a<p>If the object <em>was</em> a function, you still get its <code>__dict__</code>, but unless you're doing fun and interesting stuff its probably not very useful:</p>\u000a\u000a<pre><code>def test():\u000a    a = 1\u000a    b = 2\u000a    print test.c\u000ahuh = vars(test)       # these two lines are the same as 'test.c = 3'\u000ahuh['c'] = 3\u000atest()\u000a</code></pre>\u000a\u000a<p>which gives us:</p>\u000a\u000a<pre><code>3\u000a</code></pre>\u000a

V<p>There are a lot of different aspects of Twisted that you might find cool.</p>\u000a\u000a<p>Twisted includes lots and lots of protocol implementations, meaning that more likely than not there will be an API you can use to talk to some remote system (either client or server in most cases) - be it <a href="http://twistedmatrix.com/trac/wiki/TwistedWeb">HTTP</a>, <a href="http://twistedmatrix.com/documents/current/api/twisted.protocols.ftp.html">FTP</a>, <a href="http://twistedmatrix.com/trac/wiki/TwistedMail">SMTP, POP3, IMAP4</a>, <a href="http://twistedmatrix.com/trac/wiki/TwistedNames">DNS</a>, <a href="http://twistedmatrix.com/trac/wiki/TwistedWords">IRC, MSN, OSCAR, XMPP/Jabber</a>, <a href="http://twistedmatrix.com/trac/wiki/TwistedConch">telnet, SSH</a>, SSL, <a href="http://twistedmatrix.com/trac/wiki/TwistedNews">NNTP</a>, or one of the <strong>really</strong> obscure protocols like Finger, or ident, or one of the lower level protocol-building-protocols like <a href="http://cr.yp.to/proto/netstrings.txt">DJB's</a> <a href="http://twistedmatrix.com/documents/current/api/twisted.protocols.basic.NetstringReceiver.html">netstrings</a>, <a href="http://twistedmatrix.com/documents/current/api/twisted.protocols.basic.LineReceiver.html">simple line-oriented protocols</a>, or even one of Twisted's custom protocols like <a href="http://twistedmatrix.com/documents/10.2.0/core/howto/pb-usage.html">Perspective Broker (PB)</a> or <a href="http://amp-protocol.net/">Asynchronous Messaging Protocol (AMP)</a>.</p>\u000a\u000a<p>Another cool thing about Twisted is that on top of these low-level protocol implementations, you'll often find an abstraction that's somewhat easier to use.  For example, when writing an HTTP server, Twisted Web provides <a href="http://twistedmatrix.com/documents/current/api/twisted.web.resource.Resource.html">a "Resource" abstraction</a> which lets you construct URL hierarchies out of Python objects to define how requests will be responded to.</p>\u000a\u000a<p>All of this is tied together with cooperating APIs, mainly due to the fact that none of this functionality is implemented by blocking on the network, so you don't need to start <a href="http://www.kegel.com/c10k.html#threaded">a thread for every operation you want to do</a>.  This contributes to the scalability that people often attribute to Twisted (although it is the kind of scalability that only involves a single computer, not the kind of scalability that lets your application grow to use a whole cluster of hosts) because Twisted can handle thousands of connections in a single thread, which tends to work better than having thousands of threads, each for a single connection.</p>\u000a\u000a<p>Avoiding threading is also beneficial for testing and debugging (and hence reliability in general).  Since there is no pre-emptive context switching in a typical Twisted-based program, you generally don't need to worry about locking.  Race conditions that depend on the order of different network events happening can easily be unit tested by simulating those network events (whereas simulating a context switch isn't a feature provided by most (any?) threading libraries).</p>\u000a\u000a<p>Twisted is also really, really <a href="http://twistedmatrix.com/trac/wiki/ReviewProcess">concerned with quality</a>.  So you'll <a href="http://twistedmatrix.com/trac/wiki/CompatibilityPolicy">rarely find regressions</a> in a Twisted release, and most of the APIs just work, even if you aren't using them in the common way (because we try to test all the ways you might use them, not just the common way).  This is particularly true for all of the code added to Twisted (or modified) in the last 3 or 4 years, since 100% line coverage has been a minimum testing requirement since then.</p>\u000a\u000a<p>Another often overlooked strength of Twisted is its ten years of figuring out different platform <a href="http://itamarst.org/writings/win32sockets.html">quirks</a>.  There are lots of undocumented socket errors on different platforms and it's really hard to learn that they even exist, let alone handle them.  Twisted has gradually covered more and more of these, and it's pretty good about it at this point.  Younger projects don't have this experience, so they miss obscure failure modes that will probably only happen to users of any project you release, not to you.</p>\u000a\u000a<p>All that say, what I find coolest about Twisted is that it's a pretty boring library that lets me ignore a lot of really boring problems and just focus on the interesting and fun things. :)</p>\u000a

V<pre><code>&gt;&gt;&gt; k = [[1, 2], [4], [5, 6, 2], [1, 2], [3], [4]]\u000a&gt;&gt;&gt; import itertools\u000a&gt;&gt;&gt; k.sort()\u000a&gt;&gt;&gt; list(k for k,_ in itertools.groupby(k))\u000a[[1, 2], [3], [4], [5, 6, 2]]\u000a</code></pre>\u000a\u000a<p><a href="http://docs.python.org/library/itertools.html?highlight=itertools#module-itertools" rel="noreferrer"><code>itertools</code></a> often offers the fastest and most powerful solutions to this kind of problems, and is <strong>well</strong> worth getting intimately familiar with!-)</p>\u000a\u000a<p><strong>Edit</strong>: as I mention in a comment, normal optimization efforts are focused on large inputs (the big-O approach) because it's so much easier that it offers good returns on efforts. But sometimes (essentially for "tragically crucial bottlenecks" in deep inner loops of code that's pushing the boundaries of performance limits) one may need to go into much more detail, providing probability distributions, deciding which performance measures to optimize (maybe the upper bound or the 90th centile is more important than an average or median, depending on one's apps), performing possibly-heuristic checks at the start to pick different algorithms depending on input data characteristics, and so forth.</p>\u000a\u000a<p>Careful measurements of "point" performance (code A vs code B for a specific input) are a part of this extremely costly process, and standard library module <code>timeit</code> helps here. However, it's easier to use it at a shell prompt.  For example, here's a short module to showcase the general approach for this problem, save it as <code>nodup.py</code>:</p>\u000a\u000a<pre><code>import itertools\u000a\u000ak = [[1, 2], [4], [5, 6, 2], [1, 2], [3], [4]]\u000a\u000adef doset(k, map=map, list=list, set=set, tuple=tuple):\u000a  return map(list, set(map(tuple, k)))\u000a\u000adef dosort(k, sorted=sorted, xrange=xrange, len=len):\u000a  ks = sorted(k)\u000a  return [ks[i] for i in xrange(len(ks)) if i == 0 or ks[i] != ks[i-1]]\u000a\u000adef dogroupby(k, sorted=sorted, groupby=itertools.groupby, list=list):\u000a  ks = sorted(k)\u000a  return [i for i, _ in itertools.groupby(ks)]\u000a\u000adef donewk(k):\u000a  newk = []\u000a  for i in k:\u000a    if i not in newk:\u000a      newk.append(i)\u000a  return newk\u000a\u000a# sanity check that all functions compute the same result and don't alter k\u000aif __name__ == '__main__':\u000a  savek = list(k)\u000a  for f in doset, dosort, dogroupby, donewk:\u000a    resk = f(k)\u000a    assert k == savek\u000a    print '%10s %s' % (f.__name__, sorted(resk))\u000a</code></pre>\u000a\u000a<p>Note the sanity check (performed when you just do <code>python nodup.py</code>) and the basic hoisting technique (make constant global names local to each function for speed) to put things on equal footing.</p>\u000a\u000a<p>Now we can run checks on the tiny example list:</p>\u000a\u000a<pre><code>$ python -mtimeit -s'import nodup' 'nodup.doset(nodup.k)'\u000a100000 loops, best of 3: 11.7 usec per loop\u000a$ python -mtimeit -s'import nodup' python -mtimeit -s'import nodup' 'nodup.donewk(nodup.k)'\u000a100000 loops, best of 3: 4.44 usec per loop\u000a</code></pre>\u000a\u000a<p>confirming that the quadratic approach has small-enough constants to make it attractive for tiny lists with few duplicated values.  With a short list without duplicates:</p>\u000a\u000a<pre><code>$ python -mtimeit -s'import nodup' 'nodup.donewk([[i] for i in range(12)])'\u000a10000 loops, best of 3: 25.4 usec per loop\u000a$ python -mtimeit -s'import nodup' 'nodup.dogroupby([[i] for i in range(12)])'\u000a10000 loops, best of 3: 23.7 usec per loop\u000a$ python -mtimeit -s'import nodup' 'nodup.doset([[i] for i in range(12)])'\u000a10000 loops, best of 3: 31.3 usec per loop\u000a$ python -mtimeit -s'import nodup' 'nodup.dosort([[i] for i in range(12)])'\u000a10000 loops, best of 3: 25 usec per loop\u000a</code></pre>\u000a\u000a<p>the quadratic approach isn't bad, but the sort and groupby ones are better.  Etc, etc.</p>\u000a\u000a<p>If (as the obsession with performance suggests) this operation is at a core inner loop of your pushing-the-boundaries application, it's worth trying the same set of tests on other representative input samples, possibly detecting some simple measure that could heuristically let you pick one or the other approach (but the measure must be fast, of course).</p>\u000a\u000a<p>It's also well worth considering keeping a different representation for <code>k</code> -- why does it have to be a list of lists rather than a set of tuples in the first place? V<p>It's OK for small applications and allegedly "one-off" scripts, especially with the <code>vars</code> enhancement mentioned by @kaizer.se and the <code>.format</code> version mentioned by @RedGlyph. </p>\u000a\u000a<p>However, for large applications with a long maintenance life and many maintainers this practice can lead to maintenance headaches, and I think that's where @S.Lott's answer is coming from. Let me explain some of the issues involved, as they may not be obvious to anybody who doesn't have the scars from developing and maintaining large applications (or reusable components for such beasts).</p>\u000a\u000a<p>In a "serious" application, you would not have your format string hard-coded -- or, if you had, it would be in some form such as <code>_('Hello {name}.')</code>, where the <code>_</code> comes from <a href="http://docs.python.org/library/gettext.html" rel="noreferrer">gettext</a> or similar i18n / L10n frameworks. The point is that such an application (or reusable modules that can happen to be used in such applications) must support internationalization (AKA i18n) and locatization (AKA L10n): you want your application to be able to emit "Hello Paul" in certain countries and cultures, "Hola Paul" in some others, "Ciao Paul" in others yet, and so forth. So, the format string gets more or less automatically substituted with another at runtime, depending on the current localization settings; instead of being hardcoded, it lives in some sort of database.  For all intents and purposes, imagine that format string always being a variable, not a string literal.</p>\u000a\u000a<p>So, what you have is essentially</p>\u000a\u000a<pre><code>formatstring.format(**locals())\u000a</code></pre>\u000a\u000a<p>and you can't trivially check exactly <em>what</em> local names the formatting is going to be using. You'd have to open and peruse the L10N database, identify the format strings that are going to be used here in different settings, verify all of them.</p>\u000a\u000a<p>So in practice you don't <em>know</em> what local names are going to get used -- which horribly crimps the maintenance of the function. You dare not rename or remove any local variable, as it might horribly break the user experience for users with some (to you) obscure combinaton of language, locale and preferences</p>\u000a\u000a<p>If you have superb integration / regression testing, the breakage will be caught before the beta release -- but QA will scream at you and the release will be delayed... and, let's be honest, while aiming for 100% coverage with <em>unit</em> tests is reasonable, it really isn't with <em>integration</em> tests, once you consider the combinatorial explosion of settings [[for L10N and for many more reasons]] and supported versions of all dependencies. So, you just don't blithely go around risking breakages because "they'll be caught in QA" (if you do, you may not last long in an environment that develops large apps or reusable components;-).</p>\u000a\u000a<p>So, in practice, you'll never remove the "name" local variable even though the User Experience folks have long switched that greeting to a more appropriate "Welcome, Dread Overlord!" (and suitably L10n'ed versions thereof). All because you went for <code>locals()</code>...</p>\u000a\u000a<p>So you're accumulating cruft because of the way you've crimped your ability to maintain and edit your code -- and maybe that "name" local variable only exists because it's been fetched from a DB or the like, so keeping it (or some other local) around is not just cruft, it's reducing your performance too. Is the surface convenience of <code>locals()</code> worth <strong>that</strong>?-)</p>\u000a\u000a<p>But wait, there's worse!  Among the many useful services a <code>lint</code>-like program (like, for example, <a href="http://www.logilab.org/857" rel="noreferrer">pylint</a>) can do for you, is to warn you about unused local variables (wish it could do it for unused globals as well, but, for reusable components, that's just a tad too hard;-).  This way you'll catch most occasional misspellings such as <code>if ...: nmae = ...</code> very rapidly and cheaply, rather than by seeing a unit-test break and doing sleuth work to find out <em>why</em> it broke (you <em>do</em> have obsessive, pervasive unit tests that <strong>would</strong> catch this eventually, right?-) -- lint will tell you about an unused local variable <code>nmae</code> and you will immediately fix it.</p>\u000a\u000a<p>But if you have in your code a <code>blah.format(**locals())</code>, or equivalently a <code>blah % locals()</code>... you're SOL, pal!-)  How is poor lint going to know whether <code>nmae</code> is in fact an unused variable, or actually it does get used by whatever external function or method you're passing <code>locals()</code> to? It can't -- either it's going to warn anyway (causing a "cry wolf" effect that eventually leads you to ignore or disable such warnings), or it's never going to warn (with the same final effect: no warnings;-).</p>\u000a\u000a<p>Compare this to the "explicit is better than implicit" alternative...:</p>\u000a\u000a<pre><code>blah.format(name=name)\u000a</code></pre>\u000a\u000a<p>There -- none of the maintenance, performance, and am-I-hampering-lint worries, applies any more; bliss!  You make it immediately clear to everybody concerned (lint included;-) exactly <em>what</em> local variables are being used, and exactly for what purposes.</p>\u000a\u000a<p>I could go on, but I think this post is already pretty long;-).</p>\u000a\u000a<p>So, summarizing: "<strong>\u03b3\u03bd\u1ff6\u03b8\u03b9 \u03c3\u03b5\u03b1\u03c5\u03c4\u03cc\u03bd</strong>!" Hmm, I mean, "know thyself!". And by "thyself" I actually mean "the purpose and scope of your code". If it's a 1-off-or-thereabouts thingy, never going to be i18n'd and L10n'd, will hardly need future maintenance, will never be reused in a broader context, etc, etc, then go ahead and use <code>locals()</code> for its small but neat convenience; if you know otherwise, or even if you're not entirely certain, err on the side of caution, and make things more explicit -- suffer the small inconvenience of spelling out exactly what you're going, and enjoy all the resulting advantages.</p>\u000a\u000a<p>BTW, this is just one of the examples where Python is striving to support both "small, one-off, exploratory, maybe interactive" programming (by allowing and supporting risky conveniences that extend well beyond <code>locals()</code> -- think of <code>import *</code>, <code>eval</code>, <code>exec</code>, and several other ways you can mush up namespaces and risk maintenance impacts for the sake of convenience), as well as "large, reusable, enterprise-y" apps and components. It can do a pretty good job at both, but only if you "know thyself" and avoid using the "convenience" parts except when you're absolutely certain you can in fact afford them. More often than not, the key consideration is, "what does this do to my namespaces, and awareness of their formation and use by the compiler, lint &amp;c, human readers and maintainers, and so on?". </p>\u000a\u000a<p>Remember, "Namespaces are one honking great idea -- let's do more of those!" is how the Zen of Python concludes... but Python, as a "language for consenting adults", lets <strong>you</strong> define the boundaries of what that implies, as a consequence of your development environment, targets, and practices. Use this power responsibly!-)</p>\u000a

V<p>The answer is to use Welford's algorithm, which is very clearly defined after the "naive methods" in:</p>\u000a\u000a<ul>\u000a<li>Wikipedia: <a href="http://en.wikipedia.org/wiki/Algorithms_for_calculating_variance" rel="noreferrer">Algorithms for calculating variance</a>\u000a</ul>\u000a\u000a<p>It's more numerically stable than either the two-pass or online simple sum of squares collectors suggested in other responses.  The stability only really matters when you have lots of values that are close to each other as they lead to what is known as "<a href="http://www.cs.princeton.edu/introcs/lectures/9scientific.pdf" rel="noreferrer">catastrophic cancellation</a>" in the floating point literature.</p>\u000a\u000a<p>You might also want to brush up on the difference between dividing by the number of samples (N) and N-1 in the variance calculation (squared deviation).  Dividing by N-1 leads to an unbiased estimate of variance from the sample, whereas dividing by N on average underestimates variance (because it doesn't take into account the variance between the sample mean and the true mean).</p>\u000a\u000a<p>I wrote two blog entries on the topic which go into more details, including how to delete previous values online:</p>\u000a\u000a<ul>\u000a<li><a href="http://lingpipe-blog.com/2009/03/19/computing-sample-mean-variance-online-one-pass/" rel="noreferrer">Computing Sample Mean and Variance Online in One Pass</a>\u000a<li><a href="http://lingpipe-blog.com/2009/07/07/welford-s-algorithm-delete-online-mean-variance-deviation/" rel="noreferrer">Deleting Values in Welford\u2019s Algorithm for Online Mean and Variance</a>\u000a</ul>\u000a\u000a<p>You can also take a look at my Java implement; the javadoc, source, and unit tests are all online:</p>\u000a\u000a<ul>\u000a<li><a href="http://alias-i.com/lingpipe/docs/api/com/aliasi/stats/OnlineNormalEstimator.html" rel="noreferrer">Javadoc: <code>stats.OnlineNormalEstimator</code></a>\u000a<li><a href="http://alias-i.com/lingpipe/src/com/aliasi/stats/OnlineNormalEstimator.java" rel="noreferrer">Source: <code>stats.OnlineNormalEstimator.java</code></a>\u000a<li><a href="http://alias-i.com/lingpipe/src/com/aliasi/test/unit/stats/OnlineNormalEstimatorTest.java" rel="noreferrer">JUnit Source: <code>test.unit.stats.OnlineNormalEstimatorTest.java</code></a>\u000a<li><a href="http://alias-i.com/lingpipe" rel="noreferrer">LingPipe Home Page</a>\u000a</ul>\u000a

V<p>I have a class that handles non-interactive plotting, as a frontend to Matplotlib.  However, on occasion one wants to do interactive plotting.  With only a couple functions I found that I was able to increment the figure count, call draw manually, etc, but I needed to do these before and after every plotting call.  So to create both an interactive plotting wrapper and an offscreen plotting wrapper, I found it was more efficient to do this via metaclasses, wrapping the appropriate methods, than to do something like:</p>\u000a\u000a<pre><code>class PlottingInteractive:\u000a    add_slice = wrap_pylab_newplot(add_slice)\u000a</code></pre>\u000a\u000a<p>This method doesn't keep up with API changes and so on, but one that iterates over the class attributes in <code>__init__</code> before re-setting the class attributes is more efficient and keeps things up to date:</p>\u000a\u000a<pre><code>class _Interactify(type):\u000a    def __init__(cls, name, bases, d):\u000a        super(_Interactify, cls).__init__(name, bases, d)\u000a        for base in bases:\u000a            for attrname in dir(base):\u000a                if attrname in d: continue # If overridden, don't reset\u000a                attr = getattr(cls, attrname)\u000a                if type(attr) == types.MethodType:\u000a                    if attrname.startswith("add_"):\u000a                        setattr(cls, attrname, wrap_pylab_newplot(attr))\u000a                    elif attrname.startswith("set_"):\u000a                        setattr(cls, attrname, wrap_pylab_show(attr))\u000a</code></pre>\u000a\u000a<p>Of course, there might be better ways to do this, but I've found this to be effective.  Of course, this could also be done in <code>__new__</code> or <code>__init__</code>, but this was the solution I found the most straightforward.</p>\u000a
