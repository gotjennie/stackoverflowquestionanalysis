I usually don't use mutable default arguments unless needed, but I think with time I forgot about that.  Today in the application I added tocElements=[] in a PDF generation function's argument list and now "Table of Contents" gets longer and longer after each invocation of "generate pdf". :)</p>\u000a\u000a<p>What else should I add to my list of things to MUST avoid?</p>\u000a\u000a<ul>\u000a<li><p>Always import modules the same way, e.g. <code>from y import x</code> and <code>import x</code> are <a href="http://stackoverflow.com/questions/1459236/module-reimported-if-imported-from-different-path">treated as different modules</a>.</p></li>\u000a<li><p>Do not use <em>range</em> in place of lists because <code>range()</code> will become an iterator anyway, the following will fail:</p>\u000a\u000a<pre><code>myIndexList = [0, 1, 3]\u000aisListSorted = myIndexList == range(3)  # will fail in 3.0\u000aisListSorted = myIndexList == list(range(3))  # will not\u000a</code></pre>\u000a\u000a<p>Same thing can be mistakenly done with <em>xrange:</em></p>\u000a\u000a<pre><code>myIndexList == xrange(3)\u000a</code></pre></li>\u000a<li><p>Be careful catching multiple exception types:</p>\u000a\u000a<pre><code>try:\u000a    raise KeyError("hmm bug")\u000aexcept KeyError, TypeError:\u000a    print TypeError\u000a</code></pre>\u000a\u000a<p>This prints "hmm bug", though it is not a bug; it looks like we are catching exceptions of both types, but instead we are catching KeyError only as variable <em>TypeError,</em> use this instead:</p>\u000a\u000a<pre><code>try:\u000a    raise KeyError("hmm bug")\u000aexcept (KeyError, TypeError):\u000a    print TypeError\u000a</code></pre></li>\u000a</ul>\u000a
