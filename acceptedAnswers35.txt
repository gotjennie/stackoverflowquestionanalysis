V<p>The accepted solution has race conditions and it does not work with <code>map</code> and <code>async</code> functions.</p>\u000a\u000a<p>The correct way to handle Ctrl+C/<code>SIGINT</code> with <code>multiprocessing.Pool</code> is to:</p>\u000a\u000a<ol>\u000a<li>Make the process ignore <code>SIGINT</code> before a process <code>Pool</code> is created. This way created child processes inherit <code>SIGINT</code> handler.</li>\u000a<li>Restore the original <code>SIGINT</code> handler in the parent process after a <code>Pool</code> has been created.</li>\u000a<li>Use <code>map_async</code> and <code>apply_async</code> instead of blocking <code>map</code> and <code>apply</code>.</li>\u000a<li>Wait on the results with timeout because the default blocking waits ignore all signals. This is Python bug <a href="https://bugs.python.org/issue8296" rel="noreferrer">https://bugs.python.org/issue8296</a>.</li>\u000a</ol>\u000a\u000a<p>Putting it together:</p>\u000a\u000a<pre><code>#!/bin/env python\u000afrom __future__ import print_function\u000a\u000aimport multiprocessing\u000aimport os\u000aimport signal\u000aimport time\u000a\u000adef run_worker(delay):\u000a    print("In a worker process", os.getpid())\u000a    time.sleep(delay)\u000a\u000adef main():\u000a    print("Initializng 2 workers")\u000a    original_sigint_handler = signal.signal(signal.SIGINT, signal.SIG_IGN)\u000a    pool = multiprocessing.Pool(2)\u000a    signal.signal(signal.SIGINT, original_sigint_handler)\u000a    try:\u000a        print("Starting 2 jobs of 5 seconds each")\u000a        res = pool.map_async(run_worker, [5, 5])\u000a        print("Waiting for results")\u000a        res.get(60) # Without the timeout this blocking call ignores all signals.\u000a    except KeyboardInterrupt:\u000a        print("Caught KeyboardInterrupt, terminating workers")\u000a        pool.terminate()\u000a    else:\u000a        print("Normal termination")\u000a        pool.close()\u000a    pool.join()\u000a\u000aif __name__ == "__main__":\u000a    main()\u000a</code></pre>\u000a\u000a<p>As @YakovShklarov noted, there is a window of time between ignoring the signal and unignoring it in the parent process, during which the signal can be lost. Using <code>pthread_sigmask</code> instead to temporarily block the delivery of the signal in the parent process would prevent the signal from being lost, however, it is not available in Python-2.</p>\u000a

V<p>Martijn's answer explains what <code>None</code> is in Python, and correctly states that the book is misleading.  Since Python programmers as a rule would never say</p>\u000a\u000a<blockquote>\u000a  <p>Assigning a value of <code>None</code> to a variable is one way to reset it to\u000a  its original, empty state.</p>\u000a</blockquote>\u000a\u000a<p>it's hard to explain what Briggs means in a way which makes sense and explains why no one here seems happy with it.  One analogy which may help:</p>\u000a\u000a<p>In Python, variable names are like stickers put on objects.  Every sticker has a unique name written on it, and it can only be on one object at a time, but you could put more than one sticker on the same object, if you wanted to.  When you write</p>\u000a\u000a<pre><code>F = "fork"\u000a</code></pre>\u000a\u000a<p>you put the sticker "F" on a string object <code>"fork"</code>.  If you then write</p>\u000a\u000a<pre><code>F = None\u000a</code></pre>\u000a\u000a<p>you move the sticker to the <code>None</code> object.</p>\u000a\u000a<p>What Briggs is asking you to imagine is that you didn't <strong>write</strong> the sticker <code>"F"</code>, there was <em>already</em> an <code>F</code> sticker on the <code>None</code>, and all you did was <strong>move</strong> it, from <code>None</code> to <code>"fork"</code>.  So when you type <code>F = None</code>, you're "reset[ting] it to its original, empty state", if we decided to treat <code>None</code> as meaning <code>empty state</code>.</p>\u000a\u000a<p>I can see what he's getting at, but that's a bad way to look at it.  If you start Python and type <code>print(F)</code>, you see</p>\u000a\u000a<pre><code>&gt;&gt;&gt; print(F)\u000aTraceback (most recent call last):\u000a  File "&lt;stdin&gt;", line 1, in &lt;module&gt;\u000aNameError: name 'F' is not defined\u000a</code></pre>\u000a\u000a<p>and that <code>NameError</code> means Python doesn't recognize the name <code>F</code>, <strong>because there is no such sticker</strong>.  If Briggs were right and <code>F = None</code> resets <code>F</code> to its original state, then it should be there now, and we should see</p>\u000a\u000a<pre><code>&gt;&gt;&gt; print(F)\u000aNone\u000a</code></pre>\u000a\u000a<p>like we do after we type <code>F = None</code> and put the sticker on <code>None</code>.</p>\u000a\u000a<hr>\u000a\u000a<p>So that's all that's going on.  In reality, Python comes with some stickers already attached to objects (built-in names), but others you have to write yourself with lines like <code>F = "fork"</code> and <code>A = 2</code> and <code>c17 = 3.14</code>, and then you can stick them on other objects later (like <code>F = 10</code> or <code>F = None</code>; it's all the same.)</p>\u000a\u000a<p>Briggs is pretending that all possible stickers you might want to write were already stuck to the <code>None</code> object.</p>\u000a

V<p>The colon is there to declare the start of an indented block.</p>\u000a\u000a<p>Technically, it's not necessary; you could just indent and de-indent when the block is done. However, based on the <a href="http://www.python.org/dev/peps/pep-0020/" rel="noreferrer">Python koan</a> \u201cexplicit is better than implicit\u201d (EIBTI), I believe that Guido deliberately made the colon obligatory, so <em>any</em> statement that <em>should</em> be followed by indented code ends in a colon. (It also allows one-liners if you continue after the colon, but this style is not in wide use.)</p>\u000a\u000a<p>It also makes the work of syntax-aware auto-indenting editors easier, which also counted in the decision.</p>\u000a\u000a<hr>\u000a\u000a<p>This question turns out to be a <a href="http://docs.python.org/faq/design.html#why-are-colons-required-for-the-if-while-def-class-statements" rel="noreferrer">Python FAQ</a>, and I found one of its answers by Guido <a href="http://markmail.org/message/ve7mwqxhci4pm6lw" rel="noreferrer">here</a>:</p>\u000a\u000a<blockquote>\u000a  <p><strong>Why are colons required for the if/while/def/class statements?</strong></p>\u000a  \u000a  <p>The colon is required primarily to enhance readability (one of the results of the experimental ABC language). Consider this:</p>\u000a\u000a<pre><code>if a == b \u000a    print a\u000a</code></pre>\u000a  \u000a  <p>versus </p>\u000a\u000a<pre><code>if a == b: \u000a    print a\u000a</code></pre>\u000a  \u000a  <p>Notice how the second one is slightly easier to read. Notice further how a colon sets off the example in this FAQ answer; it\u2019s a standard usage in English.</p>\u000a  \u000a  <p>Another minor reason is that the colon makes it easier for editors with syntax highlighting; they can look for colons to decide when indentation needs to be increased instead of having to do a more elaborate parsing of the program text.</p>\u000a</blockquote>\u000a

This does not include the <code>!=</code> flag.</p>\u000a  \u000a  <p>The <code>pip install</code> command also supports a <code>--pre</code> flag that will enable installing pre-releases and development releases.</p>\u000a</blockquote>\u000a\u000a<p>The version identifiers for the <code>pytz</code> package have a format like <code>2013b</code>. <a href="http://www.python.org/dev/peps/pep-0426" rel="noreferrer">PEP426</a> uses the version identifiers described in <a href="http://www.python.org/dev/peps/pep-0440" rel="noreferrer">PEP440</a>, which specifies that:</p>\u000a\u000a<blockquote>\u000a  <p>Public version identifiers MUST comply with the following scheme:</p>\u000a\u000a<pre><code>N[.N]+[{a|b|c|rc}N][.postN][.devN]\u000a</code></pre>\u000a</blockquote>\u000a\u000a<p>Because the <code>pytz</code> versions like <code>2013b</code> do not match this format, version 1.4+ of pip is treating <em>all</em> versions of <code>pytz</code> as pre-release versions, and is not installing them by default.</p>\u000a\u000a<p>If you are only installing <code>pytz</code>, you can use the <code>--pre</code> flag to avoid this behaviour, but you wouldn't want to use this flag for installing your entire project's requirements: some packages might have unstable pre-release versions you don't want. V<p>You probably already have lxml installed on your system, perhaps installed due to a system package. Thus, the first attempt (<code>pip install lxml</code> without an active virtualenv) doesn't fail, but it also doesn't install it; it really doesn't do anything.</p>\u000a\u000a<p>In a virtualenv, by default, the system packages are ignored. Therefore, pip thinks that lxml is not installed. Therefore, it tries to install it into your virtual environment.</p>\u000a\u000a<p>lxml contains C modules that need to be compiled in order to install properly. However, the compilation of those C modules rely on your having some "development libraries" already installed as well. V<p>The NumPy function <a href="http://docs.scipy.org/doc/numpy/reference/generated/numpy.std.html"><code>np.std</code></a> takes an optional parameter <code>ddof</code>: "Delta Degrees of Freedom". By default, this is <code>0</code>. Set it to <code>1</code> to get the MATLAB result:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; np.std([1,3,4,6], ddof=1)\u000a2.0816659994661326\u000a</code></pre>\u000a\u000a<p>To add a little more context, in the calculation of the variance (of which the standard deviation is the square root) we typically divide by the number of values we have. </p>\u000a\u000a<p>But if we select a random sample of <code>N</code> elements from a larger distribution and calculate the variance, division by <code>N</code> can lead to an underestimate of the actual variance. To fix this, we can lower the number we divide by (<a href="http://en.wikipedia.org/wiki/Degrees_of_freedom_(statistics)">the degrees of freedom</a>) to a number less than <code>N</code> (usually <code>N-1</code>). The <code>ddof</code> parameter allows us change the divisor by the amount we specify.</p>\u000a\u000a<p>Unless told otherwise, NumPy will calculate the <em>biased</em> estimator for the variance (<code>ddof=0</code>, dividing by <code>N</code>). This is what you want if you are working with the entire distribution (and not a subset of values which have been randomly picked from a larger distribution). If the <code>ddof</code> parameter is given, NumPy divides by <code>N - ddof</code> instead.</p>\u000a\u000a<p>The default behaviour of MATLAB's <code>std</code> is to correct the bias for sample variance by dividing by <code>N-1</code>. This gets rid of some of (but probably not all of) of the bias in the standard deviation. This is likely to be what you want if you're using the function on a random sample of a larger distribution.</p>\u000a\u000a<p>The nice answer by @hbaderts gives further mathematical details.</p>\u000a

V<h3>Method 1: Basic registering decorator</h3>\u000a\u000a<p>I already answered this question here: <a href="http://stackoverflow.com/questions/5707589/calling-functions-by-array-index-in-python/5707605#5707605">Calling functions by array index in Python</a> =)</p>\u000a\u000a<hr>\u000a\u000a<h3>Method 2: Sourcecode parsing</h3>\u000a\u000a<p><em>If you do not have control over the <strong>class</strong> definition</em>, which is one interpretation of what you'd like to suppose, this is <strong>impossible</strong> (without code-reading-reflection), since for example the decorator could be a no-op decorator (like in my linked example) that merely returns the function unmodified. (Nevertheless if you allow yourself to wrap/redefine the decorators, see <strong>Method 3: Converting decorators to be "self-aware"</strong>, then you will find an elegant solution)</p>\u000a\u000a<p>It is a terrible terrible hack, but you could use the <code>inspect</code> module to read the sourcecode itself, and parse it. This will not work in an interactive interpreter, because the inspect module will refuse to give sourcecode in interactive mode. However, below is a proof of concept.</p>\u000a\u000a<pre><code>#!/usr/bin/python3\u000a\u000aimport inspect\u000a\u000adef deco(func):\u000a    return func\u000a\u000adef deco2():\u000a    def wrapper(func):\u000a        pass\u000a    return wrapper\u000a\u000aclass Test(object):\u000a    @deco\u000a    def method(self):\u000a        pass\u000a\u000a    @deco2()\u000a    def method2(self):\u000a        pass\u000a\u000adef methodsWithDecorator(cls, decoratorName):\u000a    sourcelines = inspect.getsourcelines(cls)[0]\u000a    for i,line in enumerate(sourcelines):\u000a        line = line.strip()\u000a        if line.split('(')[0].strip() == '@'+decoratorName: # leaving a bit out\u000a            nextLine = sourcelines[i+1]\u000a            name = nextLine.split('def')[1].split('(')[0].strip()\u000a            yield(name)\u000a</code></pre>\u000a\u000a<p>It works!:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; print(list(  methodsWithDecorator(Test, 'deco')  ))\u000a['method']\u000a</code></pre>\u000a\u000a<p>Note that one has to pay attention to parsing and the python syntax, e.g. <code>@deco</code> and <code>@deco(...</code> are valid results, but <code>@deco2</code> should not be returned if we merely ask for <code>'deco'</code>. We notice that according to the official python syntax at <a href="http://docs.python.org/reference/compound_stmts.html" rel="noreferrer">http://docs.python.org/reference/compound_stmts.html</a> decorators are as follows:</p>\u000a\u000a<pre><code>decorator      ::=  "@" dotted_name ["(" [argument_list [","]] ")"] NEWLINE\u000a</code></pre>\u000a\u000a<p>We breathe a sigh of relief at not having to deal with cases like <code>@(deco)</code>. But note that this still doesn't really help you if you have really really complicated decorators, such as <code>@getDecorator(...)</code>, e.g.</p>\u000a\u000a<pre><code>def getDecorator():\u000a    return deco\u000a</code></pre>\u000a\u000a<p>Thus, this best-that-you-can-do strategy of parsing code cannot detect cases like this. Though if you are using this method, what you're really after is what is written on top of the method in the definition, which in this case is <code>getDecorator</code>.</p>\u000a\u000a<p>According to the spec, it is also valid to have <code>@foo1.bar2.baz3(...)</code> as a decorator. You can extend this method to work with that. You might also be able to extend this method to return a <code>&lt;function object ...&gt;</code> rather than the function's name, with lots of effort. This method however is hackish and terrible.</p>\u000a\u000a<hr>\u000a\u000a<h3>Method 3: Converting decorators to be "self-aware"</h3>\u000a\u000a<p><em>If you do not have control over the <strong>decorator</strong> definition</em> (which is another interpretation of what you'd like), then all these issues go away because you have control over how the decorator is applied. Thus, you can modify the decorator by <em>wrapping</em> it, to create your <em>own</em> decorator, and use <em>that</em> to decorate your functions. Let me say that yet again: you can make a decorator that decorates the decorator you have no control over, "enlightening" it, which in our case makes it do what it was doing before but <em>also</em> append a <code>.decorator</code> metadata property to the callable it returns, allowing you to keep track of "was this function decorated or not? let's check function.decorator!". And <strong>then</strong> you can iterate over the methods of the class, and just check to see if the decorator has the appropriate <code>.decorator</code> property! =) As demonstrated here:</p>\u000a\u000a<pre><code>def makeRegisteringDecorator(foreignDecorator):\u000a    """\u000a        Returns a copy of foreignDecorator, which is identical in every\u000a        way(*), except also appends a .decorator property to the callable it\u000a        spits out.\u000a    """\u000a    def newDecorator(func):\u000a        # Call to newDecorator(method)\u000a        # Exactly like old decorator, but output keeps track of what decorated it\u000a        R = foreignDecorator(func) # apply foreignDecorator, like call to foreignDecorator(method) would have done\u000a        R.decorator = newDecorator # keep track of decorator\u000a        #R.original = func         # might as well keep track of everything!\u000a        return R\u000a\u000a    newDecorator.__name__ = foreignDecorator.__name__\u000a    newDecorator.__doc__ = foreignDecorator.__doc__\u000a    # (*)We can be somewhat "hygienic", but newDecorator still isn't signature-preserving, i.e. you will not be able to get a runtime list of parameters. For that, you need hackish libraries...but in this case, the only argument is func, so it's not a big issue\u000a\u000a    return newDecorator\u000a</code></pre>\u000a\u000a<p>Demonstration for <code>@decorator</code>:</p>\u000a\u000a<pre><code>deco = makeRegisteringDecorator(deco)\u000a\u000aclass Test2(object):\u000a    @deco\u000a    def method(self):\u000a        pass\u000a\u000a    @deco2()\u000a    def method2(self):\u000a        pass\u000a\u000adef methodsWithDecorator(cls, decorator):\u000a    """ \u000a        Returns all methods in CLS with DECORATOR as the\u000a        outermost decorator.\u000a\u000a        DECORATOR must be a "registering decorator"; one\u000a        can make any decorator "registering" via the\u000a        makeRegisteringDecorator function.\u000a    """\u000a    for maybeDecorated in cls.__dict__.values():\u000a        if hasattr(maybeDecorated, 'decorator'):\u000a            if maybeDecorated.decorator == decorator:\u000a                print(maybeDecorated)\u000a                yield maybeDecorated\u000a</code></pre>\u000a\u000a<p>It works!:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; print(list(   methodsWithDecorator(Test2, deco)   ))\u000a[&lt;function method at 0x7d62f8&gt;]\u000a</code></pre>\u000a\u000a<p>However, a "registered decorator" must be the <strong>outermost decorator</strong>, otherwise the <code>.decorator</code> attribute annotation will be lost. For example in a train of</p>\u000a\u000a<pre><code>@decoOutermost\u000a@deco\u000a@decoInnermost\u000adef func(): ...\u000a</code></pre>\u000a\u000a<p>you can only see metadata that <code>decoOutermost</code> exposes, unless we keep references to "more-inner" wrappers.</p>\u000a\u000a<p><em>sidenote: the above method can also build up a <code>.decorator</code> that keeps track of the <strong>entire stack of applied decorators and input functions and decorator-factory arguments</strong>. =) For example if you consider the commented-out line <code>R.original = func</code>, it is feasible to use a method like this to keep track of all wrapper layers. This is personally what I'd do if I wrote a decorator library, because it allows for deep introspection.</em></p>\u000a\u000a<p>There is also a difference between <code>@foo</code> and <code>@bar(...)</code>. While they are both "decorator expressons" as defined in the spec, note that <code>foo</code> is a decorator, while <code>bar(...)</code> returns a dynamically-created decorator, which is then applied. Thus you'd need a separate function <code>makeRegisteringDecoratorFactory</code>, that is somewhat like <code>makeRegisteringDecorator</code> but even MORE META:</p>\u000a\u000a<pre><code>def makeRegisteringDecoratorFactory(foreignDecoratorFactory):\u000a    def newDecoratorFactory(*args, **kw):\u000a        oldGeneratedDecorator = foreignDecoratorFactory(*args, **kw)\u000a        def newGeneratedDecorator(func):\u000a            modifiedFunc = oldGeneratedDecorator(func)\u000a            modifiedFunc.decorator = newDecoratorFactory # keep track of decorator\u000a            return modifiedFunc\u000a        return newGeneratedDecorator\u000a    newDecoratorFactory.__name__ = foreignDecoratorFactory.__name__\u000a    newDecoratorFactory.__doc__ = foreignDecoratorFactory.__doc__\u000a    return newDecoratorFactory\u000a</code></pre>\u000a\u000a<p>Demonstration for <code>@decorator(...)</code>:</p>\u000a\u000a<pre><code>def deco2():\u000a    def simpleDeco(func):\u000a        return func\u000a    return simpleDeco\u000a\u000adeco2 = makeRegisteringDecoratorFactory(deco2)\u000a\u000aprint(deco2.__name__)\u000a# RESULT: 'deco2'\u000a\u000a@deco2()\u000adef f():\u000a    pass\u000a</code></pre>\u000a\u000a<p>This generator-factory wrapper also works:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; print(f.decorator)\u000a&lt;function deco2 at 0x6a6408&gt;\u000a</code></pre>\u000a\u000a<p><strong>bonus</strong> Let's even try the following with Method #3:</p>\u000a\u000a<pre><code>def getDecorator(): # let's do some dispatching!\u000a    return deco\u000a\u000aclass Test3(object):\u000a    @getDecorator()\u000a    def method(self):\u000a        pass\u000a\u000a    @deco2()\u000a    def method2(self):\u000a        pass\u000a</code></pre>\u000a\u000a<p>Result:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; print(list(   methodsWithDecorator(Test3, deco)   ))\u000a[&lt;function method at 0x7d62f8&gt;]\u000a</code></pre>\u000a\u000a<p>As you can see, unlike method2, @deco is correctly recognized even though it was never explicitly written in the class. Unlike method2, this will also work if the method is added at runtime (manually, via a metaclass, etc.) or inherited.</p>\u000a\u000a<p>Be aware that you can also decorate a class, so if you "enlighten" a decorator that is used to both decorate methods and classes, and then write a class <em>within the body of the class you want to analyze</em>, then <code>methodsWithDecorator</code> will return decorated classes as well as decorated methods. One could consider this a feature, but you can easily write logic to ignore those by examining the argument to the decorator, i.e. <code>.original</code>, to achieve the desired semantics.</p>\u000a

V<p>Pickling is a way to convert a python object (list, dict, etc.) into a character stream. The idea is that this character stream contains all the information necessary to reconstruct the object in another python script.</p>\u000a\u000a<p>As for where the pickled information is stored, usually one would do:</p>\u000a\u000a<pre><code>with open('filename', 'wb') as f:\u000a    var = {1 : 'a' , 2 : 'b'}\u000a    pickle.dump(var, f)\u000a</code></pre>\u000a\u000a<p>That would store the pickled version of our <code>var</code> dict in the 'filename' file. Then, in another script, you could load from this file into a variable and the dictionary would be recreated:</p>\u000a\u000a<pre><code>with open('filename','rb') as f:\u000a    var = pickle.load(f)\u000a</code></pre>\u000a\u000a<p>Another use for pickling is if you need to transmit this dictionary over a network (perhaps with sockets or something.) You first need to convert it into a character stream, then you can send it over a socket connection.</p>\u000a\u000a<p>Also, there is no "compression" to speak of here...it's just a way to convert from one representation (in RAM) to another (in "text").</p>\u000a\u000a<p>About.com has a nice introduction of pickling <a href="http://python.about.com/od/pythonstandardlibrary/a/pickle_intro.htm" rel="noreferrer">here</a>.</p>\u000a

V<h2>Quick Check</h2>\u000a\u000a<p>From the signatures, we can tell that they are different:</p>\u000a\u000a<blockquote>\u000a  <p><strong>pow(x, y[, z])</strong></p>\u000a  \u000a  <p><strong>math.pow(x, y)</strong></p>\u000a</blockquote>\u000a\u000a<p>Also, trying it in the shell will give you a quick idea: </p>\u000a\u000a<pre><code>&gt;&gt;&gt; pow is math.pow\u000aFalse\u000a</code></pre>\u000a\u000a<h2>Testing the differences</h2>\u000a\u000a<p>Another way to understand the differences in behaviour between the two functions is to test for them: </p>\u000a\u000a<pre><code>import math\u000aimport traceback\u000aimport sys\u000a\u000ainf = float("inf")\u000aNaN = float("nan")\u000a\u000avals = [inf, NaN, 0.0, 1.0, 2.2, -1.0, -0.0, -2.2, -inf, 1, 0, 2]\u000a\u000atests = set([])\u000a\u000afor vala in vals:\u000a  for valb in vals:\u000a    tests.add( (vala, valb) )\u000a    tests.add( (valb, vala) )\u000a\u000a\u000afor a,b in tests:\u000a  print("math.pow(%f,%f)"%(a,b) )\u000a  try:\u000a    print("    %f "%math.pow(a,b))\u000a  except:\u000a    traceback.print_exc()\u000a\u000a  print("__builtins__.pow(%f,%f)"%(a,b) )\u000a  try:\u000a    print("    %f "%__builtins__.pow(a,b))\u000a  except:\u000a    traceback.print_exc()\u000a</code></pre>\u000a\u000a<p>We can then notice some subtle differences. For example:</p>\u000a\u000a<pre><code>math.pow(0.000000,-2.200000)\u000a    ValueError: math domain error\u000a\u000a__builtins__.pow(0.000000,-2.200000)\u000a    ZeroDivisionError: 0.0 cannot be raised to a negative power\u000a</code></pre>\u000a\u000a<p>There are other differences, and the test list above is not complete (no long numbers, no complex, etc...), but this will give us a pragmatic list of how the two functions behave differently. I would also recommend extending the above test to check for the type that each function returns. You could probably write something similar that creates a report of the differences between the two functions.</p>\u000a\u000a<h2><code>math.pow()</code></h2>\u000a\u000a<p><code>math.pow()</code> handles its arguments very differently from the builtin <code>**</code> or <code>pow()</code>. This comes at the cost of flexibility. Having a look at <a href="http://hg.python.org/cpython/file/c7163a7f7cd2/Modules/mathmodule.c#l1781" rel="noreferrer">the source</a>, we can see that the arguments to <code>math.pow()</code> are <strong>cast directly to doubles</strong>:</p>\u000a\u000a<pre><code>static PyObject *\u000amath_pow(PyObject *self, PyObject *args)\u000a{\u000a    PyObject *ox, *oy;\u000a    double r, x, y;\u000a    int odd_y;\u000a\u000a    if (! PyArg_UnpackTuple(args, "pow", 2, 2, &amp;ox, &amp;oy))\u000a        return NULL;\u000a    x = PyFloat_AsDouble(ox);\u000a    y = PyFloat_AsDouble(oy);\u000a/*...*/\u000a</code></pre>\u000a\u000a<p>The checks are then carried out against the doubles for validity, and then the result is passed to the underlying C math library. </p>\u000a\u000a<h2>builtin <code>pow()</code></h2>\u000a\u000a<p>The built-in <code>pow()</code> (same as the <code>**</code> operator) on the other hand behaves very differently, it actually uses the Objects's own implementation of the <code>**</code> operator, which can be overridden by the end user if need be by replacing a number's <code>__pow__()</code>, <code>__rpow__()</code> or <code>__ipow__()</code>,  method.</p>\u000a\u000a<p>For built-in types, it is instructive to study the difference between the power function implemented for two numeric types, for example, <a href="http://hg.python.org/cpython/file/6a60359556f9/Objects/floatobject.c#l807" rel="noreferrer">floats</a>, <a href="http://hg.python.org/cpython/file/c7163a7f7cd2/Objects/longobject.c#l3599" rel="noreferrer">long</a> and <a href="http://hg.python.org/cpython/file/c7163a7f7cd2/Objects/complexobject.c#l510" rel="noreferrer">complex</a>.</p>\u000a\u000a<h2>Overridding the default behaviour</h2>\u000a\u000a<p>Emulating numeric types is described <a href="http://docs.python.org/reference/datamodel.html#numeric-types" rel="noreferrer">here</a>. essentially, if you are creating a new type for numbers with uncertainty, what you will have to do is provide the <code>__pow__()</code>, <code>__rpow__()</code> and possibly <code>__ipow__()</code> methods for your type. This will allow your numbers to be used with the operator:</p>\u000a\u000a<pre><code>class Uncertain:\u000a  def __init__(self, x, delta=0):\u000a    self.delta = delta\u000a    self.x = x\u000a  def __pow__(self, other):\u000a    return Uncertain(\u000a      self.x**other.x, \u000a      Uncertain._propagate_power(self, other)\u000a    )\u000a  @staticmethod\u000a  def _propagate_power(A, B):\u000a    return math.sqrt(\u000a      ((B.x*(A.x**(B.x-1)))**2)*A.delta*A.delta +\u000a      (((A.x**B.x)*math.log(B.x))**2)*B.delta*B.delta\u000a    )\u000a</code></pre>\u000a\u000a<p>In order to override <code>math.pow()</code> you will have to monkey patch it to support your new type:</p>\u000a\u000a<pre><code>def new_pow(a,b):\u000a    _a = Uncertain(a)\u000a    _b = Uncertain(b)\u000a    return _a ** _b\u000a\u000amath.pow = new_pow\u000a</code></pre>\u000a\u000a<p>Note that for this to work you'll have to wrangle the <code>Uncertain</code> class to cope with an <code>Uncertain</code> instance as an input to <code>__init__()</code></p>\u000a

V<p>A more permanent solution is to subclass <code>float</code>:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; class prettyfloat(float):\u000a    def __repr__(self):\u000a        return "%0.2f" % self\u000a\u000a&gt;&gt;&gt; x\u000a[1.290192, 3.0002, 22.119199999999999, 3.4110999999999998]\u000a&gt;&gt;&gt; x = map(prettyfloat, x)\u000a&gt;&gt;&gt; x\u000a[1.29, 3.00, 22.12, 3.41]\u000a&gt;&gt;&gt; y = x[2]\u000a&gt;&gt;&gt; y\u000a22.12\u000a</code></pre>\u000a\u000a<p>The problem with subclassing <code>float</code> is that it breaks code that's explicitly looking for a variable's type. 'str':\u000a    '''An example function that fails type checking.'''\u000a    return x\u000a\u000a\u000a# For simplicity, I only do keyword args.\u000adef check_type(*args):\u000a    param, value, assert_type = args\u000a    log('Checking {0} = {1} of {2}.'.format(*args))\u000a    if not isinstance(value, assert_type):\u000a        raise AssertionError(\u000a            'Check failed - parameter {0} = {1} not {2}.'\u000a            .format(*args))\u000a    return value\u000a\u000adef decorate_func(func):    \u000a    def newf(*args, **kwargs):\u000a        for k, v in kwargs.items():\u000a            check_type(k, v, ann[k])\u000a        return check_type('&lt;return_value&gt;', func(*args, **kwargs), ann['return'])\u000a\u000a    ann = {k: eval(v) for k, v in func.__annotations__.items()}\u000a    newf.__doc__ = func.__doc__\u000a    newf.__type_checked = True\u000a    return newf\u000a\u000adef decorate_module(module = '__main__'):\u000a    '''Enforces type from annotation for all functions in module.'''\u000a    d = sys.modules[module].__dict__\u000a    for k, f in d.items():\u000a        if getattr(f, '__annotations__', {}) and not getattr(f, '__type_checked', False):\u000a            log('Decorated {0!r}.'.format(f.__name__))\u000a            d[k] = decorate_func(f)\u000a\u000a\u000aif __name__ == '__main__':\u000a    decorate_module()\u000a\u000a    # This will raise AssertionError.\u000a    func(x = 5)\u000a</code></pre>\u000a\u000a<p>Given this simplicity, it's strange at the first sight that this thing is not mainstream. However, I believe there are good reasons why it's <strong>not as useful as it might seem</strong>. Generally, type checking helps because if you add integer and dictionary, chances are you made some obvious mistake (and if you meant something reasonable, it's still <em>better to be explicit than implicit</em>).</p>\u000a\u000a<p>But in real life you often mix quantities of the same <strong>computer type</strong> as seen by compiler but clearly different <strong>human type</strong>, for example the following snippet contains an obvious mistake:</p>\u000a\u000a<pre><code>height = 1.75 # Bob's height in meters.\u000alength = len(sys.modules) # Number of modules imported by program.\u000aarea = height * length # What's that supposed to mean???\u000a</code></pre>\u000a\u000a<p>Any human should immediately see a mistake in the above line provided it knows the 'human type' of variables <code>height</code> and <code>length</code> even though it looks to computer as <strong>perfectly legal</strong> multiplication of <code>int</code> and <code>float</code>.</p>\u000a\u000a<p>There's more that can be said about possible solutions to this problem, but enforcing 'computer types' is apparently a half-solution, so, at least in my opinion, it's <em>worse than no solution at all</em>. It's the same reason why <strong>Systems Hungarian</strong> is a terrible idea while <strong>Apps Hungarian</strong> is a great one. There's more at the very informative <strong><a href="http://www.joelonsoftware.com/articles/Wrong.html" rel="noreferrer">post of Joel Spolsky</a></strong>.</p>\u000a\u000a<p>Now if somebody was to implement some kind of Pythonic third-party library that would automatically assign to real-world data its <strong>human type</strong> and then took care to transform that type like <code>width * height -&gt; area</code> and enforce that check with function annotations, I think that would be a type checking people could really use!</p>\u000a

V<p>You will still need locks if you share state between threads. The GIL only protects the interpreter internally. You can still have inconsistent updates in your own code.</p>\u000a\u000a<p>For example:</p>\u000a\u000a<pre><code>#!/usr/bin/env python\u000aimport threading\u000a\u000ashared_balance = 0\u000a\u000aclass Deposit(threading.Thread):\u000a    def run(self):\u000a        for _ in xrange(1000000):\u000a            global shared_balance\u000a            balance = shared_balance\u000a            balance += 100\u000a            shared_balance = balance\u000a\u000aclass Withdraw(threading.Thread):\u000a    def run(self):\u000a        for _ in xrange(1000000):\u000a            global shared_balance\u000a            balance = shared_balance\u000a            balance -= 100\u000a            shared_balance = balance\u000a\u000athreads = [Deposit(), Withdraw()]\u000a\u000afor thread in threads:\u000a    thread.start()\u000a\u000afor thread in threads:\u000a    thread.join()\u000a\u000aprint shared_balance\u000a</code></pre>\u000a\u000a<p>Here, your code can be interrupted between reading the shared state (<code>balance = shared_balance</code>) and writing the changed result back (<code>shared_balance = balance</code>), causing a lost update. V<p>This is <a href="http://www.stack.nl/~dimitri/doxygen/docblocks.html#pythonblocks" rel="noreferrer">documented on the doxygen website</a>, but to summarize here:</p>\u000a\u000a<p>You can use doxygen to document your Python code. You can either use the Python documentation string syntax:</p>\u000a\u000a<pre><code>"""@package docstring\u000aDocumentation for this module.\u000a\u000aMore details.\u000a"""\u000a\u000adef func():\u000a    """Documentation for a function.\u000a\u000a    More details.\u000a    """\u000a    pass\u000a</code></pre>\u000a\u000a<p>In which case the comments will be extracted by doxygen, but you won't be able to use any of the <a href="http://www.stack.nl/~dimitri/doxygen/commands.html#cmd_intro" rel="noreferrer">special doxygen commands</a>.</p>\u000a\u000a<p><strong>Or</strong> you can (similar to C-style languages under doxygen) double up the comment marker (<code>#</code>) on the first line before the member:</p>\u000a\u000a<pre><code>## @package pyexample\u000a#  Documentation for this module.\u000a#\u000a#  More details.\u000a\u000a## Documentation for a function.\u000a#\u000a#  More details.\u000adef func():\u000a    pass\u000a</code></pre>\u000a\u000a<p>In that case, you can use the special doxygen commands. There's no particular Python output mode, but you can apparently improve the results by setting <code>OPTMIZE_OUTPUT_JAVA</code> to <code>YES</code>.</p>\u000a\u000a<p>Honestly, I'm a little surprised at the difference - it seems like once doxygen can detect the comments in ## blocks or """ blocks, most of the work would be done and you'd be able to use the special commands in either case. Maybe they expect people using """ to adhere to more Pythonic documentation practices and that would interfere with the special doxygen commands?</p>\u000a
