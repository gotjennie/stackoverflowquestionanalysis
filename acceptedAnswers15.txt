V<blockquote>\u000a  <p>How does the Python <code>3.6</code> dictionary implementation perform better* than the older one while preserving element order?</p>\u000a</blockquote>\u000a\u000a<p>Essentially, by <em>keeping two arrays</em>. </p>\u000a\u000a<ul>\u000a<li><p>The first array, <a href="https://github.com/python/cpython/blob/474ef63e38726d4bcde14f6104984a742c6cb747/Objects/dictobject.c#L551" rel="noreferrer"><code>dk_entries</code></a>, holds the entries (<a href="https://github.com/python/cpython/blob/c30098c8c6014f3340a369a31df9c74bdbacc269/Objects/dict-common.h#L4" rel="noreferrer">of type <code>PyDictKeyEntry</code></a>) for the dictionary in the order that they were inserted. Preserving order is achieved by this being an append only array where new items are always inserted at the end (insertion order).</p></li>\u000a<li><p>The second, <a href="https://github.com/python/cpython/blob/c30098c8c6014f3340a369a31df9c74bdbacc269/Objects/dict-common.h#L70" rel="noreferrer"><code>dk_indices</code></a>, holds the indices for the <code>dk_entries</code> array (that is, values that indicate the position of the corresponding entry in <code>dk_entries</code>). This array acts as the hash table. When a key is hashed it leads to one of the indices stored in <code>dk_indices</code> and the corresponding entry is fetched by indexing <code>dk_entries</code>. Since only indices are kept, the type of this array depends on the overall size of the dictionary (ranging from type <a href="https://github.com/python/cpython/blob/c30098c8c6014f3340a369a31df9c74bdbacc269/Objects/dict-common.h#L64" rel="noreferrer"><code>int8_t</code></a>(<code>1</code> byte) to <a href="https://github.com/python/cpython/blob/c30098c8c6014f3340a369a31df9c74bdbacc269/Objects/dict-common.h#L66" rel="noreferrer"><code>int32_t</code></a>/<a href="https://github.com/python/cpython/blob/c30098c8c6014f3340a369a31df9c74bdbacc269/Objects/dict-common.h#L68" rel="noreferrer"><code>int64_t</code></a> (<code>4</code>/<code>8</code> bytes) on <code>32</code>/<code>64</code> bit builds) </p></li>\u000a</ul>\u000a\u000a<p>In the previous implementation, a sparse array of type <code>PyDictKeyEntry</code> and size <code>dk_size</code> had to be allocated; unfortunately, it also resulted in a lot of empty space since that array was not allowed to be more than <code>2/3 * dk_size</code> full <a href="https://github.com/python/cpython/blob/474ef63e38726d4bcde14f6104984a742c6cb747/Objects/dictobject.c#L375" rel="noreferrer">for performance reasons</a>. (and the empty space <em>still</em> had <code>PyDictKeyEntry</code> size!). </p>\u000a\u000a<p>This is not the case now since only the <em>required</em> entries are stored (those that have been inserted) and a sparse array of type <code>intX_t</code> (<code>X</code> depending on dict size) <code>2/3 * dk_size</code>s full is kept. The empty space changed from type <code>PyDictKeyEntry</code> to <code>intX_t</code>. </p>\u000a\u000a<p>So, obviously, creating a sparse array of type <code>PyDictKeyEntry</code> is much more memory demanding than a sparse array for storing <code>int</code>s.</p>\u000a\u000a<hr>\u000a\u000a<p><a href="https://mail.python.org/pipermail/python-dev/2012-December/123028.html" rel="noreferrer">In the original proposal made by Raymond Hettinger</a>, a visualization of the data structures used can be seen which captures the gist of the idea.</p>\u000a\u000a<blockquote>\u000a  <p>For example, the dictionary:</p>\u000a\u000a<pre><code>d = {'timmy': 'red', 'barry': 'green', 'guido': 'blue'}\u000a</code></pre>\u000a  \u000a  <p>is currently stored as:</p>\u000a\u000a<pre><code>entries = [['--', '--', '--'],\u000a           [-8522787127447073495, 'barry', 'green'],\u000a           ['--', '--', '--'],\u000a           ['--', '--', '--'],\u000a           ['--', '--', '--'],\u000a           [-9092791511155847987, 'timmy', 'red'],\u000a           ['--', '--', '--'],\u000a           [-6480567542315338377, 'guido', 'blue']]\u000a</code></pre>\u000a  \u000a  <p>Instead, the data should be organized as follows:</p>\u000a\u000a<pre><code>indices =  [None, 1, None, None, None, 0, None, 2]\u000aentries =  [[-9092791511155847987, 'timmy', 'red'],\u000a            [-8522787127447073495, 'barry', 'green'],\u000a            [-6480567542315338377, 'guido', 'blue']]\u000a</code></pre>\u000a</blockquote>\u000a\u000a<p>As you can visually now see, in the original proposal, a lot of space is essentially empty to reduce collisions and make look-ups faster. With the new approach, you reduce the memory required by moving the sparseness where it's really required, in the indices.</p>\u000a\u000a<hr>\u000a\u000a<p>As a small example, here's some differences in byte size as the dictionary grows. The sizes are from <code>3.5.2</code> and the latest default branch of CPython from <code>hg</code> (<code>Python 3.7.0a0</code>):</p>\u000a\u000a<p>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;<a href="https://i.stack.imgur.com/mFRgv.png" rel="noreferrer"><img src="https://i.stack.imgur.com/mFRgv.png" alt="enter image description here"></a></p>\u000a\u000a<p>In both cases the sizes where attained by using <code>getsizeof</code>. The difference in size seems to normalize around <code>25%</code>; for sizes up to <code>5465</code> elements, the values were:</p>\u000a\u000a<pre><code>Elements       Size in Bytes\u000a--------       -------------\u000a  0         0.11111111111111116\u000a  10        0.19999999999999996\u000a  15        0.2314814814814815\u000a  25        0.2647058823529411\u000a  45        0.2752525252525253\u000a  90        0.2435897435897436\u000a  175       0.24612403100775193\u000a  345       0.248378728923476\u000a  685       0.24902534113060426\u000a  1370      0.2495934959349594\u000a  2735      0.2497559785261103\u000a  5465      0.24989829956879017\u000a</code></pre>\u000a\u000a<p><sub>\u000a*The new dictionary implementations performs better <strong>memory wise</strong> by being designed more compactly; that's the main benefit here. Speed wise, the difference isn't so drastic, there's places where the new dict might introduce slight regressions (<a href="http://bugs.python.org/msg275587" rel="noreferrer">key-lookups, for example</a>) while in others (iteration and resizing come to mind) a performance boost should be present. \u000a</sub></p>\u000a\u000a<p><sub>\u000aOverall, the performance of the dictionary, especially in real-life situations, improves due to the compactness introduced.<br>\u000a</sub></p>\u000a\u000a<hr>\u000a\u000a<blockquote>\u000a  <p>Should you depend on it and/or use it?</p>\u000a</blockquote>\u000a\u000a<p><strong>NO!</strong> Anything characterized as an <em>implementation detail</em> (as noted in the documentation) should be something you <em>shouldn't</em> depend on. If you do it is going to be your fault if code you've written breaks in previous versions/different implementation and/or stops working in future releases.</p>\u000a\u000a<p>Different implementations of Python aren't required to make the dictionary ordered, rather, just support an ordered mapping where that is required (Notable examples are <em><a href="https://docs.python.org/3.6/whatsnew/3.6.html#pep-520-preserving-class-attribute-definition-order" rel="noreferrer">PEP 520: Preserving Class Attribute Definition Order</a></em> and <em><a href="https://docs.python.org/3.6/whatsnew/3.6.html#pep-468-preserving-keyword-argument-order" rel="noreferrer">PEP 468: Preserving Keyword Argument Order</a></em>)</p>\u000a\u000a<p>If you want to write code that preserves the ordering and want it to not break on previous versions/different implementations you should always use <code>OrderedDict</code>. Besides, <code>OrderedDict</code> will most likely eventually become a thin-wrapper around the new <code>dict</code> implementation.</p>\u000a

V<h1>Some basics first.</h1>\u000a\u000a<p>With objects, you need to deal with its attributes. Ordinarily we do <code>instance.attribute</code>.  Sometimes we need more control (when we do not know the name of the attribute in advance).</p>\u000a\u000a<p>For example, <code>instance.attribute</code> would become <code>getattr(instance, attribute_name)</code>.  Using this model, we can get the attribute by supplying the <em>attribute_name</em> as a string.</p>\u000a\u000a<h1>Use of <code>__getattr__</code></h1>\u000a\u000a<p>You can also tell a class how to deal with attributes which it doesn't explicitly manage and do that via <code>__getattr__</code> method.</p>\u000a\u000a<p>Python will call this method whenever you request an attribute that hasn't already been defined, so you can define what to do with it. </p>\u000a\u000a<p>A classic use case:</p>\u000a\u000a<pre><code>class A(dict):\u000a    def __getattr__(self, name):\u000a       return self[name]\u000aa = A()\u000a# Now a.somekey will give a['somekey']\u000a</code></pre>\u000a\u000a<h1>Caveats and use of <code>__getattribute__</code></h1>\u000a\u000a<p>If you need to catch every attribute <em>regardless whether it exists or not</em>, use <code>__getattribute__</code> instead.  The difference is that <code>__getattr__</code> only gets called for attributes that don't actually exist.  If you set an attribute directly, referencing that attribute will retrieve it without calling <code>__getattr__</code>. </p>\u000a\u000a<p><code>__getattribute__</code> is called all the times.</p>\u000a

V<p>Have a look at the Requests <a href="http://docs.python-requests.org/en/latest/user/quickstart/#errors-and-exceptions" rel="noreferrer">exception docs</a>.  In short:</p>\u000a\u000a<blockquote>\u000a  <p>In the event of a network problem (e.g. DNS failure, refused connection, etc), Requests will raise a <strong><code>ConnectionError</code></strong> exception.</p>\u000a  \u000a  <p>In the event of the rare invalid HTTP response, Requests will raise an <strong><code>HTTPError</code></strong> exception.</p>\u000a  \u000a  <p>If a request times out, a <strong><code>Timeout</code></strong> exception is raised.</p>\u000a  \u000a  <p>If a request exceeds the configured number of maximum redirections, a <strong><code>TooManyRedirects</code></strong> exception is raised.</p>\u000a  \u000a  <p>All exceptions that Requests explicitly raises inherit from <strong><code>requests.exceptions.RequestException</code></strong>.</p>\u000a</blockquote>\u000a\u000a<p>To answer your question, what you show will <em>not</em> cover all of your bases. You'll only catch connection-related errors, not ones that time out.</p>\u000a\u000a<p>What to do when you catch the exception is really up to the design of your script/program. Is it acceptable to exit? Can you go on and try again?  If the error is catastrophic and you can't go on, then yes, a call to <code>sys.exit()</code> is in order.</p>\u000a\u000a<p>You can either catch the base-class exception, which will handle all cases:</p>\u000a\u000a<pre><code>try:\u000a    r = requests.get(url, params={'s': thing})\u000aexcept requests.exceptions.RequestException as e:  # This is the correct syntax\u000a    print e\u000a    sys.exit(1)\u000a</code></pre>\u000a\u000a<p>Or you can catch them separately and do different things.</p>\u000a\u000a<pre><code>try:\u000a    r = requests.get(url, params={'s': thing})\u000aexcept requests.exceptions.Timeout:\u000a    # Maybe set up for a retry, or continue in a retry loop\u000aexcept requests.exceptions.TooManyRedirects:\u000a    # Tell the user their URL was bad and try a different one\u000aexcept requests.exceptions.RequestException as e:\u000a    # catastrophic error. bail.\u000a    print e\u000a    sys.exit(1)\u000a</code></pre>\u000a\u000a<hr>\u000a\u000a<p>As <a href="http://stackoverflow.com/users/456550/christian-long">Christian</a> pointed out:</p>\u000a\u000a<blockquote>\u000a  <p>If you want http errors (e.g. 401 Unauthorized) to raise exceptions, you can call <a href="http://docs.python-requests.org/en/latest/api/#requests.Response.raise_for_status" rel="noreferrer"><code>Response.raise_for_status</code></a>. That will raise an <code>HTTPError</code>, if the response was an http error.</p>\u000a</blockquote>\u000a\u000a<p>An example:</p>\u000a\u000a<pre><code>try:\u000a    r = requests.get('http://www.google.com/nothere')\u000a    r.raise_for_status()\u000aexcept requests.exceptions.HTTPError as err:\u000a    print err\u000a    sys.exit(1)\u000a</code></pre>\u000a\u000a<p>Will print:</p>\u000a\u000a<pre class="lang-none prettyprint-override"><code>404 Client Error: Not Found for url: http://www.google.com/nothere\u000a</code></pre>\u000a

V<p>Per <a href="https://docs.python.org/3/reference/lexical_analysis.html#integer-literals" rel="noreferrer">https://docs.python.org/3/reference/lexical_analysis.html#integer-literals</a>:</p>\u000a\u000a<blockquote>\u000a  <p>Integer literals are described by the following lexical definitions:</p>\u000a\u000a<pre><code>integer        ::=  decimalinteger | octinteger | hexinteger | bininteger\u000adecimalinteger ::=  nonzerodigit digit* | "0"+\u000anonzerodigit   ::=  "1"..."9"\u000adigit          ::=  "0"..."9"\u000aoctinteger     ::=  "0" ("o" | "O") octdigit+\u000ahexinteger     ::=  "0" ("x" | "X") hexdigit+\u000abininteger     ::=  "0" ("b" | "B") bindigit+\u000aoctdigit       ::=  "0"..."7"\u000ahexdigit       ::=  digit | "a"..."f" | "A"..."F"\u000abindigit       ::=  "0" | "1"\u000a</code></pre>\u000a  \u000a  <p>There is no limit for the length of integer literals apart from what\u000a  can be stored in available memory.</p>\u000a  \u000a  <p>Note that leading zeros in a non-zero decimal number are not allowed.\u000a  This is for disambiguation with C-style octal literals, which Python\u000a  used before version 3.0.</p>\u000a</blockquote>\u000a\u000a<p>As noted here, leading zeros in a <em>non-zero</em> decimal number are not allowed. <code>"0"+</code> is legal as a very special case, which <a href="https://docs.python.org/2/reference/lexical_analysis.html#integer-and-long-integer-literals" rel="noreferrer">wasn't present in Python 2</a>:</p>\u000a\u000a<pre><code>integer        ::=  decimalinteger | octinteger | hexinteger | bininteger\u000adecimalinteger ::=  nonzerodigit digit* | "0"\u000aoctinteger     ::=  "0" ("o" | "O") octdigit+ | "0" octdigit+\u000a</code></pre>\u000a\u000a<hr>\u000a\u000a<p><a href="https://mail.python.org/pipermail//python-3000-checkins/2007-June/000874.html" rel="noreferrer">SVN commit r55866</a> implemented PEP 3127 in the tokenizer, which forbids the old <code>0&lt;octal&gt;</code> numbers. However, curiously, it also adds this note:</p>\u000a\u000a<pre><code>/* in any case, allow '0' as a literal */\u000a</code></pre>\u000a\u000a<p>with a special <code>nonzero</code> flag that only throws a <code>SyntaxError</code> if the following sequence of digits contains a nonzero digit.</p>\u000a\u000a<p>This is odd because <a href="https://www.python.org/dev/peps/pep-3127/" rel="noreferrer">PEP 3127</a> does not allow this case:</p>\u000a\u000a<blockquote>\u000a  <p>This PEP proposes that the ability to specify an octal number by using a leading zero will be removed from the language in Python 3.0 (and the Python 3.0 preview mode of 2.6), and that <strong>a SyntaxError will be raised whenever a leading "0" is immediately followed by another digit</strong>. </p>\u000a</blockquote>\u000a\u000a<p>(emphasis mine)</p>\u000a\u000a<p>So, the fact that multiple zeros are allowed is <em>technically</em> violating the PEP, and was basically implemented as a special case by Georg Brandl. He made the corresponding documentation change to note that <code>"0"+</code> was a valid case for <code>decimalinteger</code> (previously that had been covered under <code>octinteger</code>).</p>\u000a\u000a<p>We'll probably never know <em>exactly</em> why Georg chose to make <code>"0"+</code> valid - it may forever remain an odd corner case in Python.</p>\u000a\u000a<hr>\u000a\u000a<p><strong>UPDATE</strong> [28 Jul 2015]: This question led to a <a href="https://mail.python.org/pipermail/python-ideas/2015-July/034631.html" rel="noreferrer">lively discussion thread</a> on python-ideas in which <a href="https://mail.python.org/pipermail/python-ideas/2015-July/034650.html" rel="noreferrer">Georg chimed in</a>:</p>\u000a\u000a<blockquote>\u000a  <p>Steven D'Aprano wrote: </p>\u000a  \u000a  <blockquote>\u000a    <p>Why was it defined that way? [...] Why would we write 0000 to get zero?</p>\u000a  </blockquote>\u000a  \u000a  <p>I could tell you, but then I'd have to kill you.</p>\u000a  \u000a  <p>Georg</p>\u000a</blockquote>\u000a\u000a<p>Later on, the thread spawned <a href="https://bugs.python.org/issue24668" rel="noreferrer">this bug report</a> aiming to get rid of this special case. Here, <a href="https://bugs.python.org/msg246945" rel="noreferrer">Georg says</a>:</p>\u000a\u000a<blockquote>\u000a  <p>I don't recall the reason for this deliberate change (as seen from the docs change). </p>\u000a  \u000a  <p>I'm unable to come up with a good reason for this change now [...]</p>\u000a</blockquote>\u000a\u000a<p>and thus we have it: the precise reason behind this inconsistency is lost to time.</p>\u000a\u000a<p>Finally, note that the bug report was rejected: leading zeros will continue to be accepted only on zero integers for the rest of Python 3.x.</p>\u000a

V<p>You are not comparing for equality. You are <strong>assigning</strong>.</p>\u000a\u000a<p>Python allows you to assign to multiple targets:</p>\u000a\u000a<pre><code>foo, bar = 1, 2\u000a</code></pre>\u000a\u000a<p>assigns the two values to <code>foo</code> and <code>bar</code>, respectively. All you need is a <em>sequence</em> or <em>iterable</em> on the right-hand side, and a list or tuple of names on the left.</p>\u000a\u000a<p>When you do:</p>\u000a\u000a<pre><code>[] = ""\u000a</code></pre>\u000a\u000a<p>you assigned an <em>empty</em> sequence (empty strings are sequences still) to an empty list of names.</p>\u000a\u000a<p>It is essentially the same thing as doing:</p>\u000a\u000a<pre><code>[foo, bar, baz] = "abc"\u000a</code></pre>\u000a\u000a<p>where you end up with <code>foo = "a"</code>, <code>bar = "b"</code> and <code>baz = "c"</code>, but with fewer characters.</p>\u000a\u000a<p>You cannot, however, assign to a string, so <code>""</code> on the left-hand side of an assignment never works and is always a syntax error.</p>\u000a\u000a<p>See the <a href="https://docs.python.org/3/reference/simple_stmts.html#assignment-statements"><em>Assignment statements</em> documentation</a>:</p>\u000a\u000a<blockquote>\u000a  <p>An assignment statement evaluates the expression list (remember that this can be a single expression or a comma-separated list, the latter yielding a tuple) and assigns the single resulting object to each of the target lists, from left to right.</p>\u000a</blockquote>\u000a\u000a<p>and</p>\u000a\u000a<blockquote>\u000a  <p>Assignment of an object to a target list, <strong>optionally enclosed in parentheses or square brackets</strong>, is recursively defined as follows.</p>\u000a</blockquote>\u000a\u000a<p><em>Emphasis mine</em>.</p>\u000a\u000a<p>That Python doesn't throw a syntax error for the empty list is actually a bit of a bug! V<p>In short, it means there are no restrictions on the object's use. It's the same as\u000aany other object.</p>\u000a\u000a<p>A first class object is an entity that can be dynamically created, destroyed, passed to a function, returned as a value, and have all the rights as other variables in the programming language have.  </p>\u000a\u000a<blockquote>\u000a  <p>Depending on the language, this can\u000a  imply:</p>\u000a  \u000a  <ul>\u000a  <li>being expressible as an anonymous literal value</li>\u000a  <li>being storable in variables</li>\u000a  <li>being storable in data structures</li>\u000a  <li>having an intrinsic identity (independent of any given name)</li>\u000a  <li>being comparable for equality with other entities</li>\u000a  <li>being passable as a parameter to a procedure/function</li>\u000a  <li>being returnable as the result of a procedure/function</li>\u000a  <li>being constructible at runtime</li>\u000a  <li>being printable</li>\u000a  <li>being readable</li>\u000a  <li>being transmissible among distributed processes</li>\u000a  <li>being storable outside running processes</li>\u000a  </ul>\u000a</blockquote>\u000a\u000a<p><a href="http://en.wikipedia.org/wiki/First-class_object" rel="noreferrer">Source</a>.</p>\u000a\u000a<p>In C++ functions themselves are not first class objects, however:</p>\u000a\u000a<ul>\u000a<li>You can override the '()' operator making it possible to have an object function, which is first class.</li>\u000a<li>Function pointers are first class. </li>\u000a<li>boost bind, lambda and function do offer first class functions</li>\u000a</ul>\u000a\u000a<p>In C++, classes are not first class objects but instances of those classes are. V<blockquote>\u000a  <p>Question:  I am using split('\u005cn') to get lines in one string, and found that\u000a  ''.split() returns empty list [], while ''.split('\u005cn') returns [''].</p>\u000a</blockquote>\u000a\u000a<p>The <a href="http://docs.python.org/2.7/library/stdtypes.html#str.split" rel="noreferrer"><em>str.split()</em></a> method has two algorithms.  If no arguments are given, it splits on repeated runs of whitespace.  However, if an argument is given, it is treated as a single delimiter with no repeated runs.</p>\u000a\u000a<p>In the case of splitting an empty string, the first mode (no argument) will return an empty list because the whitespace is eaten and there are no values to put in the result list.  </p>\u000a\u000a<p>In contrast, the second mode (with an argument such as <code>\u005cn</code>) will produce the first empty field.  Consider if you had written <code>'\u005cn'.split('\u005cn')</code>, you would get two fields (one split, gives you two halves).</p>\u000a\u000a<blockquote>\u000a  <p>Question:  Is there any specific reason for such a difference?</p>\u000a</blockquote>\u000a\u000a<p>This first mode is useful when data is aligned in columns with variable amounts of whitespace.  For example:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; data = '''\u005c\u000aShasta      California     14,200\u000aMcKinley    Alaska         20,300\u000aFuji        Japan          12,400\u000a'''\u000a&gt;&gt;&gt; for line in data.splitlines():\u000a        print line.split()\u000a\u000a['Shasta', 'California', '14,200']\u000a['McKinley', 'Alaska', '20,300']\u000a['Fuji', 'Japan', '12,400']\u000a</code></pre>\u000a\u000a<p>The second mode is useful for delimited data such as CSV where repeated commas denote empty fields.  For example:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; data = '''\u005c\u000aGuido,BDFL,,Amsterdam\u000aBarry,FLUFL,,USA\u000aTim,,,USA\u000a'''\u000a&gt;&gt;&gt; for line in data.splitlines():\u000a        print line.split(',')\u000a\u000a['Guido', 'BDFL', '', 'Amsterdam']\u000a['Barry', 'FLUFL', '', 'USA']\u000a['Tim', '', '', 'USA']\u000a</code></pre>\u000a\u000a<p>Note, the number of result fields is one greater than the number of delimiters. Think of cutting a rope.  If you make no cuts, you have one piece.  Making one cut, gives two pieces.  Making two cuts, gives three pieces.\u000aAnd so it is with Python's <em>str.split(delimiter)</em> method:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; ''.split(',')       # No cuts\u000a['']\u000a&gt;&gt;&gt; ','.split(',')      # One cut\u000a['', '']\u000a&gt;&gt;&gt; ',,'.split(',')     # Two cuts\u000a['', '', '']\u000a</code></pre>\u000a\u000a<blockquote>\u000a  <p>Question:  And is there any more convenient way to count lines in a string?</p>\u000a</blockquote>\u000a\u000a<p>Yes, there are a couple of easy ways.  One uses <a href="http://docs.python.org/2.7/library/stdtypes.html#str.count" rel="noreferrer"><em>str.count()</em></a> and the other uses <a href="http://docs.python.org/2.7/library/stdtypes.html#str.splitlines" rel="noreferrer"><em>str.splitlines()</em></a>.\u000aBoth ways will give the same answer unless the final line is missing the <code>\u005cn</code>.\u000aIf the final newline is missing, the <em>str.splitlines</em> approach will give the accurate answer.\u000aA faster technique that is also accurate uses the count method but then corrects it for the final newline:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; data = '''\u005c\u000aLine 1\u000aLine 2\u000aLine 3\u000aLine 4'''\u000a\u000a&gt;&gt;&gt; data.count('\u005cn')                               # Inaccurate\u000a3\u000a&gt;&gt;&gt; len(data.splitlines())                         # Accurate, but slow\u000a4\u000a&gt;&gt;&gt; data.count('\u005cn') + (not data.endswith('\u005cn'))   # Accurate and fast\u000a4    \u000a</code></pre>\u000a\u000a<blockquote>\u000a  <p>Question from @Kaz: Why the heck are two very different algorithms shoe-horned into a single function?</p>\u000a</blockquote>\u000a\u000a<p>The signature for <em>str.split</em> is about 20 years old, and a number of the APIs from that era are strictly pragmatic.\u000aWhile not perfect, the method signature isn't "terrible" either.\u000aFor the most part, Guido's API design choices have stood the test of time.</p>\u000a\u000a<p>The current API is not without advantages.  Consider strings such as:</p>\u000a\u000a<pre><code>ps_aux_header  = "USER               PID  %CPU %MEM      VSZ"\u000apatient_header = "name,age,height,weight"\u000a</code></pre>\u000a\u000a<p>When asked to break these strings into fields,\u000apeople tend to describe both using the same English word, "split".\u000aWhen asked to read code such as <code>fields = line.split()</code>\u000aor <code>fields = line.split(',')</code>,\u000apeople tend to correctly interpret the statements\u000aas "splits a line into fields".  </p>\u000a\u000a<p>Microsoft Excel's <a href="http://office.microsoft.com/en-us/excel-help/split-names-by-using-the-convert-text-to-columns-wizard-HA010102340.aspx" rel="noreferrer">text-to-columns tool</a> made a similar API choice and\u000aincorporates both splitting algorithms in the same tool.\u000aPeople seem to mentally model field-splitting\u000aas a single concept even though more than one algorithm is involved.</p>\u000a

V<p>There are two ways to do the redirect. Both apply to either <code>subprocess.Popen</code> or <code>subprocess.call</code>.</p>\u000a\u000a<ol>\u000a<li><p>Set the keyword argument <code>shell = True</code> or <code>executable = /path/to/the/shell</code> and specify the command just as you have it there.</p></li>\u000a<li><p>Since you're just redirecting the output to a file, set the keyword argument</p>\u000a\u000a<pre><code>stdout = an_open_writeable_file_object\u000a</code></pre>\u000a\u000a<p>where the object points to the <code>output</code> file.</p></li>\u000a</ol>\u000a\u000a<p><a href="http://docs.python.org/library/subprocess.html#subprocess.Popen" rel="noreferrer"><code>subprocess.Popen</code></a> is more general than <a href="http://docs.python.org/library/subprocess.html#subprocess.call" rel="noreferrer"><code>subprocess.call</code></a>.</p>\u000a\u000a<p><code>Popen</code> doesn't block, allowing you to interact with the process while it's running, or continue with other things in your Python program. The call to <code>Popen</code> returns a <code>Popen</code> object.</p>\u000a\u000a<p><code>call</code> <em>does</em> block. While it supports all the same arguments as the <code>Popen</code> constructor, so you can still set the process' output, environmental variables, etc., your script waits for the program to complete, and <code>call</code> returns a code representing the process' exit status. </p>\u000a\u000a<pre><code>returncode = call(*args, **kwargs) \u000a</code></pre>\u000a\u000a<p>is basically the same as calling </p>\u000a\u000a<pre><code>returncode = Popen(*args, **kwargs).wait()\u000a</code></pre>\u000a\u000a<p><code>call</code> is just a convenience function. It's implementation in CPython is in <a href="http://hg.python.org/cpython/file/e0df7db13d55/Lib/subprocess.py#l459" rel="noreferrer">subprocess.py</a>:</p>\u000a\u000a<pre><code>def call(*popenargs, timeout=None, **kwargs):\u000a    """Run command with arguments.  Wait for command to complete or\u000a    timeout, then return the returncode attribute.\u000a\u000a    The arguments are the same as for the Popen constructor.  Example:\u000a\u000a    retcode = call(["ls", "-l"])\u000a    """\u000a    with Popen(*popenargs, **kwargs) as p:\u000a        try:\u000a            return p.wait(timeout=timeout)\u000a        except:\u000a            p.kill()\u000a            p.wait()\u000a            raise\u000a</code></pre>\u000a\u000a<p>As you can see, it's a thin wrapper around <code>Popen</code>.</p>\u000a

V<p>Django forms module uses <code>__call__</code> method nicely to implement a consistent API for form validation. You can write your own validator for a form in Django as a function.</p>\u000a\u000a<pre><code>def custom_validator(value):\u000a    #your validation logic\u000a</code></pre>\u000a\u000a<p>Django has some default built-in validators such as email validators, url validators etc., which broadly fall under the umbrella of RegEx validators. To implement these cleanly, Django resorts to callable classes (instead of functions). It implements default Regex Validation logic in a RegexValidator and then extends these classes for other validations.</p>\u000a\u000a<pre><code>class RegexValidator(object):\u000a    def __call__(self, value):\u000a        # validation logic\u000a\u000aclass URLValidator(RegexValidator):\u000a    def __call__(self, value):\u000a        super(URLValidator, self).__call__(value)\u000a        #additional logic\u000a\u000aclass EmailValidator(RegexValidator):\u000a    # some logic\u000a</code></pre>\u000a\u000a<p>Now both your custom function and built-in EmailValidator can be called with the same syntax.</p>\u000a\u000a<pre><code>for v in [custom_validator, EmailValidator()]:\u000a    v(value) # &lt;-----\u000a</code></pre>\u000a\u000a<p>As you can see, this implementation in Django is similar to what others have explained in their answers below. Can this be implemented in any other way? You could, but IMHO it will not be as readable or as easily extensible for a big framework like Django.</p>\u000a

V<p>The crucial distinction between Python's <code>__init__</code> and those other languages <strong>constructors</strong> is that <code>__init__</code> is <strong>not</strong> a constructor: it's an <strong>initializer</strong> (the actual <strong>constructor</strong> (if any, but, see later;-) is <code>__new__</code> and works completely differently again).  While <strong>constructing</strong> all superclasses (and, no doubt, doing so "before" you continue constructing downwards) is obviously part of saying you're <strong>constructing</strong> a subclass's instance, that is clearly not the case for <strong>initializing</strong>, since there are many use cases in which superclasses' initialization needs to be skipped, altered, controlled -- happening, if at all, "in the middle" of the subclass initialization, and so forth.</p>\u000a\u000a<p>Basically, super-class delegation of the initializer is not automatic in Python for exactly the same reasons such delegation is also not automatic for <strong>any</strong> other methods -- and note that those "other languages" don't do automatic super-class delegation for any other method either... <em>just</em> for the constructor (and if applicable, destructor), which, as I mentioned, is <strong>not</strong> what Python's <code>__init__</code> is.  (Behavior of <code>__new__</code> is also quite peculiar, though really not directly related to your question, since <code>__new__</code> is such a peculiar constructor that it doesn't actually necessarily need to construct anything -- could perfectly well return an existing instance, or even a non-instance... clearly Python offers you a <strong>lot</strong> more control of the mechanics than the "other languages" you have in mind, which <em>also</em> includes having no automatic delegation in <code>__new__</code> itself!-).</p>\u000a

V<p>The benefit of not calling via the shell is that you are not invoking a 'mystery program.'  On POSIX, the environment variable <code>SHELL</code> controls which binary is invoked as the "shell."  On Windows, there is no bourne shell descendent, only cmd.exe.</p>\u000a\u000a<p>So invoking the shell invokes a program of the user's choosing and is platform-dependent.  Generally speaking, avoid invocations via the shell.</p>\u000a\u000a<p>Invoking via the shell does allow you to expand environment variables and file globs according to the shell's usual mechanism.  On POSIX systems, the shell expands file globs to a list of files.  On Windows, a file glob (e.g., "*.*") is not expanded by the shell, anyway (but environment variables on a command line <em>are</em> expanded by cmd.exe).</p>\u000a\u000a<p>If you think you want environment variable expansions and file globs, research the <code>ILS</code> attacks of 1992-ish on network services which performed subprogram invocations via the shell.  Examples include the various <code>sendmail</code> backdoors involving <code>ILS</code>.</p>\u000a\u000a<p>In summary, use <code>shell=False</code>.</p>\u000a
