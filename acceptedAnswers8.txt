V<p><em>Note: in pandas version 0.20.0 and above, <code>ix</code> is <a href="http://pandas-docs.github.io/pandas-docs-travis/indexing.html#ix-indexer-is-deprecated" rel="noreferrer">deprecated</a> and the use of <code>loc</code> and <code>iloc</code> is encouraged instead. I have left the parts of this answer that describe <code>ix</code> intact as a reference for users of earlier versions of pandas. Examples have been added below showing alternatives to  <code>ix</code></em>.</p>\u000a\u000a<hr>\u000a\u000a<p>First, a recap:</p>\u000a\u000a<ul>\u000a<li><code>loc</code> works on <em>labels</em> in the index. </li>\u000a<li><code>iloc</code> works on the <em>positions</em> in the index (so it only takes integers).</li>\u000a<li><code>ix</code> usually tries to behave like <code>loc</code> but falls back to behaving like <code>iloc</code> if the label is not in the index.</li>\u000a</ul>\u000a\u000a<p>It's important to note some subtleties that can make <code>ix</code> slightly tricky to use:</p>\u000a\u000a<ul>\u000a<li><p>if the index is of integer type, <code>ix</code> will only use label-based indexing and not fall back to position-based indexing. If the label is not in the index, an error is raised.</p></li>\u000a<li><p>if the index does not contain <em>only</em> integers, then given an integer, <code>ix</code> will immediately use position-based indexing rather than label-based indexing. If however <code>ix</code> is given another type (e.g. a string), it can use label-based indexing. </p></li>\u000a</ul>\u000a\u000a<hr>\u000a\u000a<p>To illustrate the differences between the three methods, consider the following Series:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; s = pd.Series(np.nan, index=[49,48,47,46,45, 1, 2, 3, 4, 5])\u000a&gt;&gt;&gt; s\u000a49   NaN\u000a48   NaN\u000a47   NaN\u000a46   NaN\u000a45   NaN\u000a1    NaN\u000a2    NaN\u000a3    NaN\u000a4    NaN\u000a5    NaN\u000a</code></pre>\u000a\u000a<p>Then <code>s.iloc[:3]</code> returns the first 3 rows (since it looks at the position) and <code>s.loc[:3]</code> returns the first 8 rows (since it looks at the labels):</p>\u000a\u000a<pre><code>&gt;&gt;&gt; s.iloc[:3] # slice the first three rows\u000a49   NaN\u000a48   NaN\u000a47   NaN\u000a\u000a&gt;&gt;&gt; s.loc[:3] # slice up to and including label 3\u000a49   NaN\u000a48   NaN\u000a47   NaN\u000a46   NaN\u000a45   NaN\u000a1    NaN\u000a2    NaN\u000a3    NaN\u000a\u000a&gt;&gt;&gt; s.ix[:3] # the integer is in the index so s.ix[:3] works like loc\u000a49   NaN\u000a48   NaN\u000a47   NaN\u000a46   NaN\u000a45   NaN\u000a1    NaN\u000a2    NaN\u000a3    NaN\u000a</code></pre>\u000a\u000a<p>Notice <code>s.ix[:3]</code> returns the same Series as <code>s.loc[:3]</code> since it looks for the label first rather than going by position (and the index is of integer type).</p>\u000a\u000a<p>What if we try with an integer label that isn't in the index (say <code>6</code>)?</p>\u000a\u000a<p>Here <code>s.iloc[:6]</code> returns the first 6 rows of the Series as expected. However, <code>s.loc[:6]</code> raises a KeyError since <code>6</code> is not in the index. </p>\u000a\u000a<pre><code>&gt;&gt;&gt; s.iloc[:6]\u000a49   NaN\u000a48   NaN\u000a47   NaN\u000a46   NaN\u000a45   NaN\u000a1    NaN\u000a\u000a&gt;&gt;&gt; s.loc[:6]\u000aKeyError: 6\u000a\u000a&gt;&gt;&gt; s.ix[:6]\u000aKeyError: 6\u000a</code></pre>\u000a\u000a<p>As per the subtleties noted above, <code>s.ix[:6]</code> now raises a KeyError because it tries to work like <code>loc</code> but can't find a <code>6</code> in the index. Because our index is of integer type it doesn't fall back to behaving like <code>iloc</code>.</p>\u000a\u000a<p>If, however, our index was of mixed type, given an integer <code>ix</code> would behave like <code>iloc</code> immediately instead of raising a KeyError:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; s2 = pd.Series(np.nan, index=['a','b','c','d','e', 1, 2, 3, 4, 5])\u000a&gt;&gt;&gt; s2.index.is_mixed() # index is mix of types\u000aTrue\u000a&gt;&gt;&gt; s2.ix[:6] # behaves like iloc given integer\u000aa   NaN\u000ab   NaN\u000ac   NaN\u000ad   NaN\u000ae   NaN\u000a1   NaN\u000a</code></pre>\u000a\u000a<p>Keep in mind that <code>ix</code> can still accept non-integers and behave like <code>loc</code>:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; s2.ix[:'c'] # behaves like loc given non-integer\u000aa   NaN\u000ab   NaN\u000ac   NaN\u000a</code></pre>\u000a\u000a<p>As general advice, if you're only indexing using labels, or only indexing using integer positions, stick with <code>loc</code> or <code>iloc</code> to avoid unexpected results - try not use <code>ix</code>.</p>\u000a\u000a<hr>\u000a\u000a<h3>Combining position-based and label-based indexing</h3>\u000a\u000a<p>Sometimes given a DataFrame, you will want to mix label and positional indexing methods for the rows and columns.</p>\u000a\u000a<p>For example, consider the following DataFrame. How best to slice the rows up to and including 'c' <em>and</em> take the first four columns?</p>\u000a\u000a<pre><code>&gt;&gt;&gt; df = pd.DataFrame(np.nan, \u000a                      index=list('abcde'),\u000a                      columns=['x','y','z', 8, 9])\u000a&gt;&gt;&gt; df\u000a    x   y   z   8   9\u000aa NaN NaN NaN NaN NaN\u000ab NaN NaN NaN NaN NaN\u000ac NaN NaN NaN NaN NaN\u000ad NaN NaN NaN NaN NaN\u000ae NaN NaN NaN NaN NaN\u000a</code></pre>\u000a\u000a<p>In earlier versions of pandas (before 0.20.0) <code>ix</code> lets you do this quite neatly - we can slice the rows by label and the columns by position (note that for the columns, <code>ix</code> default to position-based slicing since the label <code>4</code> is not a column name):</p>\u000a\u000a<pre><code>&gt;&gt;&gt; df.ix[:'c', :4]\u000a    x   y   z   8\u000aa NaN NaN NaN NaN\u000ab NaN NaN NaN NaN\u000ac NaN NaN NaN NaN\u000a</code></pre>\u000a\u000a<p>In later versions of pandas, we can achieve this result using <code>iloc</code> and the help of another method:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; df.iloc[:df.index.get_loc('c') + 1, :4]\u000a    x   y   z   8\u000aa NaN NaN NaN NaN\u000ab NaN NaN NaN NaN\u000ac NaN NaN NaN NaN\u000a</code></pre>\u000a\u000a<p><a href="http://pandas.pydata.org/pandas-docs/version/0.19.1/generated/pandas.Index.get_loc.html" rel="noreferrer"><code>get_loc()</code></a> is an index method meaning "get the position of the label in this index". Note that since slicing with <code>iloc</code> is exclusive of its endpoint, we must add 1 to this value if we want row 'c' as well.</p>\u000a\u000a<p>There are further examples in pandas' documentation <a href="http://pandas-docs.github.io/pandas-docs-travis/indexing.html#ix-indexer-is-deprecated" rel="noreferrer">here</a>.</p>\u000a

V<p>You can use <code>set_printoptions</code> to set the precision of the output:</p>\u000a\u000a<pre><code>import numpy as np\u000ax=np.random.random(10)\u000aprint(x)\u000a# [ 0.07837821  0.48002108  0.41274116  0.82993414  0.77610352  0.1023732\u000a#   0.51303098  0.4617183   0.33487207  0.71162095]\u000a\u000anp.set_printoptions(precision=3)\u000aprint(x)\u000a# [ 0.078  0.48   0.413  0.83   0.776  0.102  0.513  0.462  0.335  0.712]\u000a</code></pre>\u000a\u000a<p>And <code>suppress</code> suppresses the use of scientific notation for small numbers:</p>\u000a\u000a<pre><code>y=np.array([1.5e-10,1.5,1500])\u000aprint(y)\u000a# [  1.500e-10   1.500e+00   1.500e+03]\u000anp.set_printoptions(suppress=True)\u000aprint(y)\u000a# [    0.      1.5  1500. ]\u000a</code></pre>\u000a\u000a<p>See the <a href="http://docs.scipy.org/doc/numpy/reference/generated/numpy.set_printoptions.html" rel="noreferrer">docs for set_printoptions</a> for other options.</p>\u000a\u000a<hr>\u000a\u000a<p><strong>To apply print options locally</strong>, you could use a <a href="http://docs.python.org/2/library/contextlib.html#contextlib.contextmanager" rel="noreferrer">contextmanager</a>:</p>\u000a\u000a<pre><code>import numpy as np\u000aimport contextlib\u000a\u000a@contextlib.contextmanager\u000adef printoptions(*args, **kwargs):\u000a    original = np.get_printoptions()\u000a    np.set_printoptions(*args, **kwargs)\u000a    try:\u000a        yield\u000a    finally: \u000a        np.set_printoptions(**original)\u000a</code></pre>\u000a\u000a<p>For example, inside the <code>with-suite</code> <code>precision=3</code> and <code>suppress=True</code> are set:</p>\u000a\u000a<pre><code>x = np.random.random(10)\u000awith printoptions(precision=3, suppress=True):\u000a    print(x)\u000a    # [ 0.073  0.461  0.689  0.754  0.624  0.901  0.049  0.582  0.557  0.348]\u000a</code></pre>\u000a\u000a<p>But outside the <code>with-suite</code> the print options are back to default settings:</p>\u000a\u000a<pre><code>print(x)    \u000a# [ 0.07334334  0.46132615  0.68935231  0.75379645  0.62424021  0.90115836\u000a#   0.04879837  0.58207504  0.55694118  0.34768638]\u000a</code></pre>\u000a\u000a<hr>\u000a\u000a<p><strong>To prevent zeros from being stripped from the end of floats:</strong></p>\u000a\u000a<p><code>np.set_printoptions</code> now has a <code>formatter</code> parameter which allows you to specify a format function for each type.</p>\u000a\u000a<pre><code>np.set_printoptions(formatter={'float': '{: 0.3f}'.format})\u000aprint(x)\u000a</code></pre>\u000a\u000a<p>which prints</p>\u000a\u000a<pre><code>[ 0.078  0.480  0.413  0.830  0.776  0.102  0.513  0.462  0.335  0.712]\u000a</code></pre>\u000a\u000a<p>instead of </p>\u000a\u000a<pre><code>[ 0.078  0.48   0.413  0.83   0.776  0.102  0.513  0.462  0.335  0.712]\u000a</code></pre>\u000a

V<p>This depends entirely on the object <code>i</code>.  </p>\u000a\u000a<p><code>+=</code> calls the <a href="http://docs.python.org/2/reference/datamodel.html#object.__iadd__" rel="noreferrer"><code>__iadd__</code> method</a> (if it exists -- falling back on <code>__add__</code> if it doesn't exist) whereas <code>+</code> calls the <a href="http://docs.python.org/2/reference/datamodel.html#object.__add__" rel="noreferrer"><code>__add__</code> method</a><sup>1</sup> or the <a href="http://docs.python.org/2/reference/datamodel.html#object.__radd__" rel="noreferrer"><code>__radd__</code> method in a few cases</a><sup>2</sup>.  </p>\u000a\u000a<p>From an API perspective, <code>__iadd__</code> is supposed to be used for modifying mutable objects <em>in place</em> (returning the object which was mutated) whereas <code>__add__</code> should return a <em>new instance</em> of something.  For <em>immutable</em> objects, both methods return a new instance, but <code>__iadd__</code> will put the new instance in the current namespace with the same name that the old instance had.  This is why </p>\u000a\u000a<pre><code>i = 1\u000ai += 1\u000a</code></pre>\u000a\u000a<p>seems to increment <code>i</code>.  In reality, you get a new integer and assign it "on top of" <code>i</code> -- losing one reference to the old integer.  In this case, <code>i += 1</code> is exactly the same as <code>i = i + 1</code>.  But, with most mutable objects, it's a different story:</p>\u000a\u000a<p>As a concrete example:</p>\u000a\u000a<pre><code>a = [1, 2, 3]\u000ab = a\u000ab += [1, 2, 3]\u000aprint a  #[1, 2, 3, 1, 2, 3]\u000aprint b  #[1, 2, 3, 1, 2, 3]\u000a</code></pre>\u000a\u000a<p>compared to:</p>\u000a\u000a<pre><code>a = [1, 2, 3]\u000ab = a\u000ab = b + [1, 2, 3]\u000aprint a #[1, 2, 3]\u000aprint b #[1, 2, 3, 1, 2, 3]\u000a</code></pre>\u000a\u000a<p>notice how in the first example, since <code>b</code> and <code>a</code> reference the same object, when I use <code>+=</code> on <code>b</code>, it actually changes <code>b</code> (and <code>a</code> sees that change too -- After all, it's referencing the same list).  In the second case however, when I do <code>b = b + [1, 2, 3]</code>, this takes the list that <code>b</code> is referencing and concatenates it with a new list <code>[1, 2, 3]</code>.  It then stores the concatenated list in the current namespace as <code>b</code> -- With no regard for what <code>b</code> was the line before.</p>\u000a\u000a<hr>\u000a\u000a<p><sup><sup>1</sup>In the expression <code>x + y</code>, if <code>x.__add__</code> isn't implemented or if <code>x.__add__(y)</code> returns <code>NotImplemented</code> <strong>and</strong> <em><code>x</code> and <code>y</code> have different types</em>, then <code>x + y</code> tries to call <a href="http://docs.python.org/2/reference/datamodel.html#object.__radd__" rel="noreferrer"><code>y.__radd__(x)</code></a>.  So, in the case where you have </sup></p>\u000a\u000a<p><sup><code>foo_instance += bar_instance</code></sup></p>\u000a\u000a<p><sup>if <code>Foo</code> doesn't implement <code>__add__</code> or <code>__iadd__</code> then the result here is the same as </sup></p>\u000a\u000a<p><sup><code>foo_instance = bar_instance.__radd__(bar_instance, foo_instance)</code></sup></p>\u000a\u000a<p><sup><sup>2</sup>In the expression <code>foo_instance + bar_instance</code>, <code>bar_instance.__radd__</code> will be tried before <code>foo_instance.__add__</code> <em>if</em> the type of <code>bar_instance</code> is a subclass of the type of <code>foo_instance</code> (e.g. <code>issubclass(Bar, Foo)</code>).  The rational for this is because <code>Bar</code> is in some sense a "higher-level" object than <code>Foo</code> so <code>Bar</code> should get the option of overriding <code>Foo</code>'s behavior.</sup></p>\u000a

V<p>(This is inspired by @Mark Tolonen's answer.)</p>\u000a\u000a<p>An <code>if</code> statement runs its <code>else</code> clause if its condition evaluates to false.\u000aIdentically, a <code>while</code> loop runs the else clause if its condition evaluates to false.</p>\u000a\u000a<p>This rule matches the behavior you described:</p>\u000a\u000a<ul>\u000a<li>In normal execution, the while loop repeatedly runs until the condition evaluates to false, and therefore naturally exiting the loop runs the else clause.</li>\u000a<li>When you execute a <code>break</code> statement, you exit out of the loop without evaluating the condition, so the condition cannot evaluate to false and you never run the else clause.</li>\u000a<li>When you execute a <code>continue</code> statement, you evaluate the condition again, and do exactly what you normally would at the beginning of a loop iteration.\u000aSo, if the condition is true, you keep looping, but if it is false you run the else clause.</li>\u000a<li>Other methods of exiting the loop, such as <code>return</code>, do not evaluate the condition and therefore do not run the else clause.</li>\u000a</ul>\u000a\u000a<p><code>for</code> loops behave the same way. Just consider the condition as true if the iterator has more elements, or false otherwise.</p>\u000a

V<p>The <a href="http://www.scipy.org/Installing_SciPy/BuildingGeneral">SciPy webpage</a> used to provide build and installation instructions, but the instructions there now rely on OS binary distributions. To build SciPy (and NumPy) on operating systems without precompiled packages of the required libraries, you must build and then statically link to the Fortran libraries <a href="https://en.wikipedia.org/wiki/Basic_Linear_Algebra_Subprograms">BLAS</a> and <a href="http://en.wikipedia.org/wiki/LAPACK">LAPACK</a>:</p>\u000a\u000a<pre><code>mkdir -p ~/src/\u000acd ~/src/\u000awget http://www.netlib.org/blas/blas.tgz\u000atar xzf blas.tgz\u000acd BLAS-*\u000a\u000a## NOTE: The selected Fortran compiler must be consistent for BLAS, LAPACK, NumPy, and SciPy.\u000a## For GNU compiler on 32-bit systems:\u000a#g77 -O2 -fno-second-underscore -c *.f                     # with g77\u000a#gfortran -O2 -std=legacy -fno-second-underscore -c *.f    # with gfortran\u000a## OR for GNU compiler on 64-bit systems:\u000a#g77 -O3 -m64 -fno-second-underscore -fPIC -c *.f                     # with g77\u000agfortran -O3 -std=legacy -m64 -fno-second-underscore -fPIC -c *.f    # with gfortran\u000a## OR for Intel compiler:\u000a#ifort -FI -w90 -w95 -cm -O3 -unroll -c *.f\u000a\u000a# Continue below irrespective of compiler:\u000aar r libfblas.a *.o\u000aranlib libfblas.a\u000arm -rf *.o\u000aexport BLAS=~/src/BLAS-*/libfblas.a\u000a</code></pre>\u000a\u000a<p>Execute only one of the five g77/gfortran/ifort commands. I have commented out all, but the gfortran which I use. The subsequent LAPACK installation requires a <a href="http://en.wikipedia.org/wiki/Fortran#Fortran_90">Fortran 90</a> compiler, and since both installs should use the same Fortran compiler, g77 should not be used for BLAS.</p>\u000a\u000a<p>Next, you'll need to install the LAPACK stuff. The SciPy webpage's instructions helped me here as well, but I had to modify them to suit my environment:</p>\u000a\u000a<pre><code>mkdir -p ~/src\u000acd ~/src/\u000awget http://www.netlib.org/lapack/lapack.tgz\u000atar xzf lapack.tgz\u000acd lapack-*/\u000acp INSTALL/make.inc.gfortran make.inc          # On Linux with lapack-3.2.1 or newer\u000amake lapacklib\u000amake clean\u000aexport LAPACK=~/src/lapack-*/liblapack.a\u000a</code></pre>\u000a\u000a<p>Update on 3-Sep-2015:\u000aVerified some comments today (thanks to all): Before running <code>make lapacklib</code> edit the <code>make.inc</code> file and add <code>-fPIC</code> option to <code>OPTS</code> and <code>NOOPT</code> settings. If you are on a 64bit architecture or want to compile for one, also add <code>-m64</code>. It is important that BLAS and LAPACK are compiled with these options set to the same values. If you forget the <code>-fPIC</code> SciPy will actually give you an error about missing symbols and will recommend this switch. The specific section of <code>make.inc</code> looks like this in my setup:</p>\u000a\u000a<pre><code>FORTRAN  = gfortran \u000aOPTS     = -O2 -frecursive -fPIC -m64\u000aDRVOPTS  = $(OPTS)\u000aNOOPT    = -O0 -frecursive -fPIC -m64\u000aLOADER   = gfortran\u000a</code></pre>\u000a\u000a<p>On old machines (e.g. RedHat 5), gfortran might be installed in an older version (e.g. 4.1.2) and does not understand option <code>-frecursive</code>. Simply remove it from the <code>make.inc</code> file in such cases.</p>\u000a\u000a<p>The lapack test target of the Makefile fails in my setup because it cannot find the blas libraries. If you are thorough you can temporarily move the blas library to the specified location to test the lapack. I'm a lazy person, so I trust the devs to have it working and verify only in SciPy.</p>\u000a

V<p>Even though this isn't entirely in the SO spirit, I love this question, because I had the same trouble when I started so I'll give you a quick guide. Obviously you don't understand the principles behind them (don't take it as an offense, but if you did you wouldn't be asking). </p>\u000a\u000a<p>Django is server-side. It means, say a client goes to url you have a function inside views that renders what he sees and returns a response in html. let's break it up into examples:</p>\u000a\u000a<p>views.py</p>\u000a\u000a<pre><code>def hello(request):\u000a    return HttpResponse('Hello World!')\u000a\u000adef home(request):\u000a    return render_to_response('index.html', {'variable': 'world'})\u000a</code></pre>\u000a\u000a<p>index.html:</p>\u000a\u000a<pre class="lang-html prettyprint-override"><code>&lt;h1&gt;Hello {{ variable }}, welcome to my awesome site&lt;/h1&gt;\u000a</code></pre>\u000a\u000a<p>urls.py</p>\u000a\u000a<pre><code>url(r'^hello/', 'myapp.views.hello'),\u000aurl(r'^home/', 'myapp.views.home'),\u000a</code></pre>\u000a\u000a<p>That's an example of the simplest of usages. Going to <code>127.0.0.1:8000/hello</code> means a request to the hello function, going to <code>127.0.0.1:8000/home</code> will return the <code>index.html</code> and replace all the variables as asked (you probably know all this by now).</p>\u000a\u000a<p>Now let's talk about AJAX. AJAX calls are client-side code that does asynchronous requests. That sounds complicated, but it simply means it does a request for you in the background and then handles the response. So when you do an AJAX call for some url, you get the same data you would get as a user going to that place. </p>\u000a\u000a<p>For example, an ajax call to <code>127.0.0.1:8000/hello</code> will return the same thing it would as if you visited it. Only this time, you have it inside a js function and you can deal with it however you'd like. Let's look at a simple use case:</p>\u000a\u000a<pre class="lang-javascript prettyprint-override"><code>$.ajax({\u000a    url: '127.0.0.1:8000/hello',\u000a    type: 'get', // This is the default though, you don't actually need to always mention it\u000a    success: function(data) {\u000a        alert(data);\u000a    },\u000a    failure: function(data) { \u000a        alert('Got an error dude');\u000a    }\u000a}); \u000a</code></pre>\u000a\u000a<p>The general process is this:</p>\u000a\u000a<ol>\u000a<li>The call goes to the url <code>127.0.0.1:8000/hello</code> as if you opened a new tab and did it yourself.</li>\u000a<li>If it succeeds (status code 200), do the function for success, which will alert the data recieved.</li>\u000a<li>If fails, do a different function.</li>\u000a</ol>\u000a\u000a<p>Now what would happen here? You would get an alert with 'hello world' in it. What happens if you do an ajax call to home? Same thing, you'll get an alert stating <code>&lt;h1&gt;Hello world, welcome to my awesome site&lt;/h1&gt;</code>.</p>\u000a\u000a<p>In other words - there's nothing new about AJAX calls. They are just a way for you to let the user get data and information without leaving the page, and it makes for a smooth and very neat design of your website. A few guidelines you should take note of:</p>\u000a\u000a<ol>\u000a<li><strong>Learn jQuery</strong>. I cannot stress this enough. You're gonna have to understand it a little to know how to handle the data you receive. You'll also need to understand some basic javascript syntax (not far from python, you'll get used to it). I strongly recommend <a href="https://tutsplus.com/course/30-days-to-learn-jquery/" rel="noreferrer">Envato's video tutorials for jQuery</a>, they are great and will put you on the right path.</li>\u000a<li><strong>When to use JSON?</strong>. You're going to see a lot of examples where the data sent by the Django views is in JSON. I didn't go into detail on that, because it isn't important <em>how</em> to do it (there are plenty of explanations abound) and a lot more important <em>when</em>. And the answer to that is - JSON data is serialized data. That is, data you can manipulate. Like I mentioned, an AJAX call will fetch the response as if the user did it himself. Now say you don't want to mess with all the html, and instead want to send data (a list of objects perhaps). JSON is good for this, because it sends it as an object (JSON data looks like a python dictionary), and then you can iterate over it or do something else that removes the need to sift through useless html.</li>\u000a<li><strong>Add it last</strong>. When you build a web app and want to implement AJAX - do yourself a favor. First, build the entire app completely devoid of any AJAX. See that everything is working. Then, and only then, start writing the AJAX calls. That's a good process that helps you learn a lot as well.</li>\u000a<li><strong>Use chrome's developer tools</strong>. Since AJAX calls are done in the background it's sometimes very hard to debug them. You should use the chrome developer tools (or similar tools such as firebug) and <code>console.log</code> things to debug. I won't explain in detail, just google around and find out about it. It would be very helpful to you.</li>\u000a<li><strong>CSRF awareness</strong>. Finally, remember that post requests in Django require the <code>csrf_token</code>. With AJAX calls, a lot of times you'd like to send data without refreshing the page. You'll probably face some trouble before you'd finally remember that - wait, you forgot to send the <code>csrf_token</code>. This is a known beginner roadblock in AJAX-Django integration, but after you learn how to make it play nice, it's easy as pie.</li>\u000a</ol>\u000a\u000a<p>That's everything that comes to my head. It's a vast subject, but yeah, there's probably not enough examples out there. Just work your way there, slowly, you'll get it eventually.</p>\u000a

V<p>I don't actually think that DI/IoC are <em>that</em> uncommon in Python. What <em>is</em> uncommon, however, are DI/IoC <em>frameworks/containers</em>.</p>\u000a\u000a<p>Think about it: what does a DI container do? It allows you to </p>\u000a\u000a<ol>\u000a<li>wire together independent components into a complete application ...</li>\u000a<li>... at runtime.</li>\u000a</ol>\u000a\u000a<p>We have names for "wiring together" and "at runtime":</p>\u000a\u000a<ol>\u000a<li>scripting</li>\u000a<li>dynamic</li>\u000a</ol>\u000a\u000a<p>So, a DI container is nothing but an interpreter for a dynamic scripting language. Actually, let me rephrase that: a typical Java/.NET DI container is nothing but a crappy interpreter for a really bad dynamic scripting language with butt-ugly, often XML-based, syntax.</p>\u000a\u000a<p>When you program in Python, why would you want to use an ugly, bad scripting language when you have a beautiful, brilliant scripting language at your disposal? Actually, that's a more general question: when you program in pretty much any language, why would you want to use an ugly, bad scripting language when you have Jython and IronPython at your disposal?</p>\u000a\u000a<p>So, to recap: the <em>practice</em> of DI/IoC is just as important in Python as it is in Java, for exactly the same reasons. The <em>implementation</em> of DI/IoC however, is built into the language and often so lightweight that it completely vanishes. </p>\u000a\u000a<p>(Here's a brief aside for an analogy: in assembly, a subroutine call is a pretty major deal - you have to save your local variables and registers to memory, save your return address somewhere, change the instruction pointer to the subroutine you are calling, arrange for it to somehow jump back into your subroutine when it is finished, put the arguments somewhere where the callee can find them, and so on. IOW: in assembly, "subroutine call" is a Design Pattern, and before there were languages like Fortran which had subroutine calls built in, people were building their own "subroutine frameworks". Would you say that subroutine calls are "uncommon" in Python, just because you don't use subroutine frameworks?)</p>\u000a\u000a<p>BTW: for an example of what it looks like to take DI to its logical conclusion, take a look at <a href="http://GBracha.BlogSpot.Com/" rel="noreferrer">Gilad Bracha</a>'s <a href="http://NewspeakLanguage.Org/" rel="noreferrer">Newspeak Programming Language</a> and his writings on the subject:</p>\u000a\u000a<ul>\u000a<li><a href="http://GBracha.BlogSpot.Com/2007/06/constructors-considered-harmful.html" rel="noreferrer">Constructors Considered Harmful</a></li>\u000a<li><a href="http://GBracha.BlogSpot.Com/2007/12/some-months-ago-i-wrote-couple-of-posts.html" rel="noreferrer">Lethal Injection</a></li>\u000a<li><a href="http://GBracha.BlogSpot.Com/2009/06/ban-on-imports.html" rel="noreferrer">A Ban on Imports</a> (<a href="http://GBracha.BlogSpot.Com/2009/07/ban-on-imports-continued.html" rel="noreferrer">continued</a>)</li>\u000a</ul>\u000a

V<p>A compiled language is one where the program, once compiled, is expressed in the instructions of the target machine. For example, an addition "+" operation in your source code could be translated directly to the "ADD" instruction in machine code.</p>\u000a\u000a<p>An interpreted language is one where the instructions are not directly executed by the target machine, but instead read and executed by some other program (which normally <em>is</em> written in the language of the native machine). For example, the same "+" operation would be recognised by the interpreter at run time, which would then call its own "add(a,b)" function with the appropriate arguments, which would then execute the machine code "ADD" instruction.</p>\u000a\u000a<p>You can do anything that you can do in an interpreted language in a compiled language and vice-versa - they are both Turing complete. Both however have advantages and disadvantages for implementation and use.</p>\u000a\u000a<p>I'm going to completely generalise (purists forgive me!) but, roughly, here are the advantages of compiled languages:</p>\u000a\u000a<ul>\u000a<li>Faster performance by directly using the native code of the target machine</li>\u000a<li>Opportunity to apply quite powerful optimisations during the compile stage</li>\u000a</ul>\u000a\u000a<p>And here are the advantages of interpreted languages:</p>\u000a\u000a<ul>\u000a<li>Easier to implement (writing good compilers is very hard!!)</li>\u000a<li>No need to run a compilation stage: can execute code directly "on the fly"</li>\u000a<li>Can be more convenient for dynamic languages</li>\u000a</ul>\u000a\u000a<p>Note that modern techniques such as bytecode compilation add some extra complexity - what happens here is that the compiler targets a "virtual machine" which is not the same as the underlying hardware. These virtual machine instructions can then be compiled again at a later stage to get native code (e.g. as done by the Java JVM JIT compiler).</p>\u000a

V<h2>So what is CPython</h2>\u000a\u000a<p>CPython is the <em>original</em> Python implementation. It is the implementation you download from Python.org. People call it CPython to distinguish it from other, later, Python implementations, and to distinguish the implementation of the language engine from the Python <em>programming language</em> itself.</p>\u000a\u000a<p>The latter part is where your confusion comes from; you need to keep Python-the-language separate from whatever <em>runs</em> the Python code.</p>\u000a\u000a<p>CPython <em>happens</em> to be implemented in C. That is just an implementation detail really. CPython compiles your python code into bytecode (transparently) and interprets that bytecode in a evaluation loop.</p>\u000a\u000a<p>CPython is also the first to implement new features; Python-the-language development uses CPython as the base, other implementations follow.</p>\u000a\u000a<h2>What about Jython, etc.</h2>\u000a\u000a<p>Jython, IronPython and PyPy are the current 'other' implementations of the Python programming language; these are implemented in Java, C# and RPython (a subset of Python), respectively. Jython compiles your Python code to <em>Java</em> bytecode, so your Python code can run on the JVM. IronPython lets you run Python on the Microsoft CLR. And PyPy, being implemented in (a subset of) Python, lets you run Python code faster than CPython, which rightly should blow your mind. :-)</p>\u000a\u000a<h2>Actually compiling to C</h2>\u000a\u000a<p>So CPython does <strong>not</strong> translate your Python code to C by itself. It instead runs a interpreter loop. There <em>is</em> a project that <em>does</em> translate Python-ish code to C, and that is called <a href="http://cython.org/">Cython</a>. Cython adds a few extensions to the Python language, and lets you compile your code to C extensions, code that plugs <em>into</em> the CPython interpreter.</p>\u000a

V<p>The difference between <code>import module</code> and <code>from module import foo</code> is mainly subjective.  Pick the one you like best and be consistent in your use of it.  Here are some points to help you decide.</p>\u000a\u000a<p><code>import module</code></p>\u000a\u000a<ul>\u000a<li><strong>Pros:</strong>\u000a\u000a<ul>\u000a<li>Less maintenance of your <code>import</code> statements. Don't need to add any additional imports to start using another item from the module</li>\u000a</ul></li>\u000a<li><strong>Cons:</strong>\u000a\u000a<ul>\u000a<li>Typing <code>module.foo</code> in your code can be tedious and redundant (tedium can be minimized by using <code>import module as mo</code> then typing <code>mo.foo</code>)</li>\u000a</ul></li>\u000a</ul>\u000a\u000a<p><code>from module import foo</code></p>\u000a\u000a<ul>\u000a<li><strong>Pros:</strong>\u000a\u000a<ul>\u000a<li>Less typing to use <code>foo</code></li>\u000a<li>More control over which items of a module can be accessed</li>\u000a</ul></li>\u000a<li><strong>Cons:</strong>\u000a\u000a<ul>\u000a<li>To use a new item from the module you have to update your <code>import</code> statement</li>\u000a<li>You lose context about <code>foo</code>. For example, it's less clear what <code>ceil()</code> does compared to <code>math.ceil()</code></li>\u000a</ul></li>\u000a</ul>\u000a\u000a<p>Either method is acceptable, but <strong>don't</strong> use <code>from module import *</code>.  </p>\u000a\u000a<p>For any reasonable large set of code, if you <code>import *</code> you will likely be cementing it into the module, unable to be removed.  This is because it is difficult to determine what items used in the code are coming from 'module', making it easy to get to the point where you think you don't use the <code>import</code> any more but it's extremely difficult to be sure.</p>\u000a

V<p>On the actual behavior, there is no difference. They all return <code>None</code> and that's it. However, there is a time and  place for all of these.\u000aThe following instructions are basically how the different methods should be used (or atleast how I was taught they should be used), but they are not absolute rules so you can mix them up if you feel necessary to.</p>\u000a\u000a<hr>\u000a\u000a<p>Using <code>return None</code>.</p>\u000a\u000a<p>This tells that the function is indeed meant to return a value for later use, and in this case it returns <code>None</code>. This value <code>None</code> can then be used elsewhere. <code>return None</code> is never used if there are no other possible return values from the function.</p>\u000a\u000a<p>In the following example, we return person's mother if the person given is a human. If it's not a human, we return <code>None</code> since the "person" doesn't have a mother (let's suppose it's not an animal or so).</p>\u000a\u000a<pre><code>def get_mother(person):\u000a    if is_human(person):\u000a        return person.mother\u000a    else:\u000a        return None\u000a</code></pre>\u000a\u000a<hr>\u000a\u000a<p>Using <code>return</code>.</p>\u000a\u000a<p>This is used for the same reason as <code>break</code> in loops. The return value doesn't matter and you only want to exit the whole function. It's extremely useful in some places, even tho you don't need it that often.</p>\u000a\u000a<p>We got 15 prisoners and we know one of them has a knife. We loop through each prisoner one by one to check if they have a knife. If we hit the person with a knife, we can just exit the function cause we know there's only one knife and no reason the check rest of the prisoners. If we don't find the prisoner with a knife, we raise an the alert. This could be done in many different ways and using <code>return</code> is probably not even the best way, but it's just an example to show how to use <code>return</code> for exiting a function.</p>\u000a\u000a<pre><code>def find_prisoner_with_knife(prisoners):\u000a    for prisoner in prisoners:\u000a        if "knife" in prisoner.items:\u000a            prisoner.move_to_inquisition()\u000a            return # no need to check rest of the prisoners nor raise an alert\u000a    raise_alert()\u000a</code></pre>\u000a\u000a<p>Note: You should never do <code>var = find_prisoner_with_knife()</code>, since the return value is not meant to be caught.</p>\u000a\u000a<hr>\u000a\u000a<p>Using no return at all.</p>\u000a\u000a<p>This will also return <code>None</code>, but that value is not meant to be used or caught. It simply means that the function ended successfully. It's basically the same as <code>return</code> in <code>void</code> functions in languages such as C++ or Java.</p>\u000a\u000a<p>In the following example, we set person's mother's name, and then the function exits after completing successfully.</p>\u000a\u000a<pre><code>def set_mother(person, mother):\u000a    if is_human(person):\u000a        person.mother = mother\u000a</code></pre>\u000a\u000a<p>Note: You should never do <code>var = set_mother(my_person, my_mother)</code>, since the return value is not meant to be caught.</p>\u000a

V<p>UPDATE: <a href="http://ericlippert.com/2012/10/15/is-c-a-strongly-typed-or-a-weakly-typed-language/" rel="noreferrer">This question was the subject of my blog on the 15th of October, 2012.</a> Thanks for the great question!</p>\u000a\u000a<hr>\u000a\u000a<blockquote>\u000a  <p>What does it really mean for a language to be "weakly typed"? </p>\u000a</blockquote>\u000a\u000a<p>It means "this language uses a type system that I find distasteful". A "strongly typed" language by contrast is a language with a type system that I find pleasant.</p>\u000a\u000a<p>The terms are essentially meaningless and you should avoid them. <a href="http://en.wikipedia.org/wiki/Strong_typing" rel="noreferrer">Wikipedia</a> lists <em>eleven different meanings</em> for "strongly typed", several of which are contradictory. This indicates that the odds of confusion being created are high in any conversation involving the term "strongly typed" or "weakly typed". </p>\u000a\u000a<p>All that you can really say with any certainty is that a "strongly typed" language under discussion has some additional restriction in the type system, either at runtime or compile time, that a "weakly typed" language under discussion lacks. What that restriction might be cannot be determined without further context.</p>\u000a\u000a<p>Instead of using "strongly typed" and "weakly typed", you should describe in detail what kind of type safety you mean. For example, C# is a <em>statically typed</em> language and a <em>type safe</em> language and a <em>memory safe</em> language, <em>for the most part</em>. C# allows all three of those forms of "strong" typing to be violated.  The cast operator violates static typing; it says to the compiler "I know more about the runtime type of this expression than you do". If the developer is wrong, then the runtime will throw an exception in order to protect type safety. If the developer wishes to break type safety or memory safety, they can do so by turning off the type safety system by making an "unsafe" block. In an unsafe block you can use pointer magic to treat an int as a float (violating type safety) or to write to memory you do not own. (Violating memory safety.)</p>\u000a\u000a<p>C# imposes type restrictions that are checked at both compile-time and at runtime, thereby making it a "strongly typed" language compared to languages that do less compile-time checking or less runtime checking. C# also allows you to in special circumstances do an end-run around those restrictions, making it a "weakly typed" language compared with languages which do not allow you to do such an end-run. </p>\u000a\u000a<p>Which is it really?  It is impossible to say; it depends on the point of view of the speaker and their attitude towards the various language features. </p>\u000a

V<p>Generators give you lazy evaluation. You use them by iterating over them, either explicitly with 'for' or implicitly by passing it to any function or construct that iterates. You can think of generators as returning multiple items, as if they return a list, but instead of returning them all at once they return them one-by-one, and the generator function is paused until the next item is requested.</p>\u000a\u000a<p>Generators are good for calculating large sets of results (in particular calculations involving loops themselves) where you don't know if you are going to need all results, or where you don't want to allocate the memory for all results at the same time. Or for situations where the generator uses <em>another</em> generator, or consumes some other resource, and it's more convenient if that happened as late as possible.</p>\u000a\u000a<p>Another use for generators (that is really the same) is to replace callbacks with iteration. In some situations you want a function to do a lot of work and occasionally report back to the caller. Traditionally you'd use a callback function for this. You pass this callback to the work-function and it would periodically call this callback. The generator approach is that the work-function (now a generator) knows nothing about the callback, and merely yields whenever it wants to report something. The caller, instead of writing a separate callback and passing that to the work-function, does all the reporting work in a little 'for' loop around the generator.</p>\u000a\u000a<p>For example, say you wrote a 'filesystem search' program. You could perform the search in its entirety, collect the results and then display them one at a time. All of the results would have to be collected before you showed the first, and all of the results would be in memory at the same time. Or you could display the results while you find them, which would be more memory efficient and much friendlier towards the user. The latter could be done by passing the result-printing function to the filesystem-search function, or it could be done by just making the search function a generator and iterating over the result.</p>\u000a\u000a<p>If you want to see an example of the latter two approaches, see os.path.walk() (the old filesystem-walking function with callback) and os.walk() (the new filesystem-walking generator.) Of course, if you really wanted to collect all results in a list, the generator approach is trivial to convert to the big-list approach:</p>\u000a\u000a<pre><code>big_list = list(the_generator)\u000a</code></pre>\u000a

V<p>If you're just wanting (semi) contiguous regions, there's already an easy implementation in Python: <a href="http://en.wikipedia.org/wiki/SciPy">SciPy</a>'s <a href="http://www.scipy.org/doc/api_docs/SciPy.ndimage.morphology.html">ndimage.morphology</a> module.  This is a fairly common <a href="http://en.wikipedia.org/wiki/Mathematical_morphology">image morphology</a> operation. </p>\u000a\u000a<hr>\u000a\u000a<p>Basically, you have 5 steps:</p>\u000a\u000a<pre><code>def find_paws(data, smooth_radius=5, threshold=0.0001):\u000a    data = sp.ndimage.uniform_filter(data, smooth_radius)\u000a    thresh = data &gt; threshold\u000a    filled = sp.ndimage.morphology.binary_fill_holes(thresh)\u000a    coded_paws, num_paws = sp.ndimage.label(filled)\u000a    data_slices = sp.ndimage.find_objects(coded_paws)\u000a    return object_slices\u000a</code></pre>\u000a\u000a<ol>\u000a<li><p>Blur the input data a bit to make sure the paws have a continuous footprint.  (It would be more efficient to just use a larger kernel (the <code>structure</code> kwarg to the various <code>scipy.ndimage.morphology</code> functions) but this isn't quite working properly for some reason...) </p></li>\u000a<li><p>Threshold the array so that you have a boolean array of places where the pressure is over some threshold value (i.e. <code>thresh = data &gt; value</code>)</p></li>\u000a<li><p>Fill any internal holes, so that you have cleaner regions (<code>filled = sp.ndimage.morphology.binary_fill_holes(thresh)</code>)</p></li>\u000a<li><p>Find the separate contiguous regions (<code>coded_paws, num_paws = sp.ndimage.label(filled)</code>). This returns an array with the regions coded by number (each region is a contiguous area of a unique integer (1 up to the number of paws) with zeros everywhere else)).</p></li>\u000a<li><p>Isolate the contiguous regions using <code>data_slices = sp.ndimage.find_objects(coded_paws)</code>.  This returns a list of tuples of <code>slice</code> objects, so you could get the region of the data for each paw with <code>[data[x] for x in data_slices]</code>. Instead, we'll draw a rectangle based on these slices, which takes slightly more work.</p></li>\u000a</ol>\u000a\u000a<hr>\u000a\u000a<p>The two animations below show your "Overlapping Paws" and "Grouped Paws" example data.  This method seems to be working perfectly. (And for whatever it's worth, this runs much more smoothly than the GIF images below on my machine, so the paw detection algorithm is fairly fast...)</p>\u000a\u000a<p><img src="https://i.stack.imgur.com/1CRA6.gif" alt="Overlapping Paws">\u000a<img src="https://i.stack.imgur.com/ct8ub.gif" alt="Grouped Paws"></p>\u000a\u000a<hr>\u000a\u000a<p>Here's a full example (now with much more detailed explanations). The vast majority of this is reading the input and making an animation.  The actual paw detection is only 5 lines of code.</p>\u000a\u000a<pre><code>import numpy as np\u000aimport scipy as sp\u000aimport scipy.ndimage\u000a\u000aimport matplotlib.pyplot as plt\u000afrom matplotlib.patches import Rectangle\u000a\u000adef animate(input_filename):\u000a    """Detects paws and animates the position and raw data of each frame\u000a    in the input file"""\u000a    # V<p>A closure occurs when a function has access to a local variable from an enclosing scope that has finished its execution.</p>\u000a\u000a<pre><code>def make_printer(msg):\u000a    def printer():\u000a        print msg\u000a    return printer\u000a\u000aprinter = make_printer('Foo!')\u000aprinter()\u000a</code></pre>\u000a\u000a<p>When <code>make_printer</code> is called, a new frame is put on the stack with the compiled code for the <code>printer</code> function as a constant and the value of <code>msg</code> as a local. It then creates and returns the function. Because the function <code>printer</code> references the <code>msg</code> variable, it is kept alive after the <code>make_printer</code> function has returned. </p>\u000a\u000a<p>So, if your nested functions don't </p>\u000a\u000a<ol>\u000a<li>access variables that are local to enclosing scopes, </li>\u000a<li>do so when they are executed outside of that scope,</li>\u000a</ol>\u000a\u000a<p>then they are not closures.</p>\u000a\u000a<p>Here's an example of a nested function which is not a closure.</p>\u000a\u000a<pre><code>def make_printer(msg):\u000a    def printer(msg=msg):\u000a        print msg\u000a    return printer\u000a\u000aprinter = make_printer("Foo!")\u000aprinter()  #Output: Foo!\u000a</code></pre>\u000a\u000a<p>Here, we are binding the value to the default value of a parameter. V<p>A <code>list</code> keeps order, <code>dict</code> and <code>set</code> don't: when you care about order, therefore, you must use <code>list</code> (if your choice of containers is limited to these three, of course;-).</p>\u000a\u000a<p><code>dict</code> associates with each key a value, while <code>list</code> and <code>set</code> just contain values: very different use cases, obviously.</p>\u000a\u000a<p><code>set</code> requires items to be hashable, <code>list</code> doesn't: if you have non-hashable items, therefore, you cannot use <code>set</code> and must instead use <code>list</code>.</p>\u000a\u000a<p><code>set</code> forbids duplicates, <code>list</code> does not: also a crucial distinction.  (A "multiset", which maps duplicates into a different count for items present more than once, can be found in <code>collections.Counter</code> -- you could build one as a <code>dict</code>, if for some weird reason you couldn't import <code>collections</code>, or, in pre-2.7 Python as a <code>collections.defaultdict(int)</code>, using the items as keys and the associated value as the count).</p>\u000a\u000a<p>Checking for membership of a value in a <code>set</code> (or <code>dict</code>, for keys) is blazingly fast (taking about a constant, short time), while in a list it takes time proportional to the list's length in the average and worst cases.  So, if you have hashable items, don't care either way about order or duplicates, and want speedy membership checking, <code>set</code> is better than <code>list</code>.</p>\u000a
