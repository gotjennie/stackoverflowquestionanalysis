V<p>I advocate an object oriented approach. This is the template that I start out with:</p>\u000a\u000a<pre><code># Use Tkinter for python 2, tkinter for python 3\u000aimport tkinter as tk\u000a\u000aclass MainApplication(tk.Frame):\u000a    def __init__(self, parent, *args, **kwargs):\u000a        tk.Frame.__init__(self, parent, *args, **kwargs)\u000a        self.parent = parent\u000a\u000a        &lt;create the rest of your GUI here&gt;\u000a\u000aif __name__ == "__main__":\u000a    root = tk.Tk()\u000a    MainApplication(root).pack(side="top", fill="both", expand=True)\u000a    root.mainloop()\u000a</code></pre>\u000a\u000a<p>The important things to notice are:</p>\u000a\u000a<ul>\u000a<li><p><strong>I don't use a wildcard import.</strong> I import the package as "tk", which requires that I prefix all commands with <code>tk.</code>. This prevents global namespace pollution, plus it makes the code completely obvious when you are using Tkinter classes, ttk classes, or some of your own. </p></li>\u000a<li><p><strong>The main application is a class</strong>. This gives you a private namespace for all of your callbacks and private functions, and just generally makes it easier to organize your code. In a procedural style you have to code top-down, defining functions before using them, etc. With this method you don't since you don't actually create the main window until the very last step. I prefer inheriting from <code>tk.Frame</code> just because I typically start by creating a frame, but it is by no means necessary. </p></li>\u000a</ul>\u000a\u000a<p>If your app has additional toplevel windows, I recommend making each of those a separate class, inheriting from <code>tk.Toplevel</code>. This gives you all of the same advantages mentioned above -- the windows are atomic, they have their own namespace, and the code is well organized. Plus, it makes it easy to put each into its own module once the code starts to get large. </p>\u000a\u000a<p>Finally, you might want to consider using classes for every major portion of your interface. For example, if you're creating an app with a toolbar, a navigation pane, a statusbar, and a main area, you could make each one of those classes. This makes your main code quite small and easy to understand:</p>\u000a\u000a<pre><code>class Navbar(tk.Frame): ...\u000aclass Toolbar(tk.Frame): ...\u000aclass Statusbar(tk.Frame): ...\u000aclass Main(tk.Frame): ...\u000a\u000aclass MainApplication(tk.Frame):\u000a    def __init__(self, parent, *args, **kwargs):\u000a        tk.Frame.__init__(self, parent, *args, **kwargs)\u000a        self.statusbar = Statusbar(self, ...)\u000a        self.toolbar = Toolbar(self, ...)\u000a        self.navbar = Navbar(self, ...)\u000a        self.main = Main(self, ...)\u000a\u000a        self.statusbar.pack(side="bottom", fill="x")\u000a        self.toolbar.pack(side="top", fill="x")\u000a        self.navbar.pack(side="left", fill="y")\u000a        self.main.pack(side="right", fill="both", expand=True)\u000a</code></pre>\u000a\u000a<p>Since all of those instances share a common parent, the parent effectively becomes the "controller" part of a model-view-controller architecture. So, for example, the main window could place something on the statusbar by calling <code>self.parent.statusbar.set("Hello, world")</code>. This allows you to define a simple interface between the components, helping to keep coupling to a minimun. </p>\u000a

V<p>The equation of a line is:</p>\u000a\u000a<pre><code>f(x) = A*x + b = y\u000a</code></pre>\u000a\u000a<p>For a segment, it is exactly the same, except that x is included into an interval I.</p>\u000a\u000a<p>If you have two segments, defined as follow:</p>\u000a\u000a<pre><code>Segment1 = {(X1, Y1), (X2, Y2)}\u000aSegment2 = {(X3, Y3), (X4, Y4)}\u000a</code></pre>\u000a\u000a<p>The abcisse Xa of the potential point of intersection (Xa,Ya) must be contained in both interval I1 and I2, defined as follow :</p>\u000a\u000a<pre><code>I1 = [MIN(X1,X2), MAX(X1,X2)]\u000aI2 = [MIN(X3,X4), MAX(X3,X4)]\u000a</code></pre>\u000a\u000a<p>And we could say that Xa is included into :</p>\u000a\u000a<pre><code>Ia = [MAX( MIN(X1,X2), MIN(X3,X4) ), MIN( MAX(X1,X2), MAX(X3,X4)] )]\u000a</code></pre>\u000a\u000a<p>Now, you need to check that this interval Ia exists :</p>\u000a\u000a<pre><code>if (MAX(X1,X2) &lt; MIN(X3,X4))\u000a    return false; // There is no mutual abcisses\u000a</code></pre>\u000a\u000a<p>So, you got two line formula, and a mutual interval. Your line formulas are:</p>\u000a\u000a<pre><code>f1(x) = A1*x + b1 = y\u000af2(x) = A2*x + b2 = y\u000a</code></pre>\u000a\u000a<p>As we got two points by segment, we are able to determine A1, A2, b1 and b2:</p>\u000a\u000a<pre><code>A1 = (Y1-Y2)/(X1-X2) // Pay attention to not dividing by zero\u000aA2 = (Y3-Y4)/(X3-X4) // Pay attention to not dividing by zero\u000ab1 = Y1-A1*X1 = Y2-A1*X2\u000ab2 = Y3-A2*X3 = Y4-A2*X4\u000a</code></pre>\u000a\u000a<p>If the segments are parallel, then A1 == A2 :</p>\u000a\u000a<pre><code>if (A1 == A2)\u000a    return false; // Parallel segments\u000a</code></pre>\u000a\u000a<p>A point (Xa,Ya) standing on both line must verify both formulas f1 and f2:</p>\u000a\u000a<pre><code>Ya = A1 * Xa + b1\u000aYa = A2 * Xa + b2\u000aA1 * Xa + b1 = A2 * Xa + b2\u000aXa = (b2 - b1) / (A1 - A2) // Once again, pay attention to not dividing by zero\u000a</code></pre>\u000a\u000a<p>The last thing to do is check that Xa is included into Ia:</p>\u000a\u000a<pre><code>if ( (Xa &lt; MAX( MIN(X1,X2), MIN(X3,X4) )) ||\u000a  (Xa &gt; MIN( MAX(X1,X2), MAX(X3,X4) )) )\u000a    return false; // intersection is out of bound\u000aelse\u000a    return true;\u000a</code></pre>\u000a\u000a<p>In addition to this, you may check at startup that two of the four provided points are not equals to avoid all that testing.</p>\u000a

V<p>One classic approach to this problem is to use the "decorate, sort, undecorate" idiom, which is especially simple using python's built-in <code>zip</code> function:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; list1 = [3,2,4,1, 1]\u000a&gt;&gt;&gt; list2 = ['three', 'two', 'four', 'one', 'one2']\u000a&gt;&gt;&gt; list1, list2 = zip(*sorted(zip(list1, list2)))\u000a&gt;&gt;&gt; list1\u000a(1, 1, 2, 3, 4)\u000a&gt;&gt;&gt; list2 \u000a('one', 'one2', 'two', 'three', 'four')\u000a</code></pre>\u000a\u000a<p>These of course are no longer lists, but that's easily remedied, if it matters:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; list1, list2 = (list(t) for t in zip(*sorted(zip(list1, list2))))\u000a&gt;&gt;&gt; list1\u000a[1, 1, 2, 3, 4]\u000a&gt;&gt;&gt; list2\u000a['one', 'one2', 'two', 'three', 'four']\u000a</code></pre>\u000a\u000a<p>It's worth noting that the above may sacrifice speed for terseness; the in-place version, which takes up 3 lines, is a tad faster on my machine for small lists:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; %timeit zip(*sorted(zip(list1, list2)))\u000a100000 loops, best of 3: 3.3 us per loop\u000a&gt;&gt;&gt; %timeit tups = zip(list1, list2); tups.sort(); zip(*tups)\u000a100000 loops, best of 3: 2.84 us per loop\u000a</code></pre>\u000a\u000a<p>On the other hand, for larger lists, the one-line version could be faster:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; %timeit zip(*sorted(zip(list1, list2)))\u000a100 loops, best of 3: 8.09 ms per loop\u000a&gt;&gt;&gt; %timeit tups = zip(list1, list2); tups.sort(); zip(*tups)\u000a100 loops, best of 3: 8.51 ms per loop\u000a</code></pre>\u000a\u000a<p>As Quantum7 points out, <a href="http://stackoverflow.com/a/9764390/577088">JSF's suggestion</a> is a bit faster still, but it will probably only ever be a little bit faster, because Python uses the <a href="https://hg.python.org/cpython/file/e4c32152b25b/Objects/listobject.c#l1863" rel="noreferrer">very same DSU idiom internally</a> for all key-based sorts. It's just happening a little closer to the bare metal. (This shows just how well optimized the <code>zip</code> routines are!)</p>\u000a\u000a<p>I think the <code>zip</code>-based approach is more flexible and is a little more readable, so I prefer it.</p>\u000a

V<p>I think the issue has gotten confused regarding what you want. I imagine you're not actually trying to put the HTML in the JSON response, but rather want to alternatively return either HTML or JSON.</p>\u000a\u000a<p>First, you need to understand the core difference between the two. HTML is a presentational format. It deals more with how to display data than the data itself. JSON is the opposite. It's pure data -- basically a JavaScript representation of some Python (in this case) dataset you have. It serves as merely an interchange layer, allowing you to move data from one area of your app (the view) to another area of your app (your JavaScript) which normally don't have access to each other.</p>\u000a\u000a<p>With that in mind, you don't "render" JSON, and there's no templates involved. You merely convert whatever data is in play (most likely pretty much what you're passing as the context to your template) to JSON. V<p>Consider the following situation:</p>\u000a\u000a<pre><code>class A(object):\u000a    def __init__(self):\u000a        print('Running A.__init__')\u000a        super(A,self).__init__()\u000aclass B(A):\u000a    def __init__(self):\u000a        print('Running B.__init__')        \u000a        # super(B,self).__init__()\u000a        A.__init__(self) \u000a\u000aclass C(A):\u000a    def __init__(self):\u000a        print('Running C.__init__')\u000a        super(C,self).__init__()\u000aclass D(B,C):\u000a    def __init__(self):\u000a        print('Running D.__init__')\u000a        super(D,self).__init__()\u000a\u000afoo=D()\u000a</code></pre>\u000a\u000a<p>So the classes form a so-called inheritance diamond:</p>\u000a\u000a<pre><code>    A\u000a   / \u005c\u000a  B   C\u000a   \u005c /\u000a    D\u000a</code></pre>\u000a\u000a<p>Running the code yields</p>\u000a\u000a<pre><code>Running D.__init__\u000aRunning B.__init__\u000aRunning A.__init__\u000a</code></pre>\u000a\u000a<p>That's bad because <code>C</code>'s <code>__init__</code> is skipped. The reason for that is  because <code>B</code>'s <code>__init__</code> calls <code>A</code>'s <code>__init__</code> directly. </p>\u000a\u000a<p><strong>The purpose of <code>super</code> is to resolve inheritance diamonds</strong>. If you un-comment</p>\u000a\u000a<pre><code># super(B,self).__init__()\u000a</code></pre>\u000a\u000a<p>and comment-out</p>\u000a\u000a<pre><code>A.__init__(self) \u000a</code></pre>\u000a\u000a<p>the code yields the more desireable result:</p>\u000a\u000a<pre><code>Running D.__init__\u000aRunning B.__init__\u000aRunning C.__init__\u000aRunning A.__init__\u000a</code></pre>\u000a\u000a<p>Now all the <code>__init__</code> methods get called. Notice that at the time you define <code>B.__init__</code> you might <em>think</em> that <code>super(B,self).__init__()</code> is the same as calling <code>A.__init__(self)</code>, but you'd be wrong. In the above situation, <code>super(B,self).__init__()</code> actually calls <code>C.__init__(self)</code>. </p>\u000a\u000a<p>Holy smokes,   <code>B</code> knows nothing about <code>C</code>, and yet <code>super(B,self)</code> knows to call <code>C</code>'s <code>__init__</code>? The reason is because <code>self.__class__.mro()</code> contains <code>C</code>. In other words, <code>self</code> (or in the above, <code>foo</code>) knows about <code>C</code>.</p>\u000a\u000a<p>So be careful -- the two are not fungible. They can yield vastly different results.</p>\u000a\u000a<p>Using <code>super</code> <a href="http://fuhm.org/super-harmful/" rel="noreferrer">has pitfalls.</a> It takes a considerable level of coordination between all the classes in the inheritance diagram. (They must, for example, either have the same call signature for <code>__init__</code>, since any particular <code>__init__</code> would not know which other <code>__init__</code> <code>super</code> might call next, or \u000aelse <a href="http://rhettinger.wordpress.com/2011/05/26/super-considered-super/" rel="noreferrer">use <code>**kwargs</code></a>.) Furthermore, you must be consistent about using <code>super</code> everywhere. Skip it once (as in the above example) and you defeat the entire purpose of <code>super</code>.\u000aSee the link for more pitfalls.</p>\u000a\u000a<p>If you have full control over your class hierarchy, or you avoid inheritance diamonds, then there is no need for <code>super</code>.</p>\u000a

V<p><code>unicode</code>, which is python 3's <code>str</code>, is meant to handle <em>text</em>. Text is a sequence of <strong>code points</strong> which <em>may be bigger than a single byte</em>. Text can be <em>encoded</em> in a specific encoding to represent the text as raw bytes(e.g. <code>utf-8</code>, <code>latin-1</code>...).\u000aNote that <code>unicode</code> <em>is not encoded</em>! The internal representation used by python is an implementation detail, and you shouldn't care about it as long as it is able to represent the code points you want.</p>\u000a\u000a<p>On the contrary <code>str</code> is a plain sequence of <em>bytes</em>. It does not represent text! In fact, in python 3 <code>str</code> is called <code>bytes</code>.</p>\u000a\u000a<p>You can think of <code>unicode</code> as a general representation of some text, which can be encoded in many different ways into a sequence of binary data represented via <code>str</code>.</p>\u000a\u000a<p>Some differences that you can see:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; len(u'à')  # a single code point\u000a1\u000a&gt;&gt;&gt; len('à')   # by default utf-8 -&gt; takes two bytes\u000a2\u000a&gt;&gt;&gt; len(u'à'.encode('utf-8'))\u000a2\u000a&gt;&gt;&gt; len(u'à'.encode('latin1'))  # in latin1 it takes one byte\u000a1\u000a&gt;&gt;&gt; print u'à'.encode('utf-8')  # terminal encoding is utf-8\u000aà\u000a&gt;&gt;&gt; print u'à'.encode('latin1') # it cannot understand the latin1 byte\u000a\ufffd\u000a</code></pre>\u000a\u000a<p>Note that using <code>str</code> you have a lower-level control on the single bytes of a specific encoding representation, while using <code>unicode</code> you can only control at the code-point level. For example you can do:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; 'àèìòù'\u000a'\u005cxc3\u005cxa0\u005cxc3\u005cxa8\u005cxc3\u005cxac\u005cxc3\u005cxb2\u005cxc3\u005cxb9'\u000a&gt;&gt;&gt; print 'àèìòù'.replace('\u005cxa8', '')\u000aà\ufffdìòù\u000a</code></pre>\u000a\u000a<p>What before was valid UTF-8, isn't anymore. Using a unicode string you cannot operate in such a way that the resulting string isn't valid unicode text.\u000aYou can remove a code point, replace a code point with a different code point etc. but you cannot mess with the internal representation.</p>\u000a

V<p>Python gives the programmer a large number of tools for introspecting the running environment. <code>globals()</code> is just one of those, and it can be very useful in a debugging session to see what objects the global scope actually contains.</p>\u000a\u000a<p>The rationale behind it, I'm sure, is the same as that of using <code>locals()</code> to see the variables defined in a function, or using <code>dir</code> to see the contents of a module, or the attributes of an object.</p>\u000a\u000a<p>Coming from a C++ background, I can understand that these things seem unnecessary. In a statically linked, statically typed environment, they absolutely would be. In that case, it is known at compile time exactly what variables are global, and what members an object will have, and even what names are exported by another compilation unit.</p>\u000a\u000a<p>In a dynamic language, however, these things are not fixed; they can change depending on how code is imported, or even during run time. For that reason at least, having access to this sort of information in a debugger can be invaluable.</p>\u000a

V<p>It's a conscious design decision on Guido's part (he was even somewhat reluctant regarding the addition of the <code>collections</code> module). V<p>There are a few possible ways to do this:</p>\u000a\u000a<ul>\u000a<li>Set the environment variable <code>PYTHONPATH</code> to a colon-separated list of directories to search for imported modules.</li>\u000a<li>In your program, use <code>sys.path.append('/path/to/search')</code> to add the names of directories you want Python to search for imported modules. <a href="http://docs.python.org/library/sys.html" rel="noreferrer"><code>sys.path</code></a> is just the list of directories Python searches every time it gets asked to import a module, and you can alter it as needed (although I wouldn't recommend removing any of the standard directories!). Any directories you put in the environment variable <code>PYTHONPATH</code> will be inserted into <code>sys.path</code> when Python starts up.</li>\u000a<li>Use <a href="http://docs.python.org/library/site.html" rel="noreferrer"><code>site.addsitedir</code></a> to add a directory to <code>sys.path</code>. The difference between this and just plain appending is that when you use <code>addsitedir</code>, it also looks for <code>.pth</code> files within that directory and uses them to possibly add additional directories to <code>sys.path</code> based on the contents of the files. See the documentation for more detail.</li>\u000a</ul>\u000a\u000a<p>Which one of these you want to use depends on your situation. Remember that when you distribute your project to other users, they typically install it in such a manner that the Python code files will be automatically detected by Python's importer (i.e. packages are usually installed in the <code>site-packages</code> directory), so if you mess with <code>sys.path</code> in your code, that may be unnecessary and might even have adverse effects when that code runs on another computer. For development, I would venture a guess that setting <code>PYTHONPATH</code> is usually the best way to go.</p>\u000a\u000a<p>However, when you're using something that just runs on your own computer (or when you have nonstandard setups, e.g. sometimes in web app frameworks), it's not entirely uncommon to do something like</p>\u000a\u000a<pre><code>from os.path import dirname\u000asys.path.append(dirname(__file__))\u000a</code></pre>\u000a

V<h1>First Try</h1>\u000a\u000a<p>Using <a href="http://www.scipy.org/Weave" rel="noreferrer"><code>scipy.weave</code></a> and <a href="http://en.wikipedia.org/wiki/SSE2" rel="noreferrer">SSE2</a> intrinsics gives a marginal improvement. The first invocation is a bit slower since the code needs to be loaded from the disk and cached, subsequent invocations are faster:</p>\u000a\u000a<pre><code>import numpy\u000aimport time\u000afrom os import urandom\u000afrom scipy import weave\u000a\u000aSIZE = 2**20\u000a\u000adef faster_slow_xor(aa,bb):\u000a    b = numpy.fromstring(bb, dtype=numpy.uint64)\u000a    numpy.bitwise_xor(numpy.frombuffer(aa,dtype=numpy.uint64), b, b)\u000a    return b.tostring()\u000a\u000acode = """\u000aconst __m128i* pa = (__m128i*)a;\u000aconst __m128i* pend = (__m128i*)(a + arr_size);\u000a__m128i* pb = (__m128i*)b;\u000a__m128i xmm1, xmm2;\u000awhile (pa &lt; pend) {\u000a  xmm1 = _mm_loadu_si128(pa); // must use unaligned access \u000a  xmm2 = _mm_load_si128(pb); // numpy will align at 16 byte boundaries\u000a  _mm_store_si128(pb, _mm_xor_si128(xmm1, xmm2));\u000a  ++pa;\u000a  ++pb;\u000a}\u000a"""\u000a\u000adef inline_xor(aa, bb):\u000a    a = numpy.frombuffer(aa, dtype=numpy.uint64)\u000a    b = numpy.fromstring(bb, dtype=numpy.uint64)\u000a    arr_size = a.shape[0]\u000a    weave.inline(code, ["a", "b", "arr_size"], headers = ['"emmintrin.h"'])\u000a    return b.tostring()\u000a</code></pre>\u000a\u000a<h1>Second Try</h1>\u000a\u000a<p>Taking into account the comments, I revisited the code to find out if the copying could be avoided. Turns out I read the documentation of the string object wrong, so here goes my second try:</p>\u000a\u000a<pre><code>support = """\u000a#define ALIGNMENT 16\u000astatic void memxor(const char* in1, const char* in2, char* out, ssize_t n) {\u000a    const char* end = in1 + n;\u000a    while (in1 &lt; end) {\u000a       *out = *in1 ^ *in2;\u000a       ++in1; \u000a       ++in2;\u000a       ++out;\u000a    }\u000a}\u000a"""\u000a\u000acode2 = """\u000aPyObject* res = PyString_FromStringAndSize(NULL, real_size);\u000a\u000aconst ssize_t tail = (ssize_t)PyString_AS_STRING(res) % ALIGNMENT;\u000aconst ssize_t head = (ALIGNMENT - tail) % ALIGNMENT;\u000a\u000amemxor((const char*)a, (const char*)b, PyString_AS_STRING(res), head);\u000a\u000aconst __m128i* pa = (__m128i*)((char*)a + head);\u000aconst __m128i* pend = (__m128i*)((char*)a + real_size - tail);\u000aconst __m128i* pb = (__m128i*)((char*)b + head);\u000a__m128i xmm1, xmm2;\u000a__m128i* pc = (__m128i*)(PyString_AS_STRING(res) + head);\u000awhile (pa &lt; pend) {\u000a    xmm1 = _mm_loadu_si128(pa);\u000a    xmm2 = _mm_loadu_si128(pb);\u000a    _mm_stream_si128(pc, _mm_xor_si128(xmm1, xmm2));\u000a    ++pa;\u000a    ++pb;\u000a    ++pc;\u000a}\u000amemxor((const char*)pa, (const char*)pb, (char*)pc, tail);\u000areturn_val = res;\u000aPy_DECREF(res);\u000a"""\u000a\u000adef inline_xor_nocopy(aa, bb):\u000a    real_size = len(aa)\u000a    a = numpy.frombuffer(aa, dtype=numpy.uint64)\u000a    b = numpy.frombuffer(bb, dtype=numpy.uint64)\u000a    return weave.inline(code2, ["a", "b", "real_size"], \u000a                        headers = ['"emmintrin.h"'], \u000a                        support_code = support)\u000a</code></pre>\u000a\u000a<p>The difference is that the string is allocated inside the C code. It's impossible to have it aligned at a 16-byte-boundary as required by the SSE2 instructions, therefore the unaligned memory regions at the beginning and the end are copied using byte-wise access. </p>\u000a\u000a<p>The input data is handed in using numpy arrays anyway, because <code>weave</code> insists on copying Python <code>str</code> objects to <code>std::string</code>s. <code>frombuffer</code> doesn't copy, so this is fine, but the memory is not aligned at 16 byte, so we need to use <code>_mm_loadu_si128</code> instead of the faster <code>_mm_load_si128</code>.</p>\u000a\u000a<p>Instead of using <code>_mm_store_si128</code>, we use <code>_mm_stream_si128</code>, which will make sure that any writes are streamed to main memory as soon as possible---this way, the output array does not use up valuable cache lines. </p>\u000a\u000a<h1>Timings</h1>\u000a\u000a<p>As for the timings, the <code>slow_xor</code> entry in the first edit referred to my improved version (inline bitwise xor, <code>uint64</code>), I removed that confusion. <code>slow_xor</code> refers to the code from the original questions. V<p>I don't like repetition -- I think "DRY", "Don't Repeat Yourself", is a key programming principle.  As a consequence, I have indeed used <code>locals()</code> in similar situations. Django template rendering is far from the only situation of this kind: the general case is "a function or operator which accepts a dict, but doesn't mind if the dict contains extra entries". (For example, ordinary string formatting in Python is another such case).</p>\u000a\u000a<p>However, there's a countervailing principle: programs should be understandable in as localized a way as feasible -- that helps maintenance and refactoring (as it obviates the need to study other files to check what refactorings are acceptable). This suggests, for the <code>locals()</code> case, that it's OK if the template (or string format, etc) is a local literal (a rare case where only few variables are probably being used and thus <code>locals()</code> is not a huge win!-), but problematic in the normal case where the template lives in a different file.</p>\u000a\u000a<p>So, using <code>locals()</code>, in most cases, seriously hampers refactoring. In almost every situation in Python, local variables and their names can freely be changed as part of a local refactoring, since they have no "externally visible" effect... but using <code>locals()</code> breaks that -- suddenly you can't safely rename a variable to a different name offering better clarity, refactor code flow in a manner that removes the need for a variable, etc, etc, without each and every time studying a separate template file to check if the old name might not be needed (and possibly editing the template file, which can be non-trivial, for example if it's maintained in several different natural languages for i18n/L10n purposes).</p>\u000a\u000a<p>As a consequence, in addition to the secondary issue of performance, there is strong pressure <em>against</em> using <code>locals()</code> in "serious", "production" code -- code that does need long term maintenance and therefore easy refactoring and locality. So, when I'm "programming as best I know how", rather than "cutting corners", I'm aware I had better avoid <code>locals()</code>.</p>\u000a\u000a<p>The values that you want to have in the context in which the template is rendered are not necessarily "naturally" available as local bare-names, after all; maybe some or many of them are results of computations, items from lists or dictionaries, and the like. In this case, the temptation to "cut corners" with <code>locals()</code> is easier to avoid if you just accumulate those values into a suitable dictionary rather than assigning them local bare-names.</p>\u000a\u000a<p>It's not the easiest tradeoff, because two good principles (avoiding repetition, and having good locality) are inevitably clashing -- therefore, good question!  And not one entirely susceptible to sharp black or white answers, which is why I've tried to expand on both sides. In the end, I think it's one of those "style" aspects where a programming team might be well advised to adopt a team-uniform style guideline and stick to it -- at least it removes the need to make a decision over and over every time the issue arises, and produces a more homogeneous (and thereby maintainable) code base. [[I have to confess that this specific point has never been explicitly addressed in the style guidelines of teams I've been in, though, although many others have!-)]]</p>\u000a

V<p>SWIG generates (rather ugly) C or C++ code. It is straightforward to use for simple functions (things that can be translated directly) and reasonably easy to use for more complex functions (such as functions with output parameters that need an extra translation step to represent in Python.) For more powerful interfacing you often need to write bits of C as part of the interface file. For anything but simple use you will need to know about CPython and how it represents objects -- not hard, but something to keep in mind.</p>\u000a\u000a<p>ctypes allows you to directly access C functions, structures and other data, and load arbitrary shared libraries. You do not need to write any C for this, but you do need to understand how C works. It is, you could argue, the flip side of SWIG: it doesn't generate code and it doesn't require a compiler at runtime, but for anything but simple use it does require that you understand how things like C datatypes, casting, memory management and alignment work. You also need to manually or automatically translate C structs, unions and arrays into the equivalent ctypes datastructure, including the right memory layout.</p>\u000a\u000a<p>It is likely that in pure execution, SWIG is faster than ctypes -- because the management around the actual work is done in C at compiletime rather than in Python at runtime. However, unless you interface a lot of different C functions but each only a few times, it's unlikely the overhead will be really noticeable.</p>\u000a\u000a<p>In development time, ctypes has a much lower startup cost: you don't have to learn about interface files, you don't have to generate .c files and compile them, you don't have to check out and silence warnings. You can just jump in and start using a single C function with minimal effort, then expand it to more. And you get to test and try things out directly in the Python interpreter. Wrapping lots of code is somewhat tedious, although there are attempts to make that simpler (like ctypes-configure.)</p>\u000a\u000a<p>SWIG, on the other hand, can be used to generate wrappers for multiple languages (barring language-specific details that need filling in, like the custom C code I mentioned above.) When wrapping lots and lots of code that SWIG can handle with little help, the code generation can also be a lot simpler to set up than the ctypes equivalents.</p>\u000a
