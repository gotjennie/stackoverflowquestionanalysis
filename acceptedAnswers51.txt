V<p>As pointed out by @lgautier, there is already <a href="http://stackoverflow.com/q/5630441/1290420">another answer on this subject</a>. I have not used the other packages so this is not a comment on them, rather on the merits of rpy2 itself.</li>\u000a<li>BUT do not expect that there will be an easy way of using R in Python without learning both. I find that adding an interface between the two languages allows ease of coding when you know both, but a nightmare of debugging for someone who is deficient in one of the languages.</li>\u000a</ul>\u000a\u000a<p>My advice:</p>\u000a\u000a<ol>\u000a<li>For most applications, Python has packages that allow you to do most of the things that you want to do in R, from data wrangling to plotting. This allows you to stay within the Python environment and provides you most of the power that you need.</li>\u000a<li>At the same time, you will want R's vast range of specialised packages, so spend some time learning it in an interactive environment. I found it almost impossible to master even basic R on the command line, but <a href="http://rstudio.org/">RStudio</a> and the tutorials at <a href="http://www.statmethods.net/">Quick-R</a> and <a href="http://learnr.wordpress.com/">Learn-R</a> got me going very fast.</li>\u000a</ol>\u000a\u000a<p>Once you know both, then you will do magic with <a href="http://rpy.sourceforge.net/rpy2.html">rpy2</a> without the horrors of cross-language debugging.</p>\u000a\u000a<hr>\u000a\u000a<h2>New Resources</h2>\u000a\u000a<p><strong>Update on 29 Jan 2015</strong></p>\u000a\u000a<p>This answer has proved popular and so I thought it would be useful to point out two more recent resources:</p>\u000a\u000a<ul>\u000a<li>Ralph Heinkel gave a great talk on this subject at EuroPython 2014. The video on <a href="https://www.youtube.com/watch?v=ucJ2-5a2CAA"><em>Combining the powerful worlds of Python and R</em></a> is available on the <a href="https://www.youtube.com/channel/UCadZ6_NWdCN6YolgQdfV8Pg">EuroPython YouTube channel</a>. V<p>As my travels into the depths of PyQt5 continue, so shall I continue to update this answer with some of the shinier treasures I find. </p>\u000a\u000a<p>That being said, I am now taking a "rough draft" stab at a quick intro to PyQt5. I will also provide links to helpful resources. I am new to this framework as well, and I will elaborate on what I believe to be a good strategy for using it, as I figure that strategy out. There are likely other good strategies, so if anyone has anything to add, then please leave a comment. This is very much a work in progress.</p>\u000a\u000a<hr>\u000a\u000a<h2>Strategy</h2>\u000a\u000a<p>I've learned much from the example code as suggested in the other answer, but something the examples don't help with is PyQt5's deep magic. Frameworks with a lot of magic in them (PyQt5, Django, SQLAlchemy, ...) are great because an enormous amount of drudgery is abstracted away from you. On the flip side, it is not always clear what the hell is going on, or what you're supposed to do about it. </p>\u000a\u000a<p>Luckily, it seems we have options:</p>\u000a\u000a<ul>\u000a<li><p><a href="http://qt-project.org/doc/qt-5/designer-quick-start.html">QtDesigner</a>: For those days when your keyboard catches fire, there's a rockin' GUI-Builder called in the installation package. Yum.</p></li>\u000a<li><p><a href="http://qt-project.org/doc/qt-5/qtquick-index.html">Qt Quick</a>: The framework for QML. Model-View is an evolution of Model-View-Controller (MVC), in which the Controller has been reunited with the View. Notifies View and Controller of any changes to the underlying data. This causes updates to the display of output or available commands, respectively.</li>\u000a<li><strong><em>View</em></strong>  ~>  Displays the relevant output from the Model to the user.</li>\u000a<li><strong><em>Controller</em></strong>  ~>  Encapsulates user interaction, and notifies the Model and View of relevant events.</li>\u000a</ol>\u000a\u000a<p><strong>Model-View</strong></p>\u000a\u000a<ul>\u000a<li><a href="http://qt-project.org/doc/qt-5/graphicsview.html">The Graphics View Framework<sup>(1)</sup></a>  ~>  Represent everything (including embedded QWidgets, etc) inside a QGraphicsScene as a QGraphicsItem (or derivative thereof), including proxy classes for embedding widgets. The items are supposedly highly optimized, and integrating OpenGL support is a one-liner, which is nice. </li>\u000a</ul>\u000a\u000a<p>This design pattern puts the Controller inside the View. This way, the view is capable of handling the entirety of the user's interaction. V<p>You can assign an <a href="https://docs.python.org/3/glossary.html#term-iterable"><em>iterable</em></a> to <code>side_effect</code>, and the mock will return the next value in the sequence each time it is called:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; from unittest.mock import Mock\u000a&gt;&gt;&gt; m = Mock()\u000a&gt;&gt;&gt; m.side_effect = ['foo', 'bar', 'baz']\u000a&gt;&gt;&gt; m()\u000a'foo'\u000a&gt;&gt;&gt; m()\u000a'bar'\u000a&gt;&gt;&gt; m()\u000a'baz'\u000a</code></pre>\u000a\u000a<p>Quoting the <a href="https://docs.python.org/3/library/unittest.mock.html#unittest.mock.Mock"><code>Mock()</code> documentation</a>:</p>\u000a\u000a<blockquote>\u000a  <p>If <em>side_effect</em> is an iterable then each call to the mock will return the next value from the iterable.</p>\u000a</blockquote>\u000a\u000a<p>As an aside, the test <code>response is not 'y' or 'n' or 'yes' or 'no'</code> will <strong>not</strong> work; you are asking if the expression <code>(response is not 'y')</code> is true, or <code>'y'</code> is true (always the case, a non-empty string is always true), etc. The various expressions on either side of <code>or</code> operators are <em>independent</em>. See <a href="http://stackoverflow.com/q/15112125">How do I test one variable against multiple values?</a></p>\u000a\u000a<p>You should <em>also</em> not use <code>is</code> to test against a string. The CPython interpreter <em>may</em> reuse string objects <a href="http://stackoverflow.com/a/24245514">under certain circumstances</a>, but this is not behaviour you should count on.</p>\u000a\u000a<p>As such, use:</p>\u000a\u000a<pre><code>response not in ('y', 'n', 'yes', 'no')\u000a</code></pre>\u000a\u000a<p>instead; this will use <em>equality</em> tests (<code>==</code>) to determine if <code>response</code> references a string with the same contents (value).</p>\u000a\u000a<p>The same applies to <code>response == 'y' or 'yes'</code>; use <code>response in ('y', 'yes')</code> instead.</p>\u000a

V<p>The closest idiom for <code>*args</code> would be</p>\u000a\u000a<pre><code>function func (a, b /*, *args*/) {\u000a    var star_args = Array.prototype.slice.call (arguments, func.length);\u000a    /* now star_args[0] is the first undeclared argument */\u000a}\u000a</code></pre>\u000a\u000a<p>taking advantage of the fact that <code>Function.length</code> is the number of arguments given in the function definition.</p>\u000a\u000a<p>You could package this up in a little helper routine like</p>\u000a\u000a<pre><code>function get_star_args (func, args) {\u000a    return Array.prototype.slice.call (args, func.length);\u000a}\u000a</code></pre>\u000a\u000a<p>and then do</p>\u000a\u000a<pre><code>function func (a, b /*, *args*/) {\u000a    var star_args = get_star_args (func, arguments);\u000a    /* now star_args[0] is the first undeclared argument */\u000a}\u000a</code></pre>\u000a\u000a<p>If you're in the mood for syntactic sugar, write a function which transforms one function into another one which is called with required and optional arguments, and passes the required arguments along, with any additional optional arguments as an array in final position:</p>\u000a\u000a<pre><code>function argsify(fn){\u000a    return function(){\u000a        var args_in   = Array.prototype.slice.call (arguments);                          //args to call with\u000a        args_out.push (optional);                          //with optionals as array\u000a        return fn.apply (0, args_out);\u000a    };\u000a}\u000a</code></pre>\u000a\u000a<p>Use this as follows:</p>\u000a\u000a<pre><code>// original function\u000afunction myfunc (a, b, star_args) {\u000a     console.log (a, b, star_args[0]); Please let me know if I've misunderstood anything.</p>\u000a\u000a<p>Some general recommendation are to use a separate webserver for handling media. By separate, I mean a webserver which is not running Django. This server can be for instance:</p>\u000a\u000a<ul>\u000a<li>Lighttpd (Lighty)</li>\u000a<li>Nginx (EngineX)</li>\u000a<li>Or some other light-weight server</li>\u000a</ul>\u000a\u000a<p>Then, for Django, you can go down different paths. You can either:</p>\u000a\u000a<ul>\u000a<li><p>Serve Django via <strong>Apache</strong> and:</p>\u000a\u000a<ul>\u000a<li><p><strong>mod_python</strong></p>\u000a\u000a<p>This is the stable and recommended/well documented way. Cons: uses a lot of memory.</p></li>\u000a<li><p><strong>mod_wsgi</strong></p>\u000a\u000a<p>From what I understand, mod_wsgi is a newer alternative. It appears to be faster and easier on resources.</p></li>\u000a<li><p><strong>mod_fastcgi</strong></p>\u000a\u000a<p>When using FastCGI you are delegating the serving of Django to another process. Since mod_python includes a python interpreter in every request it uses a lot of memory. This is a way to bypass that problem. Also there is some security concerns.</p>\u000a\u000a<p>What you do is that you start your Django FastCGI server in a separate process and then configures apache via rewrites to call this process when needed.</p></li>\u000a</ul></li>\u000a</ul>\u000a\u000a<p>Or you can:</p>\u000a\u000a<ul>\u000a<li><p>Serve Django <strong>without using Apache</strong> but with another server that supports FastCGI natively:</p>\u000a\u000a<p>(The documentation mentions that you can do this if you don't have any Apache specific needs. I guess the reason must be to save memory.)</p>\u000a\u000a<ul>\u000a<li><strong>Lighttpd</strong></li>\u000a</ul>\u000a\u000a<p>This is the server that runs Youtube. It seems fast and easy to use, however i've seen reports on memoryleaks.</p>\u000a\u000a<ul>\u000a<li><strong>nginx</strong></li>\u000a</ul>\u000a\u000a<p>I've seen benchmarks claiming that this server is even faster than lighttpd. It's mostly documented in Russian though.</p></li>\u000a</ul>\u000a\u000a<p>Another thing, due to limitations in Python your server should be running in forked mode, not threaded.</p>\u000a\u000a<p>So this is my current research, but I want more opinions and experiences.</p>\u000a

V<p>This is really non-trivial.</p>\u000a\u000a<p>There are two ways to sandbox Python. V<p>IronPython classes are <em>not</em> .NET classes. They are instances of IronPython.Runtime.Types.PythonType which is the Python metaclass. This is because Python classes are dynamic and support addition and removal of methods at runtime, things you cannot do with .NET classes.</p>\u000a\u000a<p>To use Python classes in C# you will need to use the ObjectOperations class. This class allows you to operate on python types and instances in the semantics of the language itself. e.g. it uses the magic methods when appropriate, auto-promotes integers to longs etc. You can find out more about ObjectOperations by looking at the source or using reflector.</p>\u000a\u000a<p>Here is an example. V<p>For the most part, this is an attempt to look at the packaging/distribution side, not the mechanics of <code>import</code>.  Unfortunately, packaging is the place where Python provides way more than one way to do it.  I'm just trying to get the ball rolling, hopefully others will help fill what I miss or point out mistakes.  </p>\u000a\u000a<p>First of all there's some messy terminology here.  A directory containing an <code>__init__.py</code> file is a package.  However, most of what we're talking about here are specific versions of packages published on PyPI, one of it's mirrors, or in a vendor specific package management system like Debian's Apt, Redhat's Yum, Fink, Macports, Homebrew, or ActiveState's pypm.</p>\u000a\u000a<p>These published packages are what folks are trying to call "Distributions" going forward in an attempt to use "Package" only as the Python language construct.  You can see some of that usage in <a href="http://www.python.org/dev/peps/pep-0376/#how-distributions-are-installed" rel="noreferrer">PEP-376</a> <a href="http://www.python.org/dev/peps/pep-0376/#how-distributions-are-installed" rel="noreferrer">PEP-376</a>.  </p>\u000a\u000a<p>Now, your list of keywords relate to several different aspects of the Python Ecosystem:</p>\u000a\u000a<h2>Finding and publishing python distributions:</h2>\u000a\u000a<ul>\u000a<li>PyPI (aka the cheese shop)</li>\u000a<li>PyPI Mirrors</li>\u000a<li>Various package management tools / systems: apt, yum, fink, macports, homebrew</li>\u000a<li>pypm (ActiveState's alternative to PyPI)</li>\u000a</ul>\u000a\u000a<p>The above are all services that provide a place to publish Python distributions in various formats.  Some, like PyPI mirrors and apt / yum repositories can be run on your local machine or within your companies network but folks typically use the official ones.  Most, if not all provide a tool (or multiple tools in the case of PyPI) to help find and download distributions.</p>\u000a\u000a<h2>Libraries used to create and install distributions:</h2>\u000a\u000a<ul>\u000a<li><code>setuptools</code> / Distribute</li>\u000a<li><code>distutils</code></li>\u000a</ul>\u000a\u000a<p>Distutils is the standard infrastructure on which Python packages are compiled and built into distributions.  There's a ton of functionality in <code>distutils</code> but most folks just know:</p>\u000a\u000a<pre><code>from distutils.core import setup\u000a\u000asetup(name='Distutils',\u000a      version='1.0',\u000a      description='Python Distribution Utilities',\u000a      author='Greg Ward',\u000a      author_email='gward@python.net',\u000a      url='http://www.python.org/sigs/distutils-sig/',\u000a      packages=['distutils', 'distutils.command'],\u000a )\u000a</code></pre>\u000a\u000a<p>And to some extent that's a most of what you need.  With the prior 9 lines of code you have enough information to install a pure Python package and also the minimal metadata required to publish that package a distribution on PyPI.</p>\u000a\u000a<p>Setuptools provides the hooks necessary to support the Egg format and all of it's features and foibles.  Distribute is an alternative to Setuptools that adds some features while trying to be mostly backwards compatible.  I believe Distribute is going to be included in Python 3 as the successor to Distutil's <code>from distutils.core import setup</code>.</p>\u000a\u000a<p>Both Setuptools and Distribute provide a custom version of the <code>distutils</code> setup command\u000athat does useful things like support the Egg format.</p>\u000a\u000a<h2>Python Distribution Formats:</h2>\u000a\u000a<ul>\u000a<li>source</li>\u000a<li><a href="http://peak.telecommunity.com/DevCenter/PythonEggs" rel="noreferrer">eggs</a></li>\u000a</ul>\u000a\u000a<p>Distributions are typically provided either as source archives (tarball or zipfile).  The standard way to install a source distribution is by downloading and uncompressing the archive and then running the <code>setup.py</code> file inside.</p>\u000a\u000a<p>For example, the following will download, build, and install the Pygments syntax highlighting library:</p>\u000a\u000a<pre><code>curl -O -G http://pypi.python.org/packages/source/P/Pygments/Pygments-1.4.tar.gz\u000atar -zxvf Pygments-1.4.tar.gz\u000acd Pygments-1.4\u000apython setup.py build\u000asudo python setup.py install\u000a</code></pre>\u000a\u000a<p>Alternatively you can download the Egg file and install it. V<p>The following sequence of commands does remove <strong>every</strong> name from the current module:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; import sys\u000a&gt;&gt;&gt; sys.modules[__name__].__dict__.clear()\u000a</code></pre>\u000a\u000a<p>I doubt you actually DO want to do this, because "every name" includes all built-ins, so there's not much you can do after such a total wipe-out.  Remember, in Python there is really no such thing as a "variable" -- there are <em>objects</em>, of many kinds (including modules, functions, class, numbers, strings, ...), and there are <em>names</em>, bound to objects; what the sequence does is remove every name from a module (the corresponding objects go away if and only if every reference to them has just been removed).</p>\u000a\u000a<p>Maybe you want to be more selective, but it's hard to guess exactly what you mean unless you want to be more specific.  But, just to give an example:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; import sys\u000a&gt;&gt;&gt; this = sys.modules[__name__]\u000a&gt;&gt;&gt; for n in dir():\u000a...   if n[0]!='_': delattr(this, n)\u000a... \u000a&gt;&gt;&gt;\u000a</code></pre>\u000a\u000a<p>This sequence leaves alone names that are private or magical, including the <code>__builtins__</code> special name which houses all built-in names.  So, built-ins still work -- for example:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; dir()\u000a['__builtins__', '__doc__', '__name__', '__package__', 'n']\u000a&gt;&gt;&gt; \u000a</code></pre>\u000a\u000a<p>As you see, name <code>n</code> (the control variable in that <code>for</code>) also happens to stick around (as it's re-bound in the <code>for</code> clause every time through), so it might be better to name that control variable <code>_</code>, for example, to clearly show "it's special" (plus, in the interactive interpreter, name <code>_</code> is re-bound anyway after every complete expression entered at the prompt, to the value of that expression, so it won't stick around for long;-).</p>\u000a\u000a<p>Anyway, once you have determined exactly what it <em>is</em> you want to do, it's not hard to define a function for the purpose and put it in your start-up file (if you want it only in interactive sessions) or site-customize file (if you want it in every script).</p>\u000a

V<p>Choosing how to represent the inheritance is mostly a database design issue. For performance single table inheritance is usually best. From a good database design point of view, joined table inheritance is better. Joined table inheritance enables you to have foreign keys to subclasses enforced by the database, it's a lot simpler to have non-null constraints for subclass fields. Concrete table inheritance is kind of worst of both worlds.</p>\u000a\u000a<p>Single table inheritance setup with declarative looks like this:</p>\u000a\u000a<pre><code>class Building(Base):\u000a    __tablename__ = 'building'\u000a    id = Column(Integer, primary_key=True)\u000a    building_type = Column(String(32), nullable=False)\u000a    x = Column(Float, nullable=False)\u000a    y = Column(Float, nullable=False)\u000a    __mapper_args__ = {'polymorphic_on': building_type}\u000a\u000aclass Commercial(Building):\u000a    __mapper_args__ = {'polymorphic_identity': 'commercial'}\u000a    business = Column(String(50))\u000a\u000aclass Residential(Building):\u000a    __mapper_args__ = {'polymorphic_identity': 'residential'}\u000a    num_residents = Column(Integer)\u000a</code></pre>\u000a\u000a<p>To make it joined table inheritance, you'll need to add</p>\u000a\u000a<pre><code>__tablename__ = 'commercial'\u000aid = Column(None, ForeignKey('building.id'), primary_key=True)\u000a</code></pre>\u000a\u000a<p>to the subclasses.</p>\u000a\u000a<p>Querying is mostly the same with both approaches:</p>\u000a\u000a<pre><code># buildings that are within x&gt;5 and y&gt;3\u000asession.query(Building).filter((Building.x &gt; 5) &amp; (Building.y &gt; 3))\u000a# Residential buildings that have only 1 resident\u000asession.query(Residential).filter(Residential.num_residents == 1)\u000a</code></pre>\u000a\u000a<p>To control which fields are loaded you can use the <code>query.with_polymorphic()</code> method.</p>\u000a\u000a<p>The most important thing to think about using inheritance for the datamapping, is whether you actually need inheritance or can do with aggregation. Inheritance will be a pain if you will ever need to change the type of an building, or your buildings can have both commercial and residential aspects. In those cases it's usually better to have the commercial and residential aspects as related objects.</p>\u000a

V<p>On stripping assert statements: this is a standard option in the C world, where many people believe part of the definition of ASSERT is that it doesn't run in production code.  Whether stripping them out or not makes a difference depends less on how many asserts there are than on how much work those asserts do:</p>\u000a\u000a<pre><code>def foo(x):\u000a    assert x in huge_global_computation_to_check_all_possible_x_values()\u000a    # ok, go ahead and use x...\u000a</code></pre>\u000a\u000a<p>Most asserts are not like that, of course, but it's important to remember that you can do stuff like that.</p>\u000a\u000a<p>As for stripping docstrings, it does seem like a quaint holdover from a simpler time, though I guess there are memory-constrained environments where it could make a difference.</p>\u000a

V<p>There is no point in converting a data list back to a query. A query object never holds data; it just represents a query to the database. It would have to fetch everything again if you made your list to a query, and that would be redundant and very bad performance-wise.</p>\u000a\u000a<p>What you can do:</p>\u000a\u000a<ul>\u000a<li>Describe how the <code>reputation</code> field is calculated; it's probably possible to order the data in the database somehow.</li>\u000a<li>Modify the view to not require a query object. If it needs to do additional filtering etc. this should be done before any ordering, since the ordering will take less time with less entries (and less data will be fetched from the database.) So you could send the filtered query object to the sort function just before you send it to the template (which shouldn't care whether it's a query or a list.)</li>\u000a</ul>\u000a

V<p>The major guidelines are similar to any other large code project.  Apps should address a single, clearly-defined responsibility.  The name "application" is a misnomer; Django apps should be thought of more as reusable components which can be plugged together to create a real application.  Tests for each app should be contained within that app.  Apps should be decoupled from each other as much as possible, but clearly there will be dependencies, so the goal should be to keep the dependency graph as simple and sane as possible.</p>\u000a\u000a<p>I prefer to keep all the templates for a project under a single project-wide templates directory, with a subdirectory for each app (using a template subdirectory for each app is a very strong convention in Django, as it avoids template name collisions between apps).  The reason for a single project-wide templates directory is that templates, template inheritance trees, and block names can be quite project-specific, so it's hard to provide "default" app templates that can plug in to any project.  There have been some attempts to settle on standard naming conventions for base site-wide templates and the blocks they define, but I haven't seen a standard emerge yet (the way they do things over at <a href="http://pinaxproject.com" rel="noreferrer">Pinax</a> is probably the closest we have to a standard).</p>\u000a\u000a<p>Re "string externalization", if you mean i18n and l10n, Django has strong support for that and standard places where it puts the .po files - check the <a href="http://docs.djangoproject.com/en/dev/topics/i18n/#topics-i18n" rel="noreferrer">docs</a>.</p>\u000a
