V<p>This is advanced behavior which will not be needed in 90+% of the enumerations created.</p>\u000a\u000a<p>According to the docs:</p>\u000a\u000a<blockquote>\u000a  <p>The rules for what is allowed are as follows: <code>_sunder_</code> names (starting and ending with a single underscore) are reserved by enum and cannot be used; all other attributes defined within an enumeration will become members of this enumeration, with the exception of <code>__dunder__</code> names and <code>descriptors</code> (methods are also descriptors).</p>\u000a</blockquote>\u000a\u000a<p>So if you want a class constant you have several choices:</p>\u000a\u000a<ul>\u000a<li>create it in <code>__init__</code></li>\u000a<li>add it after the class has been created</li>\u000a<li>use a mixin</li>\u000a<li>create your own <code>descriptor</code></li>\u000a</ul>\u000a\u000a<p>Creating the constant in <code>__init__</code> and adding it after the class has been created both suffer from not having all the class info gathered in one place.</p>\u000a\u000a<p>Mixins can certainly be used when appropriate (<a href="http://stackoverflow.com/a/18075283/1733117">see dnozay's answer for a good example</a>), but that case can also be simplified by having a base <code>Enum</code> class with the actual constants built in.</p>\u000a\u000a<p>First, the constant that will be used in the examples below:</p>\u000a\u000a<pre><code>class Constant:  # use Constant(object) if in Python 2\u000a    def __init__(self, value):\u000a        self.value = value\u000a    def __get__(self, *args):\u000a        return self.value\u000a    def __repr__(self):\u000a        return '%s(%r)' % (self.__class__.__name__, self.value)\u000a</code></pre>\u000a\u000a<p>And the single-use Enum example:</p>\u000a\u000a<pre><code>from enum import Enum\u000a\u000aclass Planet(Enum):\u000a    MERCURY = (3.303e+23, 2.4397e6)\u000a    VENUS   = (4.869e+24, 6.0518e6)\u000a    EARTH   = (5.976e+24, 6.37814e6)\u000a    MARS    = (6.421e+23, 3.3972e6)\u000a    JUPITER = (1.9e+27,   7.1492e7)\u000a    SATURN  = (5.688e+26, 6.0268e7)\u000a    URANUS  = (8.686e+25, 2.5559e7)\u000a    NEPTUNE = (1.024e+26, 2.4746e7)\u000a\u000a    # universal gravitational constant\u000a    G = Constant(6.67300E-11)\u000a\u000a    def __init__(self, mass, radius):\u000a        self.mass = mass       # in kilograms\u000a        self.radius = radius   # in meters\u000a    @property\u000a    def surface_gravity(self):\u000a        return self.G * self.mass / (self.radius * self.radius)\u000a\u000aprint(Planet.__dict__['G'])             # Constant(6.673e-11)\u000aprint(Planet.G)                         # 6.673e-11\u000aprint(Planet.NEPTUNE.G)                 # 6.673e-11\u000aprint(Planet.SATURN.surface_gravity)    # 10.44978014597121\u000a</code></pre>\u000a\u000a<p>And, finally, the multi-use Enum example:</p>\u000a\u000a<pre><code>from enum import Enum\u000a\u000aclass AstronomicalObject(Enum):\u000a\u000a    # universal gravitational constant\u000a    G = Constant(6.67300E-11)\u000a\u000a    def __init__(self, mass, radius):\u000a        self.mass = mass\u000a        self.radius = radius\u000a    @property\u000a    def surface_gravity(self):\u000a        return self.G * self.mass / (self.radius * self.radius)\u000a\u000aclass Planet(AstronomicalObject):\u000a    MERCURY = (3.303e+23, 2.4397e6)\u000a    VENUS   = (4.869e+24, 6.0518e6)\u000a    EARTH   = (5.976e+24, 6.37814e6)\u000a    MARS    = (6.421e+23, 3.3972e6)\u000a    JUPITER = (1.9e+27,   7.1492e7)\u000a    SATURN  = (5.688e+26, 6.0268e7)\u000a    URANUS  = (8.686e+25, 2.5559e7)\u000a    NEPTUNE = (1.024e+26, 2.4746e7)\u000a\u000aclass Asteroid(AstronomicalObject):\u000a    CERES = (9.4e+20 , 4.75e+5)\u000a    PALLAS = (2.068e+20, 2.72e+5)\u000a    JUNOS = (2.82e+19, 2.29e+5)\u000a    VESTA = (2.632e+20 ,2.62e+5\u000a\u000aPlanet.MERCURY.surface_gravity    # 3.7030267229659395\u000aAsteroid.CERES.surface_gravity    # 0.27801085872576176\u000a</code></pre>\u000a\u000a<hr>\u000a\u000a<p><strong>Note</strong>:</p>\u000a\u000a<p>The <code>Constant</code> <code>G</code> really isn't.  One could rebind <code>G</code> to something else:</p>\u000a\u000a<pre><code>Planet.G = 1\u000a</code></pre>\u000a\u000a<p>If you really need it to be constant (aka not rebindable), then use the <a href="https://pypi.python.org/pypi/aenum" rel="noreferrer">new aenum library</a> [1] which will block attempts to reassign <code>constant</code>s as well as <code>Enum</code> members.</p>\u000a\u000a<hr>\u000a\u000a<p>[1] <code>aenum</code> is written by the author of <code>enum34</code>.</p>\u000a

V<p>Please refer to the doc <a href="http://pandas.pydata.org/pandas-docs/stable/indexing.html#different-choices-for-indexing" rel="noreferrer">Different Choices for Indexing</a>, it states clearly when and why you should use <strong>.loc, .iloc</strong> over <strong>.ix</strong>, it's about explicit use case:</p>\u000a\u000a<blockquote>\u000a  <p>.ix supports mixed integer and label based access. It is primarily\u000a  label based, but will fall back to integer positional access unless\u000a  the corresponding axis is of integer type. .ix is the most general and\u000a  will support any of the inputs in .loc and .iloc. .ix also supports\u000a  floating point label schemes. .ix is exceptionally useful when dealing\u000a  with mixed positional and label based hierachical indexes.</p>\u000a  \u000a  <p>However, when an axis is integer based, ONLY label based access and\u000a  not positional access is supported. Thus, in such cases, it\u2019s usually\u000a  better to be explicit and use .iloc or .loc.</p>\u000a</blockquote>\u000a\u000a<p>Hope this helps.</p>\u000a\u000a<h3>Update 22 Mar 2017</h3>\u000a\u000a<p>Thanks to comment from @Alexander, <strong>Pandas</strong> is going to deprecate <code>ix</code> in <strong>0.20</strong>, details in <a href="http://pandas-docs.github.io/pandas-docs-travis/whatsnew.html#deprecate-ix" rel="noreferrer">here</a>.</p>\u000a\u000a<p>One of the strong reason behind is because mixing indexes -- positional and label (effectively using <code>ix</code>) has been a significant source of problems for users.</p>\u000a\u000a<p>It is expected to migrate to use <code>iloc</code> and <code>loc</code> instead, here is a link on <a href="http://pandas-docs.github.io/pandas-docs-travis/indexing.html#indexing-deprecate-ix" rel="noreferrer">how to convert code</a>.</p>\u000a

V<p>To provide an answer that perhaps avoids a little of the commentary noise here, I'll try another one.</p>\u000a\u000a<p>The two pragmata in your original question really expand to:</p>\u000a\u000a<pre><code>use strict "vars";\u000ause strict "refs";\u000ause strict "subs";\u000ause warnings;\u000a</code></pre>\u000a\u000a<p>To answer each in turn:</p>\u000a\u000a<ul>\u000a<li>The effect of <code>use strict "vars"</code> is to cause a compile-time error to refer to a variable without first declaring that it exists (such as is the default in more static languages such as C, C++ and Java). Because Python does not have specific syntax to declare that a variable exists, it has no equivalent. Assigning to a name in Python always creates it if it didn't exist first. This feature of <code>strict</code> has no Python equivalent and the safety it provides cannot be recreated.</li>\u000a</ul>\u000a\u000a<p>Eg:</p>\u000a\u000a<pre><code>$ perl -c -e 'use strict "vars"; $foo = 1'\u000aGlobal symbol "$foo" requires explicit package name at -e line 1.\u000a-e had compilation errors.\u000a\u000a$ perl -c -e 'no strict "vars"; $foo = 1'\u000a-e syntax OK\u000a</code></pre>\u000a\u000a<ul>\u000a<li>The effect of <code>use strict "refs"</code> is to disallow the use of plain strings containing the name of an (existing or new) variable as a reference to the variable itself. Python does not do this so has no need to disable it.</li>\u000a</ul>\u000a\u000a<p>Eg:</p>\u000a\u000a<pre><code>$ perl -e 'use strict "refs"; ${"message"} = "hello"; print $message'\u000aCan't use string ("message") as a SCALAR ref while "strict refs" in use at -e line 1.\u000a\u000a$ perl -e 'no strict "refs"; ${"message"} = "hello"; print $message'\u000ahello\u000a</code></pre>\u000a\u000a<ul>\u000a<li>The effect of <code>use strict "subs"</code> is to cause a compile-time any attempt to call a function that is known not to exist. Python does not perform any such checking, and has no way to enable such a feature.</li>\u000a</ul>\u000a\u000a<p>Eg:</p>\u000a\u000a<pre><code>$ perl -c -e 'use strict "subs"; foo'\u000aBareword "foo" not allowed while "strict subs" in use at -e line 1.\u000a-e had compilation errors.\u000a\u000a$ perl -c -e 'no strict "subs"; foo'\u000a-e syntax OK\u000a</code></pre>\u000a\u000a<ul>\u000a<li>The effect of <code>use warnings</code> is to enable more warnings at both compile- and runtime of various categories of behaviour that was default in earlier versions, may at times be desired, or which has never been a good idea but isn't strictly an error. For example, the use of uninitialised values as numbers ought usually to give a warning, but originally it did not do so.</li>\u000a</ul>\u000a\u000a<p>Eg:</p>\u000a\u000a<pre><code>$ perl -e 'use warnings; my $u; print 2 + $u'\u000aUse of uninitialized value $u in addition (+) at -e line 1.\u000a2\u000a\u000a$ perl -e 'no warnings; my $u; print 2 + $u'\u000a2\u000a</code></pre>\u000a\u000a<p>Finally; some comments have been made that Python has similar functionality in <code>__future__</code>. However, this should not be considered similar to Perl's pragmata, as most of the latter are lexically-scoped, and can be enabled or disabled within small scopes as required; where's Python's <code>__future__</code> is only enabled for an entire source file.</p>\u000a\u000a<p>Eg.</p>\u000a\u000a<pre><code>use strict;\u000ause warnings;\u000a\u000amy $total;\u000a\u000a$total += count_things($_) for @list;\u000a\u000a{\u000a   no warnings 'uninitialized';\u000a   printf "The total is %d\u005cn", $total;\u000a}\u000a</code></pre>\u000a\u000a<p>A somewhat-contrived example, but this one demonstrates the use of <code>no warnings 'uninitialized'</code> to disable the warning about using an uninitialised value simply within the <code>printf</code> statement, while still keeping the other warnings enabled everywhere else.</p>\u000a\u000a<hr>\u000a\u000a<p>In summary then: Python does not have a <code>use strict</code> or any near-equivalent as any of the safety features it provides are either mandatory or not available in the Python language, and does not have a <code>use warnings</code>. Those features it does provide are enabled only at the file-level and cannot be selectively enabled or disabled per scope.</p>\u000a\u000a<hr>\u000a\u000a<p><em>Edit:</em> Actually I have now been informed that Python does have some controllable warning flags, that can be enabled and disabled as required.</p>\u000a

V<p><code>math.sqrt</code> is the C implementation of square root and is therefore different from using the <code>**</code> operator which implements Python's built-in <code>pow</code> function. Thus, using <code>math.sqrt</code> actually gives a different answer than using the <code>**</code> operator and there is indeed a computational reason to prefer <code>numpy</code> or <code>math</code> module implementation over the built-in. Specifically the sqrt functions are probably implemented in the most efficient way possible whereas <code>**</code> operates over a large number of bases and exponents and is probably unoptimized for the specific case of square root. On the other hand, the built-in <code>pow</code> function handles a few extra cases like "complex numbers, unbounded integer powers, and modular exponentiation".</p>\u000a\u000a<p><a href="http://stackoverflow.com/questions/842245/which-is-more-accurate-x-5-or-math-sqrtx">See this Stack Overflow question for more information on the difference  between <code>**</code> and <code>math.sqrt</code>.</a></p>\u000a\u000a<p>In terms of which is more "Pythonic", I think we need to discuss the very definition of that word. From <a href="http://docs.python.org/2/glossary.html#term-pythonic">the official Python glossary</a>, it states that a piece of code or idea is Pythonic if it "closely follows the most common idioms of the Python language, rather than implementing code using concepts common to other languages." In every single other language I can think of, there is some math module with basic square root functions. However there are languages that lack a power operator like <code>**</code> e.g. C++. So <code>**</code> is probably more Pythonic, but whether or not it's objectively better depends on the use case.</p>\u000a

V<h2><code>dir()</code> does much more than look up <code>__dict__</code></h2>\u000a\u000a<p>First of all, <code>dir()</code> is a API method that knows how to use attributes like <code>__dict__</code> to look up attributes of an object.</p>\u000a\u000a<p>Not all objects have a <code>__dict__</code> attribute though. For example, if you were to add a <a href="http://docs.python.org/2/reference/datamodel.html#slots"><code>__slots__</code> attribute</a> to your custom class, instances of that class won't have a <code>__dict__</code> attribute, yet <code>dir()</code> can still list the available attributes on those instances:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; V<p>As I said in a comment to your question, after looking at the <code>json</code> module's source code, it does not appear to lend itself to doing what you want. However the goal could be achieved by applying what is known as a <a href="https://en.wikipedia.org/wiki/Monkey-patch"><em>monkey-patch</em></a>\u000a(see question <a href="http://stackoverflow.com/questions/5626193/what-is-a-monkey-patch"><em>What is a monkey patch?</em></a>).\u000aThis could be done in your package's <code>__init__.py</code> initialization script and would affect all subsequent json serialization since modules are generally only loaded once and the result is cached in <code>sys.modules</code>.</p>\u000a\u000a<p>The patch changes the default json encoder's <code>default</code> method &mdash; the default <code>default()</code>.</p>\u000a\u000a<p>Here's an example implemented as a standalone module for simplicity's sake:</p>\u000a\u000a<p>Module: <code>make_json_serializable.py</code></p>\u000a\u000a<pre><code>""" Module that monkey-patches json module when it's imported so\u000aJSONEncoder.default() automatically checks for a special "to_json()"\u000amethod and uses it to encode the object if found.\u000a"""\u000afrom json import JSONEncoder\u000a\u000adef _default(self, obj):\u000a    return getattr(obj.__class__, "to_json", _default.default)(obj)\u000a\u000a_default.default = JSONEncoder().default  # Save unmodified default.\u000aJSONEncoder.default = _default # replacement\u000a</code></pre>\u000a\u000a<p>It's use is trivial since the patch is applied by simply importing the module.</p>\u000a\u000a<p>Sample client script:</p>\u000a\u000a<pre><code>import json\u000aimport make_json_serializable  # apply monkey-patch\u000a\u000aclass Foo(object):\u000a    def __init__(self, name):\u000a        self.name = name\u000a    def to_json(self):  # New special method.\u000a        return "{u'name': %r}" % self.name.decode('utf-8')\u000a\u000afoo = Foo('sazpaz')\u000aprint(json.dumps(foo))  # "{u'name': u'sazpaz'}"\u000a</code></pre>\u000a\u000a<p>Even better than having the replacement <code>default()</code> look for a specially named method, would be for it to be able to serialize most Python objects <em>automatically</em>, including user-defined class instances, without needing to add a special method. After researching a number of alternatives, the following seemed closest to that ideal to me:</p>\u000a\u000a<p>Module: <code>make_json_serializable2.py</code></p>\u000a\u000a<pre><code>""" Module that imports the json module and monkey-patches it so\u000aJSONEncoder.default() automatically pickles any Python objects\u000aencountered that aren't standard JSON data types.\u000a"""\u000afrom json import JSONEncoder\u000aimport pickle\u000a\u000adef _default(self, obj):\u000a    return {'_python_object': pickle.dumps(obj)}\u000a\u000aJSONEncoder.default = _default  # Replace with the above.\u000a</code></pre>\u000a\u000a<p>Of course everything can't be pickled &mdash; extension types for example. However there are ways defined to handle them via the pickle protocol by writing special methods &mdash; similar to what you suggested and I described earlier &mdash; but doing that would likely be necessary for a far fewer number of cases.</p>\u000a\u000a<p>Regardless, using the pickle protocol also means it would be fairly easy to reconstruct the original Python object by providing a custom <code>object_hook</code> function argument on any <code>json.loads()</code> calls that looked for a <code>'_python_object'</code> key in the dictionary passed in. Something like:</p>\u000a\u000a<pre><code>def as_python_object(dct):\u000a    if '_python_object' in dct:\u000a        return pickle.loads(str(dct['_python_object']))\u000a    return dct\u000a\u000apyobj = json.loads(json_str, object_hook=as_python_object)\u000a</code></pre>\u000a\u000a<p>If this has to be done in many places, it might be worthwhile to define a wrapper function that automatically supplied the extra keyword argument:</p>\u000a\u000a<pre><code>json_pkloads = functools.partial(json.loads, object_hook=as_python_object)\u000a\u000apyobj = json_pkloads(json_str)\u000a</code></pre>\u000a\u000a<p>Naturally, this could be monkey-patched it into the <code>json</code> module as well, making the function the default <code>object_hook</code> (instead of <code>None</code>).</p>\u000a\u000a<p>I got the idea for using <code>pickle</code> from an <a href="http://stackoverflow.com/a/8230373/355230">answer</a> by Raymond Hettinger to another JSON serialization question, and I consider him extremely credible as well as an official source.</p>\u000a\u000a<p>Note that this exact approach does not work as shown in Python 3 because <code>json.dumps()</code> returns a <code>bytes</code> object which the <code>JSONEncoder</code> can't handle. A simple workaround is to <code>latin1</code> "decode" the value returned from <code>pickle.dumps()</code> and then "encode" it from <code>latin1</code> before passing it on to <code>pickle.loads()</code> in the <code>as_python_object()</code> function. This works because arbitrary binary strings are valid <code>latin1</code> which can always be decoded to Unicode and then encoded back to the original string again (as pointed out in <a href="http://stackoverflow.com/a/22621777/355230">this answer</a> by Sven Marnach).</p>\u000a\u000a<p><sup>(Although the following works fine in Python 2, the decoding and encoding it does is superfluous.)</sup></p>\u000a\u000a<pre><code>class PythonObjectEncoder(json.JSONEncoder):\u000a    def default(self, obj):\u000a        return {'_python_object': pickle.dumps(obj).decode('latin1')}\u000a\u000adef as_python_object(dct):\u000a    if '_python_object' in dct:\u000a        return pickle.loads(dct['_python_object'].encode('latin1'))\u000a    return dct\u000a\u000adata = [1,2,3, set(['knights', 'who', 'say', 'ni']), {'key':'value'},\u000a        Decimal('3.14')]\u000aj = json.dumps(data, cls=PythonObjectEncoder, indent=4)\u000adata2 = json.loads(j, object_hook=as_python_object)\u000aassert data == data2  # both should be same\u000a</code></pre>\u000a

V<p>First off, if you just used <code>any(lst)</code>, then you see that <code>lst</code> is the iterable, a list of some items. If it contained <code>[0, False, '', 0.0, [], {}]</code>, all of which have boolean values of <code>False</code>, then <code>any(lst)</code> would be <code>False</code>. If <code>lst</code> contained <code>[-1, -2, 10, -4, 20]</code>, all of which evaluate to <code>True</code>, then <code>any(lst)</code> would be <code>True</code>.</p>\u000a\u000a<p>In the code you posted, <code>x &gt; 0 for x in lst</code>, this is a different kind of iterable, called a <strong>generator expression</strong>. Before generator expressions were added to Python, you would have created a <strong>list comprehension</strong>, which looks very similar, but with surrounding <code>[]</code>'s: <code>[x &gt; 0 for x in lst]</code>. From the <code>lst</code> containing <code>[-1, -2, 10, -4, 20]</code>, you would get this <em>comprehended list</em>: <code>[False, False, True, False, True]</code>. This internal value would then get passed to the <code>any</code> function, which would return <code>True</code>, since there is at least one <code>True</code> value.</p>\u000a\u000a<p>But with <strong>generator expressions</strong>, Python no longer has to create that internal list of <code>True(s)</code> and <code>False(s)</code>, the values will be generated as the <code>any</code> function iterates through the values generated one at a time by the generator expression. <em>And</em>, since <code>any</code> short-circuits, it will stop iterating as soon as it sees the first <code>True</code> value. This would be especially handy if you created <code>lst</code> using something like <code>lst = range(-1,int(1e9))</code> (or <code>xrange</code> if you are using <em>Python2.x</em>). Even though this expression will generate over a billion entries, <code>any</code> only has to go as far as the third entry when it gets to <code>1</code>, which evaluates <code>True</code> for <code>x&gt;0</code>, and so <code>any</code> can return <code>True</code>. </p>\u000a\u000a<p>If you had created a <strong>list comprehension</strong>, Python would first have had to create the billion-element list in memory, and then pass that to <code>any</code>. But by using a <strong>generator expression</strong>, you can have Python's builtin functions like <code>any</code> and <code>all</code> break out early, as soon as a <code>True</code> or <code>False</code> value is seen.</p>\u000a

V<p>you are using <code>from bar import a</code>. <code>a</code> becomes a symbol in the global scope of the importing module (or whatever scope the import statement occurs in). So when you assign a new value to <code>a</code>, you just change which value <code>a</code> points too, not the actual value. try to import <code>bar.py</code> directly with <code>import bar</code> in <code>__init__.py</code> and conduct your experiment there by setting <code>bar.a = 1</code>. This way, you are actually modifying <code>bar.__dict__['a']</code> which is the 'real' value of <code>a</code> in this context.</p>\u000a\u000a<p>It's a little convoluted with three layers but <code>bar.a = 1</code> changes the value of <code>a</code> in the module called <code>bar</code> that is actually derived from <code>__init__.py</code>. It does not change the value of <code>a</code> that <code>foobar</code> sees because <code>foobar</code> lives in the actual file <code>bar.py</code>. You could set <code>bar.bar.a</code> if you wanted to change that.</p>\u000a\u000a<p>This is one of the dangers of using the <code>from foo import bar</code> form of the <code>import</code> statement: it splits <code>bar</code> into two symbols, one visible globally from within <code>foo</code> which starts off pointing to the original value and a different symbol visible in the scope where the <code>import</code> statement is executed. Changing a where a symbol points doesn't change the value that it pointed too.</p>\u000a\u000a<p>This sort of stuff is a killer when trying to <code>reload</code> a module from the interactive interpreter.</p>\u000a

V<p>It depends a bit on which seaborn function you are using.</p>\u000a\u000a<p>The plotting functions in seaborn are broadly divided into two classes </p>\u000a\u000a<ul>\u000a<li>"Axes-level" functions, including <code>regplot</code>, <code>boxplot</code>, <code>kdeplot</code>, and many others</li>\u000a<li>"Figure-level" functions, including <code>lmplot</code>, <code>factorplot</code>, <code>jointplot</code> and one or two others</li>\u000a</ul>\u000a\u000a<p>The first group is identified by taking an explicit <code>ax</code> argument and returning an <code>Axes</code> object. As this suggests, you can use them in an "object oriented" style by passing your <code>Axes</code> to them:</p>\u000a\u000a<pre><code>f, (ax1, ax2) = plt.subplots(2)\u000asns.regplot(x, y, ax=ax1)\u000asns.kdeplot(x, ax=ax2)\u000a</code></pre>\u000a\u000a<p>Axes-level functions will only draw onto an <code>Axes</code> and won't otherwise mess with the figure, so they can coexist perfectly happily in an object-oriented matplotlib script.</p>\u000a\u000a<p>The second group of functions (Figure-level) are distinguished by the fact that the resulting plot can potentially include several Axes which are always organized in a "meaningful" way. That means that the functions need to have total control over the figure, so it isn't possible to plot, say, an <code>lmplot</code> onto one that already exists. Calling the function always initializes a figure and sets it up for the specific plot it's drawing.</p>\u000a\u000a<p>However, once you've called <code>lmplot</code>, it will return an object of the type <a href="http://stanford.edu/~mwaskom/software/seaborn/tutorial/axis_grids.html" rel="noreferrer"><code>FacetGrid</code></a>. This object has some methods for operating on the resulting plot that know a bit about the structure of the plot. It also exposes the underlying figure and array of axes at the <code>FacetGrid.fig</code> and <code>FacetGrid.axes</code> arguments. The <code>jointplot</code> function is very similar, but it uses a <a href="http://stanford.edu/~mwaskom/software/seaborn/tutorial/axis_grids.html#plotting-bivariate-data-with-jointgrid" rel="noreferrer"><code>JointGrid</code></a> object. So you can still use these functions in an object-oriented context, but all of your customization has to come after you've called the function.</p>\u000a

V<p>The other answers are looking at it from a technical point of view (i.e. what's the best way to modify a list), but I would say the (much) more important reason people are suggesting e.g. slicing is that it doesn't modify the original list.</p>\u000a\u000a<p>The reason for this in turn is that usually, the list came from somewhere. If you modify it, you can unknowningly cause very bad and hard-to-detect side effects, which can cause bugs elsewhere in the program. Or even if you don't cause a bug immediately, you'll make your program overall harder to understand and reason about, and debug.</p>\u000a\u000a<p>For example, list comprehensions/generator expressions are nice in that they never mutate the "source" list they are passed:</p>\u000a\u000a<pre><code>[x for x in lst if x != "foo"]  # creates a new list\u000a(x for x in lst if x != "foo")  # creates a lazy filtered stream\u000a</code></pre>\u000a\u000a<p>This is of course often more expensive (memory wise) because it creates a new list but a program that uses this approach is mathematically purer and easier to reason about. And with lazy lists (generators and generator expressions), even the memory overhead will disappear, and computations are only executed on demand; see <a href="http://www.dabeaz.com/generators/" rel="nofollow noreferrer">http://www.dabeaz.com/generators/</a> for an awesome introduction. And you should not think too much about optimization when designing your program (see <a href="https://softwareengineering.stackexchange.com/questions/80084/is-premature-optimization-really-the-root-of-all-evil">https://softwareengineering.stackexchange.com/questions/80084/is-premature-optimization-really-the-root-of-all-evil</a>). Also, removing an item from a list is quite expensive, unless it's a linked list (which Python's <code>list</code> isn't; for linked list, see <code>collections.deque</code>).</p>\u000a\u000a<hr>\u000a\u000a<p>In fact, side-effect free functions and <a href="https://en.wikipedia.org/wiki/Immutable_object" rel="nofollow noreferrer">immutable</a> <a href="http://www.slideshare.net/adorepump/functional-programming-with-immutable-data-structures" rel="nofollow noreferrer">data structures</a> are the basis of <strong>Functional Programming</strong>, a very powerful programming paradigm.</p>\u000a\u000a<p>However, under certain circumstances, it's OK to modify a data structure in place (even in FP, <a href="https://en.wikipedia.org/wiki/Side_effect_(computer_science)" rel="nofollow noreferrer">if the language allows it</a>), such as when it's a locally created one, or copied from the function's input:</p>\u000a\u000a<pre><code>def sorted(lst):\u000a    ret = list(lst)  # make a copy\u000a    # mutate ret\u000a    return ret\u000a</code></pre>\u000a\u000a<p>\u2014 this function appears to be a pure function from the outside because it doesn't modify its inputs (and also only depends on its arguments and nothing else (i.e. it has no (global) state), which is another requirement for something to be a <a href="https://en.wikipedia.org/wiki/Pure_function" rel="nofollow noreferrer">Pure</a> <a href="https://en.wikipedia.org/wiki/Purely_functional" rel="nofollow noreferrer">Function</a>).</p>\u000a\u000a<p>So as long as you know what you're doing, <code>del</code> is by no means bad; but use any sort of data mutation with extreme care and only when you have to. Always start out with a possibly less efficient but more correct and mathematically elegant code.</p>\u000a\u000a<p>...and learn <a href="http://en.m.wikipedia.org/wiki/Functional_programming" rel="nofollow noreferrer">Functional Programming</a> :)</p>\u000a\u000a<p><em>P.S. note that <code>del</code> can also be used to delete local variables and thus eliminate references to objects in memory, which is often useful for whatever GC related purposes.</em></p>\u000a\u000a<hr>\u000a\u000a<p><strong>Answer to your second question:</strong></p>\u000a\u000a<p>As to the second part of your question about <em><code>del</code> removing objects completely</em> \u2014 that's not the case: in fact in Python, it is not even possible to tell the interpreter/VM to remove an object from memory because Python is a garbage collected language (like Java, C#, Ruby, Haskell etc) and it's the runtime that decides what to remove and when.</p>\u000a\u000a<p>Instead, what <code>del</code> does when called on a variable (as opposed to a dictionary key or list item) like this:</p>\u000a\u000a<pre><code>del a\u000a</code></pre>\u000a\u000a<p>is that it <em>only</em> removes the local (or global) variable and <em>not</em> what the variable points to (every variable in Python holds a pointer/reference to its contents not the content itself). In fact, since locals and globals are stored as a dictionary under the hood (see <a href="https://docs.python.org/2/library/functions.html#locals" rel="nofollow noreferrer"><code>locals()</code></a> and <a href="https://docs.python.org/2/library/functions.html#globals" rel="nofollow noreferrer"><code>globals()</code></a>), <code>del a</code> is equivalent to:</p>\u000a\u000a<pre><code>del locals()['a']\u000a</code></pre>\u000a\u000a<p>or <code>del globals()['a']</code> when applied to a global. </p>\u000a\u000a<p>so if you have:</p>\u000a\u000a<pre><code>a = []\u000ab = a\u000a</code></pre>\u000a\u000a<p>you're making a list, storing a reference to it in <code>a</code> and then making another copy of that reference and storing it into <code>b</code> without copying/touching the list object itself. Therefore, these two calls affect one and the same object:</p>\u000a\u000a<pre><code>a.append(1)\u000ab.append(2)\u000a # the list will be [1, 2]\u000a</code></pre>\u000a\u000a<p>whereas deleting <code>b</code> is in no way related to touching what <code>b</code> points to:</p>\u000a\u000a<pre><code>a = []\u000ab = a\u000adel b\u000a# a is still untouched and points to a list\u000a</code></pre>\u000a\u000a<p>Also, even when you call <code>del</code> on an object attribute (e.g. <code>del self.a</code>), you're still actually modifying a dictionary <code>self.__dict__</code> just like you are actually modifying <code>locals()</code>/<code>globals()</code> when you do <code>del a</code>.</p>\u000a\u000a<p>P.S. as Sven Marcnah has pointed out that <code>del locals()['a']</code> does not actually delete the local variable <code>a</code> when inside a function, which is correct. This is probably due to <code>locals()</code> returning a copy of the actual locals. However, the answer is still generally valid.</p>\u000a

V<p>The following are rough guidelines and educated guesses based on experience. You should <code>timeit</code> or profile your concrete use case to get hard numbers, and those numbers may occasionally disagree with the below.</p>\u000a\u000a<p>A list comprehension is usually a tiny bit faster than the precisely equivalent <code>for</code> loop (that actually builds a list), most likely because it doesn't have to look up the list and its <code>append</code> method on every iteration. However, a list comprehension still does a bytecode-level loop:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; dis.dis(&lt;the code object for `[x for x in range(10)]`&gt;)\u000a 1           0 BUILD_LIST               0\u000a             3 LOAD_FAST                0 (.0)\u000a       &gt;&gt;    6 FOR_ITER                12 (to 21)\u000a             9 STORE_FAST               1 (x)\u000a            12 LOAD_FAST                1 (x)\u000a            15 LIST_APPEND              2\u000a            18 JUMP_ABSOLUTE            6\u000a       &gt;&gt;   21 RETURN_VALUE\u000a</code></pre>\u000a\u000a<p>Using a list comprehension in place of a loop that <em>doesn't</em> build a list, nonsensically accumulating a list of meaningless values and then throwing the list away, is often <em>slower</em> because of the overhead of creating and extending the list. List comprehensions aren't magic that is inherently faster than a good old loop.</p>\u000a\u000a<p>As for functional list processing functions: While these are written in C and probably outperform equivalent functions written in Python, they are <em>not</em> necessarily the fastest option. Some speed up is expected <strong>if</strong> the function is written in C too. But most cases using a <code>lambda</code> (or other Python function), the overhead of repeatedly setting up Python stack frames etc. eats up any savings. Simply doing the same work in-line, without function calls (e.g. a list comprehension instead of <code>map</code> or <code>filter</code>) is often slightly faster.</p>\u000a\u000a<blockquote>\u000a  <p>Suppose that in a game that I'm developing I need to draw complex and huge maps using for loops. This question would be definitely relevant, for if a list-comprehension, for example, is indeed faster, it would be a much better option in order to avoid lags (Despite the visual complexity of the code).</p>\u000a</blockquote>\u000a\u000a<p>Chances are, if code like this isn't already fast enough when written in good non-"optimized" Python, no amount of Python level micro optimization is going to make it fast enough and you should start thinking about dropping to C. While extensive micro optimizations can often speed up Python code considerably, there is a low (in absolute terms) limit to this. Moreover, even before you hit that ceiling, it becomes simply more cost efficient (15% speedup vs. 300% speed up with the same effort) to bite the bullet and write some C.</p>\u000a

V<p>If you don't use the results of your yields,* you can <em>always</em> turn this:</p>\u000a\u000a<pre><code>yield from foo:\u000a</code></pre>\u000a\u000a<p>\u2026 into this:</p>\u000a\u000a<pre><code>for bar in foo:\u000a    yield bar\u000a</code></pre>\u000a\u000a<p>There might be a performance cost,** but there is never a semantic difference.</p>\u000a\u000a<hr>\u000a\u000a<blockquote>\u000a  <p>Are the entries from the two halves (upper and lower) appended to each other in one list so that the parent recursion function with the yield from call and use both the halves together?</p>\u000a</blockquote>\u000a\u000a<p>No! The whole point of iterators and generators is that you don't build actual lists and append them together.</p>\u000a\u000a<p>But the <em>effect</em> is similar: you just yield from one, then yield from another.</p>\u000a\u000a<p>If you think of the upper half and the lower half as "lazy lists", then yes, you can think of this as a "lazy append" that creates a larger "lazy list". And if you call <code>list</code> on the result of the parent function, you of course <em>will</em> get an actual <code>list</code> that's equivalent to appending together the two lists you would have gotten if you'd done <code>yield list(\u2026)</code> instead of <code>yield from \u2026</code>.</p>\u000a\u000a<p>But I think it's easier to think of it the other way around: What it does is exactly the same the <code>for</code> loops do.</p>\u000a\u000a<p>If you saved the two iterators into variables, and looped over <code>itertools.chain(upper, lower)</code>, that would be the same as looping over the first and then looping over the second, right? No difference here. In fact, you could implement <code>chain</code> as just:</p>\u000a\u000a<pre><code>for arg in *args:\u000a    yield from arg\u000a</code></pre>\u000a\u000a<hr>\u000a\u000a<p>* Not the values the generator yields to its caller, the value of the yield expressions themselves, within the generator (which come from the caller using the <code>send</code> method), as described in <a href="http://www.python.org/dev/peps/pep-0342/" rel="noreferrer">PEP 342</a>. You're not using these in your examples. And I'm willing to bet you're not in your real code. But coroutine-style code often uses the value of a <code>yield from</code> expression\u2014see <a href="http://www.python.org/dev/peps/pep-3156/" rel="noreferrer">PEP 3156</a> for examples. Such code usually depends on other features of Python 3.3 generators\u2014in particular, the new <code>StopIteration.value</code> from the same <a href="http://www.python.org/dev/peps/pep-0380/" rel="noreferrer">PEP 380</a> that introduced <code>yield from</code>\u2014so it will have to be rewritten. But if not, you can use the PEP also shows you the complete horrid messy equivalent, and you can of course pare down the parts you don't care about. And if you don't use the value of the expression, it pares down to the two lines above.</p>\u000a\u000a<p>** Not a huge one, and there's nothing you can do about it short of using Python 3.3 or completely restructuring your code. It's exactly the same case as translating list comprehensions to Python 1.5 loops, or any other case when there's a new optimization in version X.Y and you need to use an older version.</p>\u000a

V<p>As has been guessed in the comments, I can confirm that the processing of the array is being done in chunks. First, I will show you where things are in the code and then I will show you how you can change the chunk size and the effect that doing so has on your benchmark.</p>\u000a\u000a<h2>Where to find the reduction processing in the Numpy source files</h2>\u000a\u000a<p>np.all(x) is the same as x.all(). all() really calls np.core.umath.logical_and.reduce(x). </p>\u000a\u000a<p>If you want to dig into the numpy source, I will try to guide you through finding that a buffer/chunk size is used. The folder with all of the code we will be looking at is numpy/core/src/umath/. </p>\u000a\u000a<p>PyUFunc_Reduce() in ufunc_object.c is the C function that handles the reduce. In PyUFunc_Reduce(), the chunk, or buffer, size is found by looking up the value for reduce in some global dictionary via the PyUFunc_GetPyValues() function (ufunc_object.c). On my machine and compiling from the development branch, the chunk size is 8192. PyUFunc_ReduceWrapper() in reduction.c is called to set-up the iterator (with a stride equal to the chunk size) and it calls the passed in loop function which is reduce_loop() in ufunc_object.c.</p>\u000a\u000a<p>reduce_loop() basically just uses the iterator and calls another innerloop() function for each chunk. V<p>"Native threads" are separate contexts of execution, managed by the operating system kernel, accessing a shared memory space and potentially executing concurrently on separate cores. Compare this with separate processes, which may execute concurrently on multiple cores but have separate memory spaces. Making sure that processes interact nicely is easy since they can only communicate with each other via the kernel. Ensuring that threads don't interact in unpredictable, buggy ways is very hard since they can read and write to the same memory in an unrestricted manner.</p>\u000a\u000a<p>The R situation is fairly straightforward: <a href="http://stackoverflow.com/questions/10835122/multithreading-with-r">R is not multithreaded</a>. Python is a little more complicated: Python does support threading, but due to the <a href="http://en.wikipedia.org/wiki/Global_Interpreter_Lock" rel="noreferrer">global interpreter lock (GIL)</a>, no actual concurrent execution of Python code is possible. Other popular open source dynamic languages are in various mixed states with respect to native threading (Ruby: <a href="http://stackoverflow.com/questions/56087/does-ruby-have-real-multithreading">no/kinda/yes?</a>; Node.js: <a href="http://stackoverflow.com/questions/5200821/grasping-the-node-js-alternative-to-multithreading">no</a>), but in general, the answer is no, they do not support fully concurrent native threading, so Julia is not alone in this.</p>\u000a\u000a<p>When we do add shared-memory parallelism to Julia, <a href="https://github.com/JuliaLang/julia/issues/1790" rel="noreferrer">as we plan to</a> \u2013 whether using native threads or multiple processes with shared memory \u2013it will be true concurrency and there will be no GIL preventing simultaneous execution of Julia code. However, this is an incredibly tricky feature to add to a language, as attested by the non-existent or limited support in other very popular, mature dynamic languages. Adding a shared-memory concurrency model is technically difficult, but the real problem is designing a programming model that will allow programmers to make effective use of hardware concurrency in a productive and safe way. This problem is generally unsolved and is a very active area of research and experimentation \u2013 there is no "gold standard" to copy. We could just add POSIX threads support, but that programming model is general considered to be dangerous and incredibly difficult to use correctly and effectively. Go has an excellent concurrency story, but it is designed for writing highly concurrent servers, not for concurrently operating on large data, so it's not at all clear that simply copying Go's model is a good idea for Julia.</p>\u000a

V<p>What you are looking for is a Producer/Consumer pattern</p>\u000a\u000a<p><strong>Basic threading example</strong></p>\u000a\u000a<p>Here is a basic example using the <a href="http://docs.python.org/library/threading.html" rel="noreferrer">threading module</a> (instead of multiprocessing)</p>\u000a\u000a<pre><code>import threading\u000aimport Queue\u000aimport sys\u000a\u000adef do_work(in_queue, out_queue):\u000a    while True:\u000a        item = in_queue.get()\u000a        # process\u000a        result = item\u000a        out_queue.put(result)\u000a        in_queue.task_done()\u000a\u000aif __name__ == "__main__":\u000a    work = Queue.Queue()\u000a    results = Queue.Queue()\u000a    total = 20\u000a\u000a    # start for workers\u000a    for i in xrange(4):\u000a        t = threading.Thread(target=do_work, args=(work, results))\u000a        t.daemon = True\u000a        t.start()\u000a\u000a    # produce data\u000a    for i in xrange(total):\u000a        work.put(i)\u000a\u000a    work.join()\u000a\u000a    # get the results\u000a    for i in xrange(total):\u000a        print results.get()\u000a\u000a    sys.exit()\u000a</code></pre>\u000a\u000a<p>You wouldn't share the file object with the threads. You would produce work for them by supplying the <a href="http://docs.python.org/library/queue.html" rel="noreferrer">queue</a> with line of data. Then each thread would pick it up and process it, then return it in the queue. </p>\u000a\u000a<p>There are some more advanced facilities built into the <a href="http://docs.python.org/library/multiprocessing.html" rel="noreferrer">multiprocessing module</a> to share data, like lists and <a href="http://docs.python.org/library/multiprocessing.html#multiprocessing.Queue" rel="noreferrer">special kind of Queue</a>. There are trade-offs to using multiprocessing vs threads and it depends on whether your work is cpu bound or IO bound. </p>\u000a\u000a<p><strong>Basic multiprocessing.Pool example</strong></p>\u000a\u000a<p>Here is a really basic example of a multiprocessing Pool</p>\u000a\u000a<pre><code>from multiprocessing import Pool\u000a\u000adef process_line(line):\u000a    return "FOO: %s" % line\u000a\u000aif __name__ == "__main__":\u000a    pool = Pool(4)\u000a    with open('file.txt') as source_file:\u000a        # chunk the work into batches of 4 lines at a time\u000a        results = pool.map(process_line, source_file, 4)\u000a\u000a    print results\u000a</code></pre>\u000a\u000a<p><a href="http://docs.python.org/library/multiprocessing.html#module-multiprocessing.pool" rel="noreferrer">A Pool</a> is a convenience object that manages its own processes. Since an open file can iterate over its lines, you can pass it to the map, which will loop over it and deliver lines to the worker function. <a href="http://docs.python.org/library/multiprocessing.html#multiprocessing.pool.multiprocessing.Pool.map" rel="noreferrer">Map</a> blocks and returns the entire result when its done. Be aware that in overly simple example, the <code>map</code> is going to consume your file all at once before dishing out work. So be aware if its larger. There are more advanced ways to design a producer/consumer setup.</p>\u000a\u000a<p><strong>Manual "pool" with limit and line re-sorting</strong></p>\u000a\u000a<p>This is a manual example of the <a href="http://docs.python.org/library/multiprocessing.html#multiprocessing.pool.multiprocessing.Pool.map" rel="noreferrer">Pool.map</a>, but instead of consuming an entire iterable, you can set a queue size so that you are only feeding it piece by piece as fast as it can process. I also added the line numbers so that you can track them and resort them if you want later on.</p>\u000a\u000a<pre><code>from multiprocessing import Process, Manager\u000aimport time\u000aimport itertools \u000a\u000adef do_work(in_queue, out_list):\u000a    while True:\u000a        item = in_queue.get()\u000a        line_no, line = item\u000a\u000a        # exit signal \u000a        if line == None:\u000a            return\u000a\u000a        # fake work\u000a        time.sleep(.5)\u000a        result = (line_no, line)\u000a\u000a        out_list.append(result)\u000a\u000a\u000aif __name__ == "__main__":\u000a    num_workers = 4\u000a\u000a    manager = Manager()\u000a    results = manager.list()\u000a    work = manager.Queue(num_workers)\u000a\u000a    # start for workers    \u000a    pool = []\u000a    for i in xrange(num_workers):\u000a        p = Process(target=do_work, args=(work, results))\u000a        p.start()\u000a        pool.append(p)\u000a\u000a    # produce data\u000a    with open("source.txt") as f:\u000a        iters = itertools.chain(f, (None,)*num_workers)\u000a        for num_and_line in enumerate(iters):\u000a            work.put(num_and_line)\u000a\u000a    for p in pool:\u000a        p.join()\u000a\u000a    # get the results\u000a    # example:  [(1, "foo"), (10, "bar"), (0, "start")]\u000a    print sorted(results)\u000a</code></pre>\u000a

V<p>Circular imports are not inherently a bad thing. It's natural for the <code>team</code> code to rely on <code>user</code> whilst the <code>user</code> does something with <code>team</code>.</p>\u000a\u000a<p>The worse practice here is <code>from module import member</code>. The <code>team</code> module is trying to get the <code>user</code> class at import-time, and the <code>user</code> module is trying to get the <code>team</code> class. But the <code>team</code> class doesn't exist yet because you're still at the first line of <code>team.py</code> when <code>user.py</code> is run.</p>\u000a\u000a<p>Instead, import only modules. This results in clearer namespacing, makes later monkey-patching possible, and solves the import problem. Because you're only importing the <em>module</em> at import-time, you don't care than the <em>class</em> inside it isn't defined yet. By the time you get around to using the class, it will be.</p>\u000a\u000a<p>So, test/users.py:</p>\u000a\u000a<pre><code>import test.teams\u000a\u000aclass User:\u000a    def setTeam(self, t):\u000a        if isinstance(t, test.teams.Team):\u000a            self.team = t\u000a</code></pre>\u000a\u000a<p>test/teams.py:</p>\u000a\u000a<pre><code>import test.users\u000a\u000aclass Team:\u000a    def setLeader(self, u):\u000a        if isinstance(u, test.users.User):\u000a            self.leader = u\u000a</code></pre>\u000a\u000a<p><code>from test import teams</code> and then <code>teams.Team</code> is also OK, if you want to write <code>test</code> less. That's still importing a module, not a module member.</p>\u000a\u000a<p>Also, if <code>Team</code> and <code>User</code> are relatively simple, put them in the same module. You don't need to follow the Java one-class-per-file idiom. The <code>isinstance</code> testing and <code>set</code> methods also scream unpythonic-Java-wart to me; depending on what you're doing you may very well be better off using a plain, non-type-checked <code>@property</code>.</p>\u000a

V<p><strong>1: Installing third party packages to hosted spaces</strong> </p>\u000a\u000a<p>You can indeed install third party packages to your hosted space. If it's a pure python package, all that's needed is to unpack it to a directory and then add that directory to your PYTHONPATH environment variable or <code>sys.path</code>.</p>\u000a\u000a<p>This can be tiring to do often, and won't work easily for compiled modules. If you have shell access to your python host, the excellent <a href="http://pypi.python.org/pypi/virtualenv" rel="noreferrer">virtualenv</a> package allows you to do set up a private python environment with its own libraries.</p>\u000a\u000a<p>To set up your virtualenv, you'll do something like this at the shell:</p>\u000a\u000a<pre><code>$ virtualenv $HOME/my_python\u000a$ $HOME/my_python/bin/easy_install numpy\u000a</code></pre>\u000a\u000a<p>You can keep running easy_install for anything else you want to install in your personal python environment.</p>\u000a\u000a<p>Now, when you write your python scripts, you will want to use your private python interpreter, if that is possible:</p>\u000a\u000a<pre><code>#!/home/myuser/my_python/bin/python\u000a\u000aimport numpy\u000a\u000a# script here\u000a</code></pre>\u000a\u000a<p>If your python env cannot be specified (such as if run by mod_wsgi), you will need to add it to the import path:</p>\u000a\u000a<pre><code>import sys\u000asys.path.insert(0, '/home/myuser/my_python/lib/python2.5/site-packages')\u000a\u000aimport numpy\u000a</code></pre>\u000a\u000a<p><strong>2: Hosting sites with numpy</strong></p>\u000a\u000a<p>I can't think of any hosting sites offhand which offer numpy pre-installed. However, <a href="http://www.dreamhost.com/" rel="noreferrer">Dreamhost</a>/Bluehost for sharedhosts provide SSH access, and with shell access you can install numpy using the methods I described above.  Any Virtual Private Server such as <a href="http://www.linode.com/" rel="noreferrer">Linode</a>/<a href="http://www.slicehost.com/" rel="noreferrer">Slicehost</a> will allow you to install whatever you desire, as well.</p>\u000a\u000a<p><strong>3: AppEngine</strong></p>\u000a\u000a<p>As mentioned above, AppEngine will not allow you to install C extensions (but pure python ones do work) so it's unlikely numpy will work for you on there, since I suspect some of its features use C speedups.</p>\u000a

V<p>pythonw.exe will run the script without a command prompt.  The problem is that the Python interpreter, Python.exe, is linked against the console subsystem to produce console output (since that's 90% of cases) -- pythonw.exe is instead linked against the GUI subsystem, and Windows will not create a console output window for it unless it asks for one.</p>\u000a\u000a<p><a href="http://www.oreillynet.com/pub/a/python/excerpts/chpt20/index.html?page=2" rel="noreferrer">This article</a> discusses GUI programming with Python, and also alludes to pythonw.exe.  It also helpfully points out that if your Python files end with <code>.pyw</code> instead of <code>.py</code>, the standard Windows installer will set up associations correctly and run your Python in pythonw.exe.</p>\u000a\u000a<p>In your case it doesn't sound like a problem, but reliance upon pythonw.exe makes your application Windows-specific -- other solutions exist to accomplish this on, say, Mac OS X.</p>\u000a

V<p>It has to do with the fact that SQLite has a page cache. According to the <a href="http://www.sqlite.org/compile.html" rel="noreferrer">Documentation</a>, the default page cache is 2000 1K pages or about 2Mb. Since this is about 75% to 90% of your data, it isn't surprising that the two number are very similar. My guess is that in addition to the SQLite page cache, the rest of the data is still in the OS disk cache. If you got SQLite to flush the page cache (and the disk cache) you would see some really significant differences.</p>\u000a

V<p>I never really understood the "Language X for Language Y developers" approach.  When I go looking to learn Language X I want to learn how to program in it the way that Language X programmers do, not the way Language Y programmers do.  I want to learn the features, idioms, etc. that are unique to the language that I am learning.  I want to be able to take advantage of the things that make the language special and use that knowledge to expand my ways of thinking and solving problems.  I don't think I would get the same sort of insights from a tutorial that was framed in the context of another language.  If you can learn your first language without a tutorial geared towards something you already know you should be able to pick up a second language the same way (and in my experience, the more languages you know the easier it is to learn new ones).</p>\u000a\u000a<p>With that said, I would recommend <a href="http://docs.python.org/tutorial/" rel="noreferrer">The Python Tutorial</a> as a good, quick, and easy way to get going with Python and <a href="http://rads.stackoverflow.com/amzn/click/1590593561" rel="noreferrer">Dive Into Python</a> as a more complete introduction, also available for free <a href="http://www.diveintopython.net/" rel="noreferrer">here</a>.  I would also agree with what others have said regarding looking at the code for the standard libraries as a source of good examples and design practices, the standard python libraries are pretty clean and easy to read.</p>\u000a
