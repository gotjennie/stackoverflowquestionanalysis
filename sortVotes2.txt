V<p>The canonical way to return multiple values in languages that support it is often <a href="http://stackoverflow.com/questions/38508/whats-the-best-way-to-return-multiple-values-from-a-function-in-python">tupling</a>. </p>\u000a\u000a<h3>Option: Using a tuple</h3>\u000a\u000a<p>Consider this trivial example:</p>\u000a\u000a<pre><code>def f(x):\u000a  y0 = x + 1\u000a  y1 = x * 3\u000a  y2 = y0 ** y3\u000a  return (y0,y1,y2)\u000a</code></pre>\u000a\u000a<p>However, this quickly gets problematic as the number of values returned increases. What if you want to return four or five values? Sure, you could keep tupling them, but it gets easy to forget which value is where. It's also rather ugly to unpack them wherever you want to receive them.</p>\u000a\u000a<h3>Option: Using a dictionary</h3>\u000a\u000a<p>The next logical step seems to be to introduce some sort of 'record notation'. In python, the obvious way to do this is by means of a <code>dict</code>. </p>\u000a\u000a<p>Consider the following:</p>\u000a\u000a<pre><code>def g(x):\u000a  y0 = x + 1\u000a  y1 = x * 3\u000a  y2 = y0 ** y3\u000a  return {'y0':y0, 'y1':y1 ,'y2':y2 }\u000a</code></pre>\u000a\u000a<p>(edit- Just to be clear, y0, y1 and y2 are just meant as abstract identifiers. As pointed out, in practice you'd use meaningful identifiers)</p>\u000a\u000a<p>Now, we have a mechanism whereby we can project out a particular member of the returned object. For example, </p>\u000a\u000a<pre><code>result['y0']\u000a</code></pre>\u000a\u000a<h3>Option: Using a class</h3>\u000a\u000a<p>However, there is another option. We could instead return a specialized structure. I've framed this in the context of Python, but I'm sure it applies to other languages as well. Indeed, if you were working in C this might very well be your only option. Here goes:</p>\u000a\u000a<pre><code>class ReturnValue(object):\u000a  def __init__(self, y0, y1, y2):\u000a     self.y0 = y0\u000a     self.y1 = y1\u000a     self.y2 = y2\u000a\u000adef g(x):\u000a  y0 = x + 1\u000a  y1 = x * 3\u000a  y2 = y0 ** y3\u000a  return ReturnValue(y0, y1, y2)\u000a</code></pre>\u000a\u000a<p>In python the previous two are perhaps very similar in terms of plumbing- After all <code>{ y0, y1, y2 }</code> just end up being entries in the internal <code>__dict__</code> of the <code>ReturnValue</code>.</p>\u000a\u000a<p>There is one additional feature provided by Python though for tiny objects, the <code>__slots__</code> attribute. The class could be expressed as:</p>\u000a\u000a<pre><code>class ReturnValue(object):\u000a  __slots__ = ["y0", "y1", "y2"]\u000a  def __init__(self, y0, y1, y2):\u000a     self.y0 = y0\u000a     self.y1 = y1\u000a     self.y2 = y2\u000a</code></pre>\u000a\u000a<p>From the <a href="http://www.network-theory.co.uk/docs/pylang/__slots__.html" rel="noreferrer">Python Reference Manual</a>:</p>\u000a\u000a<blockquote>\u000a  <p>The <code>__slots__</code> declaration takes a sequence of instance variables and reserves just enough space in each instance to hold a value for each variable. Space is saved because <code>__dict__</code> is not created for each instance.</p>\u000a</blockquote>\u000a\u000a<h3>Option: Using a list</h3>\u000a\u000a<p>Another suggestion which I'd overlooked comes from Bill the Lizard:</p>\u000a\u000a<pre><code>def h(x):\u000a  result = [x + 1]\u000a  result.append(x * 3)\u000a  result.append(y0 ** y3)\u000a  return result\u000a</code></pre>\u000a\u000a<p>This is my least favorite method though. I suppose I'm tainted by exposure to Haskell, but the idea of mixed-type lists has always felt uncomfortable to me. In this particular example the list is -not- mixed type, but it conceivably could be. A list used in this way really doesn't gain anything with respect to the tuple as far as I can tell. The only real difference between lists and tuples in Python is that lists are <a href="http://docs.python.org/library/stdtypes.html#typesseq-mutable" rel="noreferrer">mutable</a>, wheras tuples are not. I personally tend to carry over the conventions from functional programming: use lists for any number of elements of the same type, and tuples for a fixed number of elements of predetermined types.</p>\u000a\u000a<h1>Question</h1>\u000a\u000a<p>After the lengthy preamble, comes the inevitable question. Which method (do you think) is best?</p>\u000a\u000a<p>I've typically found myself going the dictionary route because it involves less set-up work. From a types perspective however, you might be better off going the class route, since that may help you avoid confusing what a dictionary represents. On the other hand, there are some in the Python community that feel <a href="http://www.canonical.org/~kragen/isinstance/" rel="noreferrer">implied interfaces should be preferred to explicit interfaces</a>, at which point the type of the object really isn't relevant, since you're basically relying on the convention that the same attribute will always have the same meaning. </p>\u000a\u000a<p>So, how do -you- return multiple values in Python?</p>\u000a
