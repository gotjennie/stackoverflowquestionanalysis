V<p>Follow these steps to change an app's name in Django:</p>\u000a\u000a<ol>\u000a<li>Rename the folder which is in your project root </li>\u000a<li>Change any references to your app in their dependencies, i.e. the app's <code>views.py</code>, <code>urls.py</code> , 'manage.py' , and <code>settings.py</code> files.</li>\u000a<li>Edit the database table <code>django_content_type</code> with the following command: <code>UPDATE django_content_type SET app_label='&lt;NewAppName&gt;' WHERE app_label='&lt;OldAppName&gt;'</code> </li>\u000a<li>Also if you have models, you will have to rename the model tables. For postgres use <code>ALTER TABLE &lt;oldAppName&gt;_modelName RENAME TO &lt;newAppName&gt;_modelName</code>. For mysql too I think it is the same (as mentioned by @null_radix)</li>\u000a<li>(For Django >= 1.7) Update the <code>django_migrations</code> table to avoid having your previous migrations re-run: <code>UPDATE django_migrations SET app='&lt;NewAppName&gt;' WHERE app='&lt;OldAppName&gt;'</code>. <strong>Note</strong>: there is some debate (in comments) if this step is required for Django 1.8+; If someone knows for sure please update here.</li>\u000a<li>If your <code>models.py</code> 's Meta Class has <code>app_name</code> listed, make sure to rename that too (mentioned by @will).</li>\u000a<li>If you've namespaced your <code>static</code> or <code>templates</code> folders inside your app, you'll also need to rename those. For example, rename <code>old_app/static/old_app</code> to <code>new_app/static/new_app</code>.</li>\u000a<li>For renaming django <code>models</code>, you'll need to change <code>django_content_type.name</code> entry in DB. For postgreSQL use <code>UPDATE django_content_type SET name='&lt;newModelName&gt;' where name='&lt;oldModelName&gt;' AND app_label='&lt;OldAppName&gt;'</code></li>\u000a</ol>\u000a\u000a<p><strong>Meta point (If using virtualenv):</strong> Worth noting, if you are renaming the directory that contains your virtualenv, there will likely be several files in your env that contain an absolute path and will also need to be updated. If you are getting errors such as <code>ImportError: No module named ...</code> this might be the culprit. (thanks to @danyamachine for providing this).</p>\u000a\u000a<p><strong>Other references:</strong> you might also want to refer the below links for a more complete picture</p>\u000a\u000a<ol>\u000a<li><a href="http://stackoverflow.com/questions/4566978/renaming-an-app-with-django-and-south">Renaming an app with Django and South</a></li>\u000a<li><a href="http://stackoverflow.com/questions/1258130/how-do-i-migrate-a-model-out-of-one-django-app-and-into-a-new-one">How do I migrate a model out of one django app and into a new one?</a></li>\u000a<li><a href="http://stackoverflow.com/questions/8408046/how-to-change-the-name-of-a-django-app?noredirect=1&amp;lq=1">How to change the name of a Django app?</a></li>\u000a<li><a href="http://stackoverflow.com/questions/5814190/backwards-migration-with-django-south?rq=1">Backwards migration with Django South</a></li>\u000a<li><a href="http://stackoverflow.com/questions/2862979/easiest-way-to-rename-a-model-using-django-south?rq=1">Easiest way to rename a model using Django/South?</a></li>\u000a</ol>\u000a

V<p><code>*x ,= p</code> is basically an obfuscated version of <code>x = list(p)</code> using <a href="https://www.python.org/dev/peps/pep-3132/" rel="noreferrer">extended iterable unpacking</a>. The comma after <code>x</code> is required to make the assignment target a tuple (it could also be a list though).</p>\u000a\u000a<p><code>*x, = p</code> <strong>is</strong> different from <code>x = p</code> because the former creates a <em>copy</em> of <code>p</code> (i.e. a new list) while the latter creates a <em>reference</em> to the original list. To illustrate:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; p = [1, 2]\u000a&gt;&gt;&gt; *x, = p \u000a&gt;&gt;&gt; x == p\u000aTrue\u000a&gt;&gt;&gt; x is p\u000aFalse\u000a&gt;&gt;&gt; x = p\u000a&gt;&gt;&gt; x == p\u000aTrue\u000a&gt;&gt;&gt; x is p\u000aTrue\u000a</code></pre>\u000a

V<p>If you're using numpy for your arrays, you can also use</p>\u000a\u000a<pre><code>x = x[numpy.logical_not(numpy.isnan(x))]\u000a</code></pre>\u000a\u000a<p>Equivalently </p>\u000a\u000a<pre><code>x = x[~numpy.isnan(x)]\u000a</code></pre>\u000a\u000a<p>[Thanks to chbrown for the added shorthand] </p>\u000a\u000a<p><strong>Explanation</strong></p>\u000a\u000a<p>The inner function, <code>numpy.isnan</code> returns a boolean/logical array which has the value <code>True</code> everywhere that <code>x</code> is not-a-number. As we want the opposite, we use the logical-not operator, <code>~</code> to get an array with <code>True</code>s everywhere that <code>x</code> <strong>is</strong> a valid number.</p>\u000a\u000a<p>Lastly we use this logical array to index into the original array <code>x</code>, to retrieve just the non-NaN values.</p>\u000a

V<p><code>r</code> is a numpy (rec)array. So <code>r["dt"] &gt;= startdate</code> is also a (boolean)\u000aarray. For numpy arrays the <code>&amp;</code> operation returns the bitwise-and of the two\u000aboolean arrays.</p>\u000a\u000a<p>The NumPy developers felt there was no one commonly understood way to evaluate\u000aan array in boolean context: it could mean <code>True</code> if <em>any</em> element is\u000a<code>True</code>, or it could mean <code>True</code> if <em>all</em> elements are <code>True</code>, or <code>True</code> if the array has non-zero length, just to name three possibilities.</p>\u000a\u000a<p>Since different users might have different needs and different assumptions, the\u000aNumPy developers refused to guess and instead decided to raise a ValueError\u000awhenever one tries to evaluate an array in boolean context. Applying <code>and</code> to\u000atwo numpy arrays causes the two arrays to be evaluated in boolean context (by\u000acalling <code>__bool__</code> in Python3 or <code>__nonzero__</code> in Python2).</p>\u000a\u000a<p>Your original code </p>\u000a\u000a<pre><code>mask = ((r["dt"] &gt;= startdate) &amp; (r["dt"] &lt;= enddate))\u000aselected = r[mask]\u000a</code></pre>\u000a\u000a<p>looks correct. However, if you do want <code>and</code>, then instead of <code>a and b</code> use <code>(a-b).any()</code> or <code>(a-b).all()</code>.</p>\u000a

V<p>From the Python grammar, we can see that <code>;</code> is not defined as <code>\u005cn</code>. The parser expects another statement after a <code>;</code>, except if there's a newline after it:</p>\u000a\u000a<pre><code>                     Semicolon w/ statement    Maybe a semicolon  Newline\u000a                          \u005c/     \u005c/               \u005c/                \u005c/\u000asimple_stmt: small_stmt (';' small_stmt)*        [';']            NEWLINE\u000a</code></pre>\u000a\u000a<p>That's why <code>x=42;;</code> doesn't work; because there isn't a statement between the two semicolons, as "nothing" isn't a statement. If there was any complete statement between them, like a <code>pass</code> or even just a <code>0</code>, the code would work.</p>\u000a\u000a<pre><code>x = 42;0; # Fine\u000ax = 42;pass; # Fine\u000ax = 42;; # Syntax error\u000a\u000aif x == 42:; print("Yes") # Syntax error - "if x == 42:" isn't a complete statement\u000a</code></pre>\u000a

V<p>The <a href="http://docs.python.org/2/reference/compound_stmts.html#function-definitions" rel="noreferrer">Python 2 documentation, <em>7.6. Function definitions</em></a> gives you a couple of ways to detect whether a caller supplied an optional parameter.</p>\u000a\u000a<p>First, you can use special formal parameter syntax <code>*</code>. If the function definition has a formal parameter preceded by a single <code>*</code>, then Python populates that parameter with any positional parameters that aren't matched by preceding formal parameters (as a tuple). If the function definition has a formal parameter preceded by <code>**</code>, then Python populates that parameter with any keyword parameters that aren't matched by preceding formal parameters (as a dict). The function's implementation can check the contents of these parameters for any "optional parameters" of the sort you want.</p>\u000a\u000a<p>For instance, here's a function <code>opt_fun</code> which takes two positional parameters <code>x1</code> and <code>x2</code>, and looks for another keyword parameter named "optional". </p>\u000a\u000a<pre><code>&gt;&gt;&gt; def opt_fun(x1, x2, *positional_parameters, **keyword_parameters):\u000a...     if ('optional' in keyword_parameters):\u000a...         print 'optional parameter found, it is ', keyword_parameters['optional']\u000a...     else:\u000a...         print 'no optional parameter, sorry'\u000a... \u000a&gt;&gt;&gt; opt_fun(1, 2)\u000ano optional parameter, sorry\u000a&gt;&gt;&gt; opt_fun(1,2, optional="yes")\u000aoptional parameter found, it is  yes\u000a&gt;&gt;&gt; opt_fun(1,2, another="yes")\u000ano optional parameter, sorry\u000a</code></pre>\u000a\u000a<p>Second, you can supply a default parameter value of some value like <code>None</code> which a caller would never use. If the parameter has this default value, you know the caller did not specify the parameter. If the parameter has a non-default value, you know it came from the caller.</p>\u000a

You use <code>select_related</code> when the object that you're going to be selecting is a single object, so <code>OneToOneField</code> or a <code>ForeignKey</code>. You use <code>prefetch_related</code> when you're going to get a "set" of things, so <code>ManyToManyField</code>s as you stated or reverse <code>ForeignKey</code>s. Just to clarify what I mean by "reverse <code>ForeignKey</code>s" here's an example:</p>\u000a\u000a<pre><code>class ModelA(models.Model):\u000a    pass\u000a\u000aclass ModelB(models.Model):\u000a    a = ForeignKey(ModelA)\u000a\u000aModelB.objects.select_related('a').all() # Forward ForeignKey relationship\u000aModelA.objects.prefetch_related('modelb_set').all() # Reverse ForeignKey relationship\u000a</code></pre>\u000a\u000a<p>The difference is that <code>select_related</code> does an SQL join and therefore gets the results back as part of the table from the SQL server. <code>prefetch_related</code> on the other hand executes another query and therefore reduces the redundant columns in the original object (<code>ModelA</code> in the above example). You may use <code>prefetch_related</code> for anything that you can use <code>select_related</code> for, it just generally doesn't make much sense since it's another query with no communications savings.</p>\u000a

In fact, I can easily trigger the exception by importing the threading module before monkey-patching threads. The following 2 lines are enough:</p>\u000a\u000a<pre><code>import threading\u000aimport gevent.monkey; gevent.monkey.patch_thread()\u000a</code></pre>\u000a\u000a<p>When executed it spits the message about ignored <code>KeyError</code>:</p>\u000a\u000a<pre><code>(env)czajnik@autosan:~$ python test.py \u000aException KeyError: KeyError(139924387112272,) in &lt;module 'threading' from '/usr/lib/python2.7/threading.pyc'&gt; ignored\u000a</code></pre>\u000a\u000a<p>If you swap the import lines, the problem is gone. </p>\u000a\u000a<h2>Long story</h2>\u000a\u000a<p>I could stop my debugging here, but I decided it's worth to understand the exact cause of the problem.</p>\u000a\u000a<p>First step was to find the code that prints the message about ignored exception. It was a little hard for me to find it (grepping for <code>Exception.*ignored</code> yielded nothing), but grepping around CPython source code I've eventually found a function called <code>void PyErr_WriteUnraisable(PyObject *obj)</code> in <a href="http://svn.python.org/projects/python/tags/r27/Python/errors.c">Python/error.c</a>, with a very interesting comment:</p>\u000a\u000a<pre><code>/* Call when an exception has occurred but there is no way for Python\u000a   to handle it. The very first call it makes is <code>wait_for_thread_shutdown()</code> - worth looking at, as we know the problem is related to threading. I decided to patch <code>threading.py</code> a bit - wrap the whole <code>_exitfunc()</code> body with <code>try</code>/<code>except</code> and print the stack trace with <a href="http://docs.python.org/library/traceback.html">traceback</a> module. The <code>_active</code> dictionary maps thread IDs (as returned by <code>_get_ident()</code>) to <code>Thread</code> instances, for all threads created. When <code>threading</code> module is loaded, an instance of <code>_MainThread</code> class is always created and added to <code>_active</code> (even if no other threads are explicitly created). </p>\u000a\u000a<p>The problem is that one of the methods patched by <code>gevent</code>'s monkey-patching is <code>_get_ident()</code> - original one maps to <code>thread.get_ident()</code>, monkey-patching replaces it with <code>green_thread.get_ident()</code>. Obviously both calls return different IDs for main thread.</p>\u000a\u000a<p>Now, if <code>threading</code> module is loaded before monkey-patching, <code>_get_ident()</code> call returns one value when <code>_MainThread</code> instance is created and added to <code>_active</code>, and another value at the time <code>_exitfunc()</code> is called - hence <code>KeyError</code> in <code>del _active[_get_ident()]</code>.</p>\u000a\u000a<p>On the contrary, if monkey-patching is done before <code>threading</code> is loaded, all is fine - at the time <code>_MainThread</code> instance is being added to <code>_active</code>, <code>_get_ident()</code> is already patched, and the same thread ID is returned at cleanup time. V<p><code>__hash__</code> should return the same value for objects that are equal. It also shouldn't change over the lifetime of the object; generally you only implement it for immutable objects.</p>\u000a\u000a<p>A trivial implementation would be to just <code>return 0</code>. This is always correct, but performs badly.</p>\u000a\u000a<p>Your solution, returning the hash of a tuple of properties, is good. But note that you don't need to list all properties that you compare in <code>__eq__</code> in the tuple. If some property usually has the same value for inequal objects, just leave it out. Don't make the hash computation any more expensive than it needs to be.</p>\u000a\u000a<p>Edit: I would recommend against using xor to mix hashes in general. When two different properties have the same value, they will have the same hash, and with xor these will cancel eachother out. Tuples use a more complex calculation to mix hashes, see <code>tuplehash</code> in <a href="http://hg.python.org/cpython/file/dcced3bd22fe/Objects/tupleobject.c" rel="noreferrer"><code>tupleobject.c</code></a>.</p>\u000a

V<p>Contrary to other answers already posted you cannot modify <code>locals()</code> directly and expect it to work.</p>\u000a\u000a<pre><code>&gt;&gt;&gt; def foo():\u000a    lcl = locals()\u000a    lcl['xyz'] = 42\u000a    print(xyz)\u000a\u000a\u000a&gt;&gt;&gt; foo()\u000a\u000aTraceback (most recent call last):\u000a  File "&lt;pyshell#6&gt;", line 1, in &lt;module&gt;\u000a    foo()\u000a  File "&lt;pyshell#5&gt;", line 4, in foo\u000a    print(xyz)\u000aNameError: global name 'xyz' is not defined\u000a</code></pre>\u000a\u000a<p>Modifying <code>locals()</code> is undefined. Outside a function when <code>locals()</code> and <code>globals()</code> are the same it will work; inside a function is will <em>usually</em> not work.</p>\u000a\u000a<p>Use a dictionary, or set an attribute on an object:</p>\u000a\u000a<pre><code>d = {}\u000ad['xyz'] = 42\u000aprint(d['xyz'])\u000a</code></pre>\u000a\u000a<p>or if you prefer, use a class:</p>\u000a\u000a<pre><code>class C: pass\u000a\u000aobj = C()\u000asetattr(obj, 'xyz', 42)\u000aprint(obj.xyz)\u000a</code></pre>\u000a\u000a<p><strong>Edit</strong>:\u000aAccess to variables in namespaces that aren't functions (so modules, class definitions, instances) are usually done by dictionary lookups (as Sven points out in the comments there are exceptions, for example classes that define <code>__slots__</code>). Function locals can be optimised for speed because the compiler (usually) knows all the names in advance, so there isn't a dictionary until you call <code>locals()</code>.</p>\u000a\u000a<p>In the C implementation of Python <code>locals()</code> (called from inside a function) creates an ordinary dictionary initialised from the current values of the local variables. Within each function any number of calls to <code>locals()</code> will return the same dictionary, but every call to <code>locals()</code> will update it with the current values of the local variables. This can give the impression that assignment to elements of the dictionary are ignored (I originally wrote that this was the case). Modifications to existing keys within the dictionary returned from <code>locals()</code> therefore only last until the next call to <code>locals()</code> in the same scope.</p>\u000a\u000a<p>In IronPython things work a bit differently. Any function that calls <code>locals()</code> inside it uses a dictionary for its local variables so assignments to local variables change the dictionary and assignments to the dictionary change the variables <strong>BUT</strong> that's only if you explicitly call <code>locals()</code> under that name. If you bind a different name to the <code>locals</code> function in IronPython then calling it gives you the local variables for the scope where the name was bound and there's no way to access the function locals through it:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; def foo():\u000a...     abc = 123\u000a...     lcl = zzz()\u000a...     lcl['abc'] = 456\u000a...     deF = 789\u000a...     print(abc)\u000a...     print(zzz())\u000a...     print(lcl)\u000a...\u000a&gt;&gt;&gt; zzz =locals\u000a&gt;&gt;&gt; foo()\u000a123\u000a{'__doc__': None, '__builtins__': &lt;module '__builtin__' (built-in)&gt;, 'zzz': &lt;built-in function locals&gt;, 'foo': &lt;function foo at 0x000000000000002B&gt;, '__name__': '__main__', 'abc': 456}\u000a{'__doc__': None, '__builtins__': &lt;module '__builtin__' (built-in)&gt;, 'zzz': &lt;built-in function locals&gt;, 'foo': &lt;function foo at 0x000000000000002B&gt;, '__name__': '__main__', 'abc': 456}\u000a&gt;&gt;&gt;\u000a</code></pre>\u000a\u000a<p>This could all change at any time. The only thing guaranteed is that you cannot depend on the results of assigning to the dictionary returned by <code>locals()</code>.</p>\u000a

I do it a lot when I want a more recent (less stable) version of something. Some packages aren't on PYPI, so same applies to them.</p>\u000a\u000a<p>Suppose you have a properly formed Python application in <code>~/src/myapp</code>. <code>~/src/myapp/setup.py</code> will have <code>install_requires</code> list that mentions one or more things that you have in your <code>/pypi</code> directory. Like so:</p>\u000a\u000a<pre><code>  install_requires=[\u000a    'boto',\u000a    'Flask',\u000a    'Werkzeug',\u000a    # and so on\u000a</code></pre>\u000a\u000a<p>If you want to be able to run your app with all the necessary dependencies while still hacking on it, you'll do something like this:</p>\u000a\u000a<pre>\u000a$ cd ~/src/myapp\u000a$ python setup.py develop --always-unzip --allow-hosts=None --find-links=/pypi\u000a</pre>\u000a\u000a<p>This way your app will be executed straight from your source directory. You can hack on things, and then rerun the app without rebuilding anything.</p>\u000a\u000a<p>If you want to install your app and its dependencies into the current python environment, you'll do something like this:</p>\u000a\u000a<pre>\u000a$ cd ~/src/myapp\u000a$ easy_install --always-unzip --allow-hosts=None --find-links=/pypi .\u000a</pre>\u000a\u000a<p>In both cases, the build will fail if one or more dependencies aren't present in <code>/pypi</code> directory. It won't attempt to promiscuously install missing things from Internet.</p>\u000a\u000a<p>I highly recommend to invoke <code>setup.py develop ...</code> and <code>easy_install ...</code> within an active <a href="http://www.virtualenv.org/en/latest/index.html">virtual environment</a> to avoid contaminating your global Python environment. It is (virtualenv that is) pretty much the way to go. Never install anything into global Python environment.</p>\u000a\u000a<p>If the machine that you've built your app has same architecture as the machine on which you want to deploy it, you can simply tarball the entire virtual environment directory into which you <code>easy_install</code>-ed everything. Just before tarballing though, you must make the virtual environment directory relocatable (see <a href="http://www.virtualenv.org/en/latest/index.html#making-environments-relocatable">--relocatable</a> option). <strong>NOTE:</strong> the destination machine needs to have the same version of Python installed, and also any C-based dependencies your app may have must be preinstalled there too (e.g. say if you depend on <a href="http://www.pythonware.com/products/pil/">PIL</a>, then libpng, libjpeg, etc must be preinstalled). </p>\u000a

Take a look at this snippet:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; class MyClass(object):\u000a...   def __init__(self):\u000a...     print 'Created MyClass@{0}'.format(id(self))\u000a... \u000a&gt;&gt;&gt; def create_instance():\u000a...   return MyClass()\u000a... \u000a&gt;&gt;&gt; x = create_instance()\u000aCreated MyClass@4299548304\u000a&gt;&gt;&gt; \u000a&gt;&gt;&gt; @mock.patch('__main__.MyClass')\u000a... def create_instance2(MyClass):\u000a... V<p>A lot of times some quick task comes up that isn't part of the main software you are developing.  Sometimes the task is one off ie compare this file to the database and let me know the differences.  It is a lot easier to do text parsing in Perl/Ruby/Python than it is in Java or C# (partially because it is a lot easier to use regular expressions).  It will probably take a lot less time to parse the text file using Perl/Ruby/Python (or maybe even vbscript <em>cringe</em> and then load it into the database than it would to create a Java/C# program to do it or to do it by hand.</p>\u000a\u000a<p>Also, due to the ease at which most of the dynamic languages parse text, they are great for code generation.  Sure your final project must be in C#/Java/Transact SQL but instead of cutting and pasting 100 times, finding errors, and cutting and pasting another 100 times it is often (but not always) easier just to use a code generator.</p>\u000a\u000a<p>A recent example at work is we needed to get data from one accounting system into our accounting system.  The system has an import format, but the old system had a completely different format (fixed width although some things had to be matched).  The task is not to create a program to migrate the data over and over again.  It is to shove the data into our system and then maintain it there going forward.  So even though we are a C# and SQL Server shop, I used Python to convert the data into the format that could be imported by our application.  Ultimately it doesn't matter that I used python, it matters that the data is in the system.  My boss was pretty impressed.</p>\u000a\u000a<p>Where I often see the dynamic languages used for is testing.  It is much easier to create a Python/Perl/Ruby program to link to a web service and throw some data against it than it is to create the equivalent Java program.  You can also use python to hit against command line programs, generate a ton of garbage (but still valid) test data, etc.. quite easily.</p>\u000a\u000a<p>The other thing that dynamic languages are big on is code generation.  Creating the C#/C++/Java code.  Some examples follow:</p>\u000a\u000a<p>The first code generation task I often see is people using dynamic languages to maintain constants in the system.  Instead of hand coding a bunch of enums, a dynamic language can be used to fairly easily parse a text file and create the Java/C# code with the enums.</p>\u000a\u000a<p>SQL is a whole other ball game but often you get better performance by cut and pasting 100 times instead of trying to do a function (due to caching of execution plans or putting complicated logic in a function causing you to go row by row instead of in a set).  In fact it is quite useful to use the table definition to create certain stored procedures automatically.</p>\u000a\u000a<p>It is always better to get buy in for a code generator.  But even if you don't, is it more fun to spend time cutting/pasting or is it more fun to create a Perl/Python/Ruby script once and then have that generate the code?  If it takes you hours to hand code something but less time to create a code generator, then even if you use it once you have saved time and hence money.  If it takes you longer to create a code generator than it takes to hand code once but you know you will have to update the code more than once, it may still make sense.  If it takes you 2 hours to hand code, 4 hours to do the generator but you know you'll have to hand code equivalent work another 5 or 6 times than it is obviously better to create the generator.</p>\u000a\u000a<p>Also some things are easier with dynamic languages than Java/C#/C/C++.  In particular regular expressions come to mind.  If you start using regular expressions in Perl and realize their value, you may suddenly start making use of the Java regular expression library if you haven't before.  If you have then there may be something else.</p>\u000a\u000a<p>I will leave you with one last example of a task that would have been great for a dynamic language.  My work mate had to take a directory full of files and burn them to various cd's for various customers.  There were a few customers but a lot of files and you had to look in them to see what they were.  He did this task by hand....A Java/C# program would have saved time, but for one time and with all the development overhead it isn't worth it.  However slapping something together in Perl/Python/Ruby probably would have been worth it.  He spent several hours doing it.  It would have taken less than one to create the Python script to inspect each file, match which customer it goes to, and then move the file to the appropriate place.....Again, not part of the standard job.  But the task came up as a one off.  Is it better to do it yourself, spend the larger amount of time to make Java/C# do the task, or spend a much smaller amount of time doing it in Python/Perl/Ruby.  If you are using C or C++ the point is even more dramatic due to the extra concerns of programming in C or C++ (pointers, no array bounds checking, etc.).</p>\u000a

V<p>In Python, everything is shared, except for function-local variables (because each function call gets its own set of locals, and threads are always separate function calls.) And even then, only the variables themselves (the names that refer to objects) are local to the function; objects themselves are always global, and anything can refer to them.\u000aThe <code>Thread</code> object for a particular thread is not a special object in this regard. If you store the <code>Thread</code> object somewhere all threads can access (like a global variable) then all threads can access that one <code>Thread</code> object. If you want to atomically modify <em>anything</em> that you didn't just create in this very same thread, and did not store anywhere another thread can get at it, you have to protect it by a lock. And all threads must of course share this very same lock, or it wouldn't be very effective.</p>\u000a\u000a<p>If you want actual thread-local storage, that's where <code>threading.local</code> comes in. Attributes of <code>threading.local</code> are not shared between threads; each thread sees only the attributes it itself placed in there. If you're curious about its implementation, the source is in <code>_threading_local.py</code> in the standard library.</p>\u000a

V<p>In production code in our company, we try to follow the following rules.</p>\u000a\u000a<p>We place imports at the beginning of the file, right after the main file's docstring, e.g.:</p>\u000a\u000a<pre><code>"""\u000aRegistry related functionality.\u000a"""\u000aimport wx\u000a# ...\u000a</code></pre>\u000a\u000a<p>Now, if we import a class that is one of few in the imported module, we import the name directly, so that in the code we only have to use the last part, e.g.:</p>\u000a\u000a<pre><code>from RegistryController import RegistryController\u000afrom ui.windows.lists import ListCtrl, DynamicListCtrl\u000a</code></pre>\u000a\u000a<p>There are modules, however, that contain dozens of classes, e.g. list of all possible exceptions. Then we import the module itself and reference to it in the code:</p>\u000a\u000a<pre><code>from main.core import Exceptions\u000a# ...\u000araise Exceptions.FileNotFound()\u000a</code></pre>\u000a\u000a<p>We use the <code>import X as Y</code> as rarely as possible, because it makes searching for usage of a particular module or class difficult. Sometimes, however, you have to use it if you wish to import two classes that have the same name, but exist in different modules, e.g.:</p>\u000a\u000a<pre><code>from Queue import Queue\u000afrom main.core.MessageQueue import Queue as MessageQueue\u000a</code></pre>\u000a\u000a<p>As a general rule, we don't do imports inside methods -- they simply make code slower and less readable. Some may find this a good way to easily resolve cyclic imports problem, but a better solution is code reorganization.</p>\u000a
