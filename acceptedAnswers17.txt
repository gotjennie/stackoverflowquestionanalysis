V<p>The order is not arbitrary, but depends on the insertion and deletion history of the dictionary or set, as well as on the specific Python implementation. For the remainder of this answer, for 'dictionary', you can also read 'set'; sets are implemented as dictionaries with just keys and no values.</p>\u000a\u000a<p>Keys are hashed, and hash values are assigned to slots in a dynamic table (it can grow or shrink based on needs). And that mapping process can lead to collisions, meaning that a key will have to be slotted in a <em>next</em> slot based on what is already there.</p>\u000a\u000a<p>Listing the contents loops over the slots, and so keys are listed in the order they <em>currently</em> reside in the table.</p>\u000a\u000a<p>Take the keys <code>'foo'</code> and <code>'bar'</code>, for example, and lets assume the table size is 8 slots. In Python 2.7, <code>hash('foo')</code> is <code>-4177197833195190597</code>, <code>hash('bar')</code> is <code>327024216814240868</code>. Modulo 8, that means these two keys are slotted in slots 3 and 4 then:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; hash('foo')\u000a-4177197833195190597\u000a&gt;&gt;&gt; hash('foo') % 8\u000a3\u000a&gt;&gt;&gt; hash('bar')\u000a327024216814240868\u000a&gt;&gt;&gt; hash('bar') % 8\u000a4\u000a</code></pre>\u000a\u000a<p>This informs their listing order:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; {'bar': None, 'foo': None}\u000a{'foo': None, 'bar': None}\u000a</code></pre>\u000a\u000a<p>All slots except 3 and 4 are empty, looping over the table first lists slot 3, then slot 4, so <code>'foo'</code> is listed before <code>'bar'</code>.</p>\u000a\u000a<p><code>bar</code> and <code>baz</code>, however, have hash values that are exactly 8 apart and thus map to the exact same slot, <code>4</code>:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; hash('bar')\u000a327024216814240868\u000a&gt;&gt;&gt; hash('baz')\u000a327024216814240876\u000a&gt;&gt;&gt; hash('bar') % 8\u000a4\u000a&gt;&gt;&gt; hash('baz') % 8\u000a4\u000a</code></pre>\u000a\u000a<p>Their order now depends on which key was slotted first; the second key will have to be moved to a next slot:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; {'baz': None, 'bar': None}\u000a{'bar': None, 'baz': None}\u000a&gt;&gt;&gt; {'bar': None, 'baz': None}\u000a{'baz': None, 'bar': None}\u000a</code></pre>\u000a\u000a<p>The table order differs here, because one or the other key was slotted first.</p>\u000a\u000a<p>The technical name for the underlying structure used by CPython (the most commonly used Python implemenation) is a <a href="http://en.wikipedia.org/wiki/hash_table" rel="noreferrer">hash table</a>, one that uses open addressing. If you are curious, and understand C well enough, take a look at the <a href="http://hg.python.org/cpython/file/tip/Objects/dictobject.c" rel="noreferrer">C implementation</a> for all the (well documented) details. You could also watch this <a href="http://pyvideo.org/video/276/the-mighty-dictionary-55" rel="noreferrer">Pycon 2010 presentation by Brandon Rhodes</a> about how CPython <code>dict</code> works, or pick up a copy of <a href="http://shop.oreilly.com/product/9780596510046.do" rel="noreferrer">Beautiful Code</a>, which includes a chapter on the implementation written by Andrew Kuchling.</p>\u000a\u000a<p>Note that as of Python 3.3, a random hash seed is used as well, making hash collisions unpredictable to prevent certain types of denial of service (where an attacker renders a Python server unresponsive by causing mass hash collisions). This means that the order of a given dictionary is then <em>also</em> dependent on the random hash seed for the current Python invocation.</p>\u000a\u000a<p>Other implementations are free to use a different structure for dictionaries, as long as they satisfy the documented Python interface for them, but I believe that all implementations so far use a variation of the hash table.</p>\u000a\u000a<p>CPython 3.6 introduces a <em>new</em> <code>dict</code> implementation that maintains insertion order, and is faster and more memory efficient to boot. Rather than keep a large sparse table where each row references the stored hash value, and the key and value objects, the new implementation adds a smaller hash <em>array</em> that only references indices in dense table (one that only contains as many rows as there are actual key-value pairs), and it is the dense table that happens to list the contained items in order. See the <a href="https://mail.python.org/pipermail/python-dev/2012-December/123028.html" rel="noreferrer">proposal to Python-Dev for more details</a>. Note that this is considered an <em>implementation detail</em>, Python-the-language does not specify that other implementations have to retain order.</p>\u000a\u000a<p>Python 2.7 and newer also provides an <a href="https://docs.python.org/2/library/collections.html#collections.OrderedDict" rel="noreferrer"><code>OrderedDict</code> class</a>, a subclass of <code>dict</code> that adds an additional data structure to record key order. At the price of some speed and extra memory, this class remembers in what order you inserted keys; listing keys, values or items will then do so in that order. It uses a doubly-linked list stored in an additional dictionary to keep the order up-to-date efficiently. See the <a href="https://mail.python.org/pipermail/python-dev/2012-December/123028.html" rel="noreferrer">post by Raymond Hettinger outlining the idea</a>. Note that the <code>set</code> type is still unordered.</p>\u000a\u000a<p>If you wanted an ordered set, you can install the <a href="https://pypi.python.org/pypi/oset" rel="noreferrer"><code>oset</code> package</a>; it works on Python 2.5 and up.</p>\u000a

V<p>The difference is that one modifies the data-structure itself (in-place operation) <code>b += 1</code> while the other just <em>reassigns</em> the variable <code>a = a + 1</code>.</p>\u000a\u000a<hr>\u000a\u000a<p>Just for completeness:</p>\u000a\u000a<p><code>x += y</code> is <strong>not always</strong> doing an in-place operation, there are (at least) three exceptions:</p>\u000a\u000a<ul>\u000a<li><p>If <code>x</code> <strong>doesn't implement</strong> an <code>__iadd__</code> method then the <code>x += y</code> statement is just a shorthand for <code>x = x + y</code>. This would be the case if <code>x</code> was something like an <code>int</code>.</p></li>\u000a<li><p>If <code>__iadd__</code> returns <code>NotImplemented</code>, Python falls back to <code>x = x + y</code>.</p></li>\u000a<li><p>The <code>__iadd__</code> method could theoretically be implemented to not work in place. It'd be really weird to do that, though.</p></li>\u000a</ul>\u000a\u000a<p>As it happens your <code>b</code>s are <code>numpy.ndarray</code>s which implements <code>__iadd__</code> and return itself so your second loop modifies the original array in-place.</p>\u000a\u000a<p>You can read more on this in the <a href="https://docs.python.org/3/reference/datamodel.html#emulating-numeric-types">Python documentation of "Emulating Numeric Types"</a>.</p>\u000a\u000a<blockquote>\u000a  <p>These [<code>__i*__</code>] methods are called to implement the augmented arithmetic assignments (<code>+=</code>, <code>-=</code>, <code>*=</code>, <code>@=</code>, <code>/=</code>, <code>//=</code>, <code>%=</code>, <code>**=</code>, <code>&lt;&lt;=</code>, <code>&gt;&gt;=</code>, <code>&amp;=</code>, <code>^=</code>, <code>|=</code>). These methods should attempt to do the operation in-place (modifying self) and return the result (which could be, but does not have to be, self). If a specific method is not defined, the augmented assignment falls back to the normal methods. For instance, if x is an instance of a class with an <code>__iadd__()</code> method, <code>x += y</code> is equivalent to <code>x = x.__iadd__(y)</code> . Otherwise, <code>x.__add__(y)</code> and <code>y.__radd__(x)</code> are considered, as with the evaluation of <code>x + y</code>. In certain situations, augmented assignment can result in unexpected errors (see <a href="https://docs.python.org/3/faq/programming.html#faq-augmented-assignment-tuple-error">Why does <code>a_tuple[i] += ["item"]</code> raise an exception when the addition works?</a>), but this behavior is in fact part of the data model.</p>\u000a</blockquote>\u000a

V<p>Numpy arrays require their length to be set explicitly at creation time, unlike python lists. This is necessary so that space for each item can be consecutively allocated in memory. Consecutive allocation is the key feature of numpy arrays: this combined with native code implementation let operations on them execute much quicker than regular lists.</p>\u000a\u000a<p>Keeping this in mind, it is technically impossible to take a generator object and turn it into an array unless you either:</p>\u000a\u000a<ol>\u000a<li><p>can predict how many elements it will yield when run:</p>\u000a\u000a<pre><code>my_array = numpy.empty(predict_length())\u000afor i, el in enumerate(gimme()): my_array[i] = el\u000a</code></pre></li>\u000a<li><p>are willing to store its elements in an intermediate list :</p>\u000a\u000a<pre><code>my_array = numpy.array(list(gimme()))\u000a</code></pre></li>\u000a<li><p>can make two identical generators, run through the first one to find the total length, initialize the array, and then run through the generator again to find each element:</p>\u000a\u000a<pre><code>length = sum(1 for el in gimme())\u000amy_array = numpy.empty(length)\u000afor i, el in enumerate(gimme()): my_array[i] = el\u000a</code></pre></li>\u000a</ol>\u000a\u000a<p><strong>1</strong> is probably what you're looking for. <strong>2</strong> is space inefficient, and <strong>3</strong> is time inefficient (you have to go through the generator twice).</p>\u000a

V<p>Python is strongly typed because every object <em>has</em> a type, every object <em>knows</em> its type, it's impossible to accidentally or deliberately use an object of a type "as if" it was an object of a <em>different</em> type, and all elementary operations on the object are delegated to its type.</p>\u000a\u000a<p>This has nothing to do with <strong>names</strong>.  A <em>name</em> in Python doesn't "have a type": if and when a name's defined, the name refers to an <em>object</em>, and the <em>object</em> does have a type (but that doesn't in fact force a type on the <em>name</em>: a name is a name).</p>\u000a\u000a<p>A name in Python can perfectly well refer to different objects at different times (as in most programming languages, though not all) -- and there is no constraint on the name such that, if it has once referred to an object of type X, it's then forevermore constrained to refer only to other objects of type X.  Constraints on <strong>names</strong> are not part of the concept of "strong typing", though some enthusiasts of <strong>static</strong> typing (where names <em>do</em> get constrained, and in a static, AKA compile-time, fashion, too) do misuse the term this way.</p>\u000a

V<p>Some answers contain the word "copy" in a context of a function call. I find it confusing.</p>\u000a\u000a<p><strong>Python doesn't copy <em>objects</em> you pass during a function call <em>ever</em>.</strong></p>\u000a\u000a<p>Function parameters are <em>names</em>. When you call a function Python binds these parameters to whatever objects you pass (via names in a caller scope).</p>\u000a\u000a<p>Objects can be mutable (like lists) or immutable (like integers, strings in Python). Mutable object you can change. You can't change a name, you just can bind it to another object.</p>\u000a\u000a<p>Your example is not about <a href="https://docs.python.org/2/tutorial/classes.html#python-scopes-and-namespaces" rel="noreferrer">scopes or namespaces</a>, it is about <a href="http://docs.python.org/reference/executionmodel.html#naming-and-binding" rel="noreferrer">naming and binding</a> and <a href="http://docs.python.org/reference/datamodel.html#objects-values-and-types" rel="noreferrer">mutability of an object</a> in Python. </p>\u000a\u000a<pre><code>def f(n, x): # these `n`, `x` have nothing to do with `n` and `x` from main()\u000a    n = 2    # put `n` label on `2` balloon\u000a    x.append(4) # call `append` method of whatever object `x` is referring to.\u000a    print 'In f():', n, x\u000a    x = []   # put `x` label on `[]` ballon\u000a    # x = [] has no effect on the original list that is passed into the function\u000a</code></pre>\u000a\u000a<p>Here are nice pictures on <a href="http://python.net/~goodger/projects/pycon/2007/idiomatic/handout.html#other-languages-have-variables" rel="noreferrer">the difference between variables in other languages and names in Python</a>.</p>\u000a

V<p>I ran into this same problem myself. This is caused by a quirk in how transactions are handled in the newer versions of Django coupled with a unittest that intentionally triggers an exception.</p>\u000a\u000a<p>I had a unittest that checked to make sure a unique column constraint was enforced by purposefully triggering an IntegrityError exception:</p>\u000a\u000a<pre><code>def test_constraint(self):\u000a    try:\u000a        # Duplicates should be prevented.\u000a        models.Question.objects.create(domain=self.domain, slug='barks')\u000a        self.assertTrue(0, 'Duplicate question allowed.')\u000a    except IntegrityError:\u000a        pass\u000a\u000a    do_more_model_stuff()\u000a</code></pre>\u000a\u000a<p>In Django 1.4, this works fine. However, in Django 1.5/1.6, each test is wrapped in a transaction, so if an exception occurs, it breaks the transaction until you explicitly roll it back. Therefore, any further ORM operations in that transaction, such as my <code>do_more_model_stuff()</code>, will fail with that <code>django.db.transaction.TransactionManagementError</code> exception.</p>\u000a\u000a<p>Like caio mentioned in the comments, the solution is to capture your exception with <code>transaction.atomic</code> like:</p>\u000a\u000a<pre><code>from django.db import transaction\u000adef test_constraint(self):\u000a    try:\u000a        # Duplicates should be prevented.\u000a        with transaction.atomic():\u000a            models.Question.objects.create(domain=self.domain, slug='barks')\u000a        self.assertTrue(0, 'Duplicate question allowed.')\u000a    except IntegrityError:\u000a        pass\u000a</code></pre>\u000a\u000a<p>That will prevent the purposefully-thrown exception from breaking the entire unittest's transaction.</p>\u000a

V<h2>Interrogate version string of already-installed distribution</h2>\u000a\u000a<p>To retrieve the version from inside your package at runtime (what your question appears to actually be asking), you can use:</p>\u000a\u000a<pre><code>import pkg_resources  # part of setuptools\u000aversion = pkg_resources.require("MyProject")[0].version\u000a</code></pre>\u000a\u000a<h2>Store version string for use during install</h2>\u000a\u000a<p>If you want to go the other way 'round (which appears to be what other answer authors here appear to have thought you were asking), put the version string in a separate file and read that file's contents in <code>setup.py</code>.</p>\u000a\u000a<p>You could make a version.py in your package with a <code>__version__</code> line, then read it from setup.py using <code>execfile('mypackage/version.py')</code>, so that it sets <code>__version__</code> in the setup.py namespace.</p>\u000a\u000a<p>If you want a much simpler way that will work with all Python versions and even non-Python languages that may need access to the version string:</p>\u000a\u000a<p>Store the version string as the sole contents of a plain text file, named e.g. <code>VERSION</code>, and read that file during <code>setup.py</code>.</p>\u000a\u000a<pre class="lang-python prettyprint-override"><code>version_file = open(os.path.join(mypackage_root_dir, 'VERSION'))\u000aversion = version_file.read().strip()\u000a</code></pre>\u000a\u000a<p>The same <code>VERSION</code> file will then work exactly as well in any other program, even non-Python ones, and you only need to change the version string in one place for all programs.</p>\u000a\u000a<h2>Warning about race condition during install</h2>\u000a\u000a<p>By the way, DO NOT import your package from your setup.py as suggested in another answer here: it will seem to work for you (because you already have your package's dependencies installed), but it will wreak havoc upon new users of your package, as they will not be able to install your package without manually installing the dependencies first.</p>\u000a

V<p>The main source of problems I've had working with unicode strings is when you mix utf-8 encoded strings with unicode ones.</p>\u000a\u000a<p>For example, consider the following scripts.</p>\u000a\u000a<p>two.py</p>\u000a\u000a<pre><code># encoding: utf-8\u000aname = 'hell wrld from two'\u000a</code></pre>\u000a\u000a<p>one.py</p>\u000a\u000a<pre><code># encoding: utf-8\u000afrom __future__ import unicode_literals\u000aimport two\u000aname = 'hell wrld from one'\u000aprint name + two.name\u000a</code></pre>\u000a\u000a<p>The output of running <code>python one.py</code> is:</p>\u000a\u000a<pre><code>Traceback (most recent call last):\u000a  File "one.py", line 5, in &lt;module&gt;\u000a    print name + two.name\u000aUnicodeDecodeError: 'ascii' codec can't decode byte 0xc3 in position 4: ordinal not in range(128)\u000a</code></pre>\u000a\u000a<p>In this example, <code>two.name</code> is an utf-8 encoded string (not unicode) since it did not import <code>unicode_literals</code>, and <code>one.name</code> is an unicode string. When you mix both, python tries to decode the encoded string (assuming it's ascii) and convert it to unicode and fails. It would work if you did <code>print name + two.name.decode('utf-8')</code>.</p>\u000a\u000a<p>The same thing can happen if you encode a string and try to mix them later.\u000aFor example, this works:</p>\u000a\u000a<pre><code># encoding: utf-8\u000ahtml = '&lt;html&gt;&lt;body&gt;hell wrld&lt;/body&gt;&lt;/html&gt;'\u000aif isinstance(html, unicode):\u000a    html = html.encode('utf-8')\u000aprint 'DEBUG: %s' % html\u000a</code></pre>\u000a\u000a<p>Output:</p>\u000a\u000a<pre><code>DEBUG: &lt;html&gt;&lt;body&gt;hell wrld&lt;/body&gt;&lt;/html&gt;\u000a</code></pre>\u000a\u000a<p>But after adding the <code>import unicode_literals</code> it does NOT:</p>\u000a\u000a<pre><code># encoding: utf-8\u000afrom __future__ import unicode_literals\u000ahtml = '&lt;html&gt;&lt;body&gt;hell wrld&lt;/body&gt;&lt;/html&gt;'\u000aif isinstance(html, unicode):\u000a    html = html.encode('utf-8')\u000aprint 'DEBUG: %s' % html\u000a</code></pre>\u000a\u000a<p>Output:</p>\u000a\u000a<pre><code>Traceback (most recent call last):\u000a  File "test.py", line 6, in &lt;module&gt;\u000a    print 'DEBUG: %s' % html\u000aUnicodeDecodeError: 'ascii' codec can't decode byte 0xc3 in position 16: ordinal not in range(128)\u000a</code></pre>\u000a\u000a<p>It fails because <code>'DEBUG: %s'</code> is an unicode string and therefore python tries to decode <code>html</code>. A couple of ways to fix the print are either doing <code>print str('DEBUG: %s') % html</code> or <code>print 'DEBUG: %s' % html.decode('utf-8')</code>.</p>\u000a\u000a<p>I hope this helps you understand the potential gotchas when using unicode strings.</p>\u000a

V<p>In the first test, Python try to convert the object to a <code>bool</code> value if it is not already one. Roughly, <strong>we are asking the object : are you meaningful or not ?</strong> This is done using the following algorithm :</p>\u000a\u000a<ol>\u000a<li><p>If the object has a <code>__nonzero__</code> special method (as do numeric built-ins, <code>int</code> and <code>float</code>), it calls this method. It must either return a <code>bool</code> value which is then directly used, or an <code>int</code> value that is considered <code>False</code> if equal to zero.</p></li>\u000a<li><p>Otherwise, if the object has a <code>__len__</code> special method (as do container built-ins, <code>list</code>, <code>dict</code>, <code>set</code>, <code>tuple</code>, ...), it calls this method, considering a container <code>False</code> if it is empty (length is zero).</p></li>\u000a<li><p>Otherwise, the object is considered <code>True</code> unless it is <code>None</code> in which case, it is considered <code>False</code>.</p></li>\u000a</ol>\u000a\u000a<p>In the second test, the object is compared for equality to <code>None</code>. Here, <strong>we are asking the object, "Are you equal to this other value?"</strong> This is done using the following algorithm :</p>\u000a\u000a<ol>\u000a<li><p>If the object has a <code>__eq__</code> method, it is called, and the return value is then converted to a <code>bool</code>value and used to determine the outcome of the <code>if</code>.</p></li>\u000a<li><p>Otherwise, if the object has a <code>__cmp__</code> method, it is called. This function must return an <code>int</code> indicating the order of the two object (<code>-1</code> if <code>self &lt; other</code>, <code>0</code> if <code>self == other</code>, <code>+1</code> if <code>self &gt; other</code>).</p></li>\u000a<li><p>Otherwise, the object are compared for identity (ie. they are reference to the same object, as can be tested by the <code>is</code> operator).</p></li>\u000a</ol>\u000a\u000a<p>There is another test possible using the <code>is</code> operator. <strong>We would be asking the object, "Are you this particular object?"</strong></p>\u000a\u000a<p>Generally, I would recommend to use the first test with non-numerical values, to use the test for equality when you want to compare objects of the same nature (two strings, two numbers, ...) and to check for identity only when using sentinel values (<code>None</code> meaning not initialized for a member field for exemple, or when using the <code>getattr</code> or the <code>__getitem__</code> methods).</p>\u000a\u000a<p>To summarize, we have :</p>\u000a\u000a<pre><code>&gt;&gt;&gt; class A(object):\u000a...    def __repr__(self):\u000a...        return 'A()'\u000a...    def __nonzero__(self):\u000a...        return False\u000a\u000a&gt;&gt;&gt; class B(object):\u000a...    def __repr__(self):\u000a...        return 'B()'\u000a...    def __len__(self):\u000a...        return 0\u000a\u000a&gt;&gt;&gt; class C(object):\u000a...    def __repr__(self):\u000a...        return 'C()'\u000a...    def __cmp__(self, other):\u000a...        return 0\u000a\u000a&gt;&gt;&gt; class D(object):\u000a...    def __repr__(self):\u000a...        return 'D()'\u000a...    def __eq__(self, other):\u000a...        return True\u000a\u000a&gt;&gt;&gt; for obj in ['', (), [], {}, 0, 0., A(), B(), C(), D(), None]:\u000a...     print '%4s: bool(obj) -&gt; %5s, obj == None -&gt; %5s, obj is None -&gt; %5s' % \u005c\u000a...         (repr(obj), bool(obj), obj == None, obj is None)\u000a  '': bool(obj) -&gt; False, obj == None -&gt; False, obj is None -&gt; False\u000a  (): bool(obj) -&gt; False, obj == None -&gt; False, obj is None -&gt; False\u000a  []: bool(obj) -&gt; False, obj == None -&gt; False, obj is None -&gt; False\u000a  {}: bool(obj) -&gt; False, obj == None -&gt; False, obj is None -&gt; False\u000a   0: bool(obj) -&gt; False, obj == None -&gt; False, obj is None -&gt; False\u000a 0.0: bool(obj) -&gt; False, obj == None -&gt; False, obj is None -&gt; False\u000a A(): bool(obj) -&gt; False, obj == None -&gt; False, obj is None -&gt; False\u000a B(): bool(obj) -&gt; False, obj == None -&gt; False, obj is None -&gt; False\u000a C(): bool(obj) -&gt;  True, obj == None -&gt;  True, obj is None -&gt; False\u000a D(): bool(obj) -&gt;  True, obj == None -&gt;  True, obj is None -&gt; False\u000aNone: bool(obj) -&gt; False, obj == None -&gt;  True, obj is None -&gt;  True\u000a</code></pre>\u000a

V<p>First, the easy cases:</p>\u000a\u000a<h2>ASCII</h2>\u000a\u000a<p>If your data contains no bytes above 0x7F, then it's ASCII.  (Or a 7-bit ISO646 encoding, but those are very obsolete.)</p>\u000a\u000a<h2>UTF-8</h2>\u000a\u000a<p>If your data validates as UTF-8, then you can safely assume it <em>is</em> UTF-8.  Due to UTF-8's strict validation rules, false positives are extremely rare.</p>\u000a\u000a<h2>ISO-8859-1 vs. windows-1252</h2>\u000a\u000a<p>The only difference between these two encodings is that ISO-8859-1 has the C1 control characters where windows-1252 has the printable characters \u20ac\u201a\u0192\u201e\u2026\u2020\u2021\u02c6\u2030\u0160\u2039\u0152\u017d\u2018\u2019\u201c\u201d\u2022\u2013\u2014\u02dc\u2122\u0161\u203a\u0153\u017e\u0178.  I've seen plenty of files that use curly quotes or dashes, but none that use C1 control characters.  So don't even bother with them, or ISO-8859-1, just detect windows-1252 instead.</p>\u000a\u000a<p>That now leaves you with only one question.</p>\u000a\u000a<h1>How do you distinguish MacRoman from cp1252?</h1>\u000a\u000a<p>This is a lot trickier.</p>\u000a\u000a<h2>Undefined characters</h2>\u000a\u000a<p>The bytes 0x81, 0x8D, 0x8F, 0x90, 0x9D are not used in windows-1252.  If they occur, then assume the data is MacRoman.</p>\u000a\u000a<h2>Identical characters</h2>\u000a\u000a<p>The bytes 0xA2 (), 0xA3 (), 0xA9 (), 0xB1 (), 0xB5 () happen to be the same in both encodings.  If these are the only non-ASCII bytes, then it doesn't matter whether you choose MacRoman or cp1252.</p>\u000a\u000a<h2>Statistical approach</h2>\u000a\u000a<p>Count character (NOT byte!) frequencies in the data you know to be UTF-8.  Determine the most frequent characters.  Then use this data to determine whether the cp1252 or MacRoman characters are more common.</p>\u000a\u000a<p>For example, in a search I just performed on 100 random English Wikipedia articles, the most common non-ASCII characters are <code>\u2022\u2013\u2019\u2014</code>.  Based on this fact,</p>\u000a\u000a<ul>\u000a<li>The bytes 0x92, 0x95, 0x96, 0x97, 0xAE, 0xB0, 0xB7, 0xE8, 0xE9, or 0xF6 suggest windows-1252.</li>\u000a<li>The bytes 0x8E, 0x8F, 0x9A, 0xA1, 0xA5, 0xA8, 0xD0, 0xD1, 0xD5, or 0xE1 suggest MacRoman.</li>\u000a</ul>\u000a\u000a<p>Count up the cp1252-suggesting bytes and the MacRoman-suggesting bytes, and go with whichever is greatest.</p>\u000a

V<p>I would second the notion that you may wish to consider a subclass instead of the approach you've outlined. However, not knowing your specific scenario, YMMV :-)</p>\u000a\u000a<p>What you're thinking of is a metaclass. The <code>__new__</code> function in a metaclass is passed the full proposed definition of the class, which it can then rewrite before the class is created. You can, at that time, sub out the constructor for a new one.</p>\u000a\u000a<p>Example:</p>\u000a\u000a<pre><code>def substitute_init(self, id, *args, **kwargs):\u000a    pass\u000a\u000aclass FooMeta(type):\u000a\u000a    def __new__(cls, name, bases, attrs):\u000a        attrs['__init__'] = substitute_init\u000a        return super(FooMeta, cls).__new__(cls, name, bases, attrs)\u000a\u000aclass Foo(object):\u000a\u000a    __metaclass__ = FooMeta\u000a\u000a    def __init__(self, value1):\u000a        pass\u000a</code></pre>\u000a\u000a<p>Replacing the constructor is perhaps a bit dramatic, but the language does provide support for this kind of deep introspection and dynamic modification.</p>\u000a

V<p><code>Queue.Queue</code> and <code>collections.deque</code> serve different purposes. Queue.Queue is intended for allowing different threads to communicate using queued messages/data, whereas <code>collections.deque</code> is simply intended as a datastructure. That's why <code>Queue.Queue</code> has methods like <code>put_nowait()</code>, <code>get_nowait()</code>, and <code>join()</code>, whereas <code>collections.deque</code> doesn't. <code>Queue.Queue</code> isn't intended to be used as a collection, which is why it lacks the likes of the <code>in</code> operator.</p>\u000a\u000a<p>It boils down to this: if you have multiple threads and you want them to be able to communicate without the need for locks, you're looking for <code>Queue.Queue</code>; if you just want a queue or a double-ended queue as a datastructure, use <code>collections.deque</code>.</p>\u000a\u000a<p>Finally, accessing and manipulating the internal deque of a <code>Queue.Queue</code> is playing with fire - you really don't want to be doing that.</p>\u000a

V<p>To exit a script you can use,</p>\u000a\u000a<pre><code>import sys\u000asys.exit()\u000a</code></pre>\u000a\u000a<p>You can also provide an exit status value, usually an integer.</p>\u000a\u000a<pre><code>import sys\u000asys.exit(0)\u000a</code></pre>\u000a\u000a<p>Exits with zero, which is generally interpreted as success.  Non-zero codes are usually treated as errors.  The default is to exit with zero.</p>\u000a\u000a<pre><code>import sys\u000asys.exit("aa! errors!")\u000a</code></pre>\u000a\u000a<p>Prints "aa! errors!" and exits with a status code of 1.</p>\u000a\u000a<p>There is also an _exit() function in the os module.  The <a href="http://www.python.org/doc/2.5.2/lib/module-sys.html" rel="noreferrer">sys.exit()</a> function raises a SystemExit exception to exit the program, so try statements and cleanup code can execute.  The os._exit() version doesn't do this.  It just ends the program without doing any cleanup or flushing output buffers, so it shouldn't normally be used.  </p>\u000a\u000a<p>The <a href="http://www.python.org/doc/2.5.2/lib/os-process.html" rel="noreferrer">Python docs</a> indicate that os._exit() is the normal way to end a child process created with a call to os.fork(), so it does have a use in certain circumstances.</p>\u000a
