V<p>Are you sure you are using Python 3.x? The syntax isn't available in Python 2.x because <code>print</code> is still a statement.</p>\u000a\u000a<pre><code>print("foo" % bar, end=" ")\u000a</code></pre>\u000a\u000a<p>in Python 2.x is identical to</p>\u000a\u000a<pre><code>print ("foo" % bar, end=" ")\u000a</code></pre>\u000a\u000a<p>or</p>\u000a\u000a<pre><code>print "foo" % bar, end=" "\u000a</code></pre>\u000a\u000a<p>i.e. as a call to print with a tuple as argument.</p>\u000a\u000a<p>That's obviously bad syntax (literals don't take keyword arguments). In Python 3.x <code>print</code> is an actual function, so it takes keyword arguments, too.</p>\u000a\u000a<p>The correct idiom in Python 2.x for <code>end=" "</code> is:</p>\u000a\u000a<pre><code>print "foo" % bar,\u000a</code></pre>\u000a\u000a<p>(note the final comma, this makes it end the line with a space rather than a linebreak)</p>\u000a\u000a<p>If you want more control over the output, consider using <code>sys.stdout</code> directly. This won't do any special magic with the output.</p>\u000a\u000a<p>Of course in somewhat recent versions of Python 2.x (2.5 should have it, not sure about 2.4), you can use the <code>__future__</code> module to enable it in your script file:</p>\u000a\u000a<pre><code>from __future__ import print_function\u000a</code></pre>\u000a\u000a<p>The same goes with <code>unicode_literals</code> and some other nice things (<code>with_statement</code>, for example). This won't work in really old versions (i.e. created before the feature was introduced) of Python 2.x, though.</p>\u000a

V<p>Your code is <em>almost right</em>!  You are right, you are just missing one step. You need to do something else.</p>\u000a\u000a<h2>Example</h2>\u000a\u000a<p>Here is the simplest way: feed the string into <code>eval()</code>.  Like so:</p>\u000a\u000a<pre><code>def reading(self):\u000a    s = open('deed.txt', 'r').read()\u000a    self.whip = eval(s)\u000a</code></pre>\u000a\u000a<p>You can do it in one line, but I think it looks messy this way:</p>\u000a\u000a<pre><code>def reading(self):\u000a    self.whip = eval(open('deed.txt', 'r').read())\u000a</code></pre>\u000a\u000a<p>But <code>eval()</code> is sometimes not recommended. In this case, you are just running <code>eval()</code> on your own file, so it should be okay.</p>\u000a\u000a<p>But because <code>eval()</code> is useful, someone made an alternative to it that is safer. V<p>There is no standard module for this, but I have written my own functions to achieve that.</p>\u000a\u000a<pre class="lang-py prettyprint-override"><code>BASE62 = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"\u000a\u000adef encode(num, alphabet=BASE62):\u000a    """Encode a positive number in Base X\u000a\u000a    Arguments:\u000a    - `num`: The number to encode\u000a    - `alphabet`: The alphabet to use for encoding\u000a    """\u000a    if num == 0:\u000a        return alphabet[0]\u000a    arr = []\u000a    base = len(alphabet)\u000a    while num:\u000a        num, rem = divmod(num, base)\u000a        arr.append(alphabet[rem])\u000a    arr.reverse()\u000a    return ''.join(arr)\u000a\u000adef decode(string, alphabet=BASE62):\u000a    """Decode a Base X encoded string into the number\u000a\u000a    Arguments:\u000a    - `string`: The encoded string\u000a    - `alphabet`: The alphabet to use for encoding\u000a    """\u000a    base = len(alphabet)\u000a    strlen = len(string)\u000a    num = 0\u000a\u000a    idx = 0\u000a    for char in string:\u000a        power = (strlen - (idx + 1))\u000a        num += alphabet.index(char) * (base ** power)\u000a        idx += 1\u000a\u000a    return num\u000a</code></pre>\u000a\u000a<p>Notice the fact that you can give it any alphabet to use for encoding and decoding. V<p>Dictionaries are implemented as hash tables and there are two important concepts when adding keys/values here: <em>hashing</em> and <em>equality</em>.</p>\u000a\u000a<p>To insert a particular key/value, Python first computes the <em>hash</em> value of the key. This hash value is used to determine the row of the table where Python should first attempt to put the key/value.</p>\u000a\u000a<p>If the row of the hash table is empty, great: the new key/value can inserted into the dictionary, filling the empty row.</p>\u000a\u000a<p>However, if there's already something in that row, Python needs to test the keys for equality. If the keys are equal (using <code>==</code>) then they're deemed to be the same key and Python just needs to update the corresponding value on that row.</p>\u000a\u000a<p>(If the keys are not equal Python looks at other rows in the table until it finds the key or reaches an empty row, but that's not relevant for this question.) </p>\u000a\u000a<hr>\u000a\u000a<p>When you write <code>{True: 'yes', 1: 'No'}</code>, you are telling Python to create a new dictionary and then fill it with two key/value pairs. These are processed left to right: <code>True: 'yes'</code> then <code>1: 'No'</code>.</p>\u000a\u000a<p>We have <code>hash(True)</code> equals 1. The key <code>True</code> goes in at row 1 in the hash table and the string <code>'yes'</code> is its value.</p>\u000a\u000a<p>For the next pair, Python sees that <code>hash(1)</code> is also 1 and so looks at row 1 of the table. There's something already there, so now Python checks the keys for equality. We have <code>1 == True</code> so <code>1</code> is deemed to be the same key as <code>True</code> and so its corresponding value is changed to the string <code>'No'</code>.</p>\u000a\u000a<p>This results in a dictionary with one entry: <code>{True: 'No'}</code>.</p>\u000a\u000a<hr>\u000a\u000a<p>If you want to peer at the guts of CPython 3.5 to see what creating a dictionary looks below the surface-Python level, here's more detail.</p>\u000a\u000a<ul>\u000a<li><p>The Python code <code>{True: 'yes', 1: 'No'}</code> is parsed into tokens and given to the compiler. Given the syntax, Python knows that a dictionary must be created using the values inside the braces. Byte code to load the four values onto the virtual machine's stack (<code>LOAD_CONST</code>) and then build the dictionary (<code>BUILD_MAP</code>) is queued up.</p></li>\u000a<li><p>The four constant values are pushed onto the top of the stack in the order that they're seen: <br><br></p>\u000a\u000a<pre><code>'No'\u000a1\u000a'yes'\u000aTrue\u000a</code></pre></li>\u000a<li><p>The opcode <code>BUILD_MAP</code> is then called with the argument <code>2</code> (Python counted two key/value pairs). This opcode is responsible for actually creating dictionary from the items on the stack. It looks like <a href="https://hg.python.org/cpython/file/3.5/Python/ceval.c#l2582">this</a>:<br><br>\u000a</p>\u000a\u000a<pre><code>TARGET(BUILD_MAP) {\u000a    int i;\u000a    PyObject *map = _PyDict_NewPresized((Py_ssize_t)oparg);\u000a    if (map == NULL)\u000a        goto error;\u000a    for (i = oparg; i &gt; 0; i--) {\u000a        int err;\u000a        PyObject *key = PEEK(2*i);\u000a        PyObject *value = PEEK(2*i - 1);\u000a        err = PyDict_SetItem(map, key, value);\u000a        if (err != 0) {\u000a            Py_DECREF(map);\u000a            goto error;\u000a        }\u000a    }\u000a\u000a    while (oparg--) {\u000a        Py_DECREF(POP());\u000a        Py_DECREF(POP());\u000a    }\u000a    PUSH(map);\u000a    DISPATCH();\u000a}\u000a</code></pre></li>\u000a</ul>\u000a\u000a<p>The three key steps here are as follows:</p>\u000a\u000a<ol>\u000a<li><p>An empty hashtable is created using <code>_PyDict_NewPresized</code>. Small dictionaries (of just a few items, like 2 in this case) need a table with eight rows. </p></li>\u000a<li><p>The <code>for</code> loop is entered, starting at 2 (in this case) and counting down to 0. <a href="https://hg.python.org/cpython/file/3.5/Python/ceval.c#l1052"><code>PEEK(n)</code></a> is a macro that points to the nth item down the stack. Therefore on the first iteration of the loop, we'll have</p></li>\u000a</ol>\u000a\u000a<pre class="lang-C prettyprint-override"><code>PyObject *key = PEEK(2*2);       /* item 4 down the stack */  \u000aPyObject *value = PEEK(2*2 - 1); /* item 3 down the stack */\u000a</code></pre>\u000a\u000a<p>This means that <code>*key</code> will be <code>True</code> and <code>*value</code> will be <code>'yes'</code> on the first loop through. On the second it will be <code>1</code> and <code>'No'</code>. </p>\u000a\u000a<ol start="3">\u000a<li><a href="https://hg.python.org/cpython/file/3.5/Objects/dictobject.c#l1208"><code>PyDict_SetItem</code></a> is called in each loop to put the current <code>*key</code> and <code>*value</code> into the dictionary. This is the same function that is called when you write <code>dictionary[key] = value</code>. It computes the hash of the key to work out where to look first in the hash table and then, if needed, compare the key to any existing key on that row (as discussed above).</li>\u000a</ol>\u000a

V<h2>Long integers:</h2>\u000a\u000a<p>There is no explicitly defined limit. The amount of available address space forms a practical limit.<br>\u000a(Taken from <a href="http://bytes.com/topic/python/answers/761897-max-long">this</a> site). See the docs on <a href="http://docs.python.org/2/library/stdtypes.html#numeric-types-int-float-long-complex">Numeric Types</a> where you'll see that <code>Long integers have unlimited precision</code>. In Python 2, Integers will automatically switch to longs when they grow beyond their limit:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; import sys\u000a&gt;&gt;&gt; type(sys.maxsize)\u000a&lt;type 'int'&gt;\u000a&gt;&gt;&gt; type(sys.maxsize+1)\u000a&lt;type 'long'&gt;\u000a</code></pre>\u000a\u000a<p><br>\u000afor integers we have</p>\u000a\u000a<h2>maxint and maxsize:</h2>\u000a\u000a<p>The maximum value of an int can be found in Python 2.x with <code>sys.maxint</code>. It was removed in Python 3, but <code>sys.maxsize</code> can often be used instead. V<p>First, the function, for those who just want some copy-and-paste code:</p>\u000a\u000a<pre><code>def truncate(f, n):\u000a    '''Truncates/pads a float f to n decimal places without rounding'''\u000a    s = '{}'.format(f)\u000a    if 'e' in s or 'E' in s:\u000a        return '{0:.{1}f}'.format(f, n)\u000a    i, p, d = s.partition('.')\u000a    return '.'.join([i, (d+'0'*n)[:n]])\u000a</code></pre>\u000a\u000a<p>This is valid in Python 2.7 and 3.1+. For older versions, it's not possible to get the same "intelligent rounding" effect (at least, not without a lot of complicated code), but rounding to 12 decimal places before truncation will work much of the time:</p>\u000a\u000a<pre><code>def truncate(f, n):\u000a    '''Truncates/pads a float f to n decimal places without rounding'''\u000a    s = '%.12f' % f\u000a    i, p, d = s.partition('.')\u000a    return '.'.join([i, (d+'0'*n)[:n]])\u000a</code></pre>\u000a\u000a<h1>Explanation</h1>\u000a\u000a<p>The core of the underlying method is to convert the value to a string at full precision and then just chop off everything beyond the desired number of characters. The latter step is easy; it can be done either with string manipulation</p>\u000a\u000a<pre><code>i, p, d = s.partition('.')\u000a'.'.join([i, (d+'0'*n)[:n]])\u000a</code></pre>\u000a\u000a<p>or the <code>decimal</code> module</p>\u000a\u000a<pre><code>str(Decimal(s).quantize(Decimal((0, (1,), -n)), rounding=ROUND_DOWN))\u000a</code></pre>\u000a\u000a<p>The first step, converting to a string, is quite difficult because there are some pairs of floating point literals (i.e. what you write in the source code) which both produce the same binary representation and yet should be truncated differently. For example, consider 0.3 and 0.29999999999999998. If you write <code>0.3</code> in a Python program, the compiler encodes it using the IEEE floating-point format into the sequence of bits (assuming a 64-bit float)</p>\u000a\u000a<pre><code>0011111111010011001100110011001100110011001100110011001100110011\u000a</code></pre>\u000a\u000a<p>This is the closest value to 0.3 that can accurately be represented as an IEEE float. But if you write <code>0.29999999999999998</code> in a Python program, the compiler translates it into <em>exactly the same value</em>. In one case, you meant it to be truncated (to one digit) as <code>0.3</code>, whereas in the other case you meant it to be truncated as <code>0.2</code>, but Python can only give one answer. This is a fundamental limitation of Python, or indeed any programming language without lazy evaluation. The truncation function only has access to the binary value stored in the computer's memory, not the string you actually typed into the source code.<sup>1</sup></p>\u000a\u000a<p>If you decode the sequence of bits back into a decimal number, again using the IEEE 64-bit floating-point format, you get</p>\u000a\u000a<pre><code>0.2999999999999999888977697537484345957637...\u000a</code></pre>\u000a\u000a<p>so a naive implementation would come up with <code>0.2</code> even though that's probably not what you want. For more on floating-point representation error, <a href="https://docs.python.org/3/tutorial/floatingpoint.html" rel="noreferrer">see the Python tutorial</a>.</p>\u000a\u000a<p>It's very rare to be working with a floating-point value that is so close to a round number and yet is <em>intentionally</em> not equal to that round number. So when truncating, it probably makes sense to choose the "nicest" decimal representation out of all that could correspond to the value in memory. Python 2.7 and up (but not 3.0) includes a <a href="https://bugs.python.org/issue1580" rel="noreferrer">sophisticated algorithm to do just that</a>, which we can access through the default string formatting operation.</p>\u000a\u000a<pre><code>'{}'.format(f)\u000a</code></pre>\u000a\u000a<p>The only caveat is that this acts like a <code>g</code> format specification, in the sense that it uses exponential notation (<code>1.23e+4</code>) if the number is large or small enough. So the method has to catch this case and handle it differently. There are a few cases where using an <code>f</code> format specification instead causes a problem, such as trying to truncate <code>3e-10</code> to 28 digits of precision (it produces <code>0.0000000002999999999999999980</code>), and I'm not yet sure how best to handle those.</p>\u000a\u000a<p>If you actually <em>are</em> working with <code>float</code>s that are very close to round numbers but intentionally not equal to them (like 0.29999999999999998 or 99.959999999999994), this will produce some false positives, i.e. it'll round numbers that you didn't want rounded. In that case the solution is to specify a fixed precision.</p>\u000a\u000a<pre><code>'{0:.{1}f}'.format(f, sys.float_info.dig + n + 2)\u000a</code></pre>\u000a\u000a<p>The number of digits of precision to use here doesn't really matter, it only needs to be large enough to ensure that any rounding performed in the string conversion doesn't "bump up" the value to its nice decimal representation. I think <code>sys.float_info.dig + n + 2</code> may be enough in all cases, but if not that <code>2</code> might have to be increased, and it doesn't hurt to do so.</p>\u000a\u000a<p>In earlier versions of Python (up to 2.6, or 3.0), the floating point number formatting was a lot more crude, and would regularly produce things like</p>\u000a\u000a<pre><code>&gt;&gt;&gt; 1.1\u000a1.1000000000000001\u000a</code></pre>\u000a\u000a<p>If this is your situation, if you <em>do</em> want to use "nice" decimal representations for truncation, all you can do (as far as I know) is pick some number of digits, less than the full precision representable by a <code>float</code>, and round the number to that many digits before truncating it. A typical choice is 12,</p>\u000a\u000a<pre><code>'%.12f' % f\u000a</code></pre>\u000a\u000a<p>but you can adjust this to suit the numbers you're using.</p>\u000a\u000a<hr>\u000a\u000a<p><sup>1</sup>Well... I lied. Technically, you <em>can</em> instruct Python to re-parse its own source code and extract the part corresponding to the first argument you pass to the truncation function. If that argument is a floating-point literal, you can just cut it off a certain number of places after the decimal point and return that. However this strategy doesn't work if the argument is a variable, which makes it fairly useless. The following is presented for entertainment value only:</p>\u000a\u000a<pre><code>def trunc_introspect(f, n):\u000a    '''Truncates/pads the float f to n decimal places by looking at the caller's source code'''\u000a    current_frame = None\u000a    caller_frame = None\u000a    s = inspect.stack()\u000a    try:\u000a        current_frame = s[0]\u000a        caller_frame = s[1]\u000a        gen = tokenize.tokenize(io.BytesIO(caller_frame[4][caller_frame[5]].encode('utf-8')).readline)\u000a        for token_type, token_string, _, _, _ in gen:\u000a            if token_type == tokenize.NAME and token_string == current_frame[3]:\u000a                next(gen) # left parenthesis\u000a                token_type, token_string, _, _, _ = next(gen) # float literal\u000a                if token_type == tokenize.NUMBER:\u000a                    try:\u000a                        cut_point = token_string.index('.') + n + 1\u000a                    except ValueError: # no decimal in string\u000a                        return token_string + '.' + '0' * n\u000a                    else:\u000a                        if len(token_string) &lt; cut_point:\u000a                            token_string += '0' * (cut_point - len(token_string))\u000a                        return token_string[:cut_point]\u000a                else:\u000a                    raise ValueError('Unable to find floating-point literal (this probably means you called {} with a variable)'.format(current_frame[3]))\u000a                break\u000a    finally:\u000a        del s, current_frame, caller_frame\u000a</code></pre>\u000a\u000a<p>Generalizing this to handle the case where you pass in a variable seems like a lost cause, since you'd have to trace backwards through the program's execution until you find the floating-point literal which gave the variable its value. V<p>I found the solution to the problem <a href="http://wirtel.be/posts/en/2014/07/29/fix_virtualenv_python_brew/">here</a>, so all credit goes to the author.</p>\u000a\u000a<p>The gist is that when you create a virtualenv, many symlinks are created to the Homebrew installed Python.</p>\u000a\u000a<p>Here is one example:</p>\u000a\u000a<pre><code>$ ls -la ~/.virtualenvs/my-virtual-env\u000a...\u000alrwxr-xr-x  1 ryan staff   78 Jun 25 13:21 .Python -&gt; /usr/local/Cellar/python/2.7.7/Frameworks/Python.framework/Versions/2.7/Python\u000a...\u000a</code></pre>\u000a\u000a<p>When you upgrade Python using Homebrew and then run <code>brew cleanup</code>, the symlinks in the virtualenv point to paths that no longer exist (because Homebrew deleted them).</p>\u000a\u000a<p>The symlinks needs to point to the newly installed Python:</p>\u000a\u000a<pre><code>lrwxr-xr-x  1 ryan staff   78 Jun 25 13:21 .Python -&gt; /usr/local/Cellar/python/2.7.8_1/Frameworks/Python.framework/Versions/2.7/Python\u000a</code></pre>\u000a\u000a<p>The solution is to remove the symlinks in the virtualenv and then recreate them:</p>\u000a\u000a<pre><code>find ~/.virtualenvs/my-virtual-env/ -type l -delete\u000avirtualenv ~/.virtualenvs/my-virtual-env\u000a</code></pre>\u000a\u000a<p>It's probably best to check what links will be deleted first before deleting them:</p>\u000a\u000a<pre><code>find ~/.virtualenvs/my-virtual-env/ -type l\u000a</code></pre>\u000a\u000a<p>In my opinion, it's even better to only delete broken symlinks. This is to avoid shadowing the <code>find</code> binary that ships with OS X.</p>\u000a

V<h2>Note</h2>\u000a\u000a<p>The below answer is <em>not</em> applicable to requests v0.13.0+. The asynchronous functionality was moved to <a href="https://github.com/kennethreitz/grequests" rel="noreferrer">grequests</a> after this question was written. However, you could just replace <code>requests</code> with <code>grequests</code> below and it should work.</p>\u000a\u000a<p>I've left this answer as is to reflect the original question which was about using requests &lt; v0.13.0.</p>\u000a\u000a<hr>\u000a\u000a<p>To do multiple tasks with <code>async.map</code> <em>asynchronously</em> you have to:</p>\u000a\u000a<ol>\u000a<li>Define a function for what you want to do with each object (your task)</li>\u000a<li>Add that function as an event hook in your request</li>\u000a<li>Call <code>async.map</code> on a list of all the requests / actions</li>\u000a</ol>\u000a\u000a<p>Example:</p>\u000a\u000a<pre><code>from requests import async\u000a# If using requests &gt; v0.13.0, use\u000a# from grequests import async\u000a\u000aurls = [\u000a    'http://python-requests.org',\u000a    'http://httpbin.org',\u000a    'http://python-guide.org',\u000a    'http://kennethreitz.com'\u000a]\u000a\u000a# A simple task to do to each response object\u000adef do_something(response):\u000a    print response.url\u000a\u000a# A list to hold our things to do via async\u000aasync_list = []\u000a\u000afor u in urls:\u000a    # The "hooks = {..." part is where you define what you want to do\u000a    # \u000a    # Note the lack of parentheses following do_something, this is\u000a    # because the response will be used as the first argument automatically\u000a    action_item = async.get(u, hooks = {'response' : do_something})\u000a\u000a    # Add the task to our list of things to do via async\u000a    async_list.append(action_item)\u000a\u000a# Do our list of things to do via async\u000aasync.map(async_list)\u000a</code></pre>\u000a

V<p>Officially, OpenCV releases two types of Python interfaces, <code>cv</code> and <code>cv2</code>.</p>\u000a\u000a<p><strong>cv:</strong></p>\u000a\u000a<p>I started working on <code>cv</code>. In this, all OpenCV data types are preserved as such. For example, when loaded, images are of format <code>cvMat</code>, same as in C++.</p>\u000a\u000a<p>For array operations, there are several functions like <code>cvSet2D</code>, <code>cvGet2D</code>, etc. And some discussions say, they are slower.</p>\u000a\u000a<p>For imageROI, you need special functions like <code>cvSetImageROI</code>.</p>\u000a\u000a<p>If you find contours, <code>cvSeq</code> structures are returned which is not so good to work with compared to Python lists or NumPy arrays.</p>\u000a\u000a<p>(And I think, soon its development will be stopped. Earlier, there was only <code>cv</code>. Later, OpenCV came with both <code>cv</code> and <code>cv2</code>. Now, there in the latest releases, there is only the <code>cv2</code> module, and <code>cv</code> is a subclass inside <code>cv2</code>. You need to call <code>import cv2.cv as cv</code> to access it.)</p>\u000a\u000a<p><strong>cv2:</strong></p>\u000a\u000a<p>And the latest one is <code>cv2</code>. In this, everything is returned as <code>NumPy</code> objects like <code>ndarray</code>  and <code>native Python</code> objects like <code>lists</code>,<code>tuples</code>,<code>dictionary</code>, etc. So due to this NumPy support, you can do any numpy operation here. <code>NumPy</code> is a highly stable and fast array processing library.</p>\u000a\u000a<p>For example, if you load an image, an <code>ndarray</code> is returned.</p>\u000a\u000a<p><code>array[i,j]</code> gives you the pixel value at (i,j) position.</p>\u000a\u000a<p>Also, for imageROI, array slicing can be used like <code>ROI=array[c1:c2,r1:r2]</code>. No need of separate functions.</p>\u000a\u000a<p>To add two images, there isn't a need to call any function, just do <code>res = img1+img2</code>. (But NumPy addition is a modulo operation for uint8 arrays like images. See the article <em><a href="http://opencvpython.blogspot.in/2012/06/difference-between-matrix-arithmetic-in.html" rel="noreferrer">Difference between Matrix Arithmetic in OpenCV and Numpy</a></em> to know more.</p>\u000a\u000a<p>Contours returned are lists of Numpy arrays. You can find a detailed discussion about Contours in <em><a href="http://opencvpython.blogspot.in/2012/06/hi-this-article-is-tutorial-which-try.html" rel="noreferrer">Contours - 1 : Getting Started</a></em>.</p>\u000a\u000a<p><strong><em>In short, with cv2 everything is simplified and pretty fast.</em></strong></p>\u000a\u000a<p>A simple discussion on how NumPy speed up <code>cv2</code> is in Stack Overflow question <em><a href="http://stackoverflow.com/questions/9357696">Performance comparison of OpenCV-Python interfaces, cv and cv2</a></em>.</p>\u000a\u000a<p><strong>pyopencv</strong>:</p>\u000a\u000a<p>I don't know much about this since I haven't used it. But it seems to have stopped further development.</p>\u000a\u000a<p>I think it would be better to stick on to official libraries.</p>\u000a\u000a<p><strong><em>In short, I would recommend you to use cv2!</em></strong></p>\u000a\u000a<p>EDIT: You can see installation procedure for the <code>cv2</code> module in <em><a href="http://opencvpython.blogspot.in/2012/05/install-opencv-in-windows-for-python.html" rel="noreferrer">Install OpenCV in Windows for Python</a></em>.</p>\u000a

V<p>This answer is similar in spirit to Douglas Leeder's, with the following changes:</p>\u000a\u000a<ul>\u000a<li>It doesn't use actual Base64, so there's no padding characters</li>\u000a<li><p>Instead of converting the number first to a byte-string (base 256), it converts it directly to base 64, which has the advantage of letting you represent negative numbers using a sign character.</p>\u000a\u000a<pre><code>import string\u000aALPHABET = string.ascii_uppercase + string.ascii_lowercase + \u005c\u000a           string.digits + '-_'\u000aALPHABET_REVERSE = dict((c, i) for (i, c) in enumerate(ALPHABET))\u000aBASE = len(ALPHABET)\u000aSIGN_CHARACTER = '$'\u000a\u000adef num_encode(n):\u000a    if n &lt; 0:\u000a        return SIGN_CHARACTER + num_encode(-n)\u000a    s = []\u000a    while True:\u000a        n, r = divmod(n, BASE)\u000a        s.append(ALPHABET[r])\u000a        if n == 0: break\u000a    return ''.join(reversed(s))\u000a\u000adef num_decode(s):\u000a    if s[0] == SIGN_CHARACTER:\u000a        return -num_decode(s[1:])\u000a    n = 0\u000a    for c in s:\u000a        n = n * BASE + ALPHABET_REVERSE[c]\u000a    return n\u000a</code></pre></li>\u000a</ul>\u000a\u000a<hr>\u000a\u000a<pre><code>    &gt;&gt;&gt; num_encode(0)\u000a    'A'\u000a    &gt;&gt;&gt; num_encode(64)\u000a    'BA'\u000a    &gt;&gt;&gt; num_encode(-(64**5-1))\u000a    '$_____'\u000a</code></pre>\u000a\u000a<hr>\u000a\u000a<p>A few side notes:</p>\u000a\u000a<ul>\u000a<li>You could (<em>marginally</em>) increase the human-readibility of the base-64 numbers by putting string.digits first in the alphabet (and making the sign character '-'); I chose the order that I did based on Python's urlsafe_b64encode.</li>\u000a<li>If you're encoding a lot of negative numbers, you could increase the efficiency by using a sign bit or one's/two's complement instead of a sign character.</li>\u000a<li>You should be able to easily adapt this code to different bases by changing the alphabet, either to restrict it to only alphanumeric characters or to add additional "URL-safe" characters.</li>\u000a<li>I would recommend <em>against</em> using a representation other than base 10 in URIs in most cases\u2014it adds complexity and makes debugging harder without significant savings compared to the overhead of HTTP\u2014unless you're going for something TinyURL-esque.</li>\u000a</ul>\u000a

V<pre><code>newlist = oldlist[::-1]\u000a</code></pre>\u000a\u000a<p>The <code>[::-1]</code> slicing (which my wife Anna likes to call "the Martian smiley";-) means: slice the whole sequence, with a step of -1, i.e., in reverse.  It works for all sequences.</p>\u000a\u000a<p>Note that this (<em>and</em> the alternatives you mentioned) is equivalent to a "shallow copy", i.e.: if the items are mutable and you call mutators on them, the mutations in the items held in the original list are also in the items in the reversed list, and vice versa.  If you need to avoid that, a <code>copy.deepcopy</code> (while always a potentially costly operation), followed in this case by a <code>.reverse</code>, is the only good option.</p>\u000a
