V<p>While this question has a python backend, the question is <strong>not</strong> tied to python itself, but rather about extension mechanisms and how to register/lookup for plugins.</p>\u000a\u000a<p>In python, the concept of entrypoints was introduced by setuptools, and is tied to the metadata of installed python distributions (called packages in other packaging systems).</p>\u000a\u000a<p>As of my understanding, one of the features provided by entrypoints is to allows an application to define a place where others can put things in, so any application wanting to use an entrypoint can get a list of registered classes/functions there. Let's take an example:</p>\u000a\u000a<ul>\u000a<li>Foo defines the entrypoint "entrypoint1" and look for plugins registered under this name.</li>\u000a<li>Bar register a callable (<code>Bar.callable</code>) on the "entrypoint1" entrypoint.</li>\u000a<li>Any python script is then able to list <code>Bar.callable</code> as one of the registered callables for "entrypoint1".</li>\u000a</ul>\u000a\u000a<p>With setuptools, applications registers entrypoints at install time and the information is stored in metadata related to packaging, called .egginfo (which usually contain information about the distribution name, its dependencies, and some more metadata about packaging).</p>\u000a\u000a<p>I have the feeling that packaging metadata is not the right place to store this kind of information, as I don't get why this information is tied to packaging.</p>\u000a\u000a<p>I'm curious to hear about such entrypoints/extensions/plugins features in other languages, and especially if the concept is tied to metadata and packaging or not. And so the question is\u2026</p>\u000a\u000a<p><strong>Do you have examples I should look at? Could you explain why the design choices have been made that way?</strong></p>\u000a\u000a<p>Can you see different ways to handle this problem? Do you know how this problem has already been solved in different tools? What are the downsides and the advantages of the current python implementation over others?</p>\u000a\u000a<hr>\u000a\u000a<p><strong>What I've found <em>so far</em></strong></p>\u000a\u000a<p>I've found in different projects, a way to create and to distribute "plugins", which are especially taking care about the "how do we make plugins". </p>\u000a\u000a<p>For instance, <a href="http://live.gnome.org/Libpeas">libpeas</a> (the gobject plugin framework) defines a set of ways to extend a default behavior by specifying plugins. While this is interesting, I am just interested by the "registering and finding" (and eventually loading) part of it.</p>\u000a\u000a<p>Here are some of my findings so far:</p>\u000a\u000a<p>Libpeas defines <a href="http://git.gnome.org/browse/libpeas/tree/peas-demo/plugins/pythonhello/pythonhello.plugin">its own metadata file</a> (*.plugin), which stores information about the type of the callable (it is possible to have different plugins in different languages). The main information here is the name of the module to load.</p>\u000a\u000a<p>Maven have <a href="https://github.com/apache/maven-3/blob/trunk/maven-core/plugin-manager.txt">a design document</a> containing information on how stuff is managed there. Maven manages plugins with their dependencies and metadata so it seems like an interesting place to look for <a href="https://github.com/apache/maven-3/tree/trunk/maven-core/src/main/java/org/apache/maven/plugin">how they implemented things</a>.</p>\u000a\u000a<p>As specified in their documentation, <a href="http://maven.apache.org/guides/plugin/guide-java-plugin-development.html">maven plugins are using annotations</a> (<code>@goal</code>) on the classes, which is then used to find all the plugins registered with a particular <code>@goal</code>. While this approach is possible in static languages, it is not in interpreted languages as we only know what are all the possible classes / callables at one given point of the time, which may change.</p>\u000a\u000a<p>Mercurial uses <a href="http://mercurial.selenic.com/wiki/UsingExtensions">a central configuration file</a> (<code>~/.hgrc</code>), containing a mapping of the name of the plugin to the path it can be found.</p>\u000a\u000a<hr>\u000a\u000a<p><strong>Some more thoughts</strong></p>\u000a\u000a<p>While that's not an answer to this question, it is also interesting to note how setuptools entrypoints are implemented, and how they compare in term of performance, with the mercurial ones.</p>\u000a\u000a<p>When you ask for a particular entry point with setuptools, <a href="https://bitbucket.org/tarek/distribute/src/b825c6440dc6/pkg_resources.py#cl-2583">all the metadata are read at run time and a list is built that way</a>. This means this reading can take some time if you have a lot of python distributions on your path. Mercurial on the other side <em>hardcode</em> this information into a single file, which means you have to specify the complete path to your callable there, then registered callables are not "discovered" but "read" directly from the configuration file. This allows finer grained configuration on what's should be available and what should not be and seems faster. </p>\u000a\u000a<p>On the other side, as the python path can be changed at runtime, this means that callables provided on such a way would have to be checked against the path in order to know if they should be returned or not in all situations.</p>\u000a\u000a<hr>\u000a\u000a<p><strong>Why entrypoints are currently tied to packaging</strong></p>\u000a\u000a<p>It is also interesting to understand why entrypoints are tied to packaging in setuptools. The main reason is that it seems useful that python distributions can register part of themselves as extending an entrypoint at installation time: then installing means also registering the entrypoints: no need for an extra registration step.</p>\u000a\u000a<p>While this works fairly well in a large majority of cases (when python distributions are actually being installed) it doesn't when they are not installed or simply not packaged. In other words, from what I understand, you can't register an entrypoint at runtime, without having an .egg-info file.</p>\u000a
