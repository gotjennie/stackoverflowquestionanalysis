V<p>Numpy matrices are strictly 2-dimensional, while numpy arrays (ndarrays) are\u000aN-dimensional.  Matrix objects are a subclass of ndarray, so they inherit all\u000athe attributes and methods of ndarrays.</p>\u000a\u000a<p>The main advantage of numpy matrices is that they provide a convenient notation\u000afor matrix multiplication: if a and b are matrices, then a*b is their matrix\u000aproduct.</p>\u000a\u000a<pre><code>import numpy as np\u000a\u000aa=np.mat('4 3; 2 1')\u000ab=np.mat('1 2; 3 4')\u000aprint(a)\u000a# [[4 3]\u000a#  [2 1]]\u000aprint(b)\u000a# [[1 2]\u000a#  [3 4]]\u000aprint(a*b)\u000a# [[13 20]\u000a#  [ 5  8]]\u000a</code></pre>\u000a\u000a<p>On the other hand, as of Python 3.5, NumPy supports infix matrix multiplication using the <code>@</code> operator, so you can achieve the same convenience of matrix multiplication with ndarrays in Python >= 3.5.</p>\u000a\u000a<pre><code>import numpy as np\u000a\u000aa=np.array([[4, 3], [2, 1]])\u000ab=np.array([[1, 2], [3, 4]])\u000aprint(a@b)\u000a# [[13 20]\u000a#  [ 5  8]]\u000a</code></pre>\u000a\u000a<p>Both matrix objects and ndarrays have <code>.T</code> to return the transpose, but matrix\u000aobjects also have <code>.H</code> for the conjugate transpose, and <code>.I</code> for the inverse.</p>\u000a\u000a<p>In contrast, numpy arrays consistently abide by the rule that operations are\u000aapplied element-wise (except for the new <code>@</code> operator). Thus, if <code>a</code> and <code>b</code> are numpy arrays, then <code>a*b</code> is the array\u000aformed by multiplying the components element-wise:</p>\u000a\u000a<pre><code>c=np.array([[4, 3], [2, 1]])\u000ad=np.array([[1, 2], [3, 4]])\u000aprint(c*d)\u000a# [[4 6]\u000a#  [6 4]]\u000a</code></pre>\u000a\u000a<p>To obtain the result of matrix multiplication, you use <code>np.dot</code> (or <code>@</code> in Python >= 3.5, as shown above):</p>\u000a\u000a<pre><code>print(np.dot(c,d))\u000a# [[13 20]\u000a#  [ 5  8]]\u000a</code></pre>\u000a\u000a<p>The <code>**</code> operator also behaves differently:</p>\u000a\u000a<pre><code>print(a**2)\u000a# [[22 15]\u000a#  [10  7]]\u000aprint(c**2)\u000a# [[16  9]\u000a#  [ 4  1]]\u000a</code></pre>\u000a\u000a<p>Since <code>a</code> is a matrix, <code>a**2</code> returns the matrix product <code>a*a</code>.\u000aSince <code>c</code> is an ndarray, <code>c**2</code> returns an ndarray with each component squared\u000aelement-wise.</p>\u000a\u000a<p>There are other technical differences between matrix objects and ndarrays\u000a(having to do with np.ravel, item selection and sequence behavior).</p>\u000a\u000a<p>The main advantage of numpy arrays is that they are more general than\u000a2-dimensional matrices. What happens when you want a 3-dimensional array? Then\u000ayou have to use an ndarray, not a matrix object. Thus, learning to use matrix\u000aobjects is more work -- you have to learn matrix object operations, and\u000andarray operations.</p>\u000a\u000a<p>Writing a program that uses both matrices and arrays makes your life difficult\u000abecause you have to keep track of what type of object your variables are, lest\u000amultiplication return something you don't expect.</p>\u000a\u000a<p>In contrast, if you stick solely with ndarrays, then you can do everything\u000amatrix objects can do, and more, except with slightly different\u000afunctions/notation.</p>\u000a\u000a<p>If you are willing to give up the visual appeal of NumPy matrix product\u000anotation (which can be achieved almost as elegantly with ndarrays in Python >= 3.5), then I think NumPy arrays are definitely the way to go.</p>\u000a\u000a<p>PS. Of course, you really don't have to choose one at the expense of the other,\u000asince <code>np.asmatrix</code> and <code>np.asarray</code> allow you to convert one to the other (as\u000along as the array is 2-dimensional).</p>\u000a

V<p>There is no simple built-in string function that does what you're looking for, but you could use the more powerful <a href="http://www.regular-expressions.info/" rel="noreferrer">regular expressions</a>:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; import re\u000a&gt;&gt;&gt; [m.start() for m in re.finditer('test', 'test test test test')]\u000a[0, 5, 10, 15]\u000a</code></pre>\u000a\u000a<p>If you want to find overlapping matches, <a href="http://www.regular-expressions.info/lookaround.html" rel="noreferrer">lookahead</a> will do that:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; [m.start() for m in re.finditer('(?=tt)', 'ttt')]\u000a[0, 1]\u000a</code></pre>\u000a\u000a<p>If you want a reverse find-all without overlaps, you can combine positive and negative lookahead into an expression like this:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; search = 'tt'\u000a&gt;&gt;&gt; [m.start() for m in re.finditer('(?=%s)(?!.{1,%d}%s)' % (search, len(search)-1, search), 'ttt')]\u000a[1]\u000a</code></pre>\u000a\u000a<p><a href="http://docs.python.org/library/re.html#re.finditer" rel="noreferrer"><code>re.finditer</code></a> returns a <a href="http://wiki.python.org/moin/Generators" rel="noreferrer">generator</a>, so you could change the <code>[]</code> in the above to <code>()</code> to get a generator instead of a list which will be more efficient if you're only iterating through the results once.</p>\u000a

V<p><code>print</code> is just a thin wrapper that formats the inputs (space between args and newline at the end) and calls the write function of a given object. By default this object is <code>sys.stdout</code>, but you can pass a file using the "chevron" form. V<p>See <a href="https://docs.python.org/2.7/reference/compound_stmts.html#function-definitions" rel="noreferrer">Function Definitions</a> in the Language Reference.</p>\u000a\u000a<blockquote>\u000a  <p>If the form <code>*identifier</code> is\u000a  present, it is initialized to a tuple\u000a  receiving any excess positional\u000a  parameters, defaulting to the empty\u000a  tuple. If the form <code>**identifier</code> is\u000a  present, it is initialized to a new\u000a  dictionary receiving any excess\u000a  keyword arguments, defaulting to a new\u000a  empty dictionary.</p>\u000a</blockquote>\u000a\u000a<p>Also, see <a href="https://docs.python.org/2.7/reference/expressions.html#calls" rel="noreferrer">Function Calls</a>.</p>\u000a\u000a<p>Assuming that one knows what positional and keyword arguments are, here are some examples:</p>\u000a\u000a<p>Example 1:</p>\u000a\u000a<pre><code># Excess keyword argument (python 2) example:\u000adef foo(a, b, c, **args):\u000a    print "a = %s" % (a,)\u000a    print "b = %s" % (b,)\u000a    print "c = %s" % (c,)\u000a    print args\u000a\u000afoo(a="testa", d="excess", c="testc", b="testb", k="another_excess")\u000a</code></pre>\u000a\u000a<p>As you can see in the above example, we only have parameters <code>a, b, c</code> in the signature of the <code>foo</code> function. Since <code>d</code> and <code>k</code> are not present, they are put into the args dictionary. The output of the program is:</p>\u000a\u000a<pre><code>a = testa\u000ab = testb\u000ac = testc\u000a{'k': 'another_excess', 'd': 'excess'}\u000a</code></pre>\u000a\u000a<p>Example 2:</p>\u000a\u000a<pre><code># Excess positional argument (python 2) example:\u000adef foo(a, b, c, *args):\u000a    print "a = %s" % (a,)\u000a    print "b = %s" % (b,)\u000a    print "c = %s" % (c,)\u000a    print args\u000a\u000afoo("testa", "testb", "testc", "excess", "another_excess")\u000a</code></pre>\u000a\u000a<p>Here, since we're testing positional arguments, the excess ones have to be on the end, and <code>*args</code> packs them into a tuple, so the output of this program is:</p>\u000a\u000a<pre><code>a = testa\u000ab = testb\u000ac = testc\u000a('excess', 'another_excess')\u000a</code></pre>\u000a\u000a<p>You can also unpack a dictionary or a tuple into arguments of a function:</p>\u000a\u000a<pre><code>def foo(a,b,c,**args):\u000a    print "a=%s" % (a,)\u000a    print "b=%s" % (b,)\u000a    print "c=%s" % (c,)\u000a    print "args=%s" % (args,)\u000a\u000aargdict = dict(a="testa", b="testb", c="testc", excessarg="string")\u000afoo(**argdict)\u000a</code></pre>\u000a\u000a<p>Prints:</p>\u000a\u000a<pre><code>a=testa\u000ab=testb\u000ac=testc\u000aargs={'excessarg': 'string'}\u000a</code></pre>\u000a\u000a<p>And</p>\u000a\u000a<pre><code>def foo(a,b,c,*args):\u000a    print "a=%s" % (a,)\u000a    print "b=%s" % (b,)\u000a    print "c=%s" % (c,)\u000a    print "args=%s" % (args,)\u000a\u000aargtuple = ("testa","testb","testc","excess")\u000afoo(*argtuple)\u000a</code></pre>\u000a\u000a<p>Prints:</p>\u000a\u000a<pre><code>a=testa\u000ab=testb\u000ac=testc\u000aargs=('excess',)\u000a</code></pre>\u000a

V<p>the answer to that question depends somewhat on the particular python implementation.  </p>\u000a\u000a<p>To understand what this is all about, pay particular attention to the actual <code>file</code> object.  In your code, that object is mentioned only once, in an expression, and becomes inaccessible immediatly after the <code>read()</code> call returns.</p>\u000a\u000a<p>This means that the file object is garbage.  The only remaining question is "When will the garbage collecter collect the file object?".</p>\u000a\u000a<p>in CPython, which uses a reference counter, this kind of garbage is noticed immediately, and so it will be collected immediately.  This is not generally true of other python implementations.</p>\u000a\u000a<p>A better solution, to make sure that the file is closed, is this pattern:</p>\u000a\u000a<pre><code>with open('Path/to/file', 'r') as content_file:\u000a    content = content_file.read()\u000a</code></pre>\u000a\u000a<p>which will always close the file immediately after the block ends; even if an exception occurs.</p>\u000a\u000a<p>Edit:  To put a finer point on it:</p>\u000a\u000a<p>Other than <code>file.__exit__()</code>, which is "automatically" called in a <code>with</code> context manager setting, the only other way that <code>file.close()</code> is automatically called (that is, other than explicitly calling it yourself,) is via <code>file.__del__()</code>.  This leads us to the question of when does <code>__del__()</code> get called?</p>\u000a\u000a<blockquote>\u000a  <p>A correctly-written program cannot assume that finalizers will ever run at any point prior to program termination.</p>\u000a</blockquote>\u000a\u000a<p>-- <a href="http://blogs.msdn.com/b/oldnewthing/archive/2010/08/09/10047586.aspx" rel="noreferrer">http://blogs.msdn.com/b/oldnewthing/archive/2010/08/09/10047586.aspx</a></p>\u000a\u000a<p>In particular:</p>\u000a\u000a<blockquote>\u000a  <p>Objects are never explicitly destroyed; however, when they become unreachable they may be garbage-collected. <strong>An implementation is allowed to postpone garbage collection or omit it altogether</strong> \u2014 it is a matter of implementation quality how garbage collection is implemented, as long as no objects are collected that are still reachable.</p>\u000a  \u000a  <p>[...]</p>\u000a  \u000a  <p>CPython currently uses a reference-counting scheme with (optional) delayed detection of cyclically linked garbage, which collects most objects as soon as they become unreachable, but is not guaranteed to collect garbage containing circular references.</p>\u000a</blockquote>\u000a\u000a<p>-- <a href="https://docs.python.org/3.5/reference/datamodel.html#objects-values-and-types" rel="noreferrer">https://docs.python.org/3.5/reference/datamodel.html#objects-values-and-types</a></p>\u000a\u000a<p>(Emphasis mine)</p>\u000a\u000a<p>but as it suggests, other implementations may have other behavior.  As an example, PyPy <a href="https://pypy.readthedocs.org/en/release-2.4.x/garbage_collection.html" rel="noreferrer">has <em>6</em> different garbage collection implementations</a>!</p>\u000a

V<blockquote>\u000a  <p>"I do not know if it is out of ignorance, but i do not like that \u000a  kind of programming, as it is using exceptions to perform flow control."</p>\u000a</blockquote>\u000a\u000a<p>In the Python world, using exceptions for flow control is common and normal.</p>\u000a\u000a<p>Even the Python core developers use exceptions for flow-control and that style is heavily baked into the language (i.e.  the iterator protocol uses <a href="http://docs.python.org/2.7/library/exceptions.html#exceptions.StopIteration"><em>StopIteration</em></a> to signal loop termination).</p>\u000a\u000a<p>In addition, the try-except-style is used to prevent the race-conditions inherent in some of the <a href="http://docs.python.org/2.7/glossary.html#term-lbyl">"look-before-you-leap"</a> constructs.   For example, testing <a href="http://docs.python.org/2.7/library/os.path.html#os.path.exists"><em>os.path.exists</em></a> results in information that may be out-of-date by the time you use it.  Likewise, <a href="http://docs.python.org/2.7/library/queue.html#Queue.Queue.full"><em>Queue.full</em></a> returns information that may be stale.  The <a href="http://docs.python.org/2.7/glossary.html#term-eafp">try-except-else style</a> will produce more reliable code in these cases.</p>\u000a\u000a<blockquote>\u000a  <p>"It my understanding that exceptions are not errors, they should only\u000a  be used for exceptional conditions"</p>\u000a</blockquote>\u000a\u000a<p>In some other languages, that rule reflects their cultural norms as reflected in their libraries.  The "rule" is also based in-part on performance considerations for those languages.</p>\u000a\u000a<p>The Python cultural norm is somewhat different.  In many cases, you <em>must</em> use exceptions for control-flow.  Also, the use of exceptions in Python does not slow the surrounding code and calling code as it does in some compiled languages (i.e. CPython already implements code for exception checking at every step, regardless of whether you actually use exceptions or not).</p>\u000a\u000a<p>In other words, your understanding that "exceptions are for the exceptional" is a rule that makes sense in some other languages, but not for Python.</p>\u000a\u000a<blockquote>\u000a  <p>"However, if it is included in the language itself, there must be a\u000a  good reason for it, isn't it?"</p>\u000a</blockquote>\u000a\u000a<p>Besides helping to avoid race-conditions, exceptions are also very useful for pulling error-handling outside loops.   This is a necessary optimization in interpreted languages which do not tend to have automatic <a href="http://en.wikipedia.org/wiki/Loop-invariant_code_motion">loop invariant code motion</a>.</p>\u000a\u000a<p>Also, exceptions can simplify code quite a bit in common situations where the ability to handle an issue is far removed from where the issue arose.  For example, it is common to have top level user-interface code calling code for business logic which in turn calls low-level routines.  Situations arising in the low-level routines (such as duplicate records for unique keys in database accesses) can only be handled in top-level code (such as asking the user for a new key that doesn't conflict with existing keys).  The use of exceptions for this kind of control-flow allows the mid-level routines to completely ignore the issue and be nicely decoupled from that aspect of flow-control.</p>\u000a\u000a<p>There is a <a href="http://uberpython.wordpress.com/2012/09/23/why-im-not-leaving-python-for-go/">nice blog post on the indispensibility of exceptions here</a>.</p>\u000a\u000a<p>Also, see this StackOverFlow answer: <a href="http://stackoverflow.com/questions/180937/are-exceptions-really-for-exceptional-errors">Are exceptions really for exceptional errors?</a></p>\u000a\u000a<blockquote>\u000a  <p>"What is the reason for the try-except-else to exist?"</p>\u000a</blockquote>\u000a\u000a<p>The else-clause itself is interesting.  It runs when there is no exception but before the finally-clause.  That is its primary purpose.  </p>\u000a\u000a<p>Without the else-clause, the only option to run additional code before finalization would be the clumsy practice of adding the code to the try-clause.  That is clumsy because it risks\u000araising exceptions in code that wasn't intended to be protected by the try-block.</p>\u000a\u000a<p>The use-case of running additional unprotected code prior to finalization doesn't arise very often.  So, don't expect to see many examples in published code.  It is somewhat rare.</p>\u000a\u000a<p>Another use-case for the else-clause is to perform actions that must occur when no exception occurs and that do not occur when exceptions are handled.  For example:</p>\u000a\u000a<pre><code>   recip = float('Inf')\u000a   try:\u000a       recip = 1 / f(x)\u000a   except ZeroDivisionError:\u000a       logging.info('Infinite result')\u000a   else:\u000a       logging.info('Finite result')\u000a</code></pre>\u000a\u000a<p>Lastly, the most common use of an else-clause in a try-block is for a bit of beautification (aligning the exceptional outcomes and non-exceptional outcomes at the same level of indentation).  This use is always optional and isn't strictly necessary.</p>\u000a

V<p>The way <em><a href="http://docs.python.org/library/timeit.html#module-timeit" rel="noreferrer">timeit</a></em> works is to run setup code once and then make repeated calls to a series of statements.  So, if you want to test sorting, some care is required so that one pass at an in-place sort doesn't affect the next pass with already sorted data (that, of course, would make the <a href="http://en.wikipedia.org/wiki/Timsort" rel="noreferrer">Timsort</a> really shine because it performs best when the data already partially ordered).</p>\u000a\u000a<p>Here is an example of how to set up a test for sorting:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; import timeit\u000a\u000a&gt;&gt;&gt; setup = '''\u000aimport random\u000a\u000arandom.seed('slartibartfast')\u000as = [random.random() for i in range(1000)]\u000atimsort = list.sort\u000a'''\u000a\u000a&gt;&gt;&gt; print min(timeit.Timer('a=s[:]; timsort(a)', setup=setup).repeat(7, 1000))\u000a0.334147930145\u000a</code></pre>\u000a\u000a<p>Note that the series of statements makes a fresh copy of the unsorted data on every pass.</p>\u000a\u000a<p>Also, note the timing technique of running the measurement suite seven times and keeping only the best time -- this can really help reduce measurement distortions due to other processes running on your system.</p>\u000a\u000a<p>Those are my tips for using timeit correctly.  Hope this helps :-)</p>\u000a

V<p>Python does <strong>not</strong> have a trailing <code>if</code> <strong>statement</strong>.</p>\u000a\u000a<p>There are two kinds of <code>if</code> in Python:</p>\u000a\u000a<ol>\u000a<li><p><code>if</code> statement:</p>\u000a\u000a<pre><code>if condition: statement\u000aif condition:\u000a    block\u000a</code></pre></li>\u000a<li><p><code>if</code> <strong>expression</strong> (introduced in Python 2.5)</p>\u000a\u000a<pre><code>expression_if_true if condition else expression_if_false\u000a</code></pre></li>\u000a</ol>\u000a\u000a<p>And note, that both <code>print a</code> and <code>b = a</code> are statements. Only the <code>a</code> part is an expression. So if you write</p>\u000a\u000a<pre><code>print a if b else 0\u000a</code></pre>\u000a\u000a<p>it means</p>\u000a\u000a<pre><code>print (a if b else 0)\u000a</code></pre>\u000a\u000a<p>and similarly when you write</p>\u000a\u000a<pre><code>x = a if b else 0\u000a</code></pre>\u000a\u000a<p>it means</p>\u000a\u000a<pre><code>x = (a if b else 0)\u000a</code></pre>\u000a\u000a<p>Now what would it print/assign if there was no <code>else</code> clause? The print/assignment is <strong>still there</strong>.</p>\u000a\u000a<p>And note, that if you don't want it to be there, you can always write the regular <code>if</code> statement on a single line, though it's less readable and there is really no reason to avoid the two-line variant.</p>\u000a

V<p>This tries to be a little more race-free than the other solutions.  (The <code>with</code> keyword is new in Python 2.5.)</p>\u000a\u000a<pre><code>import os\u000adef touch(fname, times=None):\u000a    with open(fname, 'a'):\u000a        os.utime(fname, times)\u000a</code></pre>\u000a\u000a<p>Roughly equivalent to this.</p>\u000a\u000a<pre><code>import os\u000adef touch(fname, times=None):\u000a    fhandle = open(fname, 'a')\u000a    try:\u000a        os.utime(fname, times)\u000a    finally:\u000a        fhandle.close()\u000a</code></pre>\u000a\u000a<p>Now, to really make it race-free, you need to use <a href="http://www.kernel.org/doc/man-pages/online/pages/man3/lutimes.3.html" rel="noreferrer"><code>futimes</code></a> and change the timestamp of the open filehandle, instead of opening the file and then changing the timestamp on the filename (which may have been renamed).  Unfortunately, Python doesn't seem to provide a way to call <code>futimes</code> without going through <code>ctypes</code> or similar...</p>\u000a\u000a<hr>\u000a\u000a<p><strong>EDIT</strong></p>\u000a\u000a<p>As noted by <a href="http://stackoverflow.com/users/34910/nate-parsons">Nate Parsons</a>, Python 3.3 will <a href="http://bugs.python.org/issue10812" rel="noreferrer">add</a> <a href="http://docs.python.org/dev/library/os.html#path-fd" rel="noreferrer">specifying a file descriptor</a> (when <a href="http://docs.python.org/dev/library/os.html#os.supports_fd" rel="noreferrer"><code>os.supports_fd</code></a>) to functions such as <a href="http://docs.python.org/dev/library/os.html#os.utime" rel="noreferrer"><code>os.utime</code></a>, which will use the <code>futimes</code> syscall instead of the <code>utimes</code> syscall under the hood.  In other words:</p>\u000a\u000a<pre><code>import os\u000adef touch(fname, mode=0o666, dir_fd=None, **kwargs):\u000a    flags = os.O_CREAT | os.O_APPEND\u000a    with os.fdopen(os.open(fname, flags=flags, mode=mode, dir_fd=dir_fd)) as f:\u000a        os.utime(f.fileno() if os.utime in os.supports_fd else fname,\u000a            dir_fd=None if os.supports_fd else dir_fd, **kwargs)\u000a</code></pre>\u000a

V<p><strong>Iteration</strong> is a general term for taking each item of something, one after another. Any time you use a loop, explicit or implicit, to go over a group of items, that is iteration.</p>\u000a\u000a<p>In Python, <strong>iterable</strong> and <strong>iterator</strong> have specific meanings.</p>\u000a\u000a<p>An <strong>iterable</strong> is an object that has an <code>__iter__</code> method which returns an <strong>iterator</strong>, or which defines a <code>__getitem__</code> method that can take sequential indexes starting from zero (and raises an <code>IndexError</code> when the indexes are no longer valid). So an <strong>iterable</strong> is an object that you can get an <strong>iterator</strong> from.</p>\u000a\u000a<p>An <strong>iterator</strong> is an object with a <code>next</code> (Python 2) or <code>__next__</code> (Python 3) method. </p>\u000a\u000a<p>Whenever you use a <code>for</code> loop, or <code>map</code>, or a list comprehension, etc. in Python, the <code>next</code> method is called automatically to get each item from the <strong>iterator</strong>, thus going through the process of <strong>iteration</strong>.</p>\u000a\u000a<p>A good place to start learning would be the <a href="https://docs.python.org/3/tutorial/classes.html#iterators">iterators section of the tutorial</a> and the <a href="https://docs.python.org/dev/library/stdtypes.html#iterator-types">iterator types section of the standard types page</a>. After you understand the basics, try the <a href="https://docs.python.org/dev/howto/functional.html#iterators">iterators section of the Functional Programming HOWTO</a>.</p>\u000a

V<p>Note that best practice in current versions of Python is to use <i>new-style</i> classes, i.e.</p>\u000a\u000a<pre><code>class Foo(object):\u000a...\u000a</code></pre>\u000a\u000a<p>Also, there's a difference between an 'object' and a 'class'. To build a dictionary from an arbitrary <i>object</i>, it's sufficient to use <code>__dict__</code>. Usually, you'll declare your methods at class level and your attributes at instance level, so <code>__dict__</code> should be fine. For example:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; class A(object):\u000a...   def __init__(self):\u000a...     self.b = 1\u000a...     self.c = 2\u000a...   def do_nothing(self):\u000a...     pass\u000a...\u000a&gt;&gt;&gt; a = A()\u000a&gt;&gt;&gt; a.__dict__\u000a{'c': 2, 'b': 1}\u000a</code></pre>\u000a\u000a<p>Alternatively, depending on what you want to do, it might be nice to inherit from dict. Then your class is <em>already</em> a dictionary, and if you want you can override getattr and/or setattr to call through and set the dict. For example:</p>\u000a\u000a<pre><code> class Foo(dict):\u000a     def __init__(self):\u000a         pass\u000a     def __getattr__(self, attr):\u000a         return self[attr]\u000a\u000a     # etc...\u000a</code></pre>\u000a

V<p>It's a strange construct even to seasoned Python coders. When used in conjunction with for-loops it basically means "find some item in the iterable, else if none was found do ...". As in:</p>\u000a\u000a<pre><code>found_obj = None\u000afor obj in objects:\u000a    if obj.key == search_key:\u000a        found_obj = obj\u000a        break\u000aelse:\u000a    print 'No object found.'\u000a</code></pre>\u000a\u000a<p>But anytime you see this construct, a better alternative is to either encapsulate the search in a function:</p>\u000a\u000a<pre><code>def find_obj(search_key):\u000a    for obj in objects:\u000a        if obj.key == search_key:\u000a            return obj\u000a</code></pre>\u000a\u000a<p>Or use a list comprehension:</p>\u000a\u000a<pre><code>matching_objs = [o for o in objects if o.key == search_key]\u000aif matching_objs:\u000a    print 'Found', matching_objs[0]\u000aelse:\u000a    print 'No object found.'\u000a</code></pre>\u000a\u000a<p>It is not semantically equivalent to the other two versions, but works good enough in non-performance critical code where it doesn't matter whether you iterate the whole list or not. Others may disagree, but I personally would avoid ever using the for-else or while-else blocks in production code. </p>\u000a\u000a<p>See also <a href="http://mail.python.org/pipermail/python-ideas/2009-October/006155.html" rel="noreferrer">[Python-ideas] Summary of for...else threads</a></p>\u000a

V<p>Because "append" existed long before "pop" was thought of. <a href="http://www.dalkescientific.com/writings/diary/archive/2009/03/27/python_0_9_1p1.html">Python 0.9.1</a> supported list.append in early 1991. By comparison, here's part of a <a href="http://groups.google.com/group/comp.lang.python/browse_thread/thread/48a26add2d996268/57ab953cb73bc9c7?q=list+pop+group:comp.lang.python+author:guido#">discussion on comp.lang.python</a> about adding pop in 1997. Guido wrote:</p>\u000a\u000a<blockquote>\u000a  <p>To implement a stack, one would need\u000a  to add a list.pop() primitive  (and\u000a  no, I'm not against this particular\u000a  one on the basis of any  principle). \u000a  list.push() could be added for\u000a  symmetry with list.pop()  but I'm not\u000a  a big fan of multiple names for the\u000a  same operation --  sooner or later\u000a  you're going to read code that uses\u000a  the other one, so  you need to learn\u000a  both, which is more cognitive load.</p>\u000a</blockquote>\u000a\u000a<p>You can also see he discusses the idea of if push/pop/put/pull should be at element [0] or after element [-1] where he posts a reference to Icon's list:</p>\u000a\u000a<blockquote>\u000a  <p>I stil think that all this is best\u000a  left out of the list object \u000a  implementation -- if you need a stack,\u000a  or a queue, with particular \u000a  semantics, write a little class that\u000a  uses a lists</p>\u000a</blockquote>\u000a\u000a<p>In other words, for stacks implemented directly as Python lists, which already supports fast append(), and del list[-1], it makes sense that list.pop() work by default on the last element. Even if other languages do it differently.</p>\u000a\u000a<p>Implicit here is that most people need to append to a list, but many fewer have occasion to treat lists as stacks, which is why list.append came in so much earlier.</p>\u000a
