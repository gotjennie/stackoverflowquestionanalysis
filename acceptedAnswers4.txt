V<p>There's not really any "raw <em>string</em>"; there are raw <em>string literals</em>, which are exactly the string literals marked by an <code>'r'</code> before the opening quote.</p>\u000a\u000a<p>A "raw string literal" is a slightly different syntax for a string literal, in which a backslash, <code>\u005c</code>, is taken as meaning "just a backslash" (except when it comes right before a quote that would otherwise terminate the literal) -- no "escape sequences" to represent newlines, tabs, backspaces, form-feeds, and so on.  In normal string literals, each backslash must be doubled up to avoid being taken as the start of an escape sequence.</p>\u000a\u000a<p>This syntax variant exists mostly because the syntax of regular expression patterns is heavy with backslashes (but never at the end, so the "except" clause above doesn't matter) and it looks a bit better when you avoid doubling up each of them -- that's all.  It also gained some popularity to express native Windows file paths (with backslashes instead of regular slashes like on other platforms), but that's very rarely needed (since normal slashes mostly work fine on Windows too) and imperfect (due to the "except" clause above).</p>\u000a\u000a<p><code>r'...'</code> is a byte string (in Python 2.*), <code>ur'...'</code> is a Unicode string (again, in Python 2.*), and any of the other three kinds of quoting also produces exactly the same types of strings (so for example <code>r'...'</code>, <code>r'''...'''</code>, <code>r"..."</code>, <code>r"""..."""</code> are all byte strings, and so on).</p>\u000a\u000a<p>Not sure what you mean by "going <em>back</em>" - there is no intrinsically back and forward directions, because there's no raw string <strong>type</strong>, it's just an alternative syntax to express perfectly normal string objects, byte or unicode as they may be.</p>\u000a\u000a<p>And yes, in Python 2.*, <code>u'...'</code> <strong>is</strong> of course always distinct from just <code>'...'</code> -- the former is a unicode string, the latter is a byte string. What encoding the literal might be expressed in is a completely orthogonal issue.</p>\u000a\u000a<p>E.g., consider (Python 2.6):</p>\u000a\u000a<pre><code>&gt;&gt;&gt; sys.getsizeof('ciao')\u000a28\u000a&gt;&gt;&gt; sys.getsizeof(u'ciao')\u000a34\u000a</code></pre>\u000a\u000a<p>The Unicode object of course takes more memory space (very small difference for a very short string, obviously ;-).</p>\u000a

V<p><code>_</code> has 3 main conventional uses in Python:</p>\u000a\u000a<ol>\u000a<li>To hold the result of the last executed statement in an interactive\u000ainterpreter session. V<p>Actually, a concise rule for Python Scope resolution, from <a href="http://rads.stackoverflow.com/amzn/click/0596513984" rel="noreferrer">Learning Python, 3rd. Ed.</a>. (These rules are specific to variable names, not attributes. If you reference it without a period, these rules apply)</p>\u000a\u000a<p>LEGB Rule.</p>\u000a\u000a<p><strong>L</strong>, Local \u2014 Names assigned in any way within a function (<code>def</code> or <code>lambda</code>)), and not declared global in that function.</p>\u000a\u000a<p><strong>E</strong>, Enclosing-function locals \u2014 Name in the local scope of any and all statically enclosing functions (<code>def</code> or <code>lambda</code>), from inner to outer.</p>\u000a\u000a<p><strong>G</strong>, Global (module) \u2014 Names assigned at the top-level of a module file, or by executing a <code>global</code> statement in a <code>def</code> within the file.</p>\u000a\u000a<p><strong>B</strong>, Built-in (Python) \u2014 Names preassigned in the built-in names module : <code>open</code>,<code>range</code>,<code>SyntaxError</code>,...</p>\u000a\u000a<p>So, in the case of</p>\u000a\u000a<pre><code>code1\u000aclass Foo:\u000a   code2\u000a   def spam.....\u000a      code3\u000a      for code4..:\u000a       code5\u000a       x()\u000a</code></pre>\u000a\u000a<p>The for loop does not have its own namespace. In LEGB order, the scopes would be </p>\u000a\u000a<p>L : local, in <code>def spam</code> (in <code>code3</code>, <code>code 4</code>, <code>code5</code>).</p>\u000a\u000a<p>E : Enclosed function, any enclosing functions (if the whole example were in another <code>def</code>)</p>\u000a\u000a<p>G : Global. Were there any <code>x</code> declared globally in the module (<code>code1</code>)?</p>\u000a\u000a<p>B : Any builtin <code>x</code> in Python.</p>\u000a\u000a<p><code>x</code> will never be found in <code>code2</code> (even in cases where you might expect it would, see <a href="http://stackoverflow.com/a/23471004/2810305">Antti's answer</a> or <a href="http://stackoverflow.com/q/13905741/2810305">here</a>).</p>\u000a

V<p>I'd recommend using Python's <code>with</code> statement for managing resources that need to be cleaned up. V<p>A comment in the Python source code for float objects acknowledges that:</p>\u000a\u000a<blockquote>\u000a  <p><a href="https://hg.python.org/cpython/file/ea33b61cac74/Objects/floatobject.c#l285" rel="noreferrer">Comparison is pretty much a nightmare</a></p>\u000a</blockquote>\u000a\u000a<p>This is especially true when comparing a float to an integer, because, unlike floats, integers in Python can be arbitrarily large and are always exact. Trying to cast the integer to a float might lose precision and make the comparison inaccurate. Trying to cast the float to an integer is not going to work either because any fractional part will be lost.</p>\u000a\u000a<p>To get around this problem, Python performs a series of checks, returning the result if one of the checks succeeds. It compares the signs of the two values, then whether the integer is "too big" to be a float, then compares the exponent of the float to the length of the integer. If all of these checks fail, it is necessary to construct two new Python objects to compare in order to obtain the result.</p>\u000a\u000a<p>When comparing a float <code>v</code> to an integer/long <code>w</code>, the worst case is that:</p>\u000a\u000a<ul>\u000a<li><code>v</code> and <code>w</code> have the same sign (both positive or both negative),</li>\u000a<li>the integer <code>w</code> has few enough bits that it can be held in the <a href="http://stackoverflow.com/a/2550799/3923281"><code>size_t</code></a> type (typically 32 or 64 bits),</li>\u000a<li>the integer <code>w</code> has at least 49 bits,</li>\u000a<li>the exponent of the float <code>v</code> is the same as the number of bits in <code>w</code>.</li>\u000a</ul>\u000a\u000a<p>And this is exactly what we have for the values in the question:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; import math\u000a&gt;&gt;&gt; math.frexp(562949953420000.7) # gives the float's (significand, exponent) pair\u000a(0.9999999999976706, 49)\u000a&gt;&gt;&gt; (562949953421000).bit_length()\u000a49\u000a</code></pre>\u000a\u000a<p>We see that 49 is both the exponent of the float and the number of bits in the integer. Both numbers are positive and so the four criteria above are met.</p>\u000a\u000a<p>Choosing one of the values to be larger (or smaller) can change the number of bits of the integer, or the value of the exponent, and so Python is able to determine the result of the comparison without performing the expensive final check.</p>\u000a\u000a<p>This is specific to the CPython implementation of the language.</p>\u000a\u000a<hr>\u000a\u000a<h3>The comparison in more detail</h3>\u000a\u000a<p>The <a href="https://hg.python.org/cpython/file/ea33b61cac74/Objects/floatobject.c#l301" rel="noreferrer"><code>float_richcompare</code></a> function handles the comparison between two values <code>v</code> and <code>w</code>.</p>\u000a\u000a<p>Below is a step-by-step description of the checks that the function performs. The comments in the Python source are actually very helpful when trying to understand what the function does, so I've left them in where relevant. I've also summarised these checks in a list at the foot of the answer.</p>\u000a\u000a<p>The main idea is to map the Python objects <code>v</code> and <code>w</code> to two appropriate C doubles, <code>i</code> and <code>j</code>, which can then be easily compared to give the correct result. Both Python 2 and Python 3 use the same ideas to do this (the former just handles <code>int</code> and <code>long</code> types separately).</p>\u000a\u000a<p>The first thing to do is check that <code>v</code> is definitely a Python float and map it to a C double <code>i</code>. Next the function looks at whether <code>w</code> is also a float and maps it to a C double <code>j</code>. This is the best case scenario for the function as all the other checks can be skipped. The function also checks to see whether <code>v</code> is <code>inf</code> or <code>nan</code>: </p>\u000a\u000a<pre class="lang-C prettyprint-override"><code>static PyObject*\u000afloat_richcompare(PyObject *v, PyObject *w, int op)\u000a{\u000a    double i, j;\u000a    int r = 0;\u000a    assert(PyFloat_Check(v));       \u000a    i = PyFloat_AS_DOUBLE(v);       \u000a\u000a    if (PyFloat_Check(w))           \u000a        j = PyFloat_AS_DOUBLE(w);   \u000a\u000a    else if (!Py_IS_FINITE(i)) {\u000a        if (PyLong_Check(w))\u000a            j = 0.0;\u000a        else\u000a            goto Unimplemented;\u000a    }\u000a</code></pre>\u000a\u000a<p>Now we know that if <code>w</code> failed these checks, it is not a Python float. Now the function checks if it's a Python integer. If this is the case, the easiest test is to extract the sign of <code>v</code> and the sign of <code>w</code> (return <code>0</code> if zero, <code>-1</code> if negative, <code>1</code> if positive). If the signs are different, this is all the information needed to return the result of the comparison:</p>\u000a\u000a<pre class="lang-C prettyprint-override"><code>    else if (PyLong_Check(w)) {\u000a        int vsign = i == 0.0 ? 0 : i &lt; 0.0 ? -1 : 1;\u000a        int wsign = _PyLong_Sign(w);\u000a        size_t nbits;\u000a        int exponent;\u000a\u000a        if (vsign != wsign) {\u000a            /* Magnitudes are irrelevant -- the signs alone\u000a             * determine the outcome.\u000a             */\u000a            i = (double)vsign;\u000a            j = (double)wsign;\u000a            goto Compare;\u000a        }\u000a    }   \u000a</code></pre>\u000a\u000a<p>If this check failed, then <code>v</code> and <code>w</code> have the same sign. </p>\u000a\u000a<p>The next check counts the number of bits in the integer <code>w</code>. If it has too many bits then it can't possibly be held as a float and so must be larger in magnitude than the float <code>v</code>:</p>\u000a\u000a<pre class="lang-C prettyprint-override"><code>    nbits = _PyLong_NumBits(w);\u000a    if (nbits == (size_t)-1 &amp;&amp; PyErr_Occurred()) {\u000a        /* This long is so large that size_t isn't big enough\u000a         * to hold the # of bits.  Replace with little doubles\u000a         * that give the same outcome -- w is so large that\u000a         * its magnitude must exceed the magnitude of any\u000a         * finite float.\u000a         */\u000a        PyErr_Clear();\u000a        i = (double)vsign;\u000a        assert(wsign != 0);\u000a        j = wsign * 2.0;\u000a        goto Compare;\u000a    }\u000a</code></pre>\u000a\u000a<p>On the other hand, if the integer <code>w</code> has 48 or fewer bits, it can safely turned in a C double <code>j</code> and compared:</p>\u000a\u000a<pre class="lang-C prettyprint-override"><code>    if (nbits &lt;= 48) {\u000a        j = PyLong_AsDouble(w);\u000a        /* It's impossible that &lt;= 48 bits overflowed. */\u000a        assert(j != -1.0 || ! PyErr_Occurred());\u000a        goto Compare;\u000a    }\u000a</code></pre>\u000a\u000a<p>From this point onwards, we know that <code>w</code> has 49 or more bits. It will be convenient to treat <code>w</code> as a positive integer, so change the sign and the comparison operator as necessary:</p>\u000a\u000a<pre class="lang-C prettyprint-override"><code>    if (nbits &lt;= 48) {\u000a        /* "Multiply both sides" by -1; this also swaps the\u000a         * comparator.\u000a         */\u000a        i = -i;\u000a        op = _Py_SwappedOp[op];\u000a    }\u000a</code></pre>\u000a\u000a<p>Now the function looks at the exponent of the float. Recall that a float can be written (ignoring sign) as significand * 2<sup>exponent</sup> and that the significand represents a number between 0.5 and 1:</p>\u000a\u000a<pre class="lang-C prettyprint-override"><code>    (void) frexp(i, &amp;exponent);\u000a    if (exponent &lt; 0 || (size_t)exponent &lt; nbits) {\u000a        i = 1.0;\u000a        j = 2.0;\u000a        goto Compare;\u000a    }\u000a</code></pre>\u000a\u000a<p>This checks two things. If the exponent is less than 0 then the float is smaller than 1 (and so smaller in magnitude than any integer). Or, if the exponent is less than the number of bits in <code>w</code> then we have that <code>v &lt; |w|</code> since significand * 2<sup>exponent</sup> is less than 2<sup>nbits</sup>. </p>\u000a\u000a<p>Failing these two checks, the function looks to see whether the exponent is greater than the number of bit in <code>w</code>. This shows that significand * 2<sup>exponent</sup> is greater than 2<sup>nbits</sup> and so <code>v &gt; |w|</code>:</p>\u000a\u000a<pre class="lang-C prettyprint-override"><code>    if ((size_t)exponent &gt; nbits) {\u000a        i = 2.0;\u000a        j = 1.0;\u000a        goto Compare;\u000a    }\u000a</code></pre>\u000a\u000a<p>If this check did not succeed we know that the exponent of the float <code>v</code> is the same as the number of bits in the integer <code>w</code>.</p>\u000a\u000a<p>The only way that the two values can be compared now is to construct two new Python integers from <code>v</code> and <code>w</code>. The idea is to discard the fractional part of <code>v</code>, double the integer part, and then add one. <code>w</code> is also doubled and these two new Python objects can be compared to give the correct return value. Using an example with small values, <code>4.65 &lt; 4</code> would be determined by the comparison <code>(2*4)+1 == 9 &lt; 8 == (2*4)</code> (returning false).</p>\u000a\u000a<pre class="lang-C prettyprint-override"><code>    {\u000a        double fracpart;\u000a        double intpart;\u000a        PyObject *result = NULL;\u000a        PyObject *one = NULL;\u000a        PyObject *vv = NULL;\u000a        PyObject *ww = w;\u000a\u000a        // snip\u000a\u000a        fracpart = modf(i, &amp;intpart); // split i (the double that v mapped to)\u000a        vv = PyLong_FromDouble(intpart);\u000a\u000a        // snip\u000a\u000a        if (fracpart != 0.0) {\u000a            /* Shift left, and or a 1 bit into vv\u000a             * to represent the lost fraction.\u000a             */\u000a            PyObject *temp;\u000a\u000a            one = PyLong_FromLong(1);\u000a\u000a            temp = PyNumber_Lshift(ww, one); // left-shift doubles an integer\u000a            ww = temp;\u000a\u000a            temp = PyNumber_Lshift(vv, one);\u000a            vv = temp;\u000a\u000a            temp = PyNumber_Or(vv, one); // a doubled integer is even, so this adds 1\u000a            vv = temp;\u000a        }\u000a        // snip\u000a    }\u000a}\u000a</code></pre>\u000a\u000a<p>For brevity I've left out the additional error-checking and garbage-tracking Python has to do when it creates these new objects. Needless to say, this adds additional overhead and explains why the values highlighted in the question are significantly slower to compare than others.</p>\u000a\u000a<hr>\u000a\u000a<p>Here is a summary of the checks that are performed by the comparison function.</p>\u000a\u000a<p>Let <code>v</code> be a float and cast it as a C double. If they are different then we know <code>v</code> and <code>w</code> are different and which is the greater value.</p></li>\u000a<li><p>(<em>The signs are the same.</em>) Check whether <code>w</code> has too many bits to be a float (more than <code>size_t</code>). V<p>I've had a lot of experience running a compiled regex 1000s of times versus compiling on-the-fly, and have not noticed any perceivable difference. V<p>It's cultural.  In Python, you don't write to other classes' instance or class variables.  In Java, nothing prevents you from doing the same if you <em>really</em> want to - after all, you can always edit the source of the class itself to achieve the same effect. Python drops that pretence of security and encourages programmers to be responsible. In practice, this works very nicely.</p>\u000a\u000a<p>If you want to emulate private variables for some reason, you can always use the <code>__</code> prefix from <a href="http://www.python.org/dev/peps/pep-0008/">PEP 8</a>. Python mangles the names of variables like <code>__foo</code> so that they're not easily visible to code outside the class that contains them (although you <em>can</em> get around it if you're determined enough, just like you <em>can</em> get around Java's protections if you work at it).</p>\u000a\u000a<p>By the same convention, the <code>_</code> prefix means <strong>stay away even if you're not technically prevented from doing so</strong>. You don't play around with another class's variables that look like <code>__foo</code> or <code>_bar</code>.</p>\u000a

V<p>Alright, it's the usual "<code>super()</code> cannot be used with an old-style class".</p>\u000a\u000a<p>However, the important point is that <strong>the correct test</strong> for "is this a new-style <strong>instance</strong> (i.e. object)?" is</p>\u000a\u000a<pre><code>&gt;&gt;&gt; class OldStyle: pass\u000a&gt;&gt;&gt; instance = OldStyle()\u000a&gt;&gt;&gt; issubclass(instance.__class__, object)\u000aFalse\u000a</code></pre>\u000a\u000a<p>and not (as in the question):</p>\u000a\u000a<pre><code>&gt;&gt;&gt; isinstance(instance, object)\u000aTrue\u000a</code></pre>\u000a\u000a<p>For <strong>classes</strong>, the correct "is this a new-style class" test is:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; issubclass(OldStyle, object)  # OldStyle is not a new-style class\u000aFalse\u000a&gt;&gt;&gt; issubclass(int, object)  # int is a new-style class\u000aTrue\u000a</code></pre>\u000a\u000a<p>The <strong>crucial point</strong> is that with old-style classes, the <em>class</em> of an instance and its <em>type</em> are distinct. Here, <code>OldStyle().__class__</code> is <code>OldStyle</code>, which does not inherit from <code>object</code>, while <code>type(OldStyle())</code> is the <code>instance</code> type, which <em>does</em> inherit from <code>object</code>.  Basically, an old-style class just creates objects of type <code>instance</code> (whereas a new-style class creates objects whose type is the class itself). This is probably why the instance <code>OldStyle()</code> is an <code>object</code>: its <code>type()</code> inherits from <code>object</code> (the fact that  its class does <em>not</em> inherit from <code>object</code> does not count: old-style classes merely construct new objects of type <code>instance</code>). Partial reference: <a href="http://stackoverflow.com/a/9699961/42973">http://stackoverflow.com/a/9699961/42973</a>.</p>\u000a\u000a<p>PS: The difference between a new-style class and an old-style one can also be seen with:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; type(OldStyle)  # OldStyle creates objects but is not itself a type\u000aclassobj\u000a&gt;&gt;&gt; isinstance(OldStyle, type)\u000aFalse\u000a&gt;&gt;&gt; type(int)  # A new-style class is a type\u000atype\u000a</code></pre>\u000a\u000a<p>(old-style classes are <em>not</em> types, so they cannot be the type of their instances).</p>\u000a

Here's one:</p>\u000a\u000a<pre><code>def permutations(iterable, r=None):\u000a    # permutations('ABCD', 2) --&gt; AB AC AD BA BC BD CA CB CD DA DB DC\u000a    # permutations(range(3)) --&gt; 012 021 102 120 201 210\u000a    pool = tuple(iterable)\u000a    n = len(pool)\u000a    r = n if r is None else r\u000a    if r &gt; n:\u000a        return\u000a    indices = range(n)\u000a    cycles = range(n, n-r, -1)\u000a    yield tuple(pool[i] for i in indices[:r])\u000a    while n:\u000a        for i in reversed(range(r)):\u000a            cycles[i] -= 1\u000a            if cycles[i] == 0:\u000a                indices[i:] = indices[i+1:] + indices[i:i+1]\u000a                cycles[i] = n - i\u000a            else:\u000a                j = cycles[i]\u000a                indices[i], indices[-j] = indices[-j], indices[i]\u000a                yield tuple(pool[i] for i in indices[:r])\u000a                break\u000a        else:\u000a            return\u000a</code></pre>\u000a\u000a<p>And another, based on <code>itertools.product</code>:</p>\u000a\u000a<pre><code>def permutations(iterable, r=None):\u000a    pool = tuple(iterable)\u000a    n = len(pool)\u000a    r = n if r is None else r\u000a    for indices in product(range(n), repeat=r):\u000a        if len(set(indices)) == r:\u000a            yield tuple(pool[i] for i in indices)\u000a</code></pre>\u000a

V<p>I have an approach which I think is interesting and a bit different from the rest.  The main difference in my approach, compared to some of the others, is in how the image segmentation step is performed--I used the <a href="http://scikit-learn.org/stable/modules/generated/sklearn.cluster.DBSCAN.html#sklearn.cluster.DBSCAN">DBSCAN</a> clustering algorithm from Python's scikit-learn; it's optimized for finding somewhat amorphous shapes that may not necessarily have a single clear centroid.</p>\u000a\u000a<p>At the top level, my approach is fairly simple and can be broken down into about 3 steps.  First I apply a threshold (or actually, the logical "or" of two separate and distinct thresholds).  As with many of the other answers, I assumed that the Christmas tree would be one of the brighter objects in the scene, so the first threshold is just a simple monochrome brightness test; any pixels with values above 220 on a 0-255 scale (where black is 0 and white is 255) are saved to a binary black-and-white image.  The second threshold tries to look for red and yellow lights, which are particularly prominent in the trees in the upper left and lower right of the six images, and stand out well against the blue-green background which is prevalent in most of the photos.  I convert the rgb image to hsv space, and require that the hue is either less than 0.2 on a 0.0-1.0 scale (corresponding roughly to the border between yellow and green) or greater than 0.95 (corresponding to the border between purple and red) and additionally I require bright, saturated colors: saturation and value must both be above 0.7.  The results of the two threshold procedures are logically "or"-ed together, and the resulting matrix of black-and-white binary images is shown below:</p>\u000a\u000a<p><img src="https://i.stack.imgur.com/iIkWV.png" alt="Christmas trees, after thresholding on HSV as well as monochrome brightness"></p>\u000a\u000a<p>You can clearly see that each image has one large cluster of pixels roughly corresponding to the location of each tree, plus a few of the images also have some other small clusters corresponding either to lights in the windows of some of the buildings, or to a background scene on the horizon.  The next step is to get the computer to recognize that these are separate clusters, and label each pixel correctly with a cluster membership ID number.</p>\u000a\u000a<p>For this task I chose <a href="http://en.wikipedia.org/wiki/DBSCAN">DBSCAN</a>.  There is a pretty good visual comparison of how DBSCAN typically behaves, relative to other clustering algorithms, available <a href="http://scikit-learn.org/stable/auto_examples/cluster/plot_cluster_comparison.html">here</a>.  As I said earlier, it does well with amorphous shapes.  The output of DBSCAN, with each cluster plotted in a different color, is shown here:</p>\u000a\u000a<p><img src="https://i.stack.imgur.com/rWSqn.png" alt="DBSCAN clustering output"></p>\u000a\u000a<p>There are a few things to be aware of when looking at this result.  First is that DBSCAN requires the user to set a "proximity" parameter in order to regulate its behavior, which effectively controls how separated a pair of points must be in order for the algorithm to declare a new separate cluster rather than agglomerating a test point onto an already pre-existing cluster.  I set this value to be 0.04 times the size along the diagonal of each image.  Since the images vary in size from roughly VGA up to about HD 1080, this type of scale-relative definition is critical.</p>\u000a\u000a<p>Another point worth noting is that the DBSCAN algorithm as it is implemented in scikit-learn has memory limits which are fairly challenging for some of the larger images in this sample.  Therefore, for a few of the larger images, I actually had to "decimate" (i.e., retain only every 3rd or 4th pixel and drop the others) each cluster in order to stay within this limit.  As a result of this culling process, the remaining individual sparse pixels are difficult to see on some of the larger images.  Therefore, for display purposes only, the color-coded pixels in the above images have been effectively "dilated" just slightly so that they stand out better.  It's purely a cosmetic operation for the sake of the narrative; although there are comments mentioning this dilation in my code, rest assured that it has nothing to do with any calculations that actually matter.</p>\u000a\u000a<p>Once the clusters are identified and labeled, the third and final step is easy: I simply take the largest cluster in each image (in this case, I chose to measure "size" in terms of the total number of member pixels, although one could have just as easily instead used some type of metric that gauges physical extent) and compute the convex hull for that cluster.  The convex hull then becomes the tree border.  The six convex hulls computed via this method are shown below in red:</p>\u000a\u000a<p><img src="https://i.stack.imgur.com/sl6Ar.jpg" alt="Christmas trees with their calculated borders"></p>\u000a\u000a<p>The source code is written for Python 2.7.6 and it depends on <a href="http://www.numpy.org/">numpy</a>, <a href="http://www.scipy.org/">scipy</a>, <a href="http://matplotlib.org/">matplotlib</a> and <a href="http://scikit-learn.org/stable/">scikit-learn</a>.  I've divided it into two parts.  The first part is responsible for the actual image processing:</p>\u000a\u000a<pre><code>from PIL import Image\u000aimport numpy as np\u000aimport scipy as sp\u000aimport matplotlib.colors as colors\u000afrom sklearn.cluster import DBSCAN\u000afrom math import ceil, sqrt\u000a\u000a"""\u000aInputs:\u000a\u000a    rgbimg:         [M,N,3] numpy array containing (uint, 0-255) color image\u000a\u000a    hueleftthr:     Scalar constant to select maximum allowed hue in the\u000a                    yellow-green region\u000a\u000a    huerightthr:    Scalar constant to select minimum allowed hue in the\u000a                    blue-purple region\u000a\u000a    satthr:         Scalar constant to select minimum allowed saturation\u000a\u000a    valthr:         Scalar constant to select minimum allowed value\u000a\u000a    monothr:        Scalar constant to select minimum allowed monochrome\u000a                    brightness\u000a\u000a    maxpoints:      Scalar constant maximum number of pixels to forward to\u000a                    the DBSCAN clustering algorithm\u000a\u000a    proxthresh:     Proximity threshold to use for DBSCAN, as a fraction of\u000a                    the diagonal size of the image\u000a\u000aOutputs:\u000a\u000a    borderseg:      [K,2,2] Nested list containing K pairs of x- and y- pixel\u000a                    values for drawing the tree border\u000a\u000a    X:              [P,2] List of pixels that passed the threshold step\u000a\u000a    labels:         [Q,2] List of cluster labels for points in Xslice (see\u000a                    below)\u000a\u000a    Xslice:         [Q,2] Reduced list of pixels to be passed to DBSCAN\u000a\u000a"""\u000a\u000adef findtree(rgbimg, hueleftthr=0.2, huerightthr=0.95, satthr=0.7, \u000a             valthr=0.7, monothr=220, maxpoints=5000, proxthresh=0.04):\u000a\u000a    # Make sure number of points does not exceed DBSCAN maximum capacity\u000a        Xslice = X[range(0,nsample,int(ceil(float(nsample)/maxpoints)))]\u000a\u000a    # Translate DBSCAN proximity threshold to units of pixels and run DBSCAN\u000a    pixproxthr = proxthresh * sqrt(binimg.shape[0]**2 + binimg.shape[1]**2)\u000a    db = DBSCAN(eps=pixproxthr, min_samples=10).fit(Xslice)\u000a    labels = db.labels_.astype(int)\u000a\u000a    # V<p>Using <code>os.path.split</code> or <code>os.path.basename</code> as others suggest won't work in all cases: if you're running the script on Linux and attempt to process a classic windows-style path, it will fail.</p>\u000a\u000a<p>Windows paths can use either backslash or forward slash as path separator. Therefore, the <code>ntpath</code> module (which is equivalent to os.path when running on windows) will work for all<sup>(1)</sup> paths on all platforms.</p>\u000a\u000a<pre><code>import ntpath\u000antpath.basename("a/b/c")\u000a</code></pre>\u000a\u000a<p>Of course, if the file ends with a slash, the basename will be empty, so make your own function to deal with it:</p>\u000a\u000a<pre><code>def path_leaf(path):\u000a    head, tail = ntpath.split(path)\u000a    return tail or ntpath.basename(head)\u000a</code></pre>\u000a\u000a<p>Verification:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; paths = ['a/b/c/', 'a/b/c', '\u005c\u005ca\u005c\u005cb\u005c\u005cc', '\u005c\u005ca\u005c\u005cb\u005c\u005cc\u005c\u005c', 'a\u005c\u005cb\u005c\u005cc', \u000a...     'a/b/../../a/b/c/', 'a/b/../../a/b/c']\u000a&gt;&gt;&gt; [path_leaf(path) for path in paths]\u000a['c', 'c', 'c', 'c', 'c', 'c', 'c']\u000a</code></pre>\u000a\u000a<p><br></p>\u000a\u000a<p><sup>(1) There's one caveat: Linux filenames <em>may contain backslashes</em>. So on linux, <code>r'a/b\u005cc'</code> always refers to the file <code>b\u005cc</code> in the <code>a</code> folder, while on Windows, it always refers to the <code>c</code> file in the <code>b</code> subfolder of the <code>a</code> folder. So when both forward and backward slashes are used in a path, you <em>need</em> to know the associated platform to be able to interpret it correctly. In practice it's usually safe to assume it's a windows path since backslashes are seldom used in Linux filenames, but keep this in mind when you code so you don't create accidental security holes.</sup></p>\u000a

V<p>Something to be careful about when designing a RESTful API is the conflation of GET and POST, as if they were the same thing. It's easy to make this mistake with <a href="http://www.djangoproject.com/" rel="noreferrer">Django</a>'s <a href="https://docs.djangoproject.com/en/dev/topics/http/views/" rel="noreferrer">function-based views</a> and <a href="http://www.cherrypy.org/" rel="noreferrer">CherryPy</a>'s default dispatcher, although both frameworks now provide a way around this problem (<a href="https://docs.djangoproject.com/en/dev/topics/class-based-views/" rel="noreferrer">class-based views</a> and <a href="http://docs.cherrypy.org/dev/refman/_cpdispatch.html#cherrypy._cpdispatch.MethodDispatcher" rel="noreferrer">MethodDispatcher</a>, respectively).</p>\u000a\u000a<p><a href="http://en.wikipedia.org/wiki/Representational_State_Transfer#RESTful_web_services" rel="noreferrer">HTTP-verbs are very important</a> in REST, and unless you're very careful about this, you'll end up falling into a <a href="http://www.infoq.com/articles/rest-anti-patterns" rel="noreferrer">REST anti-pattern</a>.</p>\u000a\u000a<p>Some frameworks that get it right are <a href="http://webpy.org/" rel="noreferrer">web.py</a>, <a href="http://flask.pocoo.org" rel="noreferrer">Flask</a> and <a href="http://bottlepy.org" rel="noreferrer">Bottle</a>. When combined with the <a href="https://github.com/martinblech/mimerender" rel="noreferrer">mimerender</a> library (full disclosure: I wrote it), they allow you to write nice RESTful webservices:</p>\u000a\u000a<pre><code>import web\u000aimport json\u000afrom mimerender import mimerender\u000a\u000arender_xml = lambda message: '&lt;message&gt;%s&lt;/message&gt;'%message\u000arender_json = lambda **args: json.dumps(args)\u000arender_html = lambda message: '&lt;html&gt;&lt;body&gt;%s&lt;/body&gt;&lt;/html&gt;'%message\u000arender_txt = lambda message: message\u000a\u000aurls = (\u000a    '/(.*)', 'greet'\u000a)\u000aapp = web.application(urls, globals())\u000a\u000aclass greet:\u000a    @mimerender(\u000a        default = 'html',\u000a        html = render_html,\u000a        xml  = render_xml,\u000a        json = render_json,\u000a        txt  = render_txt\u000a    )\u000a    def GET(self, name):\u000a        if not name: \u000a            name = 'world'\u000a        return {'message': 'Hello, ' + name + '!'}\u000a\u000aif __name__ == "__main__":\u000a    app.run()\u000a</code></pre>\u000a\u000a<p>The service's logic is implemented only once, and the correct representation selection (Accept header) + dispatch to the proper render function (or template) is done in a tidy, transparent way.</p>\u000a\u000a<pre><code>$ curl localhost:8080/x\u000a&lt;html&gt;&lt;body&gt;Hello, x!&lt;/body&gt;&lt;/html&gt;\u000a\u000a$ curl -H "Accept: application/html" localhost:8080/x\u000a&lt;html&gt;&lt;body&gt;Hello, x!&lt;/body&gt;&lt;/html&gt;\u000a\u000a$ curl -H "Accept: application/xml" localhost:8080/x\u000a&lt;message&gt;Hello, x!&lt;/message&gt;\u000a\u000a$ curl -H "Accept: application/json" localhost:8080/x\u000a{'message':'Hello, x!'}\u000a\u000a$ curl -H "Accept: text/plain" localhost:8080/x\u000aHello, x!\u000a</code></pre>\u000a\u000a<p><strong>Update (April 2012)</strong>: added information about Django's class-based views, CherryPy's MethodDispatcher and Flask and Bottle frameworks. Neither existed back when the question was asked.</p>\u000a

V<p>It's not because it doesn't make sense; it makes perfect sense to define "x++" as "x += 1, evaluating to the previous binding of x".</p>\u000a\u000a<p>If you want to know the original reason, you'll have to either wade through old Python mailing lists or ask somebody who was there (eg. Guido), but it's easy enough to justify after the fact:</p>\u000a\u000a<p>Simple increment and decrement aren't needed as much as in other languages.  You don't write things like <code>for(int i = 0; i &lt; 10; ++i)</code> in Python very often; instead you do things like <code>for i in range(0, 10)</code>.</p>\u000a\u000a<p>Since it's not needed nearly as often, there's much less reason to give it its own special syntax; when you do need to increment, <code>+=</code> is usually just fine.</p>\u000a\u000a<p>It's not a decision of whether it makes sense, or whether it can be done--it does, and it can.  It's a question of whether the benefit is worth adding to the core syntax of the language.  Remember, this is <em>four</em> operators--postinc, postdec, preinc, predec, and each of these would need to have its own class overloads; they all need to be specified, and tested; it would add opcodes to the language (implying a larger, and therefore slower, VM engine); every class that supports a logical increment would need to implement them (on top of <code>+=</code> and <code>-=</code>).</p>\u000a\u000a<p>This is all redundant with <code>+=</code> and <code>-=</code>, so it would become a net loss.</p>\u000a

V<p>When you use a decorator, you're replacing one function with another.  In other words, if you have a decorator</p>\u000a\u000a<pre><code>def logged(func):\u000a    def with_logging(*args, **kwargs):\u000a        print func.__name__ + " was called"\u000a        return func(*args, **kwargs)\u000a    return with_logging\u000a</code></pre>\u000a\u000a<p>then when you say</p>\u000a\u000a<pre><code>@logged\u000adef f(x):\u000a   """does some math"""\u000a   return x + x * x\u000a</code></pre>\u000a\u000a<p>it's exactly the same as saying</p>\u000a\u000a<pre><code>def f(x):\u000a    """does some math"""\u000a    return x + x * x\u000af = logged(f)\u000a</code></pre>\u000a\u000a<p>and your function f is replaced with the function with_logging.  Unfortunately, this means that if you then say</p>\u000a\u000a<pre><code>print f.__name__\u000a</code></pre>\u000a\u000a<p>it will print <code>with_logging</code> because that's the name of your new function.  In fact, if you look at the docstring for f, it will be blank because with_logging has no docstring, and so the docstring you wrote won't be there anymore.  Also, if you look at the pydoc result for that function, it won't be listed as taking one argument <code>x</code>; instead it'll be listed as taking <code>*args</code> and <code>**kwargs</code> because that's what with_logging takes.</p>\u000a\u000a<p>If using a decorator always meant losing this information about a function, it would be a serious problem.  That's why we have <code>functools.wraps</code>.  This takes a function used in a decorator and adds the functionality of copying over the function name, docstring, arguments list, etc.  And since <code>wraps</code> is itself a decorator, the following code does the correct thing:</p>\u000a\u000a<pre><code>from functools import wraps\u000adef logged(func):\u000a    @wraps(func)\u000a    def with_logging(*args, **kwargs):\u000a        print func.__name__ + " was called"\u000a        return func(*args, **kwargs)\u000a    return with_logging\u000a\u000a@logged\u000adef f(x):\u000a   """does some math"""\u000a   return x + x * x\u000a\u000aprint f.__name__  # prints 'f'\u000aprint f.__doc__   # prints 'does some math'\u000a</code></pre>\u000a
