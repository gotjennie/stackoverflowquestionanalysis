V<blockquote>\u000a  <p>Why is <code>x**4.0</code> <em>faster</em> than <code>x**4</code> in Python 3<sup>*</sup>?</p>\u000a</blockquote>\u000a\u000a<p>Python 3 <code>int</code> objects are a full fledged object designed to support an arbitrary size; due to that fact, <em>they are <a href="https://github.com/python/cpython/blob/master/Objects/longobject.c#L4051" rel="nofollow noreferrer">handled as such on the C level</a></em> (see how all variables are declared as <code>PyLongObject *</code> type in <code>long_pow</code>). This also makes their exponentiation a lot more <em>trickier</em> and <em>tedious</em> since, you need to play around with the <code>ob_digit</code> array it uses to represent its value to perform it. (<a href="https://github.com/python/cpython/blob/master/Objects/longobject.c#L4148" rel="nofollow noreferrer">Source for the brave.</a> -- See: <a href="http://stackoverflow.com/questions/40344159/understanding-memory-allocation-for-large-integers-in-python">Understanding memory allocation for large integers in Python</a> for more on <code>PyLongObject</code>s.) </p>\u000a\u000a<p>Python <code>float</code> objects, on the other hand, <em>can be transformed</em> to a C <code>double</code> type (by using <a href="https://docs.python.org/3/c-api/float.html#c.PyFloat_AsDouble" rel="nofollow noreferrer"><code>PyFloat_AsDouble</code></a>) and operations can be performed <a href="https://github.com/python/cpython/blob/master/Objects/floatobject.c#L686" rel="nofollow noreferrer">using those native types</a>. <em>This is great</em> because, after checking for relevant edge-cases, it allows Python to <a href="https://github.com/python/cpython/blob/master/Objects/floatobject.c#L784" rel="nofollow noreferrer">use the system <code>pow</code></a> (<a href="http://en.cppreference.com/w/c/numeric/math/pow" rel="nofollow noreferrer">C's <code>pow</code>, that is</a>) to handle the actual exponentiation:</p>\u000a\u000a<pre class="lang-c prettyprint-override"><code>/* Now iv and iw are finite, iw is nonzero, and iv is\u000a * positive and not equal to 1.0.  We finally allow\u000a * the platform pow to step in and do the rest.\u000a */\u000aerrno = 0;\u000aPyFPE_START_PROTECT("pow", return NULL)\u000aix = pow(iv, iw); \u000a</code></pre>\u000a\u000a<p>where <code>iv</code> and <code>iw</code> are our original <code>PyFloatObject</code>s as C <code>double</code>s.</p>\u000a\u000a<blockquote>\u000a  <p>For what it's worth: Python <code>2.7.13</code> for me is a factor <code>2~3</code> faster, and shows the inverse behaviour.</p>\u000a</blockquote>\u000a\u000a<p>The previous fact <em>also explains</em> the discrepancy between Python 2 and 3, so, I thought I'd address this comment too because it is interesting.</p>\u000a\u000a<p>In Python 2, you're using the old <code>int</code> object that differs from the <code>int</code> object in Python 3 (all <code>int</code> objects in 3.x are of <code>PyLongObject</code> type). In 2, there's a distinction that depends on the value of the object (or, if you use the suffix <code>L/l</code>):</p>\u000a\u000a<pre><code># Python 2\u000atype(30)  # &lt;type 'int'&gt;\u000atype(30L) # &lt;type 'long'&gt;\u000a</code></pre>\u000a\u000a<p>The <code>&lt;type 'int'&gt;</code> you see here <em>does the same thing <code>float</code>s do</em>, it gets safely converted into a C <code>long</code> <a href="https://github.com/python/cpython/blob/2.7/Objects/intobject.c#L726" rel="nofollow noreferrer">when operations are performed on it</a> (The <code>int_pow</code> also hints the compiler to put 'em in a register if it can do so, so that <em>could</em> make a difference):</p>\u000a\u000a<pre class="lang-c prettyprint-override"><code>static PyObject *\u000aint_pow(PyIntObject *v, PyIntObject *w, PyIntObject *z)\u000a{\u000a    register long iv, iw, iz=0, ix, temp, prev;\u000a/* Snipped for brevity */    \u000a</code></pre>\u000a\u000a<p>this allows for a good speed gain. </p>\u000a\u000a<p>To see how sluggish <code>&lt;type 'long'&gt;</code>s are in comparison to <code>&lt;type 'int'&gt;</code>s, if you wrapped the <code>x</code> name in a <code>long</code> call in Python 2 (essentially forcing it to use <code>long_pow</code> as in Python 3), the speed gain disappears:</p>\u000a\u000a<pre><code># &lt;type 'int'&gt;\u000a(python2) \u279c python -m timeit "for x in range(1000):" " x**2"       \u000a10000 loops, best of 3: 116 usec per loop\u000a# &lt;type 'long'&gt; \u000a(python2) \u279c python -m timeit "for x in range(1000):" " long(x)**2"\u000a100 loops, best of 3: 2.12 msec per loop\u000a</code></pre>\u000a\u000a<p>Take note that, though the one snippet transforms the <code>int</code> to <code>long</code> while the other does not (as pointed out by @pydsinger), this cast is not the contributing force behind the slowdown. The implementation of <code>long_pow</code> is. (Time the statements solely with <code>long(x)</code> to see).</p>\u000a\u000a<blockquote>\u000a  <p>[...] it doesn't happen outside of the loop. [...] Any idea about that?</p>\u000a</blockquote>\u000a\u000a<p>This is CPython's peephole optimizer folding the constants for you. You get the same exact timings either case since there's no actual computation to find the result of the exponentiation, only loading of values:</p>\u000a\u000a<pre><code>dis(compile('4 ** 4', '', 'exec'))\u000a  1           0 LOAD_CONST               2 (256)\u000a              3 POP_TOP\u000a              4 LOAD_CONST               1 (None)\u000a              7 RETURN_VALUE\u000a</code></pre>\u000a\u000a<p>Identical byte-code is generated for <code>'4 ** 4.'</code> with the only difference being that the <code>LOAD_CONST</code> loads the float <code>256.0</code> instead of the int <code>256</code>.</p>\u000a\u000a<p><sup>*All of the above apply solely for CPython, the reference implementation of Python. Other implementations, in different programming languages, might perform differently. </sup></p>\u000a

Fortunately that's free so the loop only repeats once. The hash table size is a power of 2, so <code>2**i</code> is the size of the hash table, <code>i</code> is the number of bits used from the hash value <code>j</code>.</p>\u000a\u000a<p>Each probe into the table can find one of these:</p>\u000a\u000a<ul>\u000a<li><p>The slot is empty, in that case the probing stops and we know the\u000avalue is not in the table.</p></li>\u000a<li><p>The slot is unused but was used in the past in which case we go try\u000athe next value calculated as above.</p></li>\u000a<li><p>The slot is full but the full hash value stored in the table isn't\u000athe same as the hash of the key we are looking for (that's what\u000ahappens in the case of <code>__builtins__</code> vs <code>without_else</code>).</p></li>\u000a<li><p>The slot is full and has exactly the hash value we want, then Python\u000achecks to see if the key and the object we are looking up are the\u000asame object (which in this case they will be because short strings\u000athat could be identifiers are interned so identical identifiers use\u000athe exact same string).</p></li>\u000a<li><p>Finally when the slot is full, the hash matches exactly, but the keys\u000aare not the identical object, then and only then will Python try\u000acomparing them for equality. V<p><strong>Script vs. Module</strong></p>\u000a\u000a<p>Here's an explanation.  The short version is that there is a big difference between directly running a Python file, and importing that file from somewhere else.  <strong>Just knowing what directory a file is in does not determine what package Python thinks it is in.</strong>  That depends, additionally, on how you load the file into Python (by running or by importing).</p>\u000a\u000a<p>There are two ways to load a Python file: as the top-level script, or as a \u000amodule.  A file is loaded as the top-level script if you execute it directly, for instance by typing <code>python myfile.py</code> on the command line.  It is loaded as a module if you do <code>python -m myfile</code>, or if it is loaded when an <code>import</code> statement is encounted inside some other file.  There can only be one top-level script at a time; the top-level script is the Python file you ran to start things off.</p>\u000a\u000a<p><strong>Naming</strong></p>\u000a\u000a<p>When a file is loaded, it is given a name (which is stored in its <code>__name__</code> attribute).  If it was loaded as the top-level script, its name is <code>__main__</code>.  If it was loaded as a module, its name is the filename, preceded by the names of any packages/subpackages of which it is a part, separated by dots.</p>\u000a\u000a<p>So for instance in your example:</p>\u000a\u000a<pre><code>package/\u000a    __init__.py\u000a    subpackage1/\u000a        __init__.py\u000a        moduleX.py\u000a    moduleA.py\u000a</code></pre>\u000a\u000a<p>if you imported <code>moduleX</code> (note: <em>imported</em>, not directly executed), its name would be <code>package.subpackage1.moduleX</code>.  If you imported <code>moduleA</code>, its name would be <code>package.moduleA</code>.  However, if you <em>directly run</em> <code>moduleX</code> from the command line, its name will instead be <code>__main__</code>, and if you directly run <code>moduleA</code> from the command line, its name will be <code>__main__</code>.  When a module is run as the top-level script, it loses its normal name and its name is instead <code>__main__</code>.</p>\u000a\u000a<p><strong>Accessing a module NOT through its containing package</strong></p>\u000a\u000a<p>There is an additional wrinkle: the module's name depends on whether it was imported "directly" from the directory it is in, or imported via a package.  This only makes a difference if you run Python in a directory, and try to import a file in that same directory (or a subdirectory of it).  For instance, if you start the Python interpreter in the directory <code>package/subpackage1</code> and then do <code>import moduleX</code>, the name of <code>moduleX</code> will just be <code>moduleX</code>, and not <code>package.subpackage1.moduleX</code>.  This is because Python adds the current directory to its search path on startup; if it finds the to-be-imported module in the current directory, it will not know that that directory is part of a package, and the package information will not become part of the module's name.</p>\u000a\u000a<p>A special case is if you run the interpreter interactively (e.g., just type <code>python</code> and start entering Python code on the fly).  In this case the name of that interactive session is <code>__main__</code>.</p>\u000a\u000a<p>Now here is the crucial thing for your error message: <strong>if a module's name has no dots, it is not considered to be part of a package</strong>.  It doesn't matter where the file actually is on disk.  All that matters is what its name is, and its name depends on how you loaded it.</p>\u000a\u000a<p>Now look at the quote you included in your question:</p>\u000a\u000a<blockquote>\u000a  <p>Relative imports use a module's name attribute to determine that module's position in the package hierarchy. If the module's name does not contain any package information (e.g. it is set to 'main') then relative imports are resolved as if the module were a top level module, regardless of where the module is actually located on the file system.</p>\u000a</blockquote>\u000a\u000a<p><strong>Relative imports...</strong></p>\u000a\u000a<p>Relative imports use the module's <em>name</em> to determine where it is in a package.  When you use a relative import like <code>from .. import foo</code>, the dots indicate to step up some number of levels in the package hierarchy.  For instance, if your current module's name is <code>package.subpackage1.moduleX</code>, then <code>..moduleA</code> would mean <code>package.moduleA</code>.  For a <code>from .. import</code> to work, the module's name must have at least as many dots as there are in the <code>import</code> statement.</p>\u000a\u000a<p><strong>... are only relative in a package</strong></p>\u000a\u000a<p>However, if your module's name is <code>__main__</code>, it is not considered to be in a package.  Its name has no dots, and therefore you cannot use <code>from .. import</code> statements inside it.  If you try to do so, you will get the "relative-import in non-package" error.</p>\u000a\u000a<p><strong>Scripts can't import relative</strong></p>\u000a\u000a<p>What you probably did is you tried to run <code>moduleX</code> or the like from the command line.  When you did this, its name was set to <code>__main__</code>, which means that relative imports within it will fail, because its name does not reveal that it is in a package. Note that this will also happen if you run Python from the same directory where a module is, and then try to import that module, because, as described above, Python will find the module in the current directory "too early" without realizing it is part of a package.</p>\u000a\u000a<p>Also remember that when you run the interactive interpreter, the "name" of that interactive session is always <code>__main__</code>.  Thus <strong>you cannot do relative imports directly from an interactive session</strong>.  Relative imports are only for use within module files.</p>\u000a\u000a<p><strong>Two solutions:</strong></p>\u000a\u000a<ol>\u000a<li><p>If you really do want to run <code>moduleX</code> directly, but you still want it to be considered part of a package, you can do <code>python -m package.subpackage.moduleX</code>.  The <code>-m</code> tells Python to load it a s a module, not as the top-level script.</p></li>\u000a<li><p>Or perhaps you don't actually want to <em>run</em> <code>moduleX</code>, you just want to run some other script, say <code>myfile.py</code>, that <em>uses</em> functions inside <code>moduleX</code>.  If that is the case, put <code>myfile.py</code> <em>somewhere else</em> --- <em>not</em> inside the <code>package</code> directory -- and run it.  If inside <code>myfile.py</code> you do things like <code>from package.moduleA import spam</code>, it will work fine.</p></li>\u000a</ol>\u000a\u000a<p><strong>Notes</strong></p>\u000a\u000a<ul>\u000a<li><p>For either of these solutions, the package directory (<code>package</code> in your example) must be accessible from the Python module search path (<code>sys.path</code>).  If it is not, you will not be able to use anything in the package reliably at all.</p></li>\u000a<li><p>since Python 2.6, the module's "name" for package-resolution purposes is determined not just by its <code>__name__</code> attributes but also by the <code>__package__</code> attribute.  That's why I'm avoiding using the explicit symbol <code>__name__</code> to refer to the module's "name".  Since Python 2.6 a module's "name" is effectively <code>__package__ + '.' + __name__</code>, or just <code>__name__</code> if <code>__package__</code> is <code>None</code>.)</p></li>\u000a</ul>\u000a

V<h2>1) Almost-English style:</h2>\u000a\u000a<p>Test for presence using the <code>in</code> operator, then apply the <code>remove</code> method.</p>\u000a\u000a<pre class="lang-py prettyprint-override"><code>if thing in some_list: some_list.remove(thing)\u000a</code></pre>\u000a\u000a<p>The <code>remove</code>method will remove only the first occurrence of <code>thing</code>, in order to remove all occurrences you can use <code>while</code> instead of <code>if</code>.</p>\u000a\u000a<pre class="lang-py prettyprint-override"><code>while thing in some_list: some_list.remove(thing)    \u000a</code></pre>\u000a\u000a<ul>\u000a<li>Simple enough, probably my choice.for small lists (can't resist one-liners)</li>\u000a</ul>\u000a\u000a<h2>2) <a href="http://docs.python.org/glossary.html#term-duck-typing" rel="noreferrer">Duck-typed</a>, <a href="http://docs.python.org/glossary.html#term-eafp" rel="noreferrer">EAFP</a> style:</h2>\u000a\u000a<p>This shoot-first-ask-questions-last attitude is common in Python. Instead of testing in advance if the object is suitable, just carry out the operation and catch relevant Exceptions:</p>\u000a\u000a<pre class="lang-py prettyprint-override"><code>try:\u000a    some_list.remove(thing)\u000aexcept ValueError:\u000a    pass # or scream: thing not in some_list!\u000aexcept AttributeError:\u000a    pass # call security, some_list not quacking like a list!\u000a</code></pre>\u000a\u000a<p>If you expect multiple occurrences of thing:</p>\u000a\u000a<pre class="lang-py prettyprint-override"><code>while True:\u000a    try:\u000a        some_list.remove(thing)\u000a    except ValueError:\u000a        break\u000a</code></pre>\u000a\u000a<ul>\u000a<li>a little verbose for this specific use case, but very idiomatic in Python.</li>\u000a<li>this performs better than #1</li>\u000a<li><a href="http://www.python.org/dev/peps/pep-0463/" rel="noreferrer">PEP 463</a> proposed a shorter syntax for try/except simple usage that would be handy here, but it was not approved.</li>\u000a</ul>\u000a\u000a<p>However, with <a href="https://docs.python.org/3/library/contextlib.html#contextlib.suppress" rel="noreferrer">contextlib's suppress() contextmanager</a> (introduced in python 3.4) the above code can be simplified to this:</p>\u000a\u000a<pre class="lang-py prettyprint-override"><code>with suppress(ValueError, AttributeError):\u000a    some_list.remove(thing)\u000a</code></pre>\u000a\u000a<p>Again, if you expect multiple occurrences of thing:</p>\u000a\u000a<pre class="lang-py prettyprint-override"><code>with suppress(ValueError):\u000a    while True:\u000a        some_list.remove(thing)\u000a</code></pre>\u000a\u000a<h2>3) Functional style:</h2>\u000a\u000a<p>Around 1993, Python got <code>lambda</code>, <code>reduce()</code>, <code>filter()</code> and <code>map()</code>, courtesy of a <a href="https://en.wikipedia.org/wiki/Lisp_%28programming_language%29" rel="noreferrer">Lisp</a> hacker who missed them and submitted working patches*. You can use <code>filter</code> to remove elements from the list:</p>\u000a\u000a<pre class="lang-py prettyprint-override"><code>is_not_thing = lambda x: x is not thing\u000acleaned_list = filter(is_not_thing, some_list)\u000a</code></pre>\u000a\u000a<p>There is a shortcut that may be useful for your case: if you want to filter out empty items (in fact items where <code>bool(item) == False</code>, like <code>None</code>, zero, empty strings or other empty collections), you can pass None as the first argument:</p>\u000a\u000a<pre class="lang-py prettyprint-override"><code>cleaned_list = filter(None, some_list)\u000a</code></pre>\u000a\u000a<ul>\u000a<li><strong>[update]</strong>: in Python 2.x, <code>filter(function, iterable)</code> used to be equivalent to <code>[item for item in iterable if function(item)]</code> (or <code>[item for item in iterable if item]</code> if the first argument is <code>None</code>); in Python 3.x, it is now equivalent to <code>(item for item in iterable if function(item))</code>. The subtle difference is that filter used to return a list, now it works like a generator expression - this is OK if you are only iterating over the cleaned list and discarding it, but if you really need a list, you have to enclose the <code>filter()</code> call with the <code>list()</code> constructor.</li>\u000a<li>*These Lispy flavored constructs are considered a little alien in Python. Around 2005, <a href="http://www.artima.com/weblogs/viewpost.jsp?thread=98196" rel="noreferrer">Guido was even talking about dropping <code>filter</code></a> - along with companions <code>map</code> and <code>reduce</code> (they are not gone yet but <code>reduce</code> was moved into the <a href="http://docs.python.org/2/library/functools.html#functools.reduce" rel="noreferrer">functools</a> module, which is worth a look if you like <a href="http://en.wikipedia.org/wiki/Higher-order_function" rel="noreferrer">high order functions</a>).</li>\u000a</ul>\u000a\u000a<h2>4) Mathematical style:</h2>\u000a\u000a<p><a href="http://en.wikipedia.org/wiki/List_comprehension" rel="noreferrer">List comprehensions</a> became the preferred style for list manipulation in Python since introduced in version 2.0 by <a href="http://www.python.org/dev/peps/pep-0202/" rel="noreferrer">PEP 202</a>. The rationale behind it is that List comprehensions provide a more concise way to create lists in situations where <code>map()</code> and <code>filter()</code> and/or nested loops would currently be used.</p>\u000a\u000a<pre class="lang-py prettyprint-override"><code>cleaned_list = [ x for x in some_list if x is not thing ]\u000a</code></pre>\u000a\u000a<p>Generator expressions were introduced in version 2.4 by <a href="http://www.python.org/dev/peps/pep-0289/" rel="noreferrer">PEP 289</a>. A generator expression is better for situations where you don't really need (or want) to have a full list created in memory - like when you just want to iterate over the elements one at a time. If you are only iterating over the list, you can think of a generator expression as a <a href="http://en.wikipedia.org/wiki/Lazy_evaluation" rel="noreferrer">lazy evaluated</a> list comprehension:</p>\u000a\u000a<pre class="lang-py prettyprint-override"><code>for item in (x for x in some_list if x is not thing):\u000a    do_your_thing_with(item)\u000a</code></pre>\u000a\u000a<ul>\u000a<li>See <a href="http://python-history.blogspot.com.br/2010/06/from-list-comprehensions-to-generator.html" rel="noreferrer">this Python history blog</a> post by <a href="http://en.wikipedia.org/wiki/Guido_van_Rossum" rel="noreferrer">GvR</a>.</li>\u000a<li>This syntax is inspired by the <a href="http://en.wikipedia.org/wiki/Set-builder_notation" rel="noreferrer">set-builder notation</a> in math.</li>\u000a<li>Python 3 has also <a href="http://www.python.org/dev/peps/pep-3100/#id103" rel="noreferrer">set</a> and <a href="http://www.python.org/dev/peps/pep-0274/" rel="noreferrer">dict comprehensions</a>.</li>\u000a</ul>\u000a\u000a<h2>Notes</h2>\u000a\u000a<ol>\u000a<li>you may want to use the inequality operator <code>!=</code> instead of <code>is not</code> (<a href="http://stackoverflow.com/q/2209755/">the difference is important</a>)</li>\u000a<li>for critics of methods implying a list copy: contrary to popular belief, generator expressions are not always more efficient than list comprehensions - please profile before complaining</li>\u000a</ol>\u000a

V<p>Both are valuable.  I use both doctest and <a href="https://pypi.python.org/pypi/nose/" rel="noreferrer">nose</a> taking the place of unittest. I use doctest for cases where the test is giving an example of usage that is actually useful as documentation.  Generally I don't make these tests comprehensive, aiming solely for informative.  I'm effectively using doctest in reverse:  not to test my code is correct based on my doctest, but to check that my documentation is correct based on the code.</p>\u000a\u000a<p>The reason is that I find comprehensive doctests will clutter your documentation far too much, so you will either end up with either unusable docstrings, or incomplete testing.</p>\u000a\u000a<p>For actually testing the <em>code</em>, the goal is to thoroughly test every case, rather than illustrate what is does by example, which is a different goal which I think is better met by other frameworks.</p>\u000a

V<p>Using these magic methods (<code>__enter__</code>, <code>__exit__</code>) allows you to implement objects which can be used easily with the <code>with</code> statement. </p>\u000a\u000a<p>The idea is that it makes it easy to build code which needs some 'cleandown' code executed (think of it as a <code>try-finally</code> block). <a href="http://effbot.org/zone/python-with-statement.htm" rel="noreferrer">Some more explanation here</a>.</p>\u000a\u000a<p>A useful example could be a database connection object (which then automagically closes the connection once the corresponding 'with'-statement goes out of scope):</p>\u000a\u000a<pre><code>class DatabaseConnection(object):\u000a\u000a    def __enter__(self):\u000a        # make a database connection and return it\u000a        ...\u000a        return self.dbconn\u000a\u000a    def __exit__(self, exc_type, exc_val, exc_tb):\u000a        # make sure the dbconnection gets closed\u000a        self.dbconn.close()\u000a        ...\u000a</code></pre>\u000a\u000a<p>As explained above, use this object with the <code>with</code> statement (you may need to do <code>from __future__ import with_statement</code> at the top of the file if you're on Python 2.5).</p>\u000a\u000a<pre><code>with DatabaseConnection() as mydbconn:\u000a    # do stuff\u000a</code></pre>\u000a\u000a<p><a href="https://www.python.org/dev/peps/pep-0343/" rel="noreferrer">PEP343 -- The 'with' statement'</a> has a nice writeup as well.</p>\u000a

V<p>PyPy is not a fork of CPython, so it could never be merged directly into CPython.</p>\u000a\u000a<p>Theoretically the Python community could universally adopt PyPy, PyPy could be made the reference implementation, and CPython could be discontinued. However, PyPy has its own weaknesses:</p>\u000a\u000a<ul>\u000a<li>CPython is easy to integrate with Python modules written in C, which is traditionally the way Python applications have handled CPU-intensive tasks (see for instance the SciPy project).</li>\u000a<li>The PyPy JIT compilation step itself costs CPU time -- it's only through repeated running of compiled code that it becomes faster overall.  This means startup times can be higher, and therefore PyPy isn't necessarily as efficient for running glue code or trivial scripts.</li>\u000a<li>PyPy and CPython behavior is not identical in all respects, especially when it comes to "implementation details" (behavior that is not specified by the language but is still important at a practical level).</li>\u000a<li>CPython runs on more architectures than PyPy and has been successfully adapted to run in embedded architectures in ways that may be impractical for PyPy.</li>\u000a<li>CPython's reference counting scheme for memory management arguably has more predictable performance impacts than PyPy's various GC systems, although this isn't necessarily true of all "pure GC" strategies.</li>\u000a<li>PyPy does not yet fully support Python 3.x, although that is an active work item.</li>\u000a</ul>\u000a\u000a<p>PyPy is a great project, but runtime speed on CPU-intensive tasks isn't everything, and in many applications it's the least of many concerns.  For instance, Django can run on PyPy and that makes templating faster, but CPython's database drivers are faster than PyPy's; in the end, which implementation is more efficient depends on where the bottleneck in a given application is.</p>\u000a\u000a<p>Another example: you'd think PyPy would be great for games, but most GC strategies like those used in PyPy cause noticeable jitter.  For CPython, most of the CPU-intensive game stuff is offloaded to the PyGame library, which PyPy can't take advantage of since PyGame is primarily implemented as a C extension (though see: pygame-cffi).  I still think PyPy can be a great platform for games, but I've never seen it actually used.</p>\u000a\u000a<p>PyPy and CPython have radically different approaches to fundamental design questions and make different tradeoffs, so neither one is "better" than the other in every case.</p>\u000a

V<p>First off, this can be done in <code>O(n)</code> in terms of the length of the list\u000aYou can notice that if you will duplicate your list 2 times (<code>[1, 2, 3]</code>) will be <code>[1, 2, 3, 1, 2, 3]</code> then your new list will definitely hold all possible cyclic lists.</p>\u000a\u000a<p>So all you need is to check whether the list you are searching is inside a 2 times of your starting list. In python you can achieve this in the following way (assuming that the lengths are the same).</p>\u000a\u000a<pre><code>list1 = [1, 1, 1, 0, 0]\u000alist2 = [1, 1, 0, 0, 1]\u000aprint ' '.join(map(str, list2)) in ' '.join(map(str, list1 * 2))\u000a</code></pre>\u000a\u000a<p>Some explanation about my oneliner:\u000a<code>list * 2</code> will combine a list with itself, <code>map(str, [1, 2])</code> convert all numbers to string and  <code>' '.join()</code> will convert array <code>['1', '2', '111']</code> into a string <code>'1 2 111'</code>.</p>\u000a\u000a<p>As pointed by some people in the comments, oneliner can potentially give some false positives, so to cover all the possible edge cases:</p>\u000a\u000a<pre><code>def isCircular(arr1, arr2):\u000a    if len(arr1) != len(arr2):\u000a        return False\u000a\u000a    str1 = ' '.join(map(str, arr1))\u000a    str2 = ' '.join(map(str, arr2))\u000a    if len(str1) != len(str2):\u000a        return False\u000a\u000a    return str1 in str2 + ' ' + str2\u000a</code></pre>\u000a\u000a<p><strong>P.S.1</strong> when speaking about time complexity, it is worth noticing that <code>O(n)</code> will be achieved if substring can be found in <code>O(n)</code> time. It is not always so and depends on the implementation in your language (<a href="http://en.wikipedia.org/wiki/String_searching_algorithm">although potentially it can be done in linear</a> time KMP for example).</p>\u000a\u000a<p><strong>P.S.2</strong> for people who are afraid strings operation and due to this fact think that the answer is not good. What important is complexity and speed. This algorithm potentially runs in <code>O(n)</code> time and <code>O(n)</code> space which makes it much better than anything in <code>O(n^2)</code> domain. To see this by yourself, you can run a small benchmark (creates a random list pops the first element and appends it to the end thus creating a cyclic list. You are free to do your own manipulations)</p>\u000a\u000a<pre><code>from random import random\u000abigList = [int(1000 * random()) for i in xrange(10**6)]\u000abigList2 = bigList[:]\u000abigList2.append(bigList2.pop(0))\u000a\u000a# then test how much time will it take to come up with an answer\u000afrom datetime import datetime\u000astartTime = datetime.now()\u000aprint isCircular(bigList, bigList2)\u000aprint datetime.now() - startTime    # please fill free to use timeit, but it will give similar results\u000a</code></pre>\u000a\u000a<p>0.3 seconds on my machine. V<p>From what I gather, <code>SettingWithCopyWarning</code> was created to flag potentially confusing "chained" assignments, such as the following, which don't always work as expected, particularly when the first selection returns a <em>copy</em>.  [see <a href="https://github.com/pydata/pandas/pull/5390" rel="noreferrer">GH5390</a> and <a href="https://github.com/pydata/pandas/issues/5597" rel="noreferrer">GH5597</a> for background discussion.]</p>\u000a\u000a<pre><code>df[df['A'] &gt; 2]['B'] = new_val  # new_val not set in df\u000a</code></pre>\u000a\u000a<p>The warning offers a suggestion to rewrite as follows:</p>\u000a\u000a<pre><code>df.loc[df['A'] &gt; 2, 'B'] = new_val\u000a</code></pre>\u000a\u000a<p>However, this doesn't fit your usage, which is equivalent to:</p>\u000a\u000a<pre><code>df = df[df['A'] &gt; 2]\u000adf['B'] = new_val\u000a</code></pre>\u000a\u000a<p>While it's clear that you don't care about writes making it back to the original frame (since you overwrote the reference to it), unfortunately this pattern can not be differentiated from the first chained assignment example, hence the (false positive) warning.  The potential for false positives is addressed in the <a href="http://pandas.pydata.org/pandas-docs/dev/indexing.html#returning-a-view-versus-a-copy" rel="noreferrer">docs on indexing</a>, if you'd like to read further.  You can safely disable this new warning with the following assignment.</p>\u000a\u000a<pre><code>pd.options.mode.chained_assignment = None  # default='warn'\u000a</code></pre>\u000a

V<p>Overriding <code>__getattr__</code> should be fine -- <code>__getattr__</code> is only called as a last resort i.e. if there are no attributes in the instance that match the name. For instance, if you access <code>foo.bar</code>, then <code>__getattr__</code> will only be called if <code>foo</code> has no attribute called <code>bar</code>. If the attribute is one you don't want to handle, raise <code>AttributeError</code>:</p>\u000a\u000a<pre><code>class Foo(object):\u000a    def __getattr__(self, name):\u000a        if some_predicate(name):\u000a            # ...\u000a        else:\u000a            # Default behaviour\u000a            raise AttributeError\u000a</code></pre>\u000a\u000a<p>However, unlike <code>__getattr__</code>, <code>__getattribute__</code> will be called first (only works for new style classes i.e. those that inherit from object). In this case, you can preserve default behaviour like so:</p>\u000a\u000a<pre><code>class Foo(object):\u000a    def __getattribute__(self, name):\u000a        if some_predicate(name):\u000a            # ...\u000a        else:\u000a            # Default behaviour\u000a            return object.__getattribute__(self, name)\u000a</code></pre>\u000a\u000a<p>See <a href="http://docs.python.org/reference/datamodel.html#object.__getattr__">the Python docs for more</a>.</p>\u000a

V<p>I have a solution that works, but you'll have to translate it to OpenCV yourself. It's written in Mathematica.</p>\u000a\u000a<p>The first step is to adjust the brightness in the image, by dividing each pixel with the result of a closing operation:</p>\u000a\u000a<pre><code>src = ColorConvert[Import["http://davemark.com/images/sudoku.jpg"], "Grayscale"];\u000awhite = Closing[src, DiskMatrix[5]];\u000asrcAdjusted = Image[ImageData[src]/ImageData[white]]\u000a</code></pre>\u000a\u000a<p><img src="https://i.stack.imgur.com/DLMIA.png" alt="enter image description here"></p>\u000a\u000a<p>The next step is to find the sudoku area, so I can ignore (mask out) the background. For that, I use connected component analysis, and select the component that's got the largest convex area:</p>\u000a\u000a<pre><code>components = \u000a  ComponentMeasurements[\u000a    ColorNegate@Binarize[srcAdjusted], {"ConvexArea", "Mask"}][[All, \u000a    2]];\u000alargestComponent = Image[SortBy[components, First][[-1, 2]]]\u000a</code></pre>\u000a\u000a<p><img src="https://i.stack.imgur.com/lGUIe.png" alt="enter image description here"></p>\u000a\u000a<p>By filling this image, I get a mask for the sudoku grid:</p>\u000a\u000a<pre><code>mask = FillingTransform[largestComponent]\u000a</code></pre>\u000a\u000a<p><img src="https://i.stack.imgur.com/jXwoT.png" alt="enter image description here"></p>\u000a\u000a<p>Now, I can use a 2nd order derivative filter to find the vertical and horizontal lines in two separate images:</p>\u000a\u000a<pre><code>lY = ImageMultiply[MorphologicalBinarize[GaussianFilter[srcAdjusted, 3, {2, 0}], {0.02, 0.05}], mask];\u000alX = ImageMultiply[MorphologicalBinarize[GaussianFilter[srcAdjusted, 3, {0, 2}], {0.02, 0.05}], mask];\u000a</code></pre>\u000a\u000a<p><img src="https://i.stack.imgur.com/ptwTi.png" alt="enter image description here"></p>\u000a\u000a<p>I use connected component analysis again to extract the grid lines from these images. The grid lines are much longer than the digits, so I can use caliper length to select only the grid lines-connected components. Sorting them by position, I get 2x10 mask images for each of the vertical/horizontal grid lines in the image:</p>\u000a\u000a<pre><code>verticalGridLineMasks = \u000a  SortBy[ComponentMeasurements[\u000a      lX, {"CaliperLength", "Centroid", "Mask"}, # &gt; 100 &amp;][[All, \u000a      2]], #[[2, 1]] &amp;][[All, 3]];\u000ahorizontalGridLineMasks = \u000a  SortBy[ComponentMeasurements[\u000a      lY, {"CaliperLength", "Centroid", "Mask"}, # &gt; 100 &amp;][[All, \u000a      2]], #[[2, 2]] &amp;][[All, 3]];\u000a</code></pre>\u000a\u000a<p><img src="https://i.stack.imgur.com/WKhhF.png" alt="enter image description here"></p>\u000a\u000a<p>Next I take each pair of vertical/horizontal grid lines, dilate them, calculate the pixel-by-pixel intersection, and calculate the center of the result. V<p>Back in the old days of Python, to call a function with arbitrary arguments, you would use <code>apply</code>:</p>\u000a\u000a<pre><code>apply(f,args,kwargs)\u000a</code></pre>\u000a\u000a<p><code>apply</code> still exists in Python2.7 though not in Python3, and is generally not used anymore. Nowadays, </p>\u000a\u000a<pre><code>f(*args,**kwargs)\u000a</code></pre>\u000a\u000a<p>is preferred. The <code>multiprocessing.Pool</code> modules tries to provide a similar interface.</p>\u000a\u000a<p><code>Pool.apply</code> is like Python <code>apply</code>, except that the function call is performed in a separate process. <code>Pool.apply</code> blocks until the function is completed.</p>\u000a\u000a<p><code>Pool.apply_async</code> is also like Python's built-in <code>apply</code>, except that the call returns immediately instead of waiting for the result. An <code>ApplyResult</code> object is returned. You call its <code>get()</code> method to retrieve the result of the function call. The <code>get()</code> method blocks until the function is completed. Thus, <code>pool.apply(func, args, kwargs)</code> is equivalent to <code>pool.apply_async(func, args, kwargs).get()</code>.</p>\u000a\u000a<p>In contrast to <code>Pool.apply</code>, the <code>Pool.apply_async</code> method also has a callback which, if supplied, is called when the function is complete. This can be used instead of calling <code>get()</code>.</p>\u000a\u000a<p>For example:</p>\u000a\u000a<pre><code>import multiprocessing as mp\u000aimport time\u000a\u000adef foo_pool(x):\u000a    time.sleep(2)\u000a    return x*x\u000a\u000aresult_list = []\u000adef log_result(result):\u000a    # This is called whenever foo_pool(i) returns a result.\u000a    # result_list is modified only by the main process, not the pool workers.\u000a    result_list.append(result)\u000a\u000adef apply_async_with_callback():\u000a    pool = mp.Pool()\u000a    for i in range(10):\u000a        pool.apply_async(foo_pool, args = (i, ), callback = log_result)\u000a    pool.close()\u000a    pool.join()\u000a    print(result_list)\u000a\u000aif __name__ == '__main__':\u000a    apply_async_with_callback()\u000a</code></pre>\u000a\u000a<p>may yield a result such as</p>\u000a\u000a<pre><code>[1, 0, 4, 9, 25, 16, 49, 36, 81, 64]\u000a</code></pre>\u000a\u000a<p>Notice, unlike <code>pool.map</code>, the order of the results may not correspond to the order in which the <code>pool.apply_async</code> calls were made.</p>\u000a\u000a<hr>\u000a\u000a<p>So, if you need to run a function in a separate process, but want the current process to <strong>block</strong> until that function returns, use <code>Pool.apply</code>. Like <code>Pool.apply</code>, <code>Pool.map</code> blocks until the complete result is returned.</p>\u000a\u000a<p>If you want the Pool of worker processes to perform many function calls asynchronously, use <code>Pool.apply_async</code>. The <strong>order</strong> of the results is not guaranteed to be the same as the order of the calls to <code>Pool.apply_async</code>.</p>\u000a\u000a<p>Notice also that you could call a number of <strong>different</strong> functions with <code>Pool.apply_async</code> (not all calls need to use the same function).</p>\u000a\u000a<p>In contrast, <code>Pool.map</code> applies the same function to many arguments.\u000aHowever, unlike <code>Pool.apply_async</code>, the results are returned in an order corresponding to the order of the arguments.</p>\u000a

V<p>In Python 2.x this is <em>not</em> guaranteed as it is possible for <code>True</code> and <code>False</code> to be reassigned.  However, even if this happens, boolean True and boolean False are still properly returned for comparisons.</p>\u000a\u000a<p>In Python 3.x <code>True</code> and <code>False</code> are keywords and will always be equal to <code>1</code> and <code>0</code>.</p>\u000a\u000a<p>Under normal circumstances in Python 2, and always in Python 3:</p>\u000a\u000a<p><code>False</code> object is of type <code>bool</code> which is a subclass of <code>int</code>:</p>\u000a\u000a<pre><code>object\u000a   |\u000a int\u000a   |\u000a bool\u000a</code></pre>\u000a\u000a<p>It is the only reason why in your example, <code>['zero', 'one'][False]</code> does work. It would not work with an object which is not a subclass of integer, because list indexing only works with integers, or objects that define a <a href="http://docs.python.org/reference/datamodel.html#object.__index__" rel="noreferrer"><code>__index__</code></a> method (thanks <a href="http://stackoverflow.com/users/270986/mark-dickinson">mark-dickinson</a>).</p>\u000a\u000a<p>Edit:</p>\u000a\u000a<p>It is true of the current python version, and of that of Python 3. The <a href="http://docs.python.org/reference/datamodel.html#the-standard-type-hierarchy" rel="noreferrer">docs for python 2.6</a> and the <a href="http://docs.python.org/release/3.0.1/reference/datamodel.html#the-standard-type-hierarchy" rel="noreferrer">docs for Python 3</a> both say:</p>\u000a\u000a<blockquote>\u000a  <p>There are two types of integers: [...] Integers (int) [...] Booleans (bool)</p>\u000a</blockquote>\u000a\u000a<p>and in the boolean subsection:</p>\u000a\u000a<blockquote>\u000a  <p>Booleans: These represent the truth values False and True [...] Boolean values behave like the values 0 and 1, respectively, in almost all contexts, the exception being that when converted to a string, the strings "False" or "True" are returned, respectively.</p>\u000a</blockquote>\u000a\u000a<p>So booleans are explicitly considered as integers in Python 2.6 and 3.</p>\u000a\u000a<p>So you're safe until Python 4 comes along. ;-)</p>\u000a

V<p>You can't directly determine the location of the main script being executed.  After all, sometimes the script didn't come from a file at all.  For example, it could come from the interactive interpreter or dynamically generated code stored only in memory.</p>\u000a\u000a<p>However, you can reliably determine the location of a module, since modules are always loaded from a file.  If you create a module with the following code and put it in the same directory as your main script, then the main script can import the module and use that to locate itself.</p>\u000a\u000a<p>some_path/module_locator.py:</p>\u000a\u000a<pre><code>def we_are_frozen():\u000a    # All of the modules are built-in to the interpreter, e.g., by py2exe\u000a    return hasattr(sys, "frozen")\u000a\u000adef module_path():\u000a    encoding = sys.getfilesystemencoding()\u000a    if we_are_frozen():\u000a        return os.path.dirname(unicode(sys.executable, encoding))\u000a    return os.path.dirname(unicode(__file__, encoding))\u000a</code></pre>\u000a\u000a<p>some_path/main.py:</p>\u000a\u000a<pre><code>import module_locator\u000amy_path = module_locator.module_path()\u000a</code></pre>\u000a\u000a<p>If you have several main scripts in different directories, you may need more than one copy of module_locator.</p>\u000a\u000a<p>Of course, if your main script is loaded by some other tool that doesn't let you import modules that are co-located with your script, then you're out of luck.  In cases like that, the information you're after simply doesn't exist anywhere in your program.  Your best bet would be to file a bug with the authors of the tool.</p>\u000a

V<p>there are two related concepts, both called "keyword arguments".</p>\u000a\u000a<p>On the calling side, which is what other commenters have mentioned, you have the ability to specify some function arguments by name. You have to mention them after all of the arguments without names (positional arguments), and there must be default values for any parameters which were not mentioned at all.</p>\u000a\u000a<p>The other concept is on the function definition side: You can define a function that takes parameters by name -- and you don't even have to specify what those names are. These are pure keyword arguments, and can't be passed positionally. The syntax is</p>\u000a\u000a<pre><code>def my_function(arg1, arg2, **kwargs)\u000a</code></pre>\u000a\u000a<p>Any keyword arguments you pass into this function will be placed into a dictionary named kwargs. You can examine the keys of this dictionary at run-time, like this:</p>\u000a\u000a<pre><code>def my_function(**kwargs):\u000a    print str(kwargs)\u000a\u000amy_function(a=12, b="abc")\u000a\u000a{'a': 12, 'b': 'abc'}\u000a</code></pre>\u000a
