V<p>Assuming Python 3 (in Python 2, this difference is a little less well-defined) - a string is a sequence of characters, ie <a href="http://en.wikipedia.org/wiki/Unicode" rel="nofollow noreferrer">unicode codepoints</a>; these are an abstract concept, and can't be directly stored on disk. A byte string is a sequence of, unsurprisingly, bytes - things that <em>can</em> be stored on disk. The mapping between them is an <em>encoding</em> - there are quite a lot of these (and infinitely many are possible) - and you need to know which applies in the particular case in order to do the conversion, since a different encoding may map the same bytes to a different string: </p>\u000a\u000a<pre><code>&gt;&gt;&gt; b'\u005cxcf\u005cx84o\u005cxcf\u005cx81\u005cxce\u005cxbdo\u005cxcf\u005cx82'.decode('utf-16')\u000a'\u84cf\ucf6f\uce81\u6fbd\u82cf'\u000a&gt;&gt;&gt; b'\u005cxcf\u005cx84o\u005cxcf\u005cx81\u005cxce\u005cxbdo\u005cxcf\u005cx82'.decode('utf-8')\u000a'\u03c4o\u03c1\u03bdo\u03c2'\u000a</code></pre>\u000a\u000a<p>Once you know which one to use, you can use the <code>.decode()</code> method of the byte string to get the right character string from it as above. For completeness, the <code>.encode()</code> method of a character string goes the opposite way:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; '\u03c4o\u03c1\u03bdo\u03c2'.encode('utf-8')\u000ab'\u005cxcf\u005cx84o\u005cxcf\u005cx81\u005cxce\u005cxbdo\u005cxcf\u005cx82'\u000a</code></pre>\u000a

V<p>Since you are calling the decorator like a function, it needs to return another function which is the actual decorator:</p>\u000a\u000a<pre><code>def my_decorator(param):\u000a    def actual_decorator(func):\u000a        print("Decorating function {}, with parameter {}".format(func.__name__, param))\u000a        return function_wrapper(func)  # assume we defined a wrapper somewhere\u000a    return actual_decorator\u000a</code></pre>\u000a\u000a<p>The outer function will be given any arguments you pass explicitly, and should return the inner function. The inner function will be passed the function to decorate, and return the modified function.</p>\u000a\u000a<p>Usually you want the decorator to change the function behavior by wrapping it in a wrapper function. Here's an example that optionally adds logging when the function is called:</p>\u000a\u000a<pre><code>def log_decorator(log_enabled):\u000a    def actual_decorator(func):\u000a        @functools.wraps(func)\u000a        def wrapper(*args, **kwargs):\u000a            if log_enabled:\u000a                print("Calling Function: " + func.__name__)\u000a            return func(*args, **kwargs)\u000a        return wrapper\u000a    return actual_decorator\u000a</code></pre>\u000a\u000a<p>The <code>functools.wraps</code> call copies things like the name and docstring to the wrapper function, to make it more similar to the original function.</p>\u000a\u000a<p>Example usage:</p>\u000a\u000a<pre><code>&gt;&gt;&gt; @log_decorator(True)\u000a... def f(x):\u000a...     return x+1\u000a...\u000a&gt;&gt;&gt; f(4)\u000aCalling Function: f\u000a5\u000a</code></pre>\u000a

V<p>The pooling and convolutional ops slide a "window" across the input tensor.  Using <a href="https://www.tensorflow.org/versions/master/api_docs/python/nn.html#conv2d"><code>tf.nn.conv2d</code></a> as an example: If the input tensor has 4 dimensions:  <code>[batch, height, width, channels]</code>, then the convolution operates on a 2D window on the <code>height, width</code> dimensions.</p>\u000a\u000a<p><code>strides</code> determines how much the window shifts by in each of the dimensions.  The typical use sets the first (the batch) and last (the depth) stride to 1.</p>\u000a\u000a<p>Let's use a very concrete example:  Running a 2-d convolution over a 32x32 greyscale input image.  I say greyscale because then the input image has depth=1, which helps keep it simple.  Let that image look like this:</p>\u000a\u000a<pre><code>00 01 02 03 04 ...\u000a10 11 12 13 14 ...\u000a20 21 22 23 24 ...\u000a30 31 32 33 34 ...\u000a...\u000a</code></pre>\u000a\u000a<p>Let's run a 2x2 convolution window over a single example (batch size = 1).  We'll give the convolution an output channel depth of 8.</p>\u000a\u000a<p>The input to the convolution has <code>shape=[1, 32, 32, 1]</code>.</p>\u000a\u000a<p>If you specify <code>strides=[1,1,1,1]</code> with <code>padding=SAME</code>, then the output of the filter will be [1, 32, 32, 8].</p>\u000a\u000a<p>The filter will first create an output for:</p>\u000a\u000a<pre><code>F(00 01\u000a  10 11)\u000a</code></pre>\u000a\u000a<p>And then for:</p>\u000a\u000a<pre><code>F(01 02\u000a  11 12)\u000a</code></pre>\u000a\u000a<p>and so on.  Then it will move to the second row, calculating:</p>\u000a\u000a<pre><code>F(10, 11\u000a  20, 21)\u000a</code></pre>\u000a\u000a<p>then</p>\u000a\u000a<pre><code>F(11, 12\u000a  21, 22)\u000a</code></pre>\u000a\u000a<p>If you specify a stride of [1, 2, 2, 1] it won't do overlapping windows.  It will compute:</p>\u000a\u000a<pre><code>F(00, 01\u000a  10, 11)\u000a</code></pre>\u000a\u000a<p>and then</p>\u000a\u000a<pre><code>F(02, 03\u000a  12, 13)\u000a</code></pre>\u000a\u000a<p>The stride operates similarly for the pooling operators.</p>\u000a\u000a<p><strong>Question 2:  Why strides [1, x, y, 1] for convnets</strong></p>\u000a\u000a<p>The first 1 is the batch:  You don't usually want to skip over examples in your batch, or you shouldn't have included them in the first place. :)</p>\u000a\u000a<p>The last 1 is the depth of the convolution:  You don't usually want to skip inputs, for the same reason.</p>\u000a\u000a<p>The conv2d operator is more general, so you <em>could</em> create convolutions that slide the window along other dimensions, but that's not a typical use in convnets.  The typical use is to use them spatially.</p>\u000a\u000a<p><strong>Why reshape to -1</strong>  -1 is a placeholder that says "adjust as necessary to match the size needed for the full tensor."  It's a way of making the code be independent of the input batch size, so that you can change your pipeline and not have to adjust the batch size everywhere in the code.</p>\u000a

It seems that seaborn is actually binning your returned <code>randint()</code> values in those bins, and there is no way to get an even spread of 110 values into 50 bins. Therefore you get those peaks where three values get put into a bin rather than the usual two values for the other bins. V<p>Generators exist in C++, just under another name: <em>Input Iterators</em>. For example, reading from <code>std::cin</code> is similar to having a generator of <code>char</code>.</p>\u000a\u000a<p>You simply need to understand what a generator does:</p>\u000a\u000a<ul>\u000a<li>there is a blob of data: the local variables define a <em>state</em></li>\u000a<li>there is an init method</li>\u000a<li>there is a "next" method</li>\u000a<li>there is a way to signal termination</li>\u000a</ul>\u000a\u000a<p>In your trivial example, it's easy enough. Conceptually:</p>\u000a\u000a<pre><code>struct State { unsigned i, j; };\u000a\u000aState make();\u000a\u000avoid next(State&amp;);\u000a\u000abool isDone(State const&amp;);\u000a</code></pre>\u000a\u000a<p>Of course, we wrap this as a proper class:</p>\u000a\u000a<pre><code>class PairSequence:\u000a    // (implicit aliases)\u000a    public std::iterator&lt;\u000a        std::input_iterator_tag,\u000a        std::pair&lt;unsigned, unsigned&gt;\u000a    &gt;\u000a{\u000a  // C++03\u000a  typedef void (PairSequence::*BoolLike)();\u000a  void non_comparable();\u000apublic:\u000a  // C++11 (explicit aliases)\u000a  using iterator_category = std::input_iterator_tag;\u000a  using value_type = std::pair&lt;unsigned, unsigned&gt;;\u000a  using reference = value_type const&amp;;\u000a  using pointer = value_type const*;\u000a  using difference_type = ptrdiff_t;\u000a\u000a  // C++03 (explicit aliases)\u000a  typedef std::input_iterator_tag iterator_category;\u000a  typedef std::pair&lt;unsigned, unsigned&gt; value_type;\u000a  typedef value_type const&amp; reference;\u000a  typedef value_type const* pointer;\u000a  typedef ptrdiff_t difference_type;\u000a\u000a  PairSequence(): done(false) {}\u000a\u000a  // C++11\u000a  explicit operator bool() const { return !done; }\u000a\u000a  // C++03\u000a  // Safe Bool idiom\u000a  operator BoolLike() const {\u000a    return done ? 0 : &amp;PairSequence::non_comparable;\u000a  }\u000a\u000a  reference operator*() const { return ij; }\u000a  pointer operator-&gt;() const { return &amp;ij; }\u000a\u000a  PairSequence&amp; operator++() {\u000a    static unsigned const Max = std::numeric_limts&lt;unsigned&gt;::max();\u000a\u000a    assert(!done);\u000a\u000a    if (ij.second != Max) { ++ij.second; return *this; }\u000a    if (ij.first != Max) { ij.second = 0; ++ij.first; return *this; }\u000a\u000a    done = true;\u000a    return *this;\u000a  }\u000a\u000a  PairSequence operator++(int) {\u000a    PairSequence const tmp(*this);\u000a    ++*this;\u000a    return tmp;\u000a  }\u000a\u000aprivate:\u000a  bool done;\u000a  value_type ij;\u000a};\u000a</code></pre>\u000a\u000a<p>So hum yeah... might be that C++ is a tad more verbose :)</p>\u000a

V<p>This limitation of the standard <code>shutil.copytree</code> seems arbitrary and annoying. Workaround:</p>\u000a\u000a<pre><code>def copytree(src, dst, symlinks=False, ignore=None):\u000a    for item in os.listdir(src):\u000a        s = os.path.join(src, item)\u000a        d = os.path.join(dst, item)\u000a        if os.path.isdir(s):\u000a            shutil.copytree(s, d, symlinks, ignore)\u000a        else:\u000a            shutil.copy2(s, d)\u000a</code></pre>\u000a\u000a<p>Note that it's not entirely consistent with the standard copytree:</p>\u000a\u000a<ul>\u000a<li>it doesn't honor <code>symlinks</code> and <code>ignore</code> parameters for the root directory of the <code>src</code> tree;</li>\u000a<li>it doesn't raise <code>shutil.Error</code> for errors at the root level of <code>src</code>;</li>\u000a<li>in case of errors during copying of a subtree, it will raise <code>shutil.Error</code> for that subtree instead of trying to copy other subtrees and raising single combined <code>shutil.Error</code>.</li>\u000a</ul>\u000a

V<h2>The Birthday Paradox, or why PRNGs produce duplicates more often than you might think.</h2>\u000a\u000a<p><br>There are a couple of issues at play in the OP's problem.  One is the <a href="http://en.wikipedia.org/wiki/Birthday_problem" rel="nofollow noreferrer">birthday paradox</a> as mentioned above and the second is the nature of what you are generating, which does not inherently guarantee that a given number will not be repeated.  </p>\u000a\u000a<p>The Birthday Paradox applies where given value can occur more than once during the period of the generator - and therefore duplicates can happen within a sample of values.  The effect of the Birthday Paradox is that the real likelihood of getting such duplicates is quite significant and the average period between them is smaller than one might otherwise have thought.  This dissonance between the percived and actual probabilities makes the Birthday Paradox a good example example of a <a href="http://en.wikipedia.org/wiki/Cognitive_bias" rel="nofollow noreferrer">cognitive bias</a>, where a naive intuitive estimate is likely to be wildly wrong.</p>\u000a\u000a<p><strong>A quick primer on Pseudo Random Number Generators (PRNGs)</strong></p>\u000a\u000a<p>The first part of your problem is that you are taking the exposed value of a random number generator and converting it to a much smaller number, so the space of possible values is reduced.  Although some <a href="http://en.wikipedia.org/wiki/Pseudorandom_number_generator" rel="nofollow noreferrer">pseudo-random number generators</a> do not repeat values during their period this transformation changes the domain to a much smaller one.  The smaller domain invalidates the 'no repeats' condition so you can expect a significant likelihood of repeats.   </p>\u000a\u000a<p>Some algorithms, such as the <a href="http://en.wikipedia.org/wiki/Linear_congruential_generator" rel="nofollow noreferrer">linear congruential PRNG</a> (<code>A'=AX|M</code>) <em>do</em> guarantee uniqueness for the entire period.  In a LCG the generated value contains the entire state of the accumulator and no additional state is held.  The generator is deterministic and cannot repeat a number within the period - any given accumulator value can imply only one possible successive value.  Therefore, each value can only occur once within the period of the generator.  However, the period of such a PRNG is relatively small - about 2^30 for typical implementations of the LCG algorithm - and cannot possibly be larger than the number of distinct values.</p>\u000a\u000a<p>Not all PRNG algorithms share this characteristic; some can repeat a given value within the period.  In the OP's problem the <a href="http://en.wikipedia.org/wiki/Mersenne_twister" rel="nofollow noreferrer">Mersenne Twister</a> algorithm (used in Python's <a href="http://docs.python.org/library/random.html" rel="nofollow noreferrer">random</a> module) has a very long period - much greater than 2^32.   Unlike a Linear Congruential PRNG, the result is not purely a function of the previous output value as the accumulator contains additional state.  With 32 bit integer output and a period of ~2^19937 it cannot possibly provide a such a guarantee.  </p>\u000a\u000a<p>The Mersenne Twister is a popular algorithm for PRNGs because it has good statistical and geometric properties and a very long period - desirable characteristics for a PRNG used on simulation models.   </p>\u000a\u000a<ul>\u000a<li><p>Good <a href="http://en.wikipedia.org/wiki/Uniform_distribution_(discrete)" rel="nofollow noreferrer">statistical properties</a> mean that the numbers generated by the algorithm are evenly distributed with no numbers having a significantly higher probability of appearing than others.  Poor statistical properties could produce unwanted skew in the results.</p></li>\u000a<li><p>Good <a href="http://random.mat.sbg.ac.at/~charly/server/node3.html" rel="nofollow noreferrer">geometric properies</a> mean that sets of N numbers do not lie on a hyperplane in N dimensional space.  Poor geometric properties can generate spurious correlations in a simulation model and distort the results.</p></li>\u000a<li><p>A long period means that you can generate a lot of numbers before the sequence wraps around to the start.  If a model needs a large number of iterations or has to be run from several seeds then the 2^30 or so discrete numbers available from a typical LCG implementation may not be sufficient.  The MT19337 algorithm has a very long period - 2^19337-1, or about 10^5821.  By comparison the total number of atoms in the universe is estimated at about 10^80.</p></li>\u000a</ul>\u000a\u000a<p>The 32 bit integer produced by a MT19337 PRNG cannot possibly represent enough discrete values to avoid repeating during such a large period.  In this case duplicate values are likely to occur and inevitable with a large enough sample.  </p>\u000a\u000a<p><strong>The Birthday Paradox in a nutshell</strong></p>\u000a\u000a<p>This problem is originally defined as the probability of any two people in the room sharing the same birthday.  The key point is that <strong><em>any two</em></strong> people in the room could share a birthday.  People tend to naively misinterpret the problem as the probability of someone in the room sharing a birthday with a specific individual, which is the source of the <a href="http://en.wikipedia.org/wiki/Cognitive_bias" rel="nofollow noreferrer">cognitive bias</a> that often causes people to underestimate the probability.  This is the incorrect assumption - there is no requirement for the match to be to a specific individual and any two individuals could match.  </p>\u000a\u000a<p><img src="https://upload.wikimedia.org/wikipedia/commons/thumb/5/56/Birthday_paradox.png/450px-Birthday_paradox.png" alt="This graph shows the probability of a shared birthday as number of people in the room increases.  For 23 people the probability of two sharing a birthday is just over 50%." title="This graph shows the probability of a shared birthday as number of people in the room increases.  For 23 people the probability of two sharing a birthday is just over 50%."></p>\u000a\u000a<p>The probability of a match occurring between any two individuals is much higher than the probability of a match to a specific individual as the match does not have to be to a specific date.  Rather, you only have to find two individuals that share the same birthday.  From this graph (which can be found on the wikipedia page on the subject), we can see that we only need 23 people in the room for there to be a 50% chance of finding two that match in this way.</p>\u000a\u000a<p>From the <a href="http://en.wikipedia.org/wiki/Birthday_problem" rel="nofollow noreferrer">Wikipedia entry on the subject</a> we can get a <a href="http://en.wikipedia.org/wiki/Birthday_problem#Understanding_the_problem" rel="nofollow noreferrer">nice summary.</a>  In the OP's problem we have 4,500 possible 'birthdays', rather than 365.  For a given number of random values generated (equating to 'people') we want to know the probability of <em>any</em> two identical values appearing within the sequence.  </p>\u000a\u000a<p><strong>Computing the likely effect of the Birthday Paradox on the OP's problem</strong></p>\u000a\u000a<p>For a sequence of 100 numbers, we have <a href="http://mathurl.com/yc4cdsr.png" rel="nofollow noreferrer" title="(100 * 99) / 2 = 4950">(100 * 99) / 2 = 4950 http://mathurl.com/yc4cdsr.png</a> pairs (see <a href="http://en.wikipedia.org/wiki/Birthday_problem#Understanding_the_problem" rel="nofollow noreferrer">Understanding the Problem</a>) that could potentially match (i.e. the first could match with the second, third etc., the second could match the third, fourth etc. and so on), so the number of <em>combinations</em> that could potentially match is rather more than just 100.  </p>\u000a\u000a<p>From <a href="http://en.wikipedia.org/wiki/Birthday_problem#Calculating_the_probability" rel="nofollow noreferrer">Calculating the Probability</a> we get an expression of <a href="http://mathurl.com/ybfweny.png" rel="nofollow noreferrer" title="1 - (4500! / (4500**100 * (4500 - 100)!)">1 - (4500! / (4500**100 * (4500 - 100)!) http://mathurl.com/ybfweny.png</a>.  The following snippet of Python code below does a naive evaluation of the probability of a matching pair occurring.  </p>\u000a\u000a<pre><code># === birthday.py ===========================================\u000a#\u000afrom math import log10, factorial\u000a\u000aPV=4500          # Number of possible values\u000aSS=100           # Sample size\u000a\u000a# These intermediate results are exceedingly large numbers;\u000a# Python automatically starts using bignums behind the scenes.\u000a#\u000anumerator = factorial (PV)          \u000adenominator = (PV ** SS) * factorial (PV - SS)\u000a\u000a# Now we need to get from bignums to floats without intermediate\u000a# values too large to cast into a double.  Taking the logs and \u000a# subtracting them is equivalent to division.\u000a#  \u000alog_prob_no_pair = log10 (numerator) - log10 (denominator)\u000a\u000a# We've just calculated the log of the probability that *NO*\u000a# two matching pairs occur in the sample.  The probability\u000a# of at least one collision is 1.0 - the probability that no \u000a# matching pairs exist.\u000a#\u000aprint 1.0 - (10 ** log_prob_no_pair)\u000a</code></pre>\u000a\u000a<p>This produces a sensible looking result of <strong><em>p=0.669</em></strong> for a match occuring within 100 numbers sampled from a population of 4500 possible values. (Maybe someone could verify this and post a comment if it's wrong).  From this we can see that the lengths of runs between matching numbers observed by the OP seem to be quite reasonable.</p>\u000a\u000a<p><strong>Footnote: using shuffling to get a unique sequence of pseudo-random numbers</strong> </p>\u000a\u000a<p>See <a href="http://stackoverflow.com/questions/2145510/python-random-is-barely-random-at-all/2145533#2145533">this answer below from S. Mark</a> for a means of getting a guaranteed unique set of random numbers.  The technique the poster refers to takes an array of numbers (which you supply, so you can make them unique) and shuffles them into random order.  Drawing the numbers in sequence from the shuffled array will give you a sequence of pseudo-random numbers that are guaranteed not to repeat.</p>\u000a\u000a<p><strong>Footnote: Cryptographically Secure PRNGs</strong> </p>\u000a\u000a<p>The MT algorithm is not <a href="http://en.wikipedia.org/wiki/Cryptographically_secure_pseudorandom_number_generator" rel="nofollow noreferrer">cryptographically secure</a> as it is relatively easy to infer the internal state of the generator by observing a sequence of numbers.  Other algorithms such as <a href="http://en.wikipedia.org/wiki/Blum_Blum_Shub" rel="nofollow noreferrer">Blum Blum Shub</a> are used for cryptographic applications, but may be unsuitable for simulation or general random number applications.  Cryptographically secure PRNGs may be expensive (perhaps requiring bignum calculations) or may not have good geometric properties.  In the case of this type of algorithm the primary requirement is that it should be computationally infeasible to infer the internal state of the generator by observing a sequence of values.</p>\u000a

V<pre><code>&gt;&gt;&gt; d = {"foo": 12, "bar": 2, "jim": 4, "bob": 17}\u000a&gt;&gt;&gt; [k for k, v in d.items() if v &gt; 6] # Use d.iteritems() on python 2.x\u000a['bob', 'foo']\u000a</code></pre>\u000a\u000a<hr>\u000a\u000a<p>I'd like to just update this answer to also showcase the solution by @glarrain which I find myself tending to use nowadays.</p>\u000a\u000a<pre><code>[k for k in d if d[k] &gt; 6]\u000a</code></pre>\u000a\u000a<p>This is completely cross compatible and doesn't require a confusing change from <code>.iteritems</code> (<code>.iteritems</code> avoids saving a list to memory on Python 2 which is fixed in Python 3) to <code>.items</code>. </p>\u000a\u000a<p>@Prof.Falken mentioned a solution to this problem</p>\u000a\u000a<pre><code>from six import iteritems\u000a</code></pre>\u000a\u000a<p>which effectively fixes the cross compatibility issues BUT requires you to download the package <code>six</code></p>\u000a\u000a<p>However I would not fully agree with @glarrain that this solution is more readable, that is up for debate and maybe just a personal preference even though Python is supposed to have only 1 way to do it. V<h3>To explain "why":</h3>\u000a\u000a<p>The <a href="http://docs.python.org/reference/datamodel.html#object.__add__" rel="noreferrer"><code>+</code> operation</a> adds the <a href="http://docs.python.org/library/array.html" rel="noreferrer">array</a> elements to the original array. The <a href="http://docs.python.org/library/array.html?highlight=append#array.array.append" rel="noreferrer"><code>array.append</code></a> operation inserts the array (or any object) into the end of the original array, which results in a <em>reference to self</em> in that spot (hence the infinite recursion).</p>\u000a\u000a<p>The difference here is that the + operation acts specific when you add an array (it's overloaded like others, see <a href="http://docs.python.org/reference/datamodel.html#emulating-container-types" rel="noreferrer">this chapter</a> on sequences) by concatenating the element. The append-method however does literally what you ask: append the object on the right-hand side that you give it (the array or any other object), instead of taking its elements.</p>\u000a\u000a<h3>An alternative</h3>\u000a\u000a<p>Use <a href="http://docs.python.org/library/array.html?highlight=append#array.array.extend" rel="noreferrer"><code>extend()</code></a> if you want to use a function that acts similar to the + operator (as others have shown here as well). It's not wise to do the opposite: to try to mimic append with the + operator for lists (see my <a href="http://docs.python.org/reference/datamodel.html#emulating-container-types" rel="noreferrer">earlier link</a> on why).</p>\u000a\u000a<h3>Little history</h3>\u000a\u000a<p>For fun, a little history: the <a href="http://ftp.ntua.gr/mirror/python/search/hypermail/python-1993/0018.html" rel="noreferrer">birth of the array module in Python</a> in February 1993. it might surprise you, but arrays were added way after sequences and lists came into existence.</p>\u000a

V<p>Django and AngularJS both have CSRF support already, your part is quite simple.</p>\u000a\u000a<p>First, you need to enable CSRF in Django, I believe you have already done so, if not, follow Django doc <a href="https://docs.djangoproject.com/en/1.5/ref/contrib/csrf/#ajax">https://docs.djangoproject.com/en/1.5/ref/contrib/csrf/#ajax</a>.</p>\u000a\u000a<p>Now, Django will set a cookie named <code>csrftoken</code> on the first GET request and expects a custom HTTP header <code>X-CSRFToken</code> on later POST/PUT/DELETE requests.</p>\u000a\u000a<p>For Angular, it expects the cookie named <code>XSRF-TOKEN</code> and will do POST/PUT/DELETE requests with <code>X-XSRF-TOKEN</code> header, so you need to do a little bit tweak to make the two go with each other:</p>\u000a\u000a<pre><code>$httpProvider.defaults.xsrfCookieName = 'csrftoken';\u000a$httpProvider.defaults.xsrfHeaderName = 'X-CSRFToken';\u000a</code></pre>\u000a\u000a<p>Add above two lines somewhere in your js code, module.config() block is a good place for this.</p>\u000a\u000a<p>That's it.</p>\u000a\u000a<p><strong>NOTE:</strong> This is for angular 1.1.5, older versions might need different approach.</p>\u000a\u000a<h2>Update:</h2>\u000a\u000a<p>Since the angular app isn't served by django, in order to let the cookie to be set, angular app needs to do a GET request to django first.</p>\u000a

V<p>Matplotlib doesn't work with pixels directly, but rather physical sizes and DPI. If you want to display a figure with a certain pixel size, you need to know the DPI of your monitor. For example <a href="http://www.infobyip.com/detectmonitordpi.php" rel="noreferrer">this link</a> will detect that for you.</p>\u000a\u000a<p>If you have an image of 3841x7195 pixels it is unlikely that you monitor will be that large, so you won't be able to show a figure of that size (matplotlib requires the figure to fit in the screen, if you ask for a size too large it will shrink to the screen size). Let's imagine you want an 800x800 pixel image just for an example. Here's how to show an 800x800 pixel image in my monitor (<code>my_dpi=96</code>):</p>\u000a\u000a<pre><code>plt.figure(figsize=(800/my_dpi, 800/my_dpi), dpi=my_dpi)\u000a</code></pre>\u000a\u000a<p>So you basically just divide the dimensions in inches by your DPI. </p>\u000a\u000a<p>If you want to save a figure of a specific size, then it is a different matter. Screen DPIs are not so important anymore (unless you ask for a figure that won't fit in the screen). Using the same example of the 800x800 pixel figure, we can save it in different resolutions using the <code>dpi</code> keyword of <code>savefig</code>. To save it in the same resolution as the screen just use the same dpi:</p>\u000a\u000a<pre><code>plt.savefig('my_fig.png', dpi=my_dpi)\u000a</code></pre>\u000a\u000a<p>To to save it as an 8000x8000 pixel image, use a dpi 10 times larger:</p>\u000a\u000a<pre><code>plt.savefig('my_fig.png', dpi=my_dpi * 10)\u000a</code></pre>\u000a\u000a<p>Note that the setting of the DPI is not supported by all backends. Here, the PNG backend is used, but the pdf and ps backends will implement the size differently. Also, changing the DPI and sizes will also affect things like fontsize. A larger DPI will keep the same relative sizes of fonts and elements, but if you want smaller fonts for a larger figure you need to increase the physical size instead of the DPI. </p>\u000a\u000a<p>Getting back to your example, if you want to save a image with 3841 x 7195 pixels, you could do the following:</p>\u000a\u000a<pre><code>plt.figure(figsize=(3.841, 7.195), dpi=100)\u000a( your code ...)\u000aplt.savefig('myfig.png', dpi=1000)\u000a</code></pre>\u000a\u000a<p>Note that I used the figure dpi of 100 to fit in most screens, but saved with <code>dpi=1000</code> to achieve the required resolution. In my system this produces a png with 3840x7190 pixels -- it seems that the DPI saved is always 0.02 pixels/inch smaller than the selected value, which will have a (small) effect on large image sizes. Some more discussion of this <a href="http://stackoverflow.com/questions/8775622/exact-figure-size-in-matplotlib-with-title-axis-labels">here</a>.</p>\u000a

V<p>You get a recursion error because you call the same function, your <code>__getattribute__</code>. V<p>I'll be the odd voice out (since all answers are decrying the use of the fact that <code>False == 0</code> and <code>True == 1</code>, as the language guarantees) as I claim that the use of this fact to simplify your code is perfectly fine.</p>\u000a\u000a<p>Historically, logical true/false operations tended to simply use <code>0</code> for false and <code>1</code> for true; in the course of Python 2.2's life-cycle, Guido noticed that too many modules started with assignments such as <code>false = 0; true = 1</code> and this produced boilerplate and useless variation (the latter because the capitalization of true and false was all over the place -- some used all-caps, some all-lowercase, some cap-initial) and so introduced the <code>bool</code> subclass of <code>int</code> and its <code>True</code> and <code>False</code> constants.</p>\u000a\u000a<p>There was quite some pushback at the time since many of us feared that the new type and constants would be used by Python newbies to <em>restrict</em> the language's abilities, but Guido was adamant that we were just being pessimistic: nobody would ever understand Python so badly, for example, as to avoid the perfectly natural use of <code>False</code> and <code>True</code> as list indices, or in a summation, or other such perfectly clear and useful idioms.</p>\u000a\u000a<p>The answers to this thread prove we were right: as we feared, a total misunderstanding of the roles of this type and constants <em>has</em> emerged, and people <em>are</em> avoiding, and, worse!, urging others to avoid, perfectly natural Python constructs in favor of useless gyrations.</p>\u000a\u000a<p>Fighting against the tide of such misunderstanding, I urge everybody to <em>use Python as Python</em>, <strong>not</strong> trying to force it into the mold of other languages whose functionality and preferred style are quite different.  <strong>In Python</strong>, True and False are 99.9% like 1 and 0, differing <em>exclusively</em> in their <code>str(...)</code> (and thereby <code>repr(...)</code>) form -- for <em>every</em> other operation except stringification, just feel free to use them without contortions.  That goes for indexing, arithmetic, bit operations, etc, etc, etc.</p>\u000a

V<p>In terms of speed, the best cross-platform IPC mechanism will be pipes.  That assumes, however, that you want cross-platform IPC on the same machine.  If you want to be able to talk to processes on remote machines, you'll want to look at using sockets instead.  Luckily, if you're talking about TCP at least, sockets and pipes behave pretty much the same behavior.  While the APIs for setting them up and connecting them are different, they both just act like streams of data.</p>\u000a\u000a<p>The difficult part, however, is not the communication channel, but the messages you pass over it.  You really want to look at something that will perform verification and parsing for you.  I recommend looking at Google's <a href="http://code.google.com/apis/protocolbuffers/docs/overview.html" rel="noreferrer">Protocol Buffers</a>. You basically create a spec file that describes the object you want to pass between processes, and there is a compiler that generates code in a number of different languages for reading and writing objects that match the spec.  It's much easier (and less bug prone) than trying to come up with a messaging protocol and parser yourself.</p>\u000a
