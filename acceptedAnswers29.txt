V<p><code>[]</code> denotes a <a href="http://docs.python.org/3/tutorial/introduction.html#lists" rel="noreferrer">list</a>, <code>()</code> denotes a <a href="http://docs.python.org/3/tutorial/datastructures.html#tuples-and-sequences" rel="noreferrer">tuple</a> and <code>{}</code> denotes a <a href="http://docs.python.org/3/tutorial/datastructures.html#dictionaries" rel="noreferrer">dictionary</a>. You should take a look at the <a href="http://docs.python.org/3/tutorial/index.html" rel="noreferrer">official Python tutorial</a> as these are the very basics of programming in Python.</p>\u000a\u000a<p>What you have is a list of strings. You can sort it like this:</p>\u000a\u000a<pre><code>In [1]: lst = ['Stem', 'constitute', 'Sedge', 'Eflux', 'Whim', 'Intrigue']\u000a\u000aIn [2]: sorted(lst)\u000aOut[2]: ['Eflux', 'Intrigue', 'Sedge', 'Stem', 'Whim', 'constitute']\u000a</code></pre>\u000a\u000a<p>As you can see, words that start with an uppercase letter get preference over those starting with a lowercase letter. If you want to sort them independently, do this:</p>\u000a\u000a<pre><code>In [4]: sorted(lst, key=str.lower)\u000aOut[4]: ['constitute', 'Eflux', 'Intrigue', 'Sedge', 'Stem', 'Whim']\u000a</code></pre>\u000a\u000a<p>You can also sort the list in reverse order by doing this:</p>\u000a\u000a<pre><code>In [12]: sorted(lst, reverse=True)\u000aOut[12]: ['constitute', 'Whim', 'Stem', 'Sedge', 'Intrigue', 'Eflux']\u000a\u000aIn [13]: sorted(lst, key=str.lower, reverse=True)\u000aOut[13]: ['Whim', 'Stem', 'Sedge', 'Intrigue', 'Eflux', 'constitute']\u000a</code></pre>\u000a\u000a<p><strong>Please note:</strong> If you work with Python 3, then <code>str</code> is the correct data type for every string that contains human-readable text. However, if you still need to work with Python 2, then you might deal with unicode strings which have the data type <code>unicode</code> in Python 2, and not <code>str</code>. In such a case, if you have a list of unicode strings, you must write <code>key=unicode.lower</code> instead of <code>key=str.lower</code>.</p>\u000a

V<pre><code>for i in ...:\u000a    for j in ...:\u000a        for k in ...:\u000a            if something:\u000a                # continue loop i\u000a</code></pre>\u000a\u000a<p>In a general case, when you have multiple levels of looping and <code>break</code> does not work for you (because you want to continue one of the upper loops, not the one right above the current one), you can do one of the following</p>\u000a\u000a<h2>Refactor the loops you want to escape from into a function</h2>\u000a\u000a<pre><code>def inner():\u000a    for j in ...:\u000a        for k in ...:\u000a            if something:\u000a                return\u000a\u000a\u000afor i in ...:\u000a    inner()\u000a</code></pre>\u000a\u000a<p>The disadvantage is that you may need to pass to that new function some variables, which were previously in scope. You can either just pass them as parameters, make them instance variables on an object (create a new object just for this function, if it makes sense), or global variables, singletons, whatever (ehm, ehm).</p>\u000a\u000a<p>Or you can define <code>inner</code> as a nested function and let it just capture what it needs (may be slower?)</p>\u000a\u000a<pre><code>for i in ...:\u000a    def inner():\u000a        for j in ...:\u000a            for k in ...:\u000a                if something:\u000a                    return\u000a    inner()\u000a</code></pre>\u000a\u000a<h2>Use exceptions</h2>\u000a\u000a<p>Philosophically, this is what exceptions are for, breaking the program flow through the structured programming building blocks (if, for, while) when necessary.</p>\u000a\u000a<p>The advantage is that you don't have to break the single piece of code into multiple parts. This is good if it is some kind of computation that you are designing while writing it in Python. Introducing abstractions at this early point may slow you down.</p>\u000a\u000a<p>Bad thing with this approach is that interpreter/compiler authors usually assume that exceptions are exceptional and optimize for them accordingly.</p>\u000a\u000a<pre><code>class ContinueI(Exception):\u000a    pass\u000a\u000a\u000acontinue_i = ContinueI()\u000a\u000afor i in ...:\u000a    try:\u000a        for j in ...:\u000a            for k in ...:\u000a                if something:\u000a                    raise continue_i\u000a    except ContinueI:\u000a        continue\u000a</code></pre>\u000a\u000a<p>Create a special exception class for this, so that you don't risk accidentally silencing some other exception.</p>\u000a\u000a<h2>Something else entirely</h2>\u000a\u000a<p>I am sure there are still other solutions.</p>\u000a

V<h3>Accessing the color cycle iterator</h3>\u000a\u000a<p>There's no "user-facing" (a.k.a. "public") method to access the underlying iterator, but you can access it through "private" (by convention) methods.  However, you'd can't get the state of an <code>iterator</code> without changing it.</p>\u000a\u000a<h3>Setting the color cycle</h3>\u000a\u000a<p>Quick aside: You can set the color/property cycle in a variety of ways (e.g. <code>ax.set_color_cycle</code> in versions &lt;1.5 or <code>ax.set_prop_cycler</code> in >=1.5).  Have a look at the <a href="http://matplotlib.org/examples/color/color_cycle_demo.html">example here for version 1.5 or greater</a>, or the <a href="http://matplotlib.org/1.4.3/examples/color/color_cycle_demo.html">previous style here</a>.</p>\u000a\u000a<h3>Accessing the underlying iterator</h3>\u000a\u000a<p>However, while there's no public-facing method to access the iterable, you can access it for a given axes object (<code>ax</code>) through the <code>_get_lines</code> helper class instance.  <code>ax._get_lines</code> is a touch confusingly named, but it's the behind-the-scenes machinery that allows the <code>plot</code> command to process all of the odd and varied ways that <code>plot</code> can be called.  Among other things, it's what keeps track of what colors to automatically assign. Similarly, there's <code>ax._get_patches_for_fill</code> to control cycling through default fill colors and patch properties.</p>\u000a\u000a<p>At any rate, the color cycle iterable is <code>ax._get_lines.color_cycle</code> for lines and <code>ax._get_patches_for_fill.color_cycle</code> for patches.  On matplotlib >=1.5, this has changed to <a href="http://matplotlib.org/cycler/">use the <code>cycler</code> library</a>, and the iterable is called <code>prop_cycler</code> instead of <code>color_cycle</code> and yields a <code>dict</code> of properties instead of only a color.</p>\u000a\u000a<p>All in all, you'd do something like:</p>\u000a\u000a<pre><code>import matplotlib.pyplot as plt\u000a\u000afig, ax = plt.subplots()\u000acolor_cycle = ax._get_lines.color_cycle\u000a# or ax._get_lines.prop_cycle on version &gt;= 1.5\u000a# Note that prop_cycler cycles over dicts, so you'll want next(cycle)['color']\u000a</code></pre>\u000a\u000a<h3>You can't view the state of an <code>iterator</code></h3>\u000a\u000a<p>However, this object is a "bare" <code>iterator</code>. We can easily get the next item (e.g. <code>next_color = next(color_cycle)</code>, but that means that the next color <em>after that</em> is what will be plotted.  By design, there's no way to get the current state of an iterator without changing it.</p>\u000a\u000a<p>In <code>v1.5</code> or greater, it would be nice to get the <code>cycler</code> object that's used, as we could infer its current state. However, the <code>cycler</code> object itself isn't accessible (publicly or privately) anywhere. Instead, only the <code>itertools.cycle</code> instance created from the <code>cycler</code> object is accessible.  Either way, there's no way to get to the underlying state of the color/property cycler.</p>\u000a\u000a<h3>Match the color of the previously plotted item instead</h3>\u000a\u000a<p>In your case, it sounds like you're wanting to match the color of something that was just plotted.  Instead of trying to determine what the color/property will be, set the color/etc of your new item based on the properties of what's plotted.</p>\u000a\u000a<p>For example, in the case you described, I'd do something like this:</p>\u000a\u000a<pre><code>import matplotlib.pyplot as plt\u000aimport numpy as np\u000a\u000adef custom_plot(x, y, **kwargs):\u000a    ax = kwargs.pop('ax', plt.gca())\u000a    base_line, = ax.plot(x, y, **kwargs)\u000a    ax.fill_between(x, 0.9*y, 1.1*y, facecolor=base_line.get_color(), alpha=0.5)\u000a\u000ax = np.linspace(0, 1, 10)\u000acustom_plot(x, x)\u000acustom_plot(x, 2*x)\u000acustom_plot(x, -x, color='yellow', lw=3)\u000a\u000aplt.show()\u000a</code></pre>\u000a\u000a<p><img src="https://i.stack.imgur.com/w4hOV.png" alt="enter image description here"></p>\u000a\u000a<p>It's not the only way, but its cleaner than trying to get the color of the plotted line before-hand, in this case.</p>\u000a

V<p>There are two key differences between <code>imap</code>/<code>imap_unordered</code> and <code>map</code>/<code>map_async</code>:</p>\u000a\u000a<ol>\u000a<li>The way they consume the iterable you pass to them.</li>\u000a<li>The way they return the result back to you.</li>\u000a</ol>\u000a\u000a<p><code>map</code> consumes your iterable by converting the iterable to a list (assuming it isn't a list already), breaking it into chunks, and sending those chunks to the worker processes in the <code>Pool</code>. Breaking the iterable into chunks performs better than passing each item in the iterable between processes one item at a time - particularly if the iterable is large. However, turning the iterable into a list in order to chunk it can have a very high memory cost, since the entire list will need to be kept in memory. </p>\u000a\u000a<p><code>imap</code> doesn't turn the iterable you give it into a list, nor does break it into chunks (by default). It will iterate over the iterable one element at a time, and send them each to a worker process. This means you don't take the memory hit of converting the whole iterable to a list, but it also means the performance is slower for large iterables, because of the lack of chunking. This can be mitigated by passing a <code>chunksize</code> argument larger than default of 1, however.</p>\u000a\u000a<p>The other major difference between <code>imap</code>/<code>imap_unordered</code> and <code>map</code>/<code>map_async</code>, is that with <code>imap</code>/<code>imap_unordered</code>, you can start receiving results from workers as soon as they're ready, rather than having to wait for all of them to be finished. With <code>map_async</code>, an <code>AsyncResult</code> is returned right away, but you can't actually retrieve results from that object until all of them have been processed, at which points it returns the same list that <code>map</code> does (<code>map</code> is actually implemented internally as <code>map_async(...).get()</code>). There's no way to get partial results; you either have the entire result, or nothing.</p>\u000a\u000a<p><code>imap</code> and <code>imap_unordered</code> both return iterables right away. With <code>imap</code>, the results will be yielded from the iterable as soon as they're ready, while still preserving the ordering of the input iterable. With <code>imap_unordered</code>, results will be yielded as soon as they're ready, regardless of the order of the input iterable. So, say you have this:</p>\u000a\u000a<pre><code>import multiprocessing\u000aimport time\u000a\u000adef func(x):\u000a    time.sleep(x)\u000a    return x + 2\u000a\u000aif __name__ == "__main__":    \u000a    p = multiprocessing.Pool()\u000a    start = time.time()\u000a    for x in p.imap(func, [1,5,3]):\u000a        print("{} (Time elapsed: {}s)".format(x, int(time.time() - start)))\u000a</code></pre>\u000a\u000a<p>This will output:</p>\u000a\u000a<pre><code>3 (Time elapsed: 1s)\u000a7 (Time elapsed: 5s)\u000a5 (Time elapsed: 5s)\u000a</code></pre>\u000a\u000a<p>If you use <code>p.imap_unordered</code> instead of <code>p.imap</code>, you'll see:</p>\u000a\u000a<pre><code>3 (Time elapsed: 1s)\u000a5 (Time elapsed: 3s)\u000a7 (Time elapsed: 5s)\u000a</code></pre>\u000a\u000a<p>If you use <code>p.map</code> or <code>p.map_async().get()</code>, you'll see:</p>\u000a\u000a<pre><code>3 (Time elapsed: 5s)\u000a7 (Time elapsed: 5s)\u000a5 (Time elapsed: 5s)\u000a</code></pre>\u000a\u000a<p>So, the primary reasons to use <code>imap</code>/<code>imap_unordered</code> over <code>map_async</code> are:</p>\u000a\u000a<ol>\u000a<li>Your iterable is large enough that converting it to a list would cause you to run out of/use too much memory.</li>\u000a<li>You want to be able to start processing the results before <em>all</em> of them are completed.</li>\u000a</ol>\u000a

V<p><strong>Edit</strong>: See also <a href="http://stackoverflow.com/a/18474773/220155">Dave Jones' answer</a>: from Python 3.3, you can use the <code>x</code> flag to <code>open()</code> to provide this function.</p>\u000a\u000a<p><strong>Original answer below</strong></p>\u000a\u000a<p>Yes, but not using Python's standard <code>open()</code> call. You'll need to use <a href="http://docs.python.org/library/os.html#os.open" rel="noreferrer"><code>os.open()</code></a> instead, which allows you to specify flags to the underlying C code.</p>\u000a\u000a<p>In particular, you want to use <code>O_CREAT | O_EXCL</code>. From the man page for <code>open(2)</code> under <code>O_EXCL</code> on my Unix system:</p>\u000a\u000a<blockquote>\u000a  <p>Ensure  that  this  call  creates the file: if this flag is specified in conjunction with <code>O_CREAT</code>, and pathname already exists, then <code>open()</code> will fail.  The behavior of <code>O_EXCL</code> is undefined if <code>O_CREAT</code> is not specified.</p>\u000a  \u000a  <p>When these two flags are specified, symbolic links are not followed: if pathname is a symbolic link, then <code>open()</code> fails  regardless of where the symbolic link points to.</p>\u000a  \u000a  <p><code>O_EXCL</code>  is only supported on NFS when using NFSv3 or later on kernel 2.6 or later.  In environments where NFS <code>O_EXCL</code> support is not provided, programs that rely on it for performing locking tasks will contain a race condition.</p>\u000a</blockquote>\u000a\u000a<p>So it's not perfect, but AFAIK it's the closest you can get to avoiding this race condition.</p>\u000a\u000a<p>Edit: the other rules of using <code>os.open()</code> instead of <code>open()</code> still apply. In particular, if you want use the returned file descriptor for reading or writing, you'll need one of the <code>O_RDONLY</code>, <code>O_WRONLY</code> or <code>O_RDWR</code> flags as well.</p>\u000a\u000a<p>All the <code>O_*</code> flags are in Python's <code>os</code> module, so you'll need to <code>import os</code> and use <code>os.O_CREAT</code> etc.</p>\u000a\u000a<h3>Example:</h3>\u000a\u000a<pre><code>import os\u000aimport errno\u000a\u000aflags = os.O_CREAT | os.O_EXCL | os.O_WRONLY\u000a\u000atry:\u000a    file_handle = os.open('filename', flags)\u000aexcept OSError as e:\u000a    if e.errno == errno.EEXIST:  # Failed as the file already exists.\u000a        pass\u000a    else:  # Something unexpected went wrong so reraise the exception.\u000a        raise\u000aelse:  # No exception, so the file must have been created successfully.\u000a    with os.fdopen(file_handle, 'w') as file_obj:\u000a        # Using `os.fdopen` converts the handle to an object that acts like a\u000a        # regular Python file object, and the `with` context manager means the\u000a        # file will be automatically closed when we're done with it.\u000a        file_obj.write("Look, ma, I'm writing to a new file!")\u000a</code></pre>\u000a

V<p>The problem is you defined <code>myList</code> from main.py, but subfile.py needs to use it. Here is a clean way to solve this problem: move all globals to a file, I call this file <code>settings.py</code>. This file is responsible for defining globals and initializing them:</p>\u000a\u000a<pre><code># settings.py\u000a\u000adef init():\u000a    global myList\u000a    myList = []\u000a</code></pre>\u000a\u000a<p>Next, your subfile can import globals:</p>\u000a\u000a<pre><code># subfile.py\u000a\u000aimport settings\u000a\u000adef stuff():\u000a    settings.myList.append('hey')\u000a</code></pre>\u000a\u000a<p>Note that subfile does not call init() -- that task belongs to main.py:</p>\u000a\u000a<pre><code># main.py\u000a\u000aimport settings\u000aimport subfile\u000a\u000asettings.init()          # Call only once\u000asubfile.stuff()         # Do stuff with global var\u000aprint settings.myList[0] # Check the result\u000a</code></pre>\u000a\u000a<p>This way, you achieve your objective while avoid initializing global variables more than once.</p>\u000a

V<p>You probably don't actually want to change your default Python.</p>\u000a\u000a<p>Your distro installed a standard system Python in <code>/usr/bin</code>, and may have scripts that depend on this being present, and selected by <code>#! /usr/bin/env python</code>. You can <em>usually</em> get away with running Python 2.6 scripts in 2.7, but do you want to risk it?</p>\u000a\u000a<p>On top of that, monkeying with <code>/usr/bin</code> can break your package manager's ability to manage packages. And changing the order of directories in your <code>PATH</code> will affect a lot of other things besides Python. (In fact, it's more common to have <code>/usr/local/bin</code> ahead of <code>/usr/bin</code>, and it may be what you actually want\u2014but if you have it the other way around, presumably there's a good reason for that.)</p>\u000a\u000a<p>But you don't <em>need</em> to change your default Python to get the system to run 2.7 when you type <code>python</code>.</p>\u000a\u000a<hr>\u000a\u000a<p>First, you can set up a shell alias:</p>\u000a\u000a<pre><code>alias python=/usr/local/bin/python2.7\u000a</code></pre>\u000a\u000a<p>Type that at a prompt, or put it in your <code>~/.bashrc</code> if you want the change to be persistent, and now when <em>you</em> type <code>python</code> it runs your chosen 2.7, but when some program on your system tries to run a script with <code>/usr/bin/env python</code> it runs the standard 2.6.</p>\u000a\u000a<hr>\u000a\u000a<p>Alternatively, just create a <a href="https://pypi.python.org/pypi/virtualenv" rel="noreferrer">virtual environment</a> out of your 2.7 (or separate venvs for different projects), and do your work inside the venv.</p>\u000a

V<p>I suppose the first question you need to ask are what permissions do you need and what sort.  By what sort, I mean do you want Model- or Object-level.  To clarify the difference say you have a model Car.  If you want to give permissions on all cars, then Model-level is appropriate, but if you want to give permissions on a per-car basis you want Object-level.  You may need both, and this isn't a problem as we'll see.  </p>\u000a\u000a<p>For Model permissions, Django handles these for you... mostly.  For each model Django will create permissions in the form 'appname.permissionname_modelname'.  If you have an app called 'drivers' with the Car model then one permission would be 'drivers.delete_car'.  The permissions that Django automatically creates will be create, change, and delete.  For some strange reason they decided not to include read permissions from CRUD, you will have to do this yourself.  Note that Django decided to change CRUD's 'update' to 'change' for some reason.  To add more permissions to a model, say read permissions, you use the Meta class:</p>\u000a\u000a<pre><code>class Car( models.Model ):\u000a    # model stuff here\u000a    class Meta:\u000a        permissions = ( \u000a            ( "read_car", "Can read Car" ),\u000a        )\u000a</code></pre>\u000a\u000a<p>Note that permissions is a set of tuples, where the tuple items are the permission as described above and a description of that permission.  You don't have to follow the permname_modelname convention but I usually stick with it.</p>\u000a\u000a<p>Finally, to check permissions, you can use has_perm:</p>\u000a\u000a<pre><code>obj.has_perm( 'drivers.read_car' )\u000a</code></pre>\u000a\u000a<p>Where obj is either a User or Group instance.  I think it is simpler to write a function for this:</p>\u000a\u000a<pre><code>def has_model_permissions( entity, model, perms, app ):\u000a    for p in perms:\u000a        if not entity.has_perm( "%s.%s_%s" % ( app, p, model.__name__ ) ):\u000a            return False\u000a        return True\u000a</code></pre>\u000a\u000a<p>Where entity is the object to check permissions on (Group or User), model is the instance of a model, perms is a list of permissions as strings to check (e.g. ['read', 'change']), and app is the application name as a string.  To do the same check as has_perm above you'd call something like this:</p>\u000a\u000a<pre><code>result = has_model_permissions( myuser, mycar, ['read'], 'drivers' )\u000a</code></pre>\u000a\u000a<p>If you need to use object or row permissions (they mean the same thing), then Django can't really help you by itself.  The nice thing is that you can use both model and object permissions side-by-side.  If you want object permissions you'll have to either <a href="http://djangoadvent.com/1.2/object-permissions/">write your own</a> (if using 1.2+) or find a project someone else has written, one I like is <a href="https://github.com/callowayproject/django-objectpermissions">django-objectpermissions</a> from washingtontimes.</p>\u000a

V<p>If you have multiple versions of a package / module, you need to be using <a href="http://www.virtualenv.org/en/latest/index.html" rel="noreferrer">virtualenv</a> (emphasis mine):</p>\u000a\u000a<blockquote>\u000a  <p><code>virtualenv</code> is a tool to create isolated Python environments.</p>\u000a  \u000a  <p>The basic problem being addressed is one of dependencies and versions, and indirectly permissions. <strong>Imagine you have an application that needs version 1 of LibFoo, but another application requires version 2. How can you use both these applications?</strong> If you install everything into <code>/usr/lib/python2.7/site-packages</code> (or whatever your platform\u2019s standard location is), it\u2019s easy to end up in a situation where you unintentionally upgrade an application that shouldn\u2019t be upgraded.</p>\u000a  \u000a  <p>Or more generally, what if you want to install an application and <em>leave it be</em>? If an application works, any change in its libraries or the versions of those libraries can break the application.</p>\u000a  \u000a  <p>Also, what if you can\u2019t install packages into the global <code>site-packages</code> directory? For instance, on a shared host.</p>\u000a  \u000a  <p>In all these cases, <code>virtualenv</code> can help you. It creates an environment that has its own installation directories, that doesn\u2019t share libraries with other virtualenv environments (and optionally doesn\u2019t access the globally installed libraries either).</p>\u000a</blockquote>\u000a\u000a<p>That's why people consider <code>insert(0,</code> to be wrong -- it's an incomplete, stopgap solution to the problem of managing multiple environments.</p>\u000a

V<p>In fact, the behaviour of <code>__import__()</code> is entirely because of the implementation of the <code>import</code> statement, which calls <code>__import__()</code>. There's basically five slightly different ways <code>__import__()</code> can be called by <code>import</code> (with two main categories):</p>\u000a\u000a<pre><code>import pkg\u000aimport pkg.mod\u000afrom pkg import mod, mod2\u000afrom pkg.mod import func, func2\u000afrom pkg.mod import submod\u000a</code></pre>\u000a\u000a<p>In the first <em>and</em> the second case, the <code>import</code> statement should assign the "left-most" module object to the "left-most" name: <code>pkg</code>. After <code>import pkg.mod</code> you can do <code>pkg.mod.func()</code> because the <code>import</code> statement introduced the local name <code>pkg</code>, which is a module object that has a <code>mod</code> attribute. So, the <code>__import__()</code> function has to return the "left-most" module object so it can be assigned to <code>pkg</code>. Those two import statements thus translate into:</p>\u000a\u000a<pre><code>pkg = __import__('pkg')\u000apkg = __import__('pkg.mod')\u000a</code></pre>\u000a\u000a<p>In the third, fourth and fifth case, the <code>import</code> statement has to do more work: it has to assign to (potentially) multiple names, which it has to get from the module object. The <code>__import__()</code> function can only return one object, and there's no real reason to make it retrieve each of those names from the module object (and it would make the implementation a lot more complicated.) So the simple approach would be something like (for the third case):</p>\u000a\u000a<pre><code>tmp = __import__('pkg')\u000amod = tmp.mod\u000amod2 = tmp.mod2\u000a</code></pre>\u000a\u000a<p>However, that won't work if <code>pkg</code> is a package and <code>mod</code> or <code>mod2</code> are modules in that package <em>that are not already imported</em>, as they are in the third and fifth case. The <code>__import__()</code> function needs to know that <code>mod</code> and <code>mod2</code> are names that the <code>import</code> statement will want to have accessible, so that it can see if they are modules and try to import them too. So the call is closer to:</p>\u000a\u000a<pre><code>tmp = __import__('pkg', fromlist=['mod', 'mod2'])\u000amod = tmp.mod\u000amod2 = tmp.mod2\u000a</code></pre>\u000a\u000a<p>which causes <code>__import__()</code> to try and load <code>pkg.mod</code> and <code>pkg.mod2</code> as well as <code>pkg</code> (but if <code>mod</code> or <code>mod2</code> don't exist, it's not an error in the <code>__import__()</code> call; producing an error is left to the <code>import</code> statement.) But that still isn't the right thing for the fourth and fifth example, because if the call were so:</p>\u000a\u000a<pre><code>tmp = __import__('pkg.mod', fromlist=['submod'])\u000asubmod = tmp.submod\u000a</code></pre>\u000a\u000a<p>then <code>tmp</code> would end up being <code>pkg</code>, as before, and not the <code>pkg.mod</code> module you want to get the <code>submod</code> attribute from. The implementation could have decided to make it so the <code>import</code> statement does extra work, splitting the package name on <code>.</code> like the <code>__import__()</code> function already does and traversing the names, but this would have meant duplicating some of the effort. So, instead, the implementation made <code>__import__()</code> return the <em>right-most</em> module instead of the <em>left-most</em> one <strong>if and only if</strong> fromlist is passed and not empty.</p>\u000a\u000a<p>(The <code>import pkg as p</code> and <code>from pkg import mod as m</code> syntax doesn't change anything about this story except which local names get assigned to -- the <code>__import__()</code> function sees nothing different when <code>as</code> is used, it all remains in the <code>import</code> statement implementation.)</p>\u000a

V<p>Make every thread except the main one a daemon (<code>t.daemon = True</code> in 2.6 or better, <code>t.setDaemon(True)</code> in 2.6 or less, for every thread object <code>t</code> before you start it).  That way, when the main thread receives the KeyboardInterrupt, if it doesn't catch it or catches it but decided to terminate anyway, the whole process will terminate.  See <a href="http://docs.python.org/library/threading.html?highlight=daemon#threading.Thread.daemon" rel="noreferrer">the docs</a>.</p>\u000a\u000a<p><strong>edit</strong>: having just seen the OP's code (not originally posted) and the claim that "it doesn't work", it appears I have to add...:</p>\u000a\u000a<p>Of course, if you want your main thread to stay responsive (e.g. to control-C), don't mire it into blocking calls, such as <code>join</code>ing another thread -- especially not totally <em>useless</em> blocking calls, such as <code>join</code>ing <strong>daemon</strong> threads.  For example, just change the final loop in the main thread from the current (utterless and damaging):</p>\u000a\u000a<pre><code>for i in range(0, thread_count):\u000a    threads[i].join()\u000a</code></pre>\u000a\u000a<p>to something more sensible like:</p>\u000a\u000a<pre><code>while threading.active_count() &gt; 0:\u000a    time.sleep(0.1)\u000a</code></pre>\u000a\u000a<p>if your main has nothing better to do than either for all threads to terminate on their own, or for a control-C (or other signal) to be received.</p>\u000a\u000a<p>Of course, there are many other usable patterns if you'd rather have your threads not terminate abruptly (as daemonic threads may) -- unless <em>they</em>, too, are mired forever in unconditionally-blocking calls, deadlocks, and the like;-).</p>\u000a

V<p>Convert both strings to timestamps (in your chosen resolution, e.g. milliseconds, seconds, hours, days, whatever), subtract the earlier from the later, multiply your random number (assuming it is distributed in the range [0, 1]) with that difference, and add again to the earlier one.  Convert the timestamp back to date string and you have a random time in that range.</p>\u000a\u000a<p>Python example (output is almost in the format you specified, other than 0 padding - blame the American time format conventions):</p>\u000a\u000a<pre><code>import random\u000aimport time\u000a\u000adef strTimeProp(start, end, format, prop):\u000a    """Get a time at a proportion of a range of two formatted times.\u000a\u000a    start and end should be strings specifying times formated in the\u000a    given format (strftime-style), giving an interval [start, end].\u000a    prop specifies how a proportion of the interval to be taken after\u000a    start.  The returned time will be in the specified format.\u000a    """\u000a\u000a    stime = time.mktime(time.strptime(start, format))\u000a    etime = time.mktime(time.strptime(end, format))\u000a\u000a    ptime = stime + prop * (etime - stime)\u000a\u000a    return time.strftime(format, time.localtime(ptime))\u000a\u000a\u000adef randomDate(start, end, prop):\u000a    return strTimeProp(start, end, '%m/%d/%Y %I:%M %p', prop)\u000a\u000aprint randomDate("1/1/2008 1:30 PM", "1/1/2009 4:50 AM", random.random())\u000a</code></pre>\u000a
