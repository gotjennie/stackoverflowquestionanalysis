V<p>Suppose you have the following situation</p>\u000a\u000a<pre><code>#include &lt;iostream&gt;\u000a\u000aclass Animal {\u000apublic:\u000a    virtual void speak() = 0;\u000a};\u000a\u000aclass Dog : public Animal {\u000a    void speak() { std::cout &lt;&lt; "woff!" &lt;&lt;std::endl; }\u000a};\u000a\u000aclass Cat : public Animal {\u000a    void speak() { std::cout &lt;&lt; "meow!" &lt;&lt;std::endl; }\u000a};\u000a\u000avoid makeSpeak(Animal &amp;a) {\u000a    a.speak();\u000a}\u000a\u000aint main() {\u000a    Dog d;\u000a    Cat c;\u000a    makeSpeak(d);\u000a    makeSpeak(c);\u000a}\u000a</code></pre>\u000a\u000a<p>As you can see, makeSpeak is a routine that accepts a generic Animal object. In this case, Animal is quite similar to a Java interface, as it contains only a pure virtual method. makeSpeak does not know the nature of the Animal it gets passed. It just sends it the signal \u201cspeak\u201d and leaves the late binding to take care of which method to call: either Cat::speak() or Dog::speak(). This means that, as far as makeSpeak is concerned, the knowledge of which subclass is actually passed is irrelevant.</p>\u000a\u000a<p>But what about Python? Let\u2019s see the code for the same case in Python. Please note that I try to be as similar as possible to the C++ case for a moment:</p>\u000a\u000a<pre><code>class Animal(object):\u000a    def speak(self):\u000a        raise NotImplementedError()\u000a\u000aclass Dog(Animal):\u000a    def speak(self):\u000a        print "woff!"\u000a\u000aclass Cat(Animal):\u000a    def speak(self):\u000a        print "meow"\u000a\u000adef makeSpeak(a):\u000a    a.speak()\u000a\u000ad=Dog()\u000ac=Cat()\u000amakeSpeak(d)\u000amakeSpeak(c)\u000a</code></pre>\u000a\u000a<p>Now, in this example you see the same strategy. You use inheritance to leverage the hierarchical concept of both Dogs and Cats being Animals. \u000aBut in Python, there\u2019s no need for this hierarchy. This works equally well</p>\u000a\u000a<pre><code>class Dog:\u000a    def speak(self):\u000a        print "woff!"\u000a\u000aclass Cat:\u000a    def speak(self):\u000a        print "meow"\u000a\u000adef makeSpeak(a):\u000a    a.speak()\u000a\u000ad=Dog()\u000ac=Cat()\u000amakeSpeak(d)\u000amakeSpeak(c)\u000a</code></pre>\u000a\u000a<p>In Python you can send the signal \u201cspeak\u201d to any object you want. If the object is able to deal with it, it will be executed, otherwise it will raise an exception. Suppose you add a class Airplane to both codes, and submit an Airplane object to makeSpeak. In the C++ case, it won\u2019t compile, as Airplane is not a derived class of Animal. In the Python case, it will raise an exception at runtime, which could even be an expected behavior.</p>\u000a\u000a<p>On the other side, suppose you add a MouthOfTruth class with a method speak(). In the C++ case, either you will have to refactor your hierarchy, or you will have to define a different makeSpeak method to accept MouthOfTruth objects, or in java you could extract the behavior into a CanSpeakIface and implement the interface for each. There are many solutions...</p>\u000a\u000a<p>What I\u2019d like to point out is that I haven\u2019t found a single reason yet to use inheritance in Python (apart of frameworks and trees of exceptions, but I guess that alternative strategies exist). you don\u2019t need to implement a base-derived hierarchy to perform polymorphically. If you want to use inheritance to reuse implementation, you can accomplish the same through containment and delegation, with the added benefit that you can alter it at runtime, and you clearly define the interface of the contained, without risking unintended side effects.</p>\u000a\u000a<p>So, in the end, the question stands: what's the point of inheritance in Python?</p>\u000a\u000a<p><strong>Edit</strong>: thanks for the very interesting answers. Indeed you can use it for code reuse, but I am always careful when reusing implementation. In general, I tend to do very shallow inheritance trees or no tree at all, and if a functionality is common I refactor it out as a common module routine and then call it from each object. I do see the advantage of having one single point of change (eg. instead of adding to Dog, Cat, Moose and so on, I just add to Animal, which is the basic advantage of inheritance), but you can achieve the same with a delegation chain (eg. a la JavaScript). I'm not claiming it's better though, just another way.</p>\u000a\u000a<p>I also found <a href="http://forums.devshed.com/python-programming-11/does-interface-only-inheritance-make-sense-in-python-82822.html" rel="noreferrer">a similar post</a> on this regard.</p>\u000a

V<p>The Zen of Python states that there should only be one way to do things- yet frequently I run into the problem of deciding when to use a function versus when to use a method.</p>\u000a\u000a<p>Let's take a trivial example- a ChessBoard object. Let's say we need some way to get all the legal King moves available on the board. Do we write ChessBoard.get_king_moves() or get_king_moves(chess_board)?</p>\u000a\u000a<p>Here are some related questions I looked at:</p>\u000a\u000a<ul>\u000a<li><a href="http://stackoverflow.com/q/2657627/577199">Why does python use &#39;magic methods&#39;?</a></li>\u000a<li><a href="http://stackoverflow.com/q/237128/577199">Is there a reason Python strings don&#39;t have a string length method?</a></li>\u000a</ul>\u000a\u000a<p>The answers I got were largely inconclusive:</p>\u000a\u000a<blockquote>\u000a  <h3>Why does Python use methods for some functionality (e.g. list.index()) but functions for other (e.g. len(list))?</h3>\u000a  \u000a  <p>The major reason is history. Functions were used for those operations that were generic for a group of types and which were\u000a  intended to work even for objects that didn\u2019t have methods at all\u000a  (e.g. tuples). It is also convenient to have a function that can\u000a  readily be applied to an amorphous collection of objects when you use\u000a  the functional features of Python (map(), apply() et al).</p>\u000a  \u000a  <p>In fact, implementing len(), max(), min() as a built-in function is actually less code than implementing them as methods for each type.\u000a  One can quibble about individual cases but it\u2019s a part of Python, and\u000a  it\u2019s too late to make such fundamental changes now. The functions have\u000a  to remain to avoid massive code breakage.</p>\u000a</blockquote>\u000a\u000a<p>While interesting, the above doesn't really say much as to what strategy to adopt.</p>\u000a\u000a<blockquote>\u000a  <p>This is one of the reasons - with custom methods, developers would be\u000a  free to choose a different method name, like getLength(), length(),\u000a  getlength() or whatsoever. Python enforces strict naming so that the\u000a  common function len() can be used.</p>\u000a</blockquote>\u000a\u000a<p>Slightly more interesting. My take is that functions are in a sense, the Pythonic version of interfaces.</p>\u000a\u000a<p>Lastly, <a href="http://mail.python.org/pipermail/python-dev/2008-January/076612.html">from Guido himself</a>:</p>\u000a\u000a<blockquote>\u000a  <blockquote>\u000a    <p><em>Talking about the Abilities/Interfaces made me think about some of our\u000a    "rogue" special method names.  In the Language Reference, it says, "A\u000a    class can implement certain operations that are invoked by special\u000a    syntax (such as arithmetic operations or subscripting and slicing) by\u000a    defining methods with special names."  But there are all these methods\u000a    with special names like <code>__len__</code> or <code>__unicode__</code> which seem to be\u000a    provided for the benefit of built-in functions, rather than for\u000a    support of syntax.  Presumably in an interface-based Python, these\u000a    methods would turn into regularly-named methods on an ABC, so that\u000a    <code>__len__</code> would become</em></p>\u000a\u000a<pre><code>class container:\u000a  ...\u000a  def len(self):\u000a    raise NotImplemented\u000a</code></pre>\u000a    \u000a    <p><em>Though, thinking about it some more, I don't see why</em> all <em>syntactic\u000a    operations wouldn't just invoke the appropriate normally-named method\u000a    on a specific ABC.  "<code>&lt;</code>", for instance, would presumably invoke\u000a    "<code>object.lessthan</code>" (or perhaps "<code>comparable.lessthan</code>").  So another\u000a    benefit would be the ability to wean Python away from this\u000a    mangled-name oddness, which seems to me an HCI improvement</em>.</p>\u000a  </blockquote>\u000a  \u000a  <p>Hm. I'm not sure I agree (figure that :-).</p>\u000a  \u000a  <p>There are two bits of "Python rationale" that I'd like to explain\u000a  first.</p>\u000a  \u000a  <p>First of all, I chose len(x) over x.len() for HCI reasons (<code>def\u000a  __len__()</code> came much later). There are two intertwined reasons actually, both HCI:</p>\u000a  \u000a  <p>(a) For some operations, prefix notation just reads better than\u000a  postfix -- prefix (and infix!) operations have a long tradition in\u000a  mathematics which likes notations where the visuals help the\u000a  mathematician thinking about a problem. Compare the easy with which we\u000a  rewrite a formula like <code>x*(a+b)</code> into <code>x*a + x*b</code> to the clumsiness of\u000a  doing the same thing using a raw OO notation.</p>\u000a  \u000a  <p>(b) When I read code that says <code>len(x)</code> I <em>know</em> that it is asking for\u000a  the length of something. This tells me two things: the result is an\u000a  integer, and the argument is some kind of container. To the contrary,\u000a  when I read <code>x.len()</code>, I have to already know that <code>x</code> is some kind of\u000a  container implementing an interface or inheriting from a class that\u000a  has a standard <code>len()</code>. Witness the confusion we occasionally have when\u000a  a class that is not implementing a mapping has a <code>get()</code> or <code>keys()</code>\u000a  method, or something that isn't a file has a <code>write()</code> method.</p>\u000a  \u000a  <p>Saying the same thing in another way, I see 'len' as a built-in\u000a  <em>operation</em>. I'd hate to lose that. I can't say for sure whether you meant that or not, but 'def len(self): ...' certainly sounds like you\u000a  want to demote it to an ordinary method. I'm strongly -1 on that.</p>\u000a  \u000a  <p>The second bit of Python rationale I promised to explain is the reason\u000a  why I chose special methods to look <code>__special__</code> and not merely\u000a  <code>special</code>. I was anticipating lots of operations that classes might want\u000a  to override, some standard (e.g. <code>__add__</code> or <code>__getitem__</code>), some not so\u000a  standard (e.g. pickle's <code>__reduce__</code> for a long time had no support in C\u000a  code at all). I didn't want these special operations to use ordinary\u000a  method names, because then pre-existing classes, or classes written by\u000a  users without an encyclopedic memory for all the special methods,\u000a  would be liable to accidentally define operations they didn't mean to\u000a  implement, with possibly disastrous consequences. Ivan Krsti\u0107\u000a  explained this more concise in his message, which arrived after I'd\u000a  written all this up.</p>\u000a  \u000a  <p>-- \u000a  --Guido van Rossum (home page: <a href="http://www.python.org/~guido/">http://www.python.org/~guido/</a>)</p>\u000a</blockquote>\u000a\u000a<p>My understanding of this is that in certain cases, prefix notation just makes more sense (ie, Duck.quack makes more sense than quack(Duck) from a linguistic standpoint.) and again, the functions allow for "interfaces".</p>\u000a\u000a<p>In such a case, my guess would be to implement get_king_moves based solely on Guido's first point.
